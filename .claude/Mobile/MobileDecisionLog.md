### Sync Mechanics — grounded in the actual code (read first-hand)

**Design premise (Nathan, confirmed — consistent with the PRD's "personal-first, single-user"):** Pommora is a **single-user app used from one location at a time** — a human can't edit on two devices simultaneously. The sync model is therefore **most-recent-wins by decision, not limitation**: no merge, no CRDT, no conflict-reconciliation UI, and **no scoping of what the phone may do** (the phone is a FULL editor — rename/cascade/move/delete like the desktop). The hazards below are real in a *multi-user / simultaneous-edit* world; for a solo, sequential user they reduce to invisible transient windows. Scope only the *necessary* precautions (atomic writes; readable-not-evicted files; don't ghost-duplicate an iCloud conflict copy) — nothing more.

The read/write/watch/lock spine, and what each means for the (solo) two-device iCloud nexus:

- **The file-watcher IS the sync-ingestion path — and it already handles "external" edits.** `main/watcher.ts`: a chokidar recursive watch on the root, 200 ms-debounced, that on ANY fs event **re-reads the WHOLE tree (`readNexus`) and pushes it to the renderer**. Deliberately **origin-blind** — no self-write suppression ("a read-only watcher can't loop; re-rendering an identical tree is a no-op"; the renderer dedupes via `stabilize()`). **So an iCloud sync-in from device B is just an "external edit," which this path already handles** (it was built for Obsidian/Finder edits) — the ingestion mechanism exists. Two things it was NOT built for: (1) it **re-walks the entire nexus per settle** (costly on a big nexus; every walked file is an iCloud read); (2) `awaitWriteFinish` is tuned to 200 ms **local** atomic writes, not multi-second iCloud downloads / dataless `.icloud` stubs → a sync-in mid-download could fire on a partial/empty file. **→ the materialization gate belongs in `readNexus`'s per-file read.**
- **The write lock is IN-PROCESS ONLY.** `io/fileLock.ts serializeOnFile` serializes overlapping RMWs inside one running app; it does nothing across devices (single-device correctness, not sync arbitration).
- **Atomic writes ✅ + byte-stable serialization ✅ are real, already-present sync assets** (no half-files uploaded; unchanged re-saves are byte-identical → no spurious conflict).
- **Multi-file cascades are THE sync hazard.** `crud/cascade.ts`: a **page rename** rewrites every inbound `[[link]]` body nexus-wide (`renameCascade`); a **context delete** strips the id from every page's tier array (`unlinkTier`) — each page an independent atomic write, **per-file, NOT cross-file atomic** ("recoverable by re-running"). Over iCloud those N writes propagate as **N independent, out-of-order sync events** → device B sees a **transient half-cascaded state**, and connections **resolve by TITLE in the body**, so mid-cascade some links are transiently broken/ambiguous. A cascade **racing a remote edit** on device B → `(1)` conflict files with **no re-run trigger on B**. `movePage`/`moveSet` are the other multi-file case. **Solo-user reality: mostly moot** — a rename completes locally in one fast operation, then all N files sync; the transient half-cascaded window exists only *during* propagation, which a one-location user isn't watching on the other device. Eventual consistency (links resolve by title in-memory → self-heal once the last file lands) is the accepted behavior.
- **Recency authority may disagree with iCloud** — the watcher trusts "newest wins"; iCloud does its own last-writer-wins + `(1)` files; nothing re-arbitrates by `modified_at` today.

**Sync decisions (resolved):**
- **S-1:** [confirmed — constraint REJECTED] The phone is a **full editor** — NOT scoped to single-file edits; it can originate renames/cascades/moves/deletes like the desktop. (My "constrain the phone" was over-engineering for a simultaneous-edit scenario a solo user doesn't create.)
- **S-2:** [confirmed] **Most-recent-wins**, no merge/reconcile UI. Sole implementation note (this IS most-recent-wins, just applied): if iCloud forks a `(1)` conflict copy (rare — both devices edited the same file while one was offline), the app **dedupes by the file's stable `id`, keeps the newest by mtime, discards the fork** → a conflict twin never surfaces as a ghost duplicate page. That's the entire conflict story.
- **S-3:** [confirmed] Cascades RECEIVED over sync → **accept eventual consistency** (self-heal once all files land).
- **S-4:** [confirmed — read-engine workstream, corrected by review M3] Reads restructure into a **staged path**: a cheap structure-only pass + **lazy per-entity materialization** (detect evicted stubs, download, show a downloading state before parse), with the watcher decoupled from full re-walks on mobile. NOT a one-line gate inside the desktop's eager `readNexus` walk — that would materialize the whole nexus on every launch and every synced change. Plus `awaitWriteFinish` retuned for iCloud latency.
