#### Lessons Learned

> **THE quirk to remember (Nathan, emphasized): why these bugs were slow to pin down — and the rules that kill each.** The meta-pattern across every Connections bug this session: **a UI symptom looked identical no matter which of four layers was broken.** A muted `[[link]]` could mean (a) not in the index, (b) resolver got the wrong string, (c) resolver worked but wasn't wired in, or (d) a stale compiled package ran old code. One symptom, four layers → effort kept landing at the wrong one. Quirk #17 ("confirm the data before blaming it") exists *because this already happened* — and it recurred. The discipline: **instrument to name the failing layer before fixing; never guess-fix a chain by its end symptom.**

- **A UIX failure can disguise itself as anything — including a data failure.** In the 06-07 navigation session, "Couldn't load this Page from disk" and "no navigation at all" both looked like data problems. The index had the page. The file was on disk. But the UIX resolution chain — `WikiLinkPageOpener` → `ConnectionFileLocator` → `resolveParent` — was broken at multiple points. Each break produced a completely different symptom: `resolveParent nil` → error screen; `WikiLinkPageOpener nil` → silent no-op; wrong ID format → silent no-op. **Rule (Quirk #17 extension):** before touching data or index, verify the end-to-end UIX chain step by step: (1) does the click handler fire? (2) does the resolver return a non-nil value? (3) does the file locator find the file? (4) does the parent resolution succeed? Name the failing step first. "Page not loading" is not a diagnosis — it's a symptom that could be any link in the chain.
- **In-memory page arrays are an unreliable proxy for vault membership.** `resolveParent` historically walked `pages(in: pageType)` (in-memory arrays populated only when a vault is expanded in the sidebar). For navigation from link-click, the target vault is usually NOT expanded → array is empty → nil → error screen. The fix was index-first: query `page_type_id`/`page_collection_id` from the SQLite index (always populated at launch), match to in-memory `PageType`/`PageCollection` by ID. **Rule:** never gate UIX on whether pages are loaded in memory. The index is the source of truth for membership; memory arrays are a display cache.
- **Strict `PageFile.load` is wrong in most navigation/resolution contexts.** Three places independently chose strict loading: `IndexBuilder`, `ConnectionFileLocator.idMatches`, `WikiLinkPageOpener`. Each broke silently for adopted pages (no frontmatter ID). The pattern: strict load fails → caller gets nil → chain aborts → symptom looks like data missing. **Rule:** outside of explicit write/save paths, use `loadLenient`. Navigation, resolution, and ID-matching all need the synthesized-ID fallback that `loadLenient` provides.
- **Stale local SPM package is hypothesis #1, not a fallback.** The in-tree `MarkdownPM` package can run a *cached* binary while source is edited — so "is my fix even running?" must be answered before debugging the fix. Burned multiple "nothing changed" cycles. **Rule:** on any MarkdownPM rendering bug, change one constant, build, confirm the binary changed first (debug builds: the real code is in `Pommora.debug.dylib`, NOT the 59KB launcher stub). **→ candidate CLAUDE.md quirk.**
- **A scan-logic fix to the index is INERT until the schema version is bumped.** The index is a *versioned regeneratable cache*; `populate` only re-runs when `currentSchemaVersion` ≠ the stored version. Both index fixes this session (lenient loader, file-level exclusion) needed a version bump (8→9→10) to take effect on an existing DB — without it the stale index is reused and the fix appears to do nothing (the exact "no change again" trap). **Rule:** any change to *which files get scanned in* or *how rows are derived* ships WITH a `PommoraIndex.currentSchemaVersion` bump + a one-line comment + the `ConnectionSchemaTests` tripwire updated. **→ candidate CLAUDE.md quirk.**
- **Two discovery contracts must MATCH.** The index scan (`IndexBuilder`) and the UI discovery (`PageContentManager.loadAll`/`descendantFiles`) read pages from disk independently and diverged FOUR ways this session (strict-vs-lenient loader; file-level folder-exclusion; `_`-prefix; non-recursive `children` vs recursive `descendantFiles`). A file visible in the sidebar but absent from the index (or vice versa) is this class of bug. **Rule:** when changing one discovery path, change the other to match; a divergence is a latent bug. (Open instance: deep ≥2-level non-collection subfolders — see Pending.)
- **Protocol-default-to-NoOp is invisible.** `pageConnectionResolver` defaults to `NoOpWikiLinkResolver`; a subtree missing injection silently gets the no-op → every link muted → looks exactly like "index empty." **Rule:** a default-to-no-op injection needs an always-on debug assert at the *injection site* (`NexusEnvironment`), not a probe at the leaf. **→ candidate CLAUDE.md quirk.**
- **Token-accessor contracts must be documented at the type, not the call site.** The placeholder-includes-brackets bug existed because `WikiLinkSelection.placeholder`'s "interior text" contract lived only in a call-site comment. **Rule:** document `displayRange` = full marker span / `placeholder` = interior on the type definition.
- **Index write paths must be integration-tested against a PRE-POPULATED index.** The cascade-wipe (`INSERT OR REPLACE` deleting children) only manifested when re-syncing a type that already had indexed children — a state from-scratch tests never create. **Rule:** test upserts against an index that already holds children, asserting they survive — not just clean-slate creation.
- **Editor lag is usually per-keystroke work in the styler.** The styler runs on every restyle and called `resolveUniqueEntity` (a synchronous main-thread `dbQueue.read`) once per link, plus 6 `os.Logger` probes fired per keystroke/render with eager string interpolation. Probes were the dominant cost (removed). If lag persists, memoize resolution per restyle-cycle (and a short-lived cross-keystroke cache invalidated on `connectionsChanged`).
