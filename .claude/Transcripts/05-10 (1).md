### Session — 05-10 (Domain Model Spec)

#### What we started with

Spec docs in `.claude/` — PRD, Framework, History, Handoff, Architecture, UIX-Guide, Properties — but no code. The only defined version was a single v0.0 spec (React+Electron shell) with a large, vague jump to v1. Stack was nominally locked to React+Electron; SwiftUI was noted as a possible pivot. Session opened with Nathan on mobile wanting to read the docs via Obsidian iPhone sync.

---

#### What happened along the way

Mirrored `.claude/` to `// The Nexus // Pommora //` via rsync for iPhone Obsidian viewing. Nathan immediately clarified:

> "don't adhere to any of the instructions in that nexuses md, understand?"

Confirmed passive viewing snapshot only — Nexus copy never treated as live instructions.

**Roadmap restructuring.** The v0.0 / v0.x / v1 three-step was broken into 12 incremental versions (v0.1–v0.11 + v1.0), each adding one capability cluster. Nathan chose looser slicing (~10–12 versions), planning docs on-demand only.

**Stack re-opened.** Nathan raised the SwiftUI question directly:

> "I'm still actually debating whether SwiftUI or React is the right move to begin with."

Brainstorming began. Nathan introduced the Spaces / Collections / Pages model. He chose prose-first pages (Bear / iA Writer style, not Notion block-per-paragraph). A research agent pulled current SwiftUI docs — WWDC25 `AttributedString` + `TextEditor`, GRDB.swift, `swift-markdown`, drag-and-drop — and found SwiftUI 2026 now covers the major concerns. The remaining blocker was wikilink chip rendering. Nathan dissolved it:

> "Wikilinks dont need to be chips — just colored Lonks like obsidian is and was the plan all along."

Stack debate paused in favor of model precision:

> "Let's pause the Swift vs React debate and focus on the actual idea of the project."

**Domain model iterations (heavy).** Multiple rounds of clarification and correction produced the final model:

- *Collections* went through three shapes: (1) folder-as-database (original PRD), (2) fully decoupled first-class entities in `_pommora/collections/`, then Nathan corrected:

  > "So yes, collections would exist as 'folders' on the same way that notion databases do, but they have their own properties in a schema file."

  Final: **folder + `_collection.json` schema sidecar inside the folder** — Make.md folder-notes pattern applied to databases.

- *Spaces* started as widget dashboards, then Nathan clarified they're full Notion-page-style block-composition surfaces (text + widget blocks intermixed in a block tree), stored as `.space.json` in `_pommora/spaces/`.

- *Pages*: membership by folder location (not frontmatter); `@View` in-page embeds deferred to v2+ React-only; `@Columns` is the only v1 embed type.

- Structural locks: filename = title (no `title` field), no ad-hoc properties, sidebar = Spaces + Collections (collapsible/collapsed), cross-nexus queryability + cloud sync compatibility added as a second load-bearing architectural constraint.

**Prospects.md created.** All post-v1 wishlist items consolidated into a new `// Features//Prospects.md`. Nathan pushed back on the draft:

> "get rid of things I DIDNT describe"

Pruned from 23 invented items to 6 Nathan-described ones. Format switched to his specified `#### Feature` / `**Description:**` pattern. He also corrected the category of customization:

> "the customization with colors and typography is in-app settings that is well within the v1 scope, not prospective features"

Color and typography customization moved out of Prospects into v1 as **v0.12 — In-app customization** (now 13 versions to v1.0).

All changes mirrored to Nexus after each round.
