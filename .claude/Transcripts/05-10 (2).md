### Session — 05-10 (Stack Research + Doc Restructure)

#### What we started with

Continuing from a compacted session that completed the initial dual-stack feasibility audit (React+Electron vs SwiftUI). Domain model was closed; both stacks remained on the table. Three top-level entities locked (Pages, Collections, Spaces); `SwiftInfo.md` and `ReactInfo.md` (newly created this session) held per-stack deep references. Stack call was and remains the only gating decision.

Session opened with Nathan asking for more research on tools and considerations for both stacks across four areas: build/distribution, Mac OS integrations, editor internals, and the state/data layer.

---

#### What happened along the way

**Research cycle 2.** Launched four parallel agents. When the summarized findings came back, Nathan rejected the summary format:

> "Thats not what I want -- Im asking for research on Both SwiftUI and React for the pheasablity and logic about whcih ot chose, and each stacks implications for this project; waht things will be possible, and waht difficultires exist"

Delivered a comprehensive feasibility comparison instead — feature-by-feature (editor, Spaces, Collections, shell, file watching, search, design system, Mac integrations, distribution, cross-platform, iPad, agentic workflow). The verbatim agent reports were also surfaced on request. The headline read: React makes the editor easy and Mac integration ~80%; SwiftUI makes Mac integration 100% and the Phase B editor a real R&D project with no shipped precedent.

**Doc restructure.** Entered plan mode; Nathan directed splitting planning into Swift and React folders. After a clarifying exchange, the actual intent became: keep existing documents but introduce a **For React** / **For Swift** labeling convention within shared docs so stack-conditional content stays visibly distinct without blurring the two paths. `ReactInfo.md` was created as a parallel mirror to `SwiftInfo.md`. Convention was applied across PRD, Architecture, Domain-Model, Prospects, UIX-Guide, CLAUDE.md, Handoff, History.

**Architecture pushback.** Two separate corrections from Nathan collapsed the three-layer enforcement model (Core / Adapter / UI with "Core has zero UI imports" rule):

> "im concerned about the core architecture design; having things be seperate for 'portability' misunderstands what I ment, I ment TRANSLATED"

> "What I ment was keeping the fucntionalities of the system able to work across both if I decide to re-build the app; not that it would automatically be rendered or cross-compatiable"

`Architecture.md` was rewritten as **conceptual portability of functionalities** — the documented decisions (file formats, schemas, directive syntax, design tokens, UX patterns) survive a rebuild; the codebase doesn't. No enforced layer separation; implementation patterns are stack-natural. `CLAUDE.md`, PRD, and `History.md` all updated to remove the three-layer framing.

**Per-entity feature docs.** After the Architecture work, Nathan directed a structural split:

> "We shiuld probably split the features documents from a breif domain-model overview; but with documets of specifics in a new md / Spaces.md / Pages.md / Collections.md"

Three per-entity docs were created from `Domain-Model.md` content. `Domain-Model.md` trimmed to a brief overview linking to the per-entity files, retaining cross-cutting topics (linking model, sidebar, resolved decisions). `Pages.md` received the full Swift editor strategy: Phase A (v1 — native `TextEditor` + quick fork for H4-H6 + toggles); Phase B (committed post-v1 core feature — hover-on-selection bubble toolbar). Nathan clarified Phase B directly:

> "its a must, but not a must for V1"

Nathan moved `Resources.md`, `ReactInfo.md`, and `SwiftInfo.md` from `// Features//` to `.claude/` top level during the session; all path references were updated across the project.

**End of session.** Nathan began a project rename (Pommora → Nexus / NexusOS) but interrupted it; deferred to the next session so the `.claude/` routing isn't broken before the folder rename. Handoff.md was rewritten to a merged format (simple immediate-decisions structure on top, historical session log preserved below).
