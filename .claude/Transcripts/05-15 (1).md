# Session — 05-15 (Sidebar Selection Polish)

## What we started with

v0.1a Nexus Foundation had just shipped (15 commits on main since v0.0; sandbox, security-scoped bookmark persistence, `.nexus/` init, FolderTree-driven sidebar, 25 unit tests). Latest commit was `807057d` ("Move inspector toggle to main toolbar; kill inspector-segment glass") — but the glass was still visible on screen, and Nathan opened the session asking to revert it. Test folder was empty; he also wanted it populated with sample Items / Spaces / Collections content. Visual-scaffolding work was the goal — iterating selection chrome and layout on placeholders, with the de-scaffold back to FolderTree deferred to a later session.

## What happened along the way

Reverted 807057d via `git checkout 807057d^ -- ...` (inspector toggle back inside `.inspector { }.toolbar { }`; segment glass accepted as a known v0.0 gap). Copied the PommoraNonEmpty fixture (Tasks Items collection, Papers / Projects Pages collections, Inbox loose folder, Quick note, Bookmark) into the test nexus root, plus a `.nexus/spaces/Welcome.space.json`.

Then Nathan pivoted from real data to scaffold:

> "what I want is 3 placeholder items below the search bar; then a heading toggle with Spaces (3 placeholders items), then a heading toggle with collections (3 collection folders) three placeholders each"

Rewrote `SidebarView` to drop FolderTree and render hardcoded placeholders. `FolderTree.swift` / `SidebarNode.swift` / `SidebarRow.swift` stay in the target, dormant.

Built a custom `SelectableRow` with tap-driven selection — `List(selection:)` doesn't allow the gray-fill-plus-accent-foreground combo Nathan wanted, so a custom row was required. Iterated through black@11%, white@11%, white@9%, gray@11% before landing. Added `.brightness(0.11)` on foreground, then tuned layout (full-row width via `.frame(maxWidth: .infinity)`, fill via `.listRowBackground`, `.symbolRenderingMode(.monochrome)` so the icon respects foregroundStyle).

Cross-context shading bug surfaced — loose Items rendered slightly brighter than Section rows; Placeholders inside `DisclosureGroup` rendered slightly dimmer. Three rounds of attempted fixes (Section wrap, `Color.clear` row background, per-context brightness/opacity scalars) all got reverted as Nathan corrected the direction:

> "we're focused on the color"
> "NO! we're talking about the saturation and fill NOT spacing"
> "Space Two is the style I want on everything"

The structural attempts kept missing the point — he wanted color-only. He eventually diagnosed and fixed it himself:

> "I fixed it; we needed to set the brightness to zero while keepoing the text brightness the same"

SF Symbol `.brightness()` composites inconsistently across Section vs DisclosureGroup vs direct-List; `Text.brightness()` is predictable. Removing brightness from the icon (text-only `+0.12`) equalized all three contexts. Padding tuned: fill rect inset to 11pt horizontal (aligns with search field), row content padding to 4pt leading (icon at chevron-distance from sidebar edge).

`EmptyPane` wrapper dropped from `ContentView` — `detail:` is bare `Color.clear`. Doc pass through `Sidebar.md`, `Handoff.md`, `History.md`. Two commits to main and pushed: `3824138` (sidebar scaffold + selection chrome + 807057d revert + EmptyPane removal) and `06307b7` (padding refinement + Vault hierarchy doc).

End-of-session Nathan floated an architectural reconsideration:

> "I'm also considering Spaces, Vaults, Collections, Items as the heiarchy"
> "Don't implement any of this yet, just give it to me straight"

Capacities-style 4-entity model — Vaults as databases (schema holders), Collections as structural sub-categories inside Vaults sharing the Vault's schema. Captured as a pre-decision section in Handoff with six open questions and the recommended decision approach:

> "Keep this in the handoff as something to think about; I use remote control to plan while im at the gym"
