## What happened along the way

Planning ran in plan mode: two Explore agents mapped the current list code, a Plan agent designed the architecture, four decisions were locked via question rounds (portable `- item` source, Shift+Enter hard exit, mid-line CommonMark continuation, bare-marker Enter trigger).

Implementation hit `MarkdownListHandler.swift` with six targeted edits: `import Markdown`, added `bareMarkerRegex` + `ListContext` + `detectListContext` (three-stage code-block / regex prefilter / AST confirmation), deleted the space-trigger block + dead `numberRegex`, rewrote the `\n` branch with Case 1 (bare-marker trigger) + Case 4 (Enter continuation) + an edge guard, added `handleShiftEnter`, wired `insertLineBreak:` in `doCommandBy`. Build verify caught a pre-existing typo (`hcaierarchicalColor:`) at `MarkdownTextLayoutFragment.swift:534` — flagged as a parallel-session change; Nathan authorized the fix. `ContextMenu.swift` got a parallel cleanup pass: broader `isSelectionList` detection, prefix-stripping in `applyList`, `didMarkdownUnorderedList` writes `- ` instead of `\t• `.

First test round surfaced two regressions and Nathan called them out:

> "the space to create new list item works; but it doesnt work to initialize the list item"
> "an empty list and enter should create a list too"
> "lets also remove the in-line shift into non-bulleted indented item. Lets just make it create a new list item -- the current version is a bit buggy"

Case 2 (empty-item exit) and Case 3 (mid-line continuation indent) were removed in response — the model collapsed to "Enter always creates a new list item; exit gesture is Shift+Enter only."

Shift+Enter then refused to fire. Discovered macOS's default key binding maps Shift+Return → `insertNewline:` (the `insertLineBreak:` selector only fires on Ctrl+\). Rewrote detection as a modifier-flag check at the top of the `\n` block via `NSApp.currentEvent.modifierFlags.contains(.shift)`. Removed the now-dead `doCommandBy` hook + `handleShiftEnter` function.

Visual regression: list items no longer indented from the page margin (the old `\t• ` provided indent for free). Added `firstLineHeadIndent = indentPerLevel + depthIndent` in `paragraphAttributes`. Broadened `bulletListPattern` from `[-•]` to `[-*+•]` so `*`/`+` markers also styled.

Bullet glyph substitution (`-` → `•` visually) was the last attempt — hide the dash via styler attributes, overdraw `•` via new `drawListBullets` in the layout fragment. Nathan tested and the bullets came out invisible:

> "Im going to revert the code since that only made the bullet invisible. Lets keep this as 'bullets arent rendering; defer fix' -- its not a big issue for now"

He reverted that pair of changes himself. Build + lint pass post-revert. Also flagged a small bug for tomorrow: typed `->` transforms to `→` correctly, but typed `<-` doesn't become `←` and `<->` doesn't become `↔` — though pasted versions render fine. Closed the session with:

> "We solved the seperator issue FINALLY"

Docs updated: Handoff, History (Session 13 entry), PageEditor.md, Framework.md, CLAUDE.md Active Version table — all reflect HR + Lists shipped, bullet glyph deferred.
