#### A — Pinned Model  *(RESOLVED)*
- **A-1:** [confirmed] Pins are now **durable + user-reorderable** — Nathan explicitly overrode the "temporary" nature ("Pinned are no longer temporary, i'm overriding that"). The gallery is one continuous surface: pinned at top (reorderable), recents below, **no divider**.
- **A-2:** [confirmed] **Favorites are out of scope this pass** ("pinned only for now"). The existing favorites list + rail stay untouched; the pins↔favorites reconciliation (they now overlap heavily) is deferred, NOT decided. Don't fold, remove, or repurpose favorites here.
- **A-4:** [confirmed] **Nav-click reconciles the target by id before opening (review #2).** A durable pin outlives folder reorganizations, but its stored `path` doesn't — and today `go` → `select` case `'page'` opens `target.path` with **no reconcile** (store.ts:532-540), so clicking a pin whose entity was moved/renamed hits a dead path → error pane (the id-based card still renders, masking it). Fix: `go` (useNavData.ts:59) runs `reconcileSelection(tree, target)` — which refreshes a moved page/set's path by id (selection.ts:44) — before `select`, as Back/Forward do (store.ts:236). **Two impl notes (R2):** (a) `reconcileSelection` can return `{kind:'none'}` for a target deleted between render and click (fs-watcher race; J-2 render-prunes the *rendered* set, but a click can still land on a just-deleted target) — `go` must **bail/no-op on `none`**, not pass it to `select` (whose `SelectTarget` rejects `none`); stepHistory's `continue`-skip has no single-click analog. (b) `go`'s `useCallback` currently deps `[select]` — reconciling needs the live `tree`, so add `tree` to the deps or it closes over a stale tree. Input side is safe: `isTreeTarget` already narrows away task/event before reconcile.
- **A-3:** [confirmed] Durable ordered pins = an **in-memory ordered `pins: NavTarget[]` list** (recents dedupe against it; recents stream stays pure MRU history; pins own their membership + order). Ratified by Nathan ("ordered pin list, yes"). Rejected alternative (flag + `pinnedOrder` overlay) stays in Considered&Rejected. **Persistence is per-pin files, not one array — see A-5.**
- **A-5:** [confirmed] **Sync-safe pin persistence — no pin lost to a concurrent add (Nathan: "don't accept sync loss").** Each pin is its OWN file: `.nexus/pins/<sanitized-navKey>.json` = `{ ...NavTarget, order }`. Rationale: the specified sync model is last-writer-wins **per file** (Navigation.md); a single `pins` array file means two machines each rewrite the whole array → the later write clobbers the earlier's additions → a durable pin silently lost (the review-#6 / favorites LWW hole). Splitting to one file per pin makes concurrent add (new file, machine A) / add (new file, machine B) touch **disjoint files** → file-sync keeps both → **no distinct pin is lost to a concurrent add.** That's the core win over the single-array file; it does NOT make every op conflict-free (see the resolution rules — the earlier "no pin *ever* lost" over-claimed). `order` is a **numeric fractional key** — the simplest realization of the fractional-index concept: insert-between picks the **midpoint** `(prev+next)/2` of its neighbors (no base-62 LexoRank string needed for a small pin set; greenfield either way). A reorder rewrites **only the moved pin's file**. In-memory `pins` = the files loaded, sorted by **`(order, navKey)`** — navKey is the deterministic tiebreak so two machines' concurrent between-inserts that compute an equal `order` still sort identically everywhere (review R3-F3), not by nondeterministic readdir order. This is the filesystem-as-merge pattern; it fits Pommora's per-entity-file canonical model.
- **A-5b:** [confirmed] **Conflict resolution (R3 folds):**
  - **Unpin vs reorder of the SAME pin (R3-F1):** delete-vs-rewrite race on one path can *resurrect* a removed pin. Resolve with a **tombstone unpin** — unpin writes `{ ...target, order, deleted: true }` rather than unlinking; delete-vs-reorder becomes an ordinary same-file LWW on a flag (no resurrection), and a full load reaps tombstones (drops them from the in-memory set, deletes the files). This is a future-sync-correctness rule; single-machine it degenerates to a plain delete.
  - **Precision ceiling (R3 obs):** many between-inserts at one gap exhaust the fractional key; a rebalance (rewrite all pin files) is the standard remedy but re-incurs the N-write conflict surface — **out of scope, accept the ceiling** (a pin set is small; exhaustion is remote).
  - **Malformed/partial file on load (R3-F5):** `loadPins` ignores non-`.json`/`*.tmp` names (never reads an in-flight temp); a parse failure on an existing `.json` **logs, not silently drops** — a corrupt read must not shrink the set.
  - **Live refresh (R3-F2) → see I-3:** the running app must re-read pins on a `.nexus/pins/` watcher event, else a synced-in pin only appears on reload.

#### B — Card Image  *(RESOLVED — rendered page thumbnail)*
- **B-1:** [confirmed] The card image is a **rendered thumbnail of the actual pane**, zoomed out so the entity's inline title banner reads at ~1/3 the image height. Not the raw `banner` asset — the real rendered view.
- **B-2:** [confirmed] Production = **capture-on-open** (Nathan: "cache on opening the page… one-time snapshot"). The gallery only ever shows **visited** entities (recents = history; a pin was visited to be pinned), so a thumbnail is captured the moment each entity settles in the pane — no offscreen mass-render, no per-gallery-render work (respects "never expensive on every X"). **Overwrite-on-open** keeps it fresh — no mtime/hash staleness check needed (a thumb can only be stale relative to the last time the user actually looked, which is exactly recapture time).
- **B-2b:** [confirmed] **Refresh-the-source-on-nav-open** (Nathan): re-capture the thumbnail of the entity the nav was invoked *from* (current `selection`) so the card you're most likely to leave from is current. **Ordering is load-bearing (review #1, tightened R2):** the NavPane is a `GlassPane` overlay in the *same* single window (`NavPane.tsx:134`), opened centered over the pane — so the source-capture MUST fire on the **`navOpen: false→true` transition itself**, before the overlay mounts, else `capturePage` reads the overlay glass composited on top and syncs a contaminated thumbnail. **Two mutators flip `navOpen` true** — `openNav` (store.ts:478-481, ribbon click) AND `toggleNav` (store.ts:483-486, the toggle-nav keyboard command) — so the capture-before-open logic must sit on the shared transition (or in both), NOT hooked to `openNav` alone; the keyboard path would otherwise reintroduce the contaminated capture.
- **B-3 — the researched pipeline** [confirmed, phase-D best practice, sources in Sources]:
  - **Capture:** main `win.webContents.capturePage()` (native compositor read — the ONLY option that captures glass/`backdrop-filter` + CM6 correctly; `html2canvas`/`html-to-image` drop the blur and add a dep → rejected). Main-process API → fits "main owns native + fs"; renderer never touches capture.
  - **Retina-safe crop:** capture the **full page** (no `rect` arg — the `rect` overload has a long-standing HiDPI crop bug), then `nativeImage.crop(paneRect × scaleFactor).resize({ width: ~2× card px, quality: 'best' })` in main. Renderer sends the pane element's `getBoundingClientRect()` (DIP) + `scaleFactor`; main does the pixel math.
  - **Timing (no paint event exists):** renderer waits `document.fonts.ready` + images + **double-`requestAnimationFrame`**, **post open/close transition**, then signals; ~150–250ms debounce so rapid open/close doesn't thrash. Guard so capture fires once per settled open, never on scroll/HMR.
  - **Hot-path cost (review #4):** capture-on-open fires on *every* navigation, so it must be **fully non-blocking + idle** — the renderer signals after settle, main does the capture/crop/resize/encode/write off the interaction path (idle), never synchronously gating the navigation. Navigation is a low-frequency user action (not the per-frame/scroll trigger the "never expensive on every X" rule targets), so at-idle capture is acceptable; but a pure list-mode user still pays it. **Prospect gate** if cost ever shows: only capture once gallery mode has been used this session (trade: first gallery open shows placeholders until entities are re-visited). Default = ungated + idle.
  - **Skip capture on failure/blank states (review obs):** don't cache an error-state thumbnail — skip capture when `pageStatus === 'error'`. Contexts render near-blank; prefer the placeholder tile over a blank capture for context kinds.
  - **Format:** `toJPEG(~78)` — smallest for many small UI shots, and no alpha channel (dodges the transparent→black resize bug). WebP unsupported by `nativeImage`. No dependency.
  - **Filename (review #3):** `navKey` is `` `${kind}:${id}` `` (navRecents.ts:14) — the **colon is illegal on Windows/NTFS** and a legacy `:`↔`/` hazard on macOS, which would break the very cross-platform sync this synced cache exists for. Sanitize navKey → a colon-free filename (`` `${kind}-${id}` ``) for the file only; the in-memory navKey stays unchanged.
  - **Cache location:** **`.nexus/assets/<nexusID>/thumbnails/<sanitized-navKey>.jpg` — synced** (Nathan's call). Lives under the existing `.nexus/assets/<nexusID>/` tree so the established **`nexus-asset://` privileged protocol** (confined to `.nexus/assets/`) serves it to `<img>` with no new protocol path — and it's inherently nexus-scoped + synced (assets already sync). Unlike the SQLite index (rebuilds from one fs walk → stays off the synced path), a thumbnail can only be regenerated by *actually re-opening that entity* — so it's precious, not cheaply regeneratable, and syncing means a second machine/platform shows real previews immediately instead of placeholders until every entity is re-visited. Lives in the nexus's own `.nexus/` (inherently nexus-scoped — J-1's cross-nexus collision dissolves) alongside the already-synced `.nexus/assets/` binaries; LWW like the other sidecars. At ~20–40KB × ≤100 recents ≈ a few MB/nexus — trivial sync weight.
  - **IPC:** main captures → resizes → encodes → **writes the file → returns the path** (or `file://`). Never ship dataURL bytes across the bridge (base64 bloat + big copy). Gallery renders the path; placeholder on miss.
  - **Eviction — membership-tied, not a byte-cap sweep (review #5):** an atime-LRU sweep is unreliable (`noatime`/iCloud volumes) and a MB cap contradicts the "few MB" reality. Instead, **tie thumbnail deletion to membership**: when an entity rolls off recents (`capRecents`, navRecents.ts:41) AND is **absent from the durable `pins` list** (A-5 — not the retired `pinned` flag), delete its thumbnail. This bounds the synced folder to the live working set (≤ recents cap + pins) with no orphan accumulation — and since it's synced, a delete propagates as a sync delete, which is exactly right (the entity is no longer surfaced anywhere). `capRecents` is an in-memory splice today → add an fs-delete hook for the evicted keys.

### Build Risks (capture pipeline — designed-out, not patched)
- **Blank/black capture:** only capture the visible foreground window (occluded windows return black on some platforms); gate on the settle signal; JPEG/opaque avoids the alpha→black path.
- **Mid-transition frame:** suppress capture until the pane open/close (Bloom) transition ends — capture on transition-end, not on open.
- **HiDPI crop:** full-page-then-crop with `rect × scaleFactor`; never trust `capturePage(rect)` directly.
- **Memory:** capture → resize → encode → release in one handler; never hold an array of full-size `NativeImage`s. Naturally one-at-a-time under capture-on-open.
- **Cost rule:** exactly one capture per settled open (debounced) — never on scroll, HMR, or every render.

- **Gallery surface:** card grid off `resolvedRecents`; responsive columns; all-kinds cards; pinned-top + reflow reorder; active-item accent border; hover-pin + pinned accent pin; List⇄Gallery toggle; title/location eclipse-scroll; **no per-row tree walk**; placeholder tile so the grid is never broken.

#### Considered & Rejected
- **Banner asset as the card image** — rejected: Nathan wants the rendered "actual window," and pages have no banner (only a planned `cover`); banners alone would leave most pages imageless.
- **Offscreen mass-render / live render on gallery open** — rejected: O(N) full renders on a high-frequency surface; violates the "never expensive on every X" rule. Capture-on-visit amortizes to one capture per navigation.
- **DOM-to-image (html2canvas/toDataURL)** for thumbnails — rejected vs. native `capturePage`: flaky over glass/CM6/retina, heavier, and a new dependency; the native screenshot is exact.
- **Pin flag + `pinnedOrder` overlay** (vs. a durable `pins` list) — kept as the fallback data model; rejected as primary for having two sources of truth for pin state.
- **Single `pins` array sidecar file** (vs. per-pin files) — rejected: whole-file LWW sync silently drops a concurrent pin from another machine (review #6). Per-pin files (A-5) make concurrent add/remove touch disjoint files → no pin loss. Nathan: "don't accept sync loss."
- **Integer/dense pin order** (vs. fractional index) — rejected: a reorder reindexes every pin → N file writes and a wider concurrent-reorder conflict surface. Fractional order rewrites only the moved pin's file.

#### Lessons
- A confident past-session sentence ("pins are temporary/float by MRU") is a ratified decision only until the user overrides it — Nathan flipped pins to durable on the spot. Verify intent, don't inherit wording. → routes to Guidelines.
- **A guarantee's *scope* must be attacked, not just its intent.** "Disjoint files → no pin ever lost" was true for concurrent *adds* but false for same-pin *unpin-vs-reorder* — the headline was broader than the mechanism. State exactly which ops a correctness claim covers.
- **In a single-window app, any capture reads the whole composited window** — an overlay (NavPane glass) contaminates a screenshot of "the page." Capture must precede the overlay's mount, and against the state transition, not one caller (both `openNav` + `toggleNav` flip it).
- **Nav state loads once at open and rides no watcher** — a synced-in sidecar/pin needs an explicit narrow reload, never a tree re-walk. Any "it syncs" claim for nav data is reload-only until that refresh exists.
