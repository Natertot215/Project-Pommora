## Unified Subfield + Scan-Promote — Decision Log

> **Status:** ⭐ **RATIFIED** (Nathan). Adversarial round 1 folded (F1 dissolved; DF-7 resolved — coexistence stays, most-recent-wins). Plan-review round folded: the preview stats source is the preview's **own local body**, never the shared `liveBody` slot (a second writer would evict the main pane's live count — see Z-2); view modes are **two separate persisted slices** (DF-2). Ready to build.

### Frame

- **Purpose:** Collapse the floating ↔ full-pane surface split so the Subfield footer and the "scan = promote floating into its full in-pane equivalent" semantics are shared, not re-implemented per surface.
- **Core Value:** One Subfield component serving the main pane, the page preview, and NavView (scoped only where it must be — the preview) — so a preview reads like a mini full-page and NavView gains its view-type toggle in a real footer, with the scan glyph meaning the same thing everywhere.
- **Success Criteria:** (1) A page preview shows the real toggle-able Subfield (location + live word/char/line for *its* page). (2) NavView carries the detail-pane Subfield hosting the List/Gallery toggle. (3) List-mode rows in NavView + NavWindow get view-like insets so pinned icons breathe. (4) The shared toolbar's Open action, from the NavWindow map flavor, promotes NavWindow → NavView, one-time-copying the view mode. (5) Nothing regresses on the existing main-pane Subfield, preview, or NavWindow.

### Decisions

#### Z — Enabler: Scope the Subfield
- **Z-0:** [confirmed by Nathan] The **detail-pane Subfield is shared across everything the detail pane routes** — Pages AND NavView — one footer whose *content shifts by what's open* (page stats vs the view-type toggle), one collapse toggle. NavView is "not a completely different entity"; it's a detail-pane resident. **Consequence:** the detail-pane Subfield stays coupled to global `selection` (correct — it describes the active tab); scoping (below) is needed **only for the floating preview**, which shows a *different* page than the detail pane. This splits the work: NavView = a light registry extension of the existing Subfield; preview = the scoped instance.
- **Z-1:** [confirmed] Subfield is a **props-less global-store consumer** — reads `selection`/`tree`/`trail`/`pageDetail`/`subfieldOrder`, sole mount at `DetailPane.tsx:151`. For the **preview** it must take a **scope** (the preview's page). No existing scoped/multi-mount code and `setSubfieldOrder` has **zero renderer callers** (no reorder UI to migrate) — so the read-side scoping model can be defined freely, low reconciliation risk. *(Grounded.)*
- **Z-2:** [confirmed — live-as-you-type, LOCAL body] The preview counts **live-as-you-type**, sourced from the preview window's **own local state** — never the global `liveBody` slot. Grounding: `liveBody` is a **single-owner slot** (the active main editor writes it; all 3 readers — `subfieldItems.tsx:35`, `store.ts:600` warm fold, `useNavThumbnails.ts:84` — guard on `liveBody.path === selection.path`). A second writer doesn't get *isolated* by those guards, it **evicts** the main pane: the instant the preview wrote `{path: previewPath, …}`, the main footer's guard would fail and it would fall back to `pageDetail.body` (the last-**saved** snapshot), silently dropping the main editor's unsaved live count. So the preview must **not** touch the slot. The design = `PageEmbed` exposes its editor body through a controlled `body`/`onBody` prop; `PreviewWindow` holds it in **local state** (debounced, like `PageView`) and hands it to its scoped Subfield. No store contract touched; the main pane keeps sole ownership of `liveBody`. This suffices because the preview's Subfield only ever needs *its own* page's count, which the local buffer already holds.
- **Z-3:** [confirmed by Nathan] The **preview keeps its own collapse toggle + flag, separate from the detail pane's**. The detail pane's one shared flag stays as-is (Pages + NavView collapse together). NavWindow has **no Subfield at all** — its view-type toggle stays on the SidePane rail (`navwindow-style-toggle`). So collapse flags: detail-pane (shared, existing) + preview (its own, new); NavWindow n/a.
- **Z-4:** [confirmed-finding] The **toggle button + hover reveal zone are host chrome** (`DetailPane.tsx:141-149` + `.subfield-near`), NOT part of Subfield. So the preview window and NavView each must grow their OWN toggle affordance + reveal — mounting the component alone isn't enough. Adjacency: three hosts (DetailPane, PreviewWindow, NavView) each own toggle chrome. *(Grounded.)*

#### P2 — NavView Subfield + Toggle
- **P2-1:** [confirmed, reframed — corrected by F5] NavView uses the **existing detail-pane Subfield** (same mount, same collapse toggle) — NOT a separate scoped instance (Z-0). The work: populate the **existing `none` registry entry** (`subfieldItems.tsx:20` — the key is `none`, there is NO `newtab` *kind*; `newtab` is only a tab target) with a **new view-type-toggle `SubfieldItemId`**, un-gate `showSubfield` to include **`none` AND `tree` present** (NOT bare `none` — F4: `case 'none'` also renders the no-nexus blank prompt, `DetailPane.tsx:22-31`), and wire the toggle to NavView's (new) list/gallery mode.
- **P2-2:** [corrected by F2] NavView is **gallery-only today** — no list mode, no toggle (`NavView.tsx:69`). Both are net-new. `none` kind has **no container path → no breadcrumb**; content = **just the view-type toggle** (empty left). **The view-mode must be a STORE slice, NOT a module var:** the toggle lives in the Subfield and its consumer (NavView) is a **sibling subtree** — a `savedViewMode`-style module-var mutation won't re-render NavView (that pattern only works for NavWindow because its toggle is in-component). So P2 is **not "light"** — it needs a persisted store slice (DF-3) that both the Subfield toggle item and `promote()` (DF-2) read/write.
- **P2-3:** [resolved] The view-type toggle appears in the **detail-pane Subfield** for NavView and on the **rail** for NavWindow — the split stands because NavWindow is the separate floating surface (keeps `navwindow-style-toggle`); the detail pane hosts its own via the shared Subfield.

#### DF — Don't-Forget Sweep (interactive + structural)
- **DF-1:** [DROPPED — no reshape, no shared write] No `liveBody` change at all. The preview sources its count from local state (Z-2), so it neither reshapes nor writes the slot. F1's "3-reader/2-writer contract" is real, and it's exactly why a second writer is unsafe: the readers' `=== selection.path` guards would **evict** the main pane's live count the moment the preview wrote (a guard miss falls back to the saved body), so the preview stays off the slot entirely. (F1's cited `store.ts:1469` write is the MAIN pane's warm-tab restore in `select`, not the preview — verified.)
- **DF-7:** [DISPROVEN empirically — the exclusion does NOT exist] Verified in the running app: selected page X in the detail, called `openPreview(X)` → a preview opened with `preview.originId === selection.id` (True). So a PagePreview and the detail page **CAN** hold the same page at once — there is no guard (only the same-*origin* re-summon no-op; the connection `open` handler `PageView.tsx:41-44` previews unconditionally). Nathan's belief that it's blocked is not in the code. **Implication:** the same-file-two-editors case is already possible today (independent of this feature). With Z-2's local-body design the two surfaces read **independent** stat sources (main = `liveBody`, preview = its own local buffer), so each Subfield shows its own live count with no cross-talk. **Resolved by Nathan:** leave the coexistence — no exclusion guard. **Most-recent-wins is the app's core philosophy**, so two editors on one file resolving to the last save is *by design*, not a bug. Closed.
- **DF-2:** [confirmed by Nathan] View modes stay **separate per surface** — NavWindow and NavView each own theirs; they are NOT one shared state. The promote (NavWindow → NavView) does a **one-time copy** of NavWindow's current mode into NavView **for that single transaction only**; afterward they're independent again. So: no shared-state plumbing — the promote reads NavWindow's mode and sets NavView's once.
- **DF-3:** [confirmed by Nathan] The view-mode **persists cross-session** — NavWindow's current `savedViewMode` (a session-only module var, resets on relaunch) "should never have been session-only"; fix it to persist to disk, and NavView's own mode persists likewise. **Two separate persisted slices** (`navWindowMode` + `navViewMode`, per DF-2), stored under a **standalone settings key** — NOT folded into the `subfield` config key, whose existing `setSubfield*` setters rebuild `{order, expanded}` and would clobber an added field. *(Preview's own collapse flag: session-only `useState` in the preview window — not persisted; it resets with the window, which is fine for a transient floating surface.)*
- **DF-4:** [open — visual, defer to design] Layout/z-order: where the preview Subfield sits vs the toolbar (top) + right inspector overlay + one-scroller body; NavView's gallery scroll must **reserve the Subfield footer height** (the existing `subfield-open` shrink on `.detail-pane-view` governs this uniformly — F-review confirms NavView rides it). 
- **DF-6:** [confirmed by F6 — build note] The preview/NavView collapse **reveal zone must clear the bottom-right `FloatingResizeCorners` handle** (`NavWindow.tsx:345`) + the right inspector `--io` edge — the ported DetailPane reveal rectangle (`:129-134`, BR 260×120) sits exactly on the resize corner in a floating host. Fix: trigger the reveal off the toggle chevron's own hover box, not a fixed BR corner rectangle. *(DetailPane has no resize corner, so the current pattern is safe there; only the floating hosts conflict.)*
- **DF-5:** [note] Promote is **one-way** (no inverse) — consistent with the existing page promote; re-summoning NavWindow is the way back. Every other action has its inverse (collapse↔expand, list↔gallery).

#### Lessons
- The Subfield was already built to vary items by view kind and its roadmap anticipated scoping — the "unify" ask ran with the component's grain, not against it. → durable: when a footer/registry is keyed by context, new surfaces are registry entries, not new components.
- An adversarial "HIGH" (F1) dissolved once a domain fact was applied (path-guarded readers + most-recent-wins): the reviewer was right about the contract shape but wrong about the blast radius. → verify a finding's *consequence* against the domain, not just its facts, before it drives scope.
- Most-recent-wins is a load-bearing product axiom here — it turns "two editors on one file" from a data-loss bug into intended behavior, dissolving a whole class of guards. → when a coexistence looks unsafe, check the philosophy before adding a lock.
- A path-guarded single-owner slot (`liveBody`) *reads* safe but *writes* unsafe: a guard miss doesn't isolate a second writer, it makes the reader fall back to a stale saved snapshot — an eviction, not isolation. → a "shared slot with path guards" is a one-writer contract; give a second surface its own local buffer rather than time-sharing the slot.
