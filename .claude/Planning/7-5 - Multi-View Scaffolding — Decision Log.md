## Multi-View Scaffolding — Decision Log

### Frame

- **Purpose:** Scaffold multi-view support on the existing SavedView data layer — the ViewPane → SettingsPane rename (Configuration bucket added, Layout as the flat door), a per-storage (Collection / depth-1 Set) ViewDropdown switcher next to the toolbar trio, the new ViewPane dropdown (view rows · footer), the shared two-door ViewSettings editor, the view-creation method, and the menu-consolidation pre-work everything builds on.
- **Core Value:** A container's saved views become visible, switchable, and creatable from the toolbar.
- **Success Criteria:** The button shows on Collection/depth-1-Set surfaces with the active view's icon (table by default), opens a dropdown of view rows, switching persists to `activeViews.json`, new views mint title-only via `+`, both doors open the same ViewSettings editor, and no container with visible views is ever view-less on disk (G-1).

### Decisions

#### A — Naming & Pane Split

- **A-1:** [confirmed] Current `ViewPane` renames to **SettingsPane** — it keeps the view-configuration leafs (Group, Sort, Filter, plus the built Properties + Visibility); docs reconcile to the new naming.
- **A-2:** [confirmed] The container icon + inline-rename header stays in SettingsPane as-is — it's the per-Collection/Set identity, nothing to do with views. ViewSettings carries its own icon + title field for view identity.
- **A-3:** [confirmed] The SettingsPane root gains **Configuration** above Properties (the per-collection bucket — future page formatting; glyph `sliders-horizontal`; its leaf ships with one row this cycle: Open In — F-1). **Layout stays Layout** (glyph `layout-panel-left`) — its leaf becomes the flat door into the active view's ViewSettings (D-1). Root order: Configuration · Properties · Visibility · Layout · Group · Filter · Sort.
- **A-4:** [confirmed] The SettingsPane's Group/Sort/Filter rows keep opening their empty leafs exactly as today — only the "pending" captions blank (H-10). ViewSettings ships without those rows entirely this cycle.

#### B — The Switcher Button (ViewDropdown)

- **B-1:** [confirmed] Named **ViewDropdown**, living in the `Toolbar/` folder beside the trio; per-storage routed (Collection / depth-1 Set), standalone left of the trio, slightly wider than a trio segment, with a left-right padding knob Nathan can toggle. Same glass/styling as the trio.
- **B-2:** [confirmed] The button's glyph is the active saved view's `icon`.
- **B-3:** [confirmed] Lucide's **`table`** registers as THE table glyph — the view icon, the view rows, the ViewDropdown button, AND the Table tile in ViewSettings' 3×2 type grid all use it (one glyph per concept, everywhere); `mintDefaultView`'s `'tablecells'` name reconciles to it.
- **B-4:** [confirmed] Both presentation settings live in the ViewDropdown's **right-click native context menu** — never an in-pane row (a row inside the dropdown would be unreachable once `view_style` sits at Toolbar; the menu rides the button/bar itself, reachable in either mode). It's a new main-process `Menu.popup` site (the fifth native-menu site) on the **returning-picker pattern** (the `optionMenu.ts`/`propertyMenu` precedent): a dynamic **Show Title / Hide Title** toggle item over a **Style ▸ Dropdown / Toolbar** submenu, current values checked. The popup resolves the picked action to the renderer, and the ViewDropdown performs the write through F-2's container-settings op with its own container path + kind — one write path serves the menu and any future control. Native-only, no glass fallback — users drive native menus fine everywhere; only agent tooling can't (plan verification note: the menu gets Nathan's manual pass, not a CDP one).
- **B-5:** [confirmed] **Show/Hide Title** (`view_button`: icon-only default vs icon + text) is per-container. The labeled variant holds one **fixed uniform width** with the name overflow-scrolling inside (OverflowScroll DRY-ed on); each variant carries its own clear sizing knob.
- **B-6:** [confirmed] **Dropdown / Toolbar** (`view_style` — the ViewBar concept) **toggles full-stop** — persists per-container, reads back — and the ViewDropdown carries a **minimal presentation seam** (a real branch on `viewStyle`) so ViewBar later slots into existing wiring; until ViewBar lands, both branches render the dropdown button.
- **B-7:** [confirmed] The container settings (`open_in` · `view_button` · `view_style`) persist in the sidecar (design intent that syncs, unlike per-machine `activeViews`); `view_button`/`view_style` are per-container, `open_in` is Collection-owned (F-1).

#### C — The ViewPane (View-List Dropdown)

- **C-1:** [confirmed] Named **ViewPane**; discloses from the ViewDropdown. It is a **navigation menu**: a row per saved view (icon + name + push chevron) and a footer BottomRow (`+` create pinned left, `…` pinned right) — **no options block** (per-container settings live in the context menu (B-4) and the SettingsPane (F-1)). Two zones: ViewRows descend from the top; the Footer closes the pane.
- **C-2:** [confirmed] Clicking a view row switches the active view (persisting via `activeViews`) and closes the pane — nav-menu semantics. The row's chevron pushes into ViewSettings instead.
- **C-3:** [confirmed] View rows are drag-reorderable — the list is a direct reference to the container's `views[]` order, persisting through the shipped `views:reorder` IPC.
- **C-4:** [confirmed] The footer `…` is a future-functionality placeholder — ships as chrome, no menu yet. The `+` is the creation entry (E-1).
- **C-5:** [confirmed] The pane ships on the plain shared glass — no per-row card fills. Shell grammar reuses the settings dropdown's verbatim: **MenuSurface** (NotchedPane, radius 12, `MENU_GUTTER`), **useExitPresence**, the Bloom open/close pair, with its own notch-inset math to the ViewDropdown's center.
- **C-6:** [confirmed] Pane sizing: level-1 (view list) and level-2 (ViewSettings) share an enforced min-max width; per-type sub-panes (level-3) may expand/contract beyond it — e.g. a List view's Layout pane widens into two columns for leading/trailing property assignment. All three levels ride the **same PaneSlider** and size-change animation.
- **C-7:** [confirmed] Only Collections and depth-1 Sets carry views ([[PageSets]] §Depth-1 View Rule). **Sub-Sets are expand-only disclosure groups — no detail view, never the selection** — so no Sub-Set enforcement or fallback exists anywhere in this feature: a `set` selection is depth-1 by construction (the scope helper's structural test stays as a cheap guard, not a behavior). Reparenting flips view-eligibility move-safely: a depth-1 Set moved deeper keeps its `views[]` **dormant** in the sidecar (never stripped, never rendered), re-surfacing when lifted back.

#### D — ViewSettings (the Shared Per-View Editor)

- **D-1:** [confirmed] **One view config, two doors — one shared component: ViewSettings** (a PropertiesPane-style shared pane: the type grid + the view-type-specific interface). The **ViewPane door** (full — a view row's chevron) carries the leaf rows and the header ⋮; the **SettingsPane Layout door** (flat — the SAME pane for the active view) drops the leafs and the ⋮ entirely. Purpose: one view config reachable both from view management (ViewPane) and from container settings (SettingsPane) — same data, two shapes.
- **D-2:** [confirmed] Header: back label + trailing ⋮ (full door only), then an icon square + editable title field. Empty rename keeps the prior name (`InlineEditHeader` convention); "Untitled" is only the mint default.
- **D-3:** [confirmed] The ⋮ holds **Duplicate / Delete** (rename lives in the title field). Duplicate keeps the source's name verbatim — views are id-keyed everywhere (activeViews, reorder, upsert) so name collisions are cosmetic — and lands directly after the original. Delete mutes/disables on a container's only view (`deleteView` already refuses the last one; the ⋮ mirrors the rule rather than surfacing an error). Deleting the active view closes the dropdown — `pickView` already falls back to `views[0]`, and the button glyph + list highlight follow on the same refetch beat.
- **D-4:** [confirmed] Below the header: the **3×2 type grid** — six rounded-rectangle tiles, wider than tall (the Figma frame's ~4:3 proportion), each holding ONLY its type's glyph in `label-tertiary` (titles inside cells deferred). The selected tile's 2px border is `var(--accent)` (tint-primary); unselected tiles a 2px border in `separator.border` (NOT segment), 6px radius. **Clear named knobs:** `edge-paddingX/Y` (grid insets against divider and neighbors), `grid-paddingX/Y` (between cells). Non-implemented types render at full visual weight — no muting — their click is a no-op. Selecting a type drives which type-specific rows render below.
- **D-5:** [confirmed] **Per-type grid glyphs:** Table `table` (B-3) · List a custom list glyph (dots replaced with rounded circles — becomes THE default list icon) · Gallery `layout-dashboard` · Calendar `calendar-days` (glyph edits deferred) · Timeline `chart-gantt` · Cards a custom 2×3 grid of 2:3-aspect tiles (a flatter grid). Custom glyphs live as registry-conforming SVG components; sources checked Lucide-first, Tabler second (`strokeWidth 1.75`), custom last.
- **D-6:** [confirmed] The type roster goes **five → six**: **Table, Cards, List, Gallery, Calendar, Timeline** — Board dissolves into a future Cards format/style. **Non-binding roster:** the type list is an extensible registry, so a new view type scopes in as an add-on (grid + enum extend without rework). Changes the shared `ViewType` enum + Views.md's five-type model.
- **D-7:** [confirmed] Below the grid, for Table: the **Layout leaf** (blank chrome this cycle) + the **Format row**. The leaf zone is the per-type pattern — a future Gallery might carry a "Size" toggle there instead. **Visibility folds INTO the Layout leaf** — order + visibility as one shared section — while remaining its own separate SettingsPane entry (same `property_order`/`hidden_properties` data, two access points, one shared component). **The Layout leaf's section is deferred to the next session and gated on Nathan's own Figma redesign**; until then the SettingsPane's Visibility entry remains the working source — always reachable regardless of view state.
- **D-8:** [confirmed] **"Format" is a two-option control row below the Layout leaf** (label + trailing `chevrons-up-down`) — **Standard / Compact**. It SHIPS this cycle but **toggles nothing visually** — Compact lands later as a full-scoped secondary CSS style for tables, larger than a zoom factor alone (the `--zoom` density knob in `table-tokens.css` — Standard = 1, Compact = 0.9 — is the starting substrate, not the whole style); wiring it is NOT this cycle's job. The picker **persists** (`format` key on SavedView) AND is targeted so the future hookup is CSS-only: the table root applies a format class/attribute from the active view's value; no Compact rules bind to it yet.
- **D-9:** [confirmed] **In-pane control rows (the Format toggle) get dual wiring:** a mac-native menu (main-process `Menu.popup`) on macOS, AND a glass `PickerMenu` wired to the same row so a non-mac user gets a picker. `chevrons-up-down` registers as the control's trailing glyph. (Verification note: the native path gets Nathan's manual pass; the PickerMenu path is CDP-verifiable.)
- **D-10:** [confirmed] **Back-label map** — a pushed pane's ‹ row names its back destination: ViewPane list → ViewSettings reads **‹ Views**; ViewSettings → Layout (or any view-setting leaf) reads **‹ Settings**; the SettingsPane's leafs keep their existing **‹ Settings**.
- **D-11:** [confirmed] **Tone map:** view rows — icon `label-secondary`, title `label-control`, chevron `label-secondary`; secondary panes — `chevron-right` and `chevrons-up-down` in `label-secondary`; every interactive element carries the `state-hover` effect. Control-row selection details render `footnote-emphasized` in `label-secondary` — menu.css's existing `detail` + `side` pairing, consumed, not re-authored.

#### E — Creation

- **E-1:** [confirmed] The footer `+` creates a new view **without** navigating to it by default — "open view on creation" is a per-nexus config setting (home: the `personalization` object in `.nexus/settings.json`; read-allowlist side in F-4).
- **E-2:** [confirmed] A new view mints as **"Untitled"**, type Table, at the bottom of the views list, **title-only** — every assigned property AND the default-on tier columns hidden; Title always shows because `resolveColumns` guarantees it, so hiding the schema ids plus the three tiers is the complete mint. The mint routes through a **per-type default-rules seam** (a per-ViewType mint config) so a future type can default differently; only the seam ships. The G-1 seeded/entry-minted *default* view stays **all-shown** — the invariant formalizes what a container already renders, never visibly changes it.
- **E-3:** [confirmed] The new row appears via a disclosure-fold animation expanding out of the bottom of the last view row — the same beat tokens (`duration.base` + `easing.standard`) as the All Properties unfold; same reveal mechanics, new call site.

#### F — Container Settings (Configuration & Open In)

- **F-1:** [confirmed] "Open In" is the **per-collection default page-opening behavior** — clicking a page opens full-view or in a hovering preview window (Swift's page-preview precedent; the React preview surface builds later). **Its home is the Configuration leaf — its first content, shipping this cycle** (Configuration ships non-blank): it configures the collection and its pages, not a view. The on-disk enum renames to `open_in: full-page | page-preview`, with legacy values coerced on read (`window` → `full-page`, `compact` → `page-preview`); exact UI labels are Nathan's call at build time. **Open In is Collection-owned:** a Set's SettingsPane shows the same row as a proxy that toggles its parent Collection's setting, never its own. Full-view works today; preview is the future target.
- **F-2:** [confirmed] The container-settings **persistence layer is net-new work on BOTH sides**. Write: no mutate op or IPC exists for a container sidecar's non-view settings (`updateFolderSidecar` exists in main but only `reorder.ts` consumes it, nothing preload-exposed) — the plan adds the op/handler + preload surface. Read: the sidecar read walk is an allowlist (readNexus builds nodes from named fields only) — the plan adds the node fields on `CollectionNode`/`SetNode` AND the readNexus branches that populate them, or the settings are write-only and dead. `open_in` already exists on the collection zod schema (`compact | window`, collection-only — confirming the Collection-owned write target) but has NO read or write surface today. **`view_button`, `view_style`, and `format` are new END-TO-END:** the first two need container zod schema fields + node fields + readNexus branches; `format` needs the `savedView` codec field (which IS its read side — parseViews round-trips the codec) plus its consumer, the table root's class switch. None of the three exists in any schema today.
- **F-3:** [confirmed] Dropping `'board'` from the enum: `savedView.type`'s `.catch('table')` means a hand-authored board view silently coerces to table — and the next `saveView` RMW **persists** the coercion to disk. Accepted (no board views exist in practice), but the plan notes it since the format sells agent/hand legibility. New sidecar keys ride the existing foreign-key-preserving sidecar RMW.
- **F-4:** [confirmed] `openViewOnCreate` isn't just "homed in personalization" — `readPersonalization` (`readNexus.ts`) is a hard allowlist that drops unknown keys on read, so the plan must add it to the `Personalization` interface (`types.ts`) AND a coercion branch in the reader, or the key is write-only and dead.

#### G — Invariants & Concurrency

- **G-1:** [confirmed] **Views are never empty where views can be seen — two enforcement sites.** **(1) Creation-seed:** a container-specific branch in the `createContainer` handler seeds the default Table view through `createFolderEntity`'s existing `extra` sidecar-keys param (the ULID mints in main, where `newId` lives — `shared/mintDefaultView`'s sentinel can't); an app-created container is born complete on disk. **(2) Entry-mint:** the store's `select` action (store.ts:350 — the single path every container-open flows through), on landing a Collection / depth-1 Set whose `views[]` is empty, persists the minted default via `views:save` and refetches. Entry-mint covers legacy containers, external/agent-created folders, and reparent lifts (`moveSet` is a pure folder move — mutate.ts:373) in one site — **no session-open backfill walk, no move-op hook**. Entry always precedes any view UI by construction (the ViewDropdown exists only for the current selection), so no surface can meet a phantom; the in-memory sentinel remains the render fallback for the beat between select and save-complete. **Entry-mint is the SOLE mint site**: the store tracks the mint in flight (keyed by container id — the guard against re-select re-fires during the watcher's settle window, and the promise other writers await), and every other view writer is **adopt-only**: any writer that can hold the sentinel awaits the in-flight mint and saves against the real id, never minting its own — today that's TableView's `persistView`, HiddenPane's save, and ViewSettings' writers (the title edit, the type-grid selection, the Format persist, Duplicate; the flat door is reachable during the mint beat since SettingsDropdown derives from live selection and `pickView` serves the sentinel). Delete needs no enrollment — it's muted on the sentinel by D-3's last-view rule. `select`'s collection/set cases go async-side-effecting for exactly this branch (they're synchronous store-sets today — store.ts:371). The check is O(1) on the already-loaded node and writes at most once per container lifetime (perf law); the accepted cost: an externally-created folder never entered stays viewless on disk until entered — unobservable by any UI. Opening the dropdown never writes; `activeViews` writes only on view selection. Depth-1 gating is structural (only Collections and `collection.sets` qualify — `readSet` parses `views[]` at any depth, so nothing may ever write views into Sub-Set sidecars).
- **G-2:** [confirmed] **The `views:save/reorder/delete` IPC handlers are the only sidecar RMWs in main WITHOUT `serializeOnFile`** (index.ts ~446/462/476 — settings, mutate ops, and page writes all ride the fileLock; views never joined). They get wrapped on the container sidecar path as part of this cycle — a lost-update under any two overlapping writers otherwise. The per-machine pointer writes (`writeActiveViews`, `writeViewOrders`) are the same unserialized read-modify-write class and wrap in the same pass. Serialization fixes the lost-update; the double-MINT is fixed separately by G-1's single-mint-site rule (a second sentinel save would still mint sequentially even when serialized). The new container-settings write path (F-2) rides the same chain from day one.
- **G-3:** [confirmed] Only one dropdown opens at a time, so the two doors can't race each other — but that alone serializes nothing: **the table behind any dropdown is itself a live view writer** (`persistView`, fire-and-forget). Concurrent-writer safety comes from G-2 (serialized handlers) + G-1 (single mint site, adopt-only writers), not from dropdown exclusivity.
- **G-4:** [confirmed] The view list never renders empty — by the time the ViewPane can open, its container was selected, and entry-mint has persisted the default or is mid-flight; the sentinel row covers only that in-flight beat.

#### H — Build Materials (components + CSS, grounded)

- **H-1:** [confirmed] The ViewDropdown button builds on the **Segmented pair**; `glass` default keeps the trio's GlassControls pill; icons render through the `Icon` registry. **Segmented collapses into the control component as a yes/no** — it currently assumes multiple buttons; a single toolbar button shouldn't need a "segmented" wrapper. One control primitive, `segmented` as a boolean. The button's sizing is **DRY-ed CSS-var variants, not a local constant**: the icon-only variant is an icon affordance + x-padding var; the labeled variant a fixed uniform width + x-padding (B-5). **Toolbar spacing primitive:** the x-distance between toolbar buttons/groups is one shared knob in the general toolbar area (start 10px) that any future button or cluster inherits.
- **H-2:** [confirmed] **Menu consolidation is pre-work, before the new ViewPane builds.** `viewPane.css.ts` hand-rolls five near-identical icon-button styles (`topRowAction` · `rowPlus` · `optionsAdd` · `eyeButton` · `paletteButton` — one parameterized primitive pretending to be five) and re-declares tones menu.css already owns. `menu.css.ts` becomes THE shared source — **both the existing SettingsPane and the new ViewPane route to it, and it absorbs the existing pane's coloring, spacing, and fills as shared row defaults** (the tone roles: row TITLES = `label-control` — view titles, leaf rows — as a **dropdown-scoped default**: the `item` primitive also serves the sidebar, so the control tone rides the dropdown surfaces, never a global `item` flip; HeadingLabel = `label-secondary`; ActionLabel = `label-tertiary`; the hover fill + radius; and the COLOR block's shareable entries), one size-parameterized icon-button primitive named **AccessoryButton** (the TopRow's trailing buttons — ellipsis, plus, eye, palette…), the **whole TopRow assembly DRY-ed as one general scheme** (‹ chevron + heading + optional trailing AccessoryButton — replacing the five hand-assembled `MenuTopRow` + separator copies across the panes), a new **BottomRow** (the footer scheme mirroring TopRow — nothing existed beyond `flushAffordance`'s footer geometry; `+` left / `…` right per the Figma frame), and the divider padding — with the glass shell staying single-sourced through MenuSurface. SettingsPane refactors to consume; the new ViewPane is born consuming. **A code-only before/after line diff is reported** (comments + blanks excluded). The icon-button duplication is contained to viewPane.css.ts (PickerMenu's and chip's bare-button styles are text-row families, not icon buttons); the five clones share box/flex/reset/tone/hover-fill and differ only in size + rest-opacity behavior (eye: ghost-rest · palette: hidden-until-row-hover · groupAdd composes a sixth). The hoist also removes a load-order dependency: `topRowPad`'s tone beats `flushAffordance`'s only by CSS source order today — the shared TopRow scheme owns the tone directly.

But `viewPane.css.ts` does NOT simply rename alongside its component — it's a shared knob file imported by 11 consumers (Toolbar, InlineEditHeader, paneDnd, HiddenPane, the property editors…); the rename is a **split**: the anchor hoists out, the file renames with its import graph updated, nothing re-tunes.

DRY mandates: side-to-side window padding, corner rounding, and the TopRow — inherited from the existing pane and shared across both surfaces.

#### Considered & Rejected

- **Fourth trio segment** for the switcher — rejected; the trio's beak/width math is load-bearing and the button is semantically its own control. Standalone won.
- **Inline view tabs replacing the dropdown** (tab strip at low view counts) — not rejected outright but demoted: it survives as the ViewBar behind the per-container Dropdown/Toolbar toggle; only the ViewDropdown builds now.
- **Deep Sub-Sets inheriting their ancestor's views** — rejected; Sub-Sets simply don't carry views, and the ViewDropdown doesn't render there.
- **Board as its own view type** — rejected; Board becomes a Cards format/style.
- **An options block at the ViewPane root (Open In · Format · Style rows)** — rejected; per-container settings are collection/page configuration (→ SettingsPane) or button presentation (→ the right-click context menu), and an in-pane row would be unreachable in Toolbar mode. The ViewPane stays a pure navigation menu.
- **Card-like per-row background fills** (the Figma frame's wrap-row fills) — rejected; not canonical, plain glass ships.
- **Blocking entry-mint** (await mint + refetch before rendering the selection, deleting the sentinel/adopt-only machinery) — rejected; the refetch is a full tree walk, so first entry into every legacy container would eat a visible click-to-render delay. The non-blocking sentinel path keeps selection instant.

#### Lessons

- An invariant is only as real as its named enforcement sites — "views are never empty" meant nothing until creation-seed and entry-mint (the sole mint site) each had a named mechanism; a `[confirmed]` tag on an unenforced rule is the exact false-foundation the review loop exists to catch.
- Every new config/sidecar key needs BOTH sides of its allowlist named — the read walk and the personalization reader both drop unknown keys, so "persist it in X" without a read-surfacing branch specs a write-only dead key. Caught twice in one spec (F-4, then F-2's read half).
