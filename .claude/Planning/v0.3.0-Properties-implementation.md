#### Post-ParadigmV2 status (2026-05-22)

This doc was drafted against the pre-ParadigmV2 vocabulary (single kind-agnostic `Vault` containing Pages + Items; unified `AgendaItem`; `Sub-topic`s as tier-3 Contexts). The ParadigmV2 refactor (`// Planning//ParadigmV2.md`) split the operational layer into symmetric Pages-side (Page Type → Page Collection → Page) and Items-side (Item Type → Item Collection → Item), split `AgendaItem` into `AgendaTask` + `AgendaEvent`, renamed Sub-topics to Projects, and unified schema sidecars to `_schema.json`. v0.3.0 Properties ships on top of the post-ParadigmV2 foundation; this section captures what changed in this doc and what's still outstanding.

- **Updated** — full terminology sweep applied below:

  - `Collection` (container concept) → `Page Collection` / `Item Collection` per side; `Collection` generic.
  - `_vault.json` / `_collection.json` / `_agenda.json` → `_schema.json` everywhere (per-side: under `<nexus>/Pages/<Type>/`, `<nexus>/Items/<Type>/`, `<nexus>/Agenda/Tasks/`, `<nexus>/Agenda/Events/`).

  - `VaultView` (Swift struct) → `SavedView` — shared across Page Type AND Item Type.
  - `RelationScope`: `vault(id)` / `collection(id)` → four scopes `page_type(id)` / `item_type(id)` / `page_collection(id)` / `item_collection(id)` (all dual-mandatory); `context_tier(N)` unchanged (one-way).

- **Still aligned** — the substantive decisions survive ParadigmV2 unchanged:

  - `panel_hidden_properties` per-entity scope wired v0.3.0; UI later.

1. **"Updated" = auto-tracked Last-Edited-Time.** First-class sortable property; read-only, derived from `modifiedAt`. Top of property panel.

3. **Relation scope = Notion-style single target.** Each Relation targets ONE container at creation: a specific Page Type, a specific Item Type, a specific Page Collection, a specific Item Collection, or a Context tier (1/2/3). Multiple containers → multiple Relation properties.

#### Status property type — full spec

Notion-parity type with **3 EventKit-compatible structural groups**, each containing user-editable options. Single-pick value (like Select).

##### Model

```swift
// PropertyDefinition.swift — additions

var statusGroups: [StatusGroup]?  // populated only when type == .status

struct StatusGroup: Codable, Equatable, Hashable, Identifiable, Sendable {
    var id: StatusGroupID         // structural: .upcoming / .inProgress / .done
    var label: String             // user-facing, renamable; defaults: "Upcoming" / "In Progress" / "Done"
    var color: SelectColor        // default by id: .gray / .blue / .green
    var options: [StatusOption]   // ordered; first option in each group is the default for that group
}

enum StatusGroupID: String, Codable, CaseIterable, Hashable, Sendable {
    case upcoming     = "upcoming"
    case inProgress   = "in_progress"
    case done         = "done"
}

struct StatusOption: Codable, Equatable, Hashable, Identifiable, Sendable {
    var value: String          // canonical key (immutable post-create) — used in stored values
    var label: String          // user-facing; renamable freely
    var color: SelectColor?    // optional override; nil means inherit group color
    var groupId: StatusGroupID // which group this option belongs to (load-bearing — drives sort + EventKit + display color)
    var id: String { value }
}
```

**Three structural groups, EventKit-aligned.** Group IDs (`upcoming` / `in_progress` / `done`) match EventKit's effective workflow model (see "EventKit semantics" below). Adding/removing groups NOT supported. Group LABELS are renamable (e.g., `"Upcoming"` → `"Queued"`). `StatusGroupID` is the load-bearing identity; label is presentation.

**`groupId` on StatusOption is source-of-truth for group membership.** Moving an option between groups rewrites its `groupId` (see "Moving options between groups").

**Multi-option-per-group fully supported.** Default seed ships ONE option per group; user adds more via schema editor or right-click "Add option…" popover. Sort + filter honor per-group option order.

**Default seed** (via `PropertyDefinition.statusDefault(name:)` static helper):

```json
{
  "name": "status",
  "type": "status",
  "status_groups": [
    {
      "id": "upcoming",
      "label": "Upcoming",
      "color": "gray",
      "options": [
        { "value": "not_started", "label": "Not started", "group_id": "upcoming" }
      ]
    },
    {
      "id": "in_progress",
      "label": "In Progress",
      "color": "blue",
      "options": [
        { "value": "in_progress", "label": "In progress", "color": "blue", "group_id": "in_progress" }
      ]
    },
    {
      "id": "done",
      "label": "Done",
      "color": "green",
      "options": [
        { "value": "done", "label": "Done", "color": "green", "group_id": "done" }
      ]
    }
  ]
}
```

##### Within-group option order (sort + drag)

Options within a group form an ordered array. Drag-reordering:
- **Drives sort order** (ascending: first-listed sorts first; same as Select)
- **Drives filter UI ordering** (v0.6.0)
- **No confirmation dialog** (within-group order is presentation, not workflow position)

##### Moving options between groups (RC-2026-05-19 — corrected semantics)

Moving an option **rewrites its `group_id`** and moves the entry between `options[]` arrays. The `value` stays immutable (entity-level `status: "<value>"` still resolves), but downstream semantics change:

- **Sort** — entities sort by NEW group's position.
- **Display color** — new group's pill color (option-level override still applies).
- **EventKit mapping** (v0.7.0) — flips. In Progress → Done means AgendaTasks sync as `EKReminder.isCompleted = true` / AgendaEvents sync as `EKEvent.status = .confirmed (past)`. Data-affecting.
- **Type Table grouping** (v0.6.0 Group By) — entities reshuffle visually.

**Implementation:** `PageTypeManager.moveStatusOption(propertyName:, optionValue:, toGroup: StatusGroupID, in pageType:)` (Phase 1.6) and the parallel `ItemTypeManager.moveStatusOption(...)` (Items side) and `AgendaTaskManager.moveStatusOption(...)` (Agenda Tasks). Schema-only file write (no entity rewrites; `value` unchanged) but semantically data-affecting. **Always shows a confirmation dialog** noting downstream effects (e.g., "Moving 'Blocked' from In Progress to Upcoming will affect N entities — they will now sort as Upcoming and (at v0.7.0) sync as not-completed reminders.").

##### Value storage

Page frontmatter / Item JSON / AgendaTask JSON / AgendaEvent JSON:

```yaml
properties:
  status: "in_progress"       # the option's canonical value
```

Single string value (option's `value`). Resolves at render time to option's label + color (option.color > group.color).

##### PropertyValue case

New case in `PropertyValue` (analogous to `.select(String)`):

```swift
case status(String)   // option's canonical value
```

Encodes as bare string. Disambiguated from `.select` at decode via **schema lookup** — the manager knows the parent Type's schema. Standalone decode (no schema context) defaults to `.select`; manager promotes to `.status` when binding to schema.

##### UI behavior

**Picker:** Menu-style grouped popover; three section headers (group labels) with options beneath. "+" button at bottom of each group section adds an option.

**Pill rendering:** resolved color (option.color > group.color). Same `Capsule().fill(color)` style as the refactored MultiSelectChips.

**Sort:** group position first (Upcoming < In Progress < Done ascending), then option order within group (Notion-style).

##### Schema mutations

- **Rename group label** — schema-only write (groups identified by structural `StatusGroupID`)
- **Add option** — appended to chosen group; schema-only write
- **Rename option label** — schema-only write (`value` immutable; values reference `value` not `label`)
- **Move option between groups** — schema-only write (`value` unchanged; only group reassigned)
- **Delete option** — removes + cleans up member values referencing it
- **Add/remove a group** — NOT supported (structural).

##### Where Status appears as built-in

1. **AgendaTask schema** (`<nexus>/Agenda/Tasks/_schema.json`) — required; cannot be deleted. Default seed adds `status` alongside `type` Select. EventKit sync (v0.7.0) maps Status group → `EKReminder.isCompleted`.
2. **AgendaEvent schema** (`<nexus>/Agenda/Events/_schema.json`) — NOT built-in. Completion isn't an event concept; users can add a Status property manually if a custom workflow is wanted.
3. **Page Types / Item Types** — NOT auto-seeded. User adds via Type Settings → Edit Properties.

##### EventKit semantics — load-bearing design rationale

The 3-group structure exists for clean EventKit mapping. Group IDs mirror EventKit's effective workflow position, not Notion's nominal "To-do / In Progress / Complete":

| Pommora StatusGroupID | `EKEvent.status` | `EKReminder.isCompleted` | Semantic meaning |
|---|---|---|---|
| `upcoming` | `.tentative` (future-dated) | `false` (not started, future due) | Not yet started |
| `in_progress` | `.confirmed` (currently happening) | `false` (today / actively due) | Actively in flight |
| `done` | `.confirmed` (past) | `true` (completed) | Complete |

Bidirectional mirror at v0.7.0 when EventKit sync ships. v0.3.0 just gets the property type live. **3 group slots locked in code** — a 4th group would create EventKit incompatibility. Workflow customization happens via adding options inside each group, not by inventing new groups.

v0.3.0 keeps the data scaffold compatible so the future feature lands as additive. **Reservations:**

##### Storage location reserved

```
<nexus>/.nexus/templates/<ulid>.json     (per-template definition file)
<nexus>/.nexus/templates/_templates.json (registry: which template applies to which Type)
```

Directory created on first nexus init (alongside `<nexus>/.nexus/spaces/` and `topics/`). Empty until templates ship.

##### No UI surface in v0.3.0

No "+ New Page from template…" entry, no template browser/editor. Page/Item creation flows unchanged. Template surface lands post-v1 as additive.

##### Why content templates over Type templates

Notion's templates are per-CONTENT not per-TYPE: a Type's schema is fundamental and rarely changes — you don't want a template auto-rewriting it. Templates pre-filling BODY + PROPERTY VALUES at content creation accelerate the repetitive case (new "Weekly Review" Page with same outline) without modifying the schema layer.

**0.3 — Add dual-relation config to `PropertyDefinition`**

```swift
var allowsMultiple: Bool = true     // single vs multi-pick relation
var dualProperty: DualPropertyConfig?  // REQUIRED when relationScope is .pageType / .itemType / .pageCollection / .itemCollection; nil when .contextTier

struct DualPropertyConfig: Codable, Equatable, Hashable, Sendable {
    var syncedPropertyName: String              // name of the reverse property on the target Type
    var syncedPropertyDefinedOnTypeId: String   // target Type ULID (PageType for page_* scopes; ItemType for item_* scopes — always the parent Type, even when the source scope is a Collection)

```

**Type/Collection scopes REQUIRE `dualProperty != nil`**; schema editor makes reverse-name a required input. **Context-tier scopes REQUIRE `dualProperty == nil`** (no `properties[]` schema to write reverse into). Validator rejects either violation. For Collection-scoped Relations, the reverse property lives on the parent Type's `_schema.json` — Collections don't have their own `properties[]`.

When a Relation with `dualProperty != nil` is saved on Page Type A:
1. Target Page Type B's schema gets a new Relation property (named per `syncedPropertyName`) with its own `dualProperty` pointing back at A
2. The two definitions are paired — modify one (rename/delete/type change) cascades to the other
3. Setting a value on Page A1 mirrors a back-pointer on B-side
4. Removing on either side removes on both

Symmetric on the Items side via Item Type ↔ Item Type. Cross-side (Item Type ↔ Page Type) NOT supported v0.3.0 — validator rejects.

**1.2 — Multi-file transaction support (new file `AtomicIO/SchemaTransaction.swift`)**

```swift
struct SchemaTransaction {
    let schemaURL: URL                // path to the Type's _schema.json
    let originalType: TypeSnapshot    // before-state for rollback (Codable enum: .page(PageType) | .item(ItemType))
    let newType: TypeSnapshot         // after-state
    let pageMutations: [(URL, PageFile) -> PageFile]
    let itemMutations: [(URL, Item) -> Item]

    func commit() async throws  // Two-phase: write all to temp files first, then atomic rename in batch
    func rollback()              // Restores originalType to _schema.json if commit fails midway
}

enum TypeSnapshot {
    case page(PageType)
    case item(ItemType)
}
```

Two-phase commit:
- Phase A: write all new content to `.tmp-<uuid>` siblings
- Phase B: atomically rename each `.tmp` → final in sequence

If Phase B fails partway: already-renamed files stay; remaining `.tmp` deleted; `_schema.json` restored via rollback. Partial inconsistency only possible in catastrophic OS failure; v0.3.3 SQLite adds launch-time consistency check.

The same transaction primitive serves AgendaTask / AgendaEvent schema mutations (their `_schema.json` files sit under `<nexus>/Agenda/Tasks/` and `<nexus>/Agenda/Events/`).

**1.5 — Dual-relation maintenance hook (mandatory path for Type/Collection)**

Every Type/Collection-scoped Relation has `dualProperty != nil`. On member value change, the hook fires:
- Resolve target Type from `dualProperty.syncedPropertyDefinedOnTypeId` (always the parent Type — PageType for page_* scopes, ItemType for item_* scopes — even when the source scope is a Collection)
- For each added target ID: add source entity's ULID to target's reverse-property value (paired name from `dualProperty.syncedPropertyName`)
- For each removed target ID: remove source entity's ULID from target's reverse-property value
- Target rewrites use AtomicYAMLMarkdown / AtomicJSON
- Reverse property's own `dualProperty` points back — terminal hop, no further cascade

**Pair-creation** (Pages side: `PageTypeManager.createPairedRelationProperty(sourcePageType:targetScope:sourceName:reverseName:allowsMultiple:)`; Items side: parallel on `ItemTypeManager`):
1. Validate target exists + reverse name unique in target schema
2. Append source-side PropertyDefinition with `dualProperty = DualPropertyConfig(syncedPropertyName: reverseName, syncedPropertyDefinedOnTypeId: targetTypeId)`
3. Append target-side mirror
4. Both writes through SchemaTransaction (rollback if either fails)

**Pair-deletion:** removes BOTH definitions + clears mirrored values from both Types. Confirmation dialog ("Deleting this property will also remove '<reverseName>' from <Type X>. Continue?").

Implemented as `propagateDualRelation(source:propertyName:oldValue:newValue:)` in PageContentManager+CRUD (Pages side) and ItemContentManager+CRUD (Items side). Called from `updatePageFrontmatter` / `updateItemProperties` after local write succeeds.

**Context-tier relations skip the dual hook** (`dualProperty == nil`). Reverse navigation uses existing tier1/2/3 backlink-via-query pattern.

**Cross-side guard:** validator rejects a source-Pages-side definition referencing an Item Type / Item Collection (and vice versa). Pickers in the schema editor stay side-locked.

##### Relation property creation flow

Guided multi-step flow; side-locked (Pages-side wizard never lists Item targets; Items-side wizard never lists Page targets):

```
Step 1: Pick scope kind
  ◯ Specific Type   (Page Type if source is Pages-side; Item Type if source is Items-side)
  ◯ Specific Collection   (Page Collection / Item Collection respectively)
  ◯ Context tier (Space / Topic / Project)

Step 2 (if Type): Pick target Type
  Searchable list of all Types on this side in the nexus

Step 2 (if Collection): Pick target Collection
  Searchable list of (Type, Collection) pairs on this side

Step 2 (if Context tier): Pick tier number
  ◯ Spaces (tier 1)
  ◯ Topics (tier 2)
  ◯ Projects (tier 3)

Step 3: Name the property as it appears in THIS Type
  TextField: "Property name (in <this type>)"
  Example: "Source Materials"

Step 4 (Type/Collection only): Reverse property name in TARGET
  TextField: "Reverse property name (in <target type>)"
  Required, non-empty, unique in target's _schema.json.
  Omitted for Context-tier scope (one-way).
  Note: for Collection-scoped Relations, the reverse property is stored
  on the parent Type's _schema.json (Collections don't carry their own).

Step 5: Allow multiple values? (toggle; default ON)

Step 6: Save
  Type/Collection: PageTypeManager.createPairedRelationProperty(...) or
                   ItemTypeManager.createPairedRelationProperty(...) adds BOTH sides atomically.
  Context-tier: addProperty(...) on source only.
```
