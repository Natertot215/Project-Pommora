#### Decisions locked

1. **"Updated" = auto-tracked Last-Edited-Time.** Promoted from collapsed footer to first-class sortable property. Read-only, derived from each entity's `modifiedAt` field. Listed at the top of the property panel.
2. **Number included.** Catalog gets full Notion catalog parity for the listed types.
3. **Relation scope = Notion-style single target.** Each Relation property targets exactly ONE container at creation time. Choices: a specific Vault, a specific Collection, or a Context tier (1/2/3). To target multiple containers, create multiple Relation properties.
4. **Dual relations MANDATORY for Vault/Collection scopes (RC-2026-05-19 refinement).** Every Relation property scoped to a Vault or Collection automatically creates a paired reverse property on the target — no opt-out toggle. The user must provide BOTH names at creation time (the property's name in the source Vault AND the reverse property's name in the target). Mirrored values stay synced automatically. **Context-tier scopes (Spaces / Topics / Sub-topics) stay one-way** because Contexts don't have a per-tier `properties[]` schema to write the reverse into; the "reverse view" is derived via query, same pattern as the existing `tier1`/`tier2`/`tier3` backlinks today. The previously-considered `.anywhere` scope is **removed** — every Relation property has a definite target (Vault, Collection, or Context tier).
5. **Property names are the key** (current spec preserved). Renaming triggers cross-member rewrite using a transactional pattern. Stable property IDs were considered and rejected — they'd violate Pommora load-bearing constraint #3 (agent legibility of frontmatter).
6. **Wikilinks are NOT a property type.** They're a body-text feature stored as `wikilinks: [...]` in frontmatter (derived, not user-edited). The schema editor does NOT offer Wikilink as a creatable type. Implementation lands v0.3.2.
7. **Status is a new property type — EventKit-aligned groups (RC-2026-05-19 refinement).** Single-pick from 3 fixed workflow groups: **Upcoming / In Progress / Done**. Group IDs and names are chosen to be EventKit-compatible: every group maps cleanly onto EventKit's reminder + event semantics so the v0.7.0 sync layer doesn't have to negotiate translation. Each group has user-editable options. Group LABELS can be renamed but the 3 structural slots are fixed in code. Notion ref: `Schema.status` API via Context7; group naming diverges from Notion's "To-do / In Progress / Complete" intentionally to match EventKit terminology.
8. **Multi-select pills render their color.** Current `MultiSelectChips.swift` ignores `SelectOption.color`; v0.3.0 refactors the chip primitive to consume per-option colors. Same color palette as Select (9 Notion-palette colors via `PropertyDefinition.SelectColor`).
9. **NO Vault templates. (RC-2026-05-19 — supersedes earlier toggle proposal.)** Vault creation does NOT seed any default properties. Users add properties manually via the schema editor — including Status if they want it on a Vault. Status is built-in ONLY on Agenda (where EventKit needs it). Future **content templates** (Page-from-template, Item-from-template, etc. — Notion-style) are reserved for post-v1; v0.3.0 keeps the data scaffold compatible without shipping any template surface. See "Content templates (post-v1 reservation)" section below.
10. **Schema CRUD coverage explicit (re-confirmed RC-2026-05-19):** v0.3.0's schema editor MUST support add, rename, and remove (delete) for every property type — including Status (where rename/remove operates on options + group labels, not the structural 3-group skeleton).
11. **Property icons.** Every property can carry an optional SF Symbol icon. Displayed next to the property name in: the schema editor list, the property panel rows (next to label), and the column header in Vault Table views. Settable via the schema editor's per-property config (`IconPickerField` reused from the existing SymbolPicker integration). Additive — no migration concerns.
12. **Vault Settings is the central edit surface — six sections (RC-2026-05-19).** v0.3.0 ships the consolidated `VaultSettingsSheet` with six tab/section anchors. Three light up fully at v0.3.0; three are placeholder shells filled in at v0.6.0 Vault Views:
    - **Edit Properties** — v0.3.0 functional (schema editor: add/rename/delete/reorder/icon/per-type config)
    - **Sort** — v0.3.0 functional (per-Vault default; single-criterion; multi-criterion at v0.6.0)
    - **Property Visibility** — v0.3.0 functional (per-Vault show/hide; per-saved-view at v0.6.0)
    - **Filter** — v0.3.0 placeholder ("Coming v0.6.0 with Vault Views")
    - **Group By** — v0.3.0 placeholder
    - **Layout** — v0.3.0 placeholder ("Current: Table view; 5-type picker arrives v0.6.0")
    Reached via VaultDetailView toolbar gear button + Vault row right-click → "Vault Settings…". Replaces the previously-sketched "Edit Schema…" right-click entry — that opens the same sheet, just defaulting to the Properties section.
13. **Collection picker UX — two-step Vault→Collection at schema editor; Collection-grouped value picker at value time (RC-2026-05-19).**
    - **Target picker** (schema editor configuring a Collection-scoped Relation): user picks Vault first, then sees that Vault's Collections to pick from. Avoids dumping every Collection in the nexus into one flat list.
    - **Value picker for Vault-scoped relations**: shows the Vault's content grouped by Collection (each Collection as a section header, with members listed beneath; Vault-root content as its own "(Vault root)" section at top).
    - **Value picker for Collection-scoped relations**: flat list of just that Collection's members (scope already narrowed; no navigation needed).
    - **Value picker for Context-tier-scoped relations**: flat or grouped by parent (tier-2 Topics group by parent Space; tier-3 Sub-topics group by parent Topic).
14. **No inline option creation in value pickers; structured inline-add from property labels (RC-2026-05-19, refined).** Value pickers (Select dropdown, Multi-select chip rows, Status grouped popover) never accept typed-in new options. **But** right-clicking a property LABEL (column header in Vault Table, labeled row in Page inspector / Item Window) exposes "Add option…" — opens a small structured popover (Name + Color + Group-for-Status + Save/Cancel) that commits the new option to the schema on its own Save. This is the inline path Nathan explicitly carved out: "tables and lines where there's a clear place to put the property, via the context menu dropdown." Distinct from typing-creates-option (still forbidden). Three commit paths total:
    - **Vault Settings → Edit Properties → "+ Add option"** — batched with sheet-level Save (canonical surface)
    - **Right-click property LABEL → "Add option…"** — structured popover with own Save (single-option commit shortcut)
    - **Right-click property VALUE** (pill/chip/status indicator) → "Edit options…" only (jumps to Vault Settings; no add-from-value-context)
    - **"Manage options…" link** at value picker bottom → jumps to Vault Settings at this property
    `MultiSelectChips.allowsAddingOptions` flag stays removed; `isSchemaEditMode` gate stays removed. The structured "Add option…" popover is a NEW small commit path, not a return to typed inline-add.

15. **Vault Settings = save-required + concurrent-disabled (RC-2026-05-19, Q1 + Q2 resolved).** VaultSettingsSheet is a **batched edit boundary** — all edits in the sheet stage into a draft; commits happen only on the sheet-level Save button. Cancel discards the entire draft. SchemaTransaction wraps the flush so all property edits land atomically. Concurrent open is **forbidden** — second invocation on a Vault that already has Settings open shows a disabled state ("Settings already open in <window title>"). Implemented via `@MainActor` singleton `VaultSettingsRegistry`. The inline "Add option…" popover (Decision #14) is a SEPARATE commit boundary with its own Save — does NOT batch with VaultSettingsSheet.

16. **`modified_at` trigger semantics (Q3 resolved).** Pommora's frontmatter `modified_at` updates on **any content or frontmatter edit** (body change, property value change, title rename, icon change, tier1/2/3 edit, panel-visibility toggle, etc.). View-only actions (opening / viewing a Page or Item) never update `modified_at`. The Last Edited Time property displays this frontmatter value. External edits (someone modifies the file outside Pommora) update file mtime but NOT the frontmatter `modified_at` until v0.3.3 watcher detects + propagates — documented gap.

17. **Hidden property used for sort = auto-show (Q5 resolved).** If a user has hidden a property via Property Visibility and then sets the default sort to it (or sorts a Table column on it directly), the column auto-shows. Sort precedence beats visibility because the user can't see direction indicators without the header. Implementation: when SortDescriptor references a property name in `hidden_properties`, the Table re-shows it (single-write removes from `hidden_properties`).

18. **Per-entity property disclosure wiring (RC-2026-05-19 — UI deferred, wiring required NOW).** Pages, Items, and Agenda items each carry an optional **per-entity hidden-properties list** — a UI-preference field controlling which Vault-schema properties show in THIS entity's inspector / Item Window panel. Distinct from `Vault.hidden_properties` (which controls Vault Table column visibility — Vault-wide). The per-entity field is `panel_hidden_properties: [String]` (default empty array; legacy entities decode as empty). **UI affordances** (right-click "Hide on this Page/Item" on inspector row + "+ Add property" footer button — un-hide currently-hidden OR add brand-new to Vault schema) **ship later.** v0.3.0 ships the data scaffolding + manager method signature support (`ContentManager.updatePageFrontmatter(_:, panelHiddenProperties: [String]?)` and parallel for Items, always passed nil v0.3.0) so the future UI lands as additive. Validator: each name must resolve to a Vault-schema property; orphans silently dropped on next save (preserves agent-edit tolerance).

    Two visibility scopes, never confused:

    | Scope | Field | Effect |
    |---|---|---|
    | Per-Vault | `Vault.hidden_properties: [String]` (Vault Settings → Property Visibility section) | Hides column in Vault Table view |
    | Per-Entity | `<entity>.panel_hidden_properties: [String]` (right-click on inspector row, UI ships post-v0.3.0) | Hides property row in THIS entity's inspector / Item Window only |

19. **Universal void-on-delete for property options (Q4 resolved, RC-2026-05-19).** When the user deletes an option from a Select / Status property's catalog, **every entity referencing that option's `value` gets its property value voided (set to `.null`)**. Universal rule across property types: deleting a property variant always voids — no quarantine, no auto-reassignment, no refuse-to-delete. Confirmation dialog before delete lists affected entity count. **Multi-select case is slightly different:** the deleted option's `value` is removed from each entity's array (the array shrinks; the entity's property value isn't fully voided unless the array becomes empty, in which case it becomes `.null`). Same delete cascade pattern applies. Implementation: `VaultManager.deleteSelectOption(...)` / `deleteStatusOption(...)` / `deleteMultiSelectOption(...)` all cascade through SchemaTransaction to rewrite affected entity files.

20. **Vault Settings gains a 7th "Templates" section — placeholder + unwired in v0.3.0 (Q7 resolved).** Adds a UI anchor for future content templates (Page templates, Item templates — Notion-style, pre-fill body + properties at content creation time). v0.3.0 ships the section header + placeholder ContentUnavailableView ("Templates — Pre-configured layouts for new Pages and Items in this Vault. Coming post-v1."). **No `template: ContentTemplate? = nil` parameter is added to `ContentManager.createPage(...)` in v0.3.0** — that addition lands when content templates actually ship. The placeholder section preserves the wiring concept (Vault Settings IS the template-management home) without dead code in the manager layer. The previously-defined ContentTemplate Codable sketch + `<nexus>/.nexus/templates/` storage location remain reserved.

    VaultSettingsSheet is now a **7-section sheet**:
    1. Edit Properties (functional v0.3.0)
    2. Sort (functional v0.3.0)
    3. Property Visibility (functional v0.3.0)
    4. Filter (placeholder — "Coming v0.6.0 with Vault Views")

#### Status property type — full spec

Notion-parity property type with **3 structural groups** chosen to be EventKit-compatible. Each group contains **user-editable options**. Single-pick on a Page/Item/Agenda value (one option selected at a time, like Select — not Multi-select).

##### Model

```swift
// PropertyDefinition.swift — additions

var statusGroups: [StatusGroup]?  // populated only when type == .status

struct StatusGroup: Codable, Equatable, Hashable, Identifiable, Sendable {
    var id: StatusGroupID         // structural: .upcoming / .inProgress / .done
    var label: String             // user-facing, renamable; defaults: "Upcoming" / "In Progress" / "Done"
    var color: SelectColor        // default by id: .gray / .blue / .green
    var options: [StatusOption]   // ordered; first option in each group is the default for that group
}

enum StatusGroupID: String, Codable, CaseIterable, Hashable, Sendable {
    case upcoming     = "upcoming"
    case inProgress   = "in_progress"
    case done         = "done"
}

struct StatusOption: Codable, Equatable, Hashable, Identifiable, Sendable {
    var value: String          // canonical key (immutable post-create) — used in stored values
    var label: String          // user-facing; renamable freely
    var color: SelectColor?    // optional override; nil means inherit group color
    var groupId: StatusGroupID // which group this option belongs to (load-bearing — drives sort + EventKit + display color)
    var id: String { value }
}
```

**Three structural groups, EventKit-aligned.** The group IDs (`upcoming` / `in_progress` / `done`) are chosen explicitly to match EventKit's effective workflow model — see "EventKit semantics" below. Adding/removing groups is NOT supported. Group LABELS are renamable (e.g., a user could rename `"Upcoming"` → `"Queued"` or `"Backlog"`). The `StatusGroupID` enum is the load-bearing identity; the label is presentation.

**The `groupId` field on StatusOption is the source-of-truth for which group an option belongs to.** Moving an option between groups means rewriting its `groupId` — see "Moving options between groups" below.

**Multi-option-per-group is fully supported.** The default seed ships ONE option per group; the user adds more via the schema editor or the right-click "Add option…" inline popover. Inside a single group (e.g., In Progress), the user can have multiple options like "In progress" + "Active" + "Paused", drag-reorder them, and sort + filter behavior both honor the per-group option order.

**Default seed when Status property is created** (via `PropertyDefinition.statusDefault(name:)` static helper):

```json
{
  "name": "status",
  "type": "status",
  "status_groups": [
    {
      "id": "upcoming",
      "label": "Upcoming",
      "color": "gray",
      "options": [
        { "value": "not_started", "label": "Not started", "group_id": "upcoming" }
      ]
    },
    {
      "id": "in_progress",
      "label": "In Progress",
      "color": "blue",
      "options": [
        { "value": "in_progress", "label": "In progress", "color": "blue", "group_id": "in_progress" }
      ]
    },
    {
      "id": "done",
      "label": "Done",
      "color": "green",
      "options": [
        { "value": "done", "label": "Done", "color": "green", "group_id": "done" }
      ]
    }
  ]
}
```

##### Within-group option order (sort + drag)

Inside any single group, options form an ordered array. Drag-reordering changes that array's order, which:
- **Drives sort order** (ascending: first-listed sorts first; descending: last-listed sorts first — same as Select)
- **Drives filter UI ordering** (v0.6.0 — option list in filter selectors uses the same array order)
- **Does NOT trigger a confirmation dialog** (within-group order changes are presentation, not workflow position)

Example: user creates Status, then adds "Active" + "Paused" to the In Progress group. They drag Paused above Active. Sort ascending across the property now reads: Upcoming options → Paused → Active → In progress → Done options. Filter "Status is one of {In Progress group}" returns rows in [Paused, Active, In progress] order.

##### Moving options between groups (RC-2026-05-19 — corrected semantics)

Moving an option from one group to another **rewrites the option's `group_id` field** and **moves the option entry from one group's `options[]` array to another**. The option's `value` stays immutable (so stored entity-level frontmatter `status: "<value>"` still resolves), but **every downstream semantic of that option changes**:

- **Sort behavior** — entities with that option's `value` now sort according to the NEW group's position (Upcoming/InProgress/Done order). An entity with `status: "blocked"` that was in "In Progress" group now sorts as "Upcoming" if the option was moved.
- **Display color** — entities show the new group's pill color (option-level color override still applies if set).
- **EventKit mapping** (v0.7.0) — flips. Moving an option from "In Progress" → "Done" means every Agenda item with `status: "<that option's value>"` now syncs as `EKReminder.isCompleted = true` / `EKEvent.status = .confirmed (past)`. This IS a data-affecting operation.
- **Vault Table grouping** (v0.6.0 Group By feature) — entities reshuffle visually into the new group's column/section.

**Implementation:** the move operation is implemented as `VaultManager.moveStatusOption(propertyName:, optionValue:, toGroup: StatusGroupID, in vault:)` (Phase 1.6). It's a **schema-only write** at the file level (no entity rewrites — entity-level values reference the option by `value` which is unchanged). But the user sees behavioral changes everywhere the option appears because group reassignment changes semantic meaning. **The move should always show a confirmation dialog** noting the downstream effects ("Moving 'Blocked' from In Progress to Upcoming will affect N entities — they will now sort as Upcoming and (when EventKit sync is enabled at v0.7.0) sync as not-completed reminders.").

##### Value storage

Page frontmatter / Item JSON / Agenda JSON:

```yaml
properties:
  status: "in_progress"       # the option's canonical value
```

Single string value (the option's `value`). Resolves at render time to find the option's label + the option's color (or its group's color if option color is nil).

##### PropertyValue case

New case `.status(String)` in `PropertyValue` (analogous to `.select(String)`):

```swift
case status(String)   // option's canonical value
```

Codable round-trip: encodes as a bare string. Disambiguated from `.select` at decode time by **schema lookup** — when decoding a Page's properties, the manager knows the parent Vault's schema and which key corresponds to a Status property. (At standalone decode, default to `.select`; the manager promotes to `.status` when binding values to schema.)

##### UI behavior

**Picker:** Menu-style grouped popover. Three section headers (group labels) with their options listed beneath. Currently-selected option highlighted. Click an option to select. Adding a new option happens via a "+" button at the bottom of each group's section.

**Pill rendering:** uses the resolved color (option's color > group's color). Same `Capsule().fill(color)` style as the soon-to-be-refactored MultiSelectChips. Foreground text high-contrast.

**Sort behavior:** Status sorts by **group position first** (Awaiting < InProgress < Done in ascending), then by **option order within group** (Notion-style). This matches user workflow intuition (oldest workflow position first).

##### Schema mutations

- **Rename a group label** — schema-only write, no member rewrite (groups identified by structural `StatusGroupID`)
- **Add an option** — appended to a chosen group; schema-only write
- **Rename an option's label** — schema-only write (option `value` immutable; values reference `value` not `label`)
- **Move an option between groups** — schema-only write (the option's `value` and stored member values don't change; only the group it belongs to)
- **Delete an option** — removes from group; cleans up any member values that referenced the deleted option's `value`
- **Add/remove a group** — NOT supported. Three groups are structural.

##### Where Status appears as built-in

1. **Agenda layer** (`<nexus>/Agenda/_agenda.json`) — Status is required; cannot be deleted. Default seed adds `status` alongside the existing `type` Select. EventKit sync (at v0.7.0) maps Status group to `EKEvent.status` / `EKReminder.isCompleted` (Done group → completed; others → not).
2. **Vault templates** — see "Vault templates" section below. Status ships as the default v0.3.0 template option.

##### EventKit semantics — load-bearing design rationale

The 3-group structure exists specifically for clean EventKit mapping. The group IDs (`upcoming` / `in_progress` / `done`) are chosen to mirror EventKit's effective workflow position, not Notion's nominal "To-do / In Progress / Complete":

| Pommora StatusGroupID | `EKEvent.status` | `EKReminder.isCompleted` | Semantic meaning |
|---|---|---|---|
| `upcoming` | `.tentative` (future-dated) | `false` (not started, future due) | Not yet started |
| `in_progress` | `.confirmed` (currently happening) | `false` (today / actively due) | Actively in flight |
| `done` | `.confirmed` (past) | `true` (completed) | Complete |

Stored as bidirectional mirror at v0.7.0 when EventKit sync ships. v0.3.0 just gets the property type live; mapping is wired at v0.7.0. **The 3 group slots are deliberately structural — locked in code — because adding a 4th group later would create EventKit incompatibility** (no clean target on the EventKit side). User customization of WORKFLOW happens via adding options inside each group ("Backlog", "Queued", "Triaged" all inside Upcoming), not by inventing new groups.

#### Content templates (post-v1 reservation)

**v0.3.0 does NOT ship templates.** Vault-level templates (the previously-sketched "seed a Vault's schema with default properties at creation") were considered and rejected RC-2026-05-19 in favor of Notion-style **content-level templates** — Page templates, Item templates, etc. that pre-populate body + frontmatter / properties when creating a NEW Page or Item, not a new Vault. Content templates are scoped to a Vault (so the schema matches) and apply at content-creation time.

v0.3.0 keeps the data scaffold compatible so the future feature lands as additive expansion, not a rewrite. **The reservations:**

##### Storage location reserved

```
<nexus>/.nexus/templates/<ulid>.json     (per-template definition file)
<nexus>/.nexus/templates/_templates.json (registry: which template applies to which Vault)
```

`<nexus>/.nexus/templates/` directory created on first nexus init (alongside the existing `<nexus>/.nexus/spaces/` and `<nexus>/.nexus/topics/`). Empty until templates ship.

##### Codable struct (sketch — not implemented v0.3.0)

```swift
struct ContentTemplate: Codable, Equatable, Hashable, Identifiable, Sendable {
    var id: String                    // ULID
    var name: String                  // user-facing template name
    var icon: String?                 // SF Symbol
    var targetKind: ContentKind       // .page | .item | .agendaItem
    var targetVaultId: String?        // nil = applies to any Vault of targetKind
    var bodyDefault: String?          // for .page — default Markdown body
    var descriptionDefault: String?   // for .item — default 250-char description
    var propertyDefaults: [String: PropertyValue]  // property-name → default value
    var tier1Default: [String]        // optional tier1 IDs to seed
    var tier2Default: [String]
    var tier3Default: [String]
    var iconDefault: String?          // SF Symbol on the created Page/Item
    var createdAt: Date
}

enum ContentKind: String, Codable, Sendable {
    case page, item, agendaItem
}
```

##### Manager API signature reservation

`ContentManager.createPage(...)` and `createItem(...)` already exist. They can be extended to accept an optional `template: ContentTemplate? = nil` parameter without breaking callers. v0.3.0 implementations always pass nil. Future template manager calls `createPage(..., template: someTemplate)` to seed defaults.

```swift
@MainActor
func createPage(
    in collection: Pommora.Collection?,
    vault: Vault,
    title: String,
    template: ContentTemplate? = nil   // RESERVED — always nil v0.3.0
) async throws -> PageMeta
```

##### No UI surface in v0.3.0

No "+ New Page from template…" menu entry, no template browser, no template editor. Page/Item creation flows stay as-is. The template surface lands post-v1 as additive.

##### Why content templates over Vault templates

Notion's templates are per-CONTENT not per-VAULT for good reason: a Vault's schema is fundamental and rarely changes; you don't want a template auto-rewriting the Vault's structure. Templates that pre-fill BODY + PROPERTY VALUES at content creation are higher-leverage — they accelerate the repetitive case (creating a new "Weekly Review" Page with the same outline + properties) without modifying the schema layer. v0.3.0 preserves this distinction by not muddying Vault creation with template-like seeding.

    case collection(String)       // target Collection ULID
    case contextTier(Int)         // tier number 1/2/3

    //                  "vault_id": "...", "collection_id": "...", "tier": N }

**0.3 — Add dual-relation config to `PropertyDefinition`**

Add fields:

```swift

var dualProperty: DualPropertyConfig?  // REQUIRED when relationScope is .vault / .collection; nil when .contextTier

struct DualPropertyConfig: Codable, Equatable, Hashable, Sendable {
    var syncedPropertyName: String   // name of the reverse property on the target
    var syncedPropertyDefinedOnVaultId: String   // target Vault ULID (or owning Vault of the Collection)
}
```

**Vault/Collection scopes REQUIRE `dualProperty != nil`** (validation enforces; the schema editor flow makes the reverse-name input a required field, not an optional toggle). **Context-tier scopes REQUIRE `dualProperty == nil`** (Contexts have no `properties[]` schema to write the reverse into). Validator rejects either violation at save time.

When a Relation property with `dualProperty != nil` is saved on Vault A:
1. The target Vault B's schema gets a new Relation property added (named per `syncedPropertyName`), with its own `dualProperty` pointing back at A's property
2. The two property definitions are paired — modifying one (rename, delete, type change) cascades to the other
3. Setting a value on Page A1 in Vault A → mirrors a value on whichever Page in B the relation points at (back-pointer to A1's ULID added to B-side property)
4. Removing a value on either side removes it on both

##### Phase 1 — Manager / transaction layer

Where schema CRUD lives. The transaction support pattern lands here too — multi-file rewrites for rename/type-change/delete.

**1.1 — VaultManager schema CRUD (`Vaults/VaultManager.swift`, add at end)**

Five new `@MainActor` methods:

```swift
func addProperty(name: String, type: PropertyType, config: PropertyConfigDraft, to vault: Vault) async throws
func renameProperty(from oldName: String, to newName: String, in vault: Vault) async throws
func changePropertyType(name: String, to newType: PropertyType, in vault: Vault) async throws
func deleteProperty(name: String, in vault: Vault) async throws
func updatePropertyOptions(name: String, options: [PropertyDefinition.SelectOption], in vault: Vault) async throws
```

`PropertyConfigDraft` is a sum type matching `PropertyDefinition`'s optional fields per type — used at create time when the type-specific config is being assembled.

Each method:
1. Mutates the Vault's `properties[]` in-memory
2. Writes `_vault.json` via `Vault.save(to:)` (AtomicJSON)
3. Cascades to every Page + Item in the Vault (for rename/type-change/delete)
4. On any failure mid-cascade: rolls back the Vault write + reports `RenameAtomicityError` to `pendingError` (existing pattern from v0.2.0 sidebar toast)

**1.2 — Multi-file transaction support (new file `AtomicIO/SchemaTransaction.swift`)**

The cross-file rewrite needs a coordinator. Pattern:

```swift
struct SchemaTransaction {
    let vaultURL: URL
    let originalVault: Vault
    let newVault: Vault
    let pageMutations: [(URL, PageFile) -> PageFile]
    let itemMutations: [(URL, Item) -> Item]

    func commit() async throws  // Two-phase: write all to temp files first, then atomic rename in batch
    func rollback()              // Restores originalVault to _vault.json if commit fails midway
}
```

Two-phase commit:
- Phase A: write all new content to `.tmp-<uuid>` siblings of each target file
- Phase B: atomically rename each `.tmp` → final name in sequence

If Phase B fails partway: the already-renamed files stay; remaining `.tmp` files are deleted; `_vault.json` is restored to the original via rollback. Result: a partial inconsistency between vault schema and member files possible only in catastrophic OS failure (power loss mid-rename batch). v0.3.3 SQLite landing adds an extra consistency check on launch.

**1.3 — `ContentManager.updatePageFrontmatter`** (`Content/ContentManager+CRUD.swift`, new method)

**1.5 — Dual-relation maintenance hook (mandatory path for Vault/Collection)**

Every Relation property scoped to Vault/Collection has `dualProperty != nil`. Whenever the value changes on a member (Page/Item), the dual-relation hook fires:
- Resolve the target Vault from `dualProperty.syncedPropertyDefinedOnVaultId`
- For each newly-added relation target ID: add the source entity's ULID to the target entity's reverse-property value (paired prop name from `dualProperty.syncedPropertyName`)
- For each removed relation target ID: remove the source entity's ULID from the target entity's reverse-property value
- Each target rewrite uses the same atomic-write pattern as direct edits (AtomicYAMLMarkdown / AtomicJSON)
- The reverse property's own `dualProperty` points back at us — no further cascade needed (terminal hop)

**Pair-creation flow** (new helper `VaultManager.createPairedRelationProperty(sourceVault:targetScope:sourceName:reverseName:allowsMultiple:)`):
1. Validate target Vault/Collection exists + reverse name is unique within target's schema
2. Append source-side PropertyDefinition to source Vault's `properties[]` with `dualProperty = DualPropertyConfig(syncedPropertyName: reverseName, syncedPropertyDefinedOnVaultId: targetVaultId)`
3. Append target-side PropertyDefinition to target Vault's `properties[]` with `dualProperty = DualPropertyConfig(syncedPropertyName: sourceName, syncedPropertyDefinedOnVaultId: sourceVaultId)`
4. Both writes through SchemaTransaction (two-phase commit; rollback if either fails)

**Pair-deletion flow:** deleting either side of a dual pair removes BOTH property definitions AND clears mirrored values from every member of BOTH vaults. Confirmed via dialog ("Deleting this property will also remove '<reverseName>' from <Vault X>. Continue?").

Implemented as `propagateDualRelation(source:propertyName:oldValue:newValue:)` in ContentManager+CRUD. Called from `updatePageFrontmatter` + `updateItemProperties` after the local file write succeeds.

**Context-tier relation paths skip the dual hook entirely** — they have `dualProperty == nil`. Reverse navigation uses the existing tier1/2/3 backlink-via-query pattern (no stored reverse field).

**1.6 — Status-specific manager helpers**

Add to VaultManager:

> [Cancel]  [Move 12+3 entities to Upcoming]"
