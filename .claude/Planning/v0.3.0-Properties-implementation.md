#### Decisions locked

1. **"Updated" = auto-tracked Last-Edited-Time.** First-class sortable property; read-only, derived from `modifiedAt`. Top of property panel.
2. **Number included.** Catalog gets full Notion parity for the listed types.
3. **Relation scope = Notion-style single target.** Each Relation targets ONE container at creation: a specific Vault, a specific Collection, or a Context tier (1/2/3). Multiple containers → multiple Relation properties.
4. **Dual relations MANDATORY for Vault/Collection scopes (RC-2026-05-19).** Every Vault/Collection-scoped Relation auto-creates a paired reverse property — no opt-out. User provides BOTH names at creation. Mirrored values stay synced. **Context-tier scopes stay one-way** because Contexts have no per-tier `properties[]` schema; reverse view is derived via query (same pattern as `tier1/2/3` backlinks today). Previously-considered `.anywhere` scope **removed** — every Relation has a definite target.
5. **Property names are the key.** Renaming triggers cross-member rewrite (transactional). Stable IDs rejected — violate load-bearing constraint #3 (agent legibility of frontmatter).
6. **Wikilinks are NOT a property type.** Body-text feature stored as derived `wikilinks: [...]` in frontmatter. Schema editor does NOT offer Wikilink as creatable. Implementation v0.3.2.
7. **Status is a new property type — EventKit-aligned groups (RC-2026-05-19).** Single-pick from 3 fixed workflow groups: **Upcoming / In Progress / Done**. IDs chosen for EventKit compatibility so v0.7.0 sync needs no translation. Per-group user-editable options. Group LABELS renamable; the 3 structural slots fixed in code. Notion ref: `Schema.status` (Context7); naming diverges from Notion's "To-do / In Progress / Complete" intentionally.
8. **Multi-select pills render per-option color.** `MultiSelectChips.swift` currently ignores `SelectOption.color`; v0.3.0 refactors the primitive. Same 9-color Notion palette as Select.
9. **NO Vault templates (RC-2026-05-19).** Supersedes earlier toggle proposal. Vault creation seeds NO default properties. Users add manually. Status is built-in ONLY on Agenda (EventKit-required). Future **content templates** (Page/Item-from-template, Notion-style) reserved post-v1; v0.3.0 keeps data scaffold compatible without shipping any template surface.
10. **Schema CRUD coverage explicit:** v0.3.0 schema editor MUST support add/rename/delete for every property type — including Status (operates on options + group labels, not the structural 3-group skeleton).
11. **Property icons.** Optional SF Symbol per property. Displayed in schema editor list, property panel rows, Vault Table column headers. Settable via per-property config (`IconPickerField` reuses existing SymbolPicker). Additive — no migration.
12. **Vault Settings = central edit surface, six sections (RC-2026-05-19).** Consolidated `VaultSettingsSheet`. Three fully functional at v0.3.0; three placeholder shells for v0.6.0:
    - **Edit Properties** — v0.3.0 functional (add/rename/delete/reorder/icon/per-type config)
    - **Sort** — v0.3.0 functional (per-Vault default, single-criterion; multi at v0.6.0)
    - **Property Visibility** — v0.3.0 functional (per-Vault show/hide; per-saved-view at v0.6.0)
    - **Filter** — placeholder ("Coming v0.6.0 with Vault Views")
    - **Group By** — placeholder
    - **Layout** — placeholder ("Current: Table; 5-type picker arrives v0.6.0")
    Entry: VaultDetailView toolbar gear + Vault-row right-click → "Vault Settings…". Replaces the previously-sketched "Edit Schema…" entry (now defaults to Properties section).
13. **Collection picker UX (RC-2026-05-19).** Two-step Vault→Collection at schema editor; Collection-grouped value picker at value time:
    - **Target picker** (schema editor, Collection-scoped Relation): pick Vault first, then its Collections. Avoids flat nexus-wide list.
    - **Value picker (Vault-scoped)**: Vault's content grouped by Collection (each as section header; Vault-root in "(Vault root)" section at top).
    - **Value picker (Collection-scoped)**: flat list of that Collection's members.
    - **Value picker (Context-tier)**: flat or grouped by parent (tier-2 by parent Space; tier-3 by parent Topic).
14. **No inline option creation in value pickers; structured inline-add from property LABELS (RC-2026-05-19).** Value pickers (Select / Multi-select / Status) never accept typed new options. Right-clicking a property LABEL (column header, inspector row, Item Window row) exposes "Add option…" — small structured popover (Name + Color + Group-for-Status + Save/Cancel) committing on its own Save. Nathan's carved-out path: "tables and lines where there's a clear place to put the property, via the context menu dropdown." Distinct from typed inline-add (still forbidden). Three commit paths:
    - **Vault Settings → Edit Properties → "+ Add option"** — batched with sheet Save (canonical)
    - **Right-click property LABEL → "Add option…"** — structured popover with own Save
    - **Right-click property VALUE** → "Edit options…" only (jumps to Vault Settings)
    - **"Manage options…" link** at value picker bottom → jumps to Vault Settings at this property
    `MultiSelectChips.allowsAddingOptions` + `isSchemaEditMode` stay removed. Structured "Add option…" popover is a NEW small commit path, not typed inline-add.

15. **Vault Settings = save-required + concurrent-disabled (RC-2026-05-19, Q1 + Q2).** VaultSettingsSheet is a **batched edit boundary** — edits stage into a draft; commit only on sheet Save. Cancel discards. SchemaTransaction wraps the flush atomically. Concurrent open **forbidden** — second invocation shows disabled state ("Settings already open in <window title>") via `@MainActor` singleton `VaultSettingsRegistry`. The inline "Add option…" popover (Decision #14) is a SEPARATE commit boundary with its own Save.

16. **`modified_at` trigger semantics (Q3).** Updates on **any content or frontmatter edit** (body, property value, title rename, icon, tier1/2/3, panel-visibility toggle). View-only actions never update. Last Edited Time property displays this. External edits update file mtime but NOT frontmatter `modified_at` until v0.3.3 watcher closes the gap — documented.

17. **Hidden property used for sort = auto-show (Q5).** Setting default sort to a hidden property (or column-sorting on it) auto-shows the column. Sort precedence beats visibility (no direction indicators without header). Single-write removes from `hidden_properties`.

18. **Per-entity property disclosure wiring (RC-2026-05-19 — UI deferred, wiring NOW).** Pages / Items / Agenda items carry `panel_hidden_properties: [String]` (default empty; legacy decodes empty). Distinct from `Vault.hidden_properties` (Table column visibility, Vault-wide). **UI affordances** (right-click "Hide on this Page/Item" + "+ Add property" footer) **ship later.** v0.3.0 ships data scaffolding + manager method signature (`ContentManager.updatePageFrontmatter(_:, panelHiddenProperties: [String]?)`, always passed nil v0.3.0) so future UI is additive. Validator: orphan names silently dropped on save.

    Two visibility scopes, never confused:

    | Scope | Field | Effect |
    |---|---|---|
    | Per-Vault | `Vault.hidden_properties: [String]` (Vault Settings → Property Visibility) | Hides column in Vault Table |
    | Per-Entity | `<entity>.panel_hidden_properties: [String]` (right-click inspector row; UI post-v0.3.0) | Hides row in THIS entity's inspector/Item Window only |

19. **Universal void-on-delete for property options (Q4, RC-2026-05-19).** Deleting an option from Select/Status voids every referencing entity's value (`.null`). Universal rule — no quarantine, no auto-reassignment, no refuse-to-delete. Confirmation dialog lists affected entity count. **Multi-select differs:** the option's `value` is removed from each array (shrinks; `.null` only if empty). Same cascade pattern. Implementation: `VaultManager.deleteSelectOption(...)` / `deleteStatusOption(...)` / `deleteMultiSelectOption(...)` all cascade through SchemaTransaction.

20. **Vault Settings 7th "Templates" section — placeholder + unwired (Q7).** UI anchor for future content templates. v0.3.0 ships the section header + ContentUnavailableView ("Templates — Pre-configured layouts for new Pages and Items in this Vault. Coming post-v1."). **No `template: ContentTemplate? = nil` parameter added to `ContentManager.createPage(...)`** — lands when templates actually ship. Placeholder preserves the wiring concept (Vault Settings IS the template home) without dead code. ContentTemplate Codable sketch + `<nexus>/.nexus/templates/` location remain reserved.

    VaultSettingsSheet is now a **7-section sheet**:
    1. Edit Properties (functional v0.3.0)
    2. Sort (functional v0.3.0)
    3. Property Visibility (functional v0.3.0)
    4. Filter (placeholder — "Coming v0.6.0 with Vault Views")

#### Status property type — full spec

Notion-parity type with **3 EventKit-compatible structural groups**, each containing user-editable options. Single-pick value (like Select).

##### Model

```swift
// PropertyDefinition.swift — additions

var statusGroups: [StatusGroup]?  // populated only when type == .status

struct StatusGroup: Codable, Equatable, Hashable, Identifiable, Sendable {
    var id: StatusGroupID         // structural: .upcoming / .inProgress / .done
    var label: String             // user-facing, renamable; defaults: "Upcoming" / "In Progress" / "Done"
    var color: SelectColor        // default by id: .gray / .blue / .green
    var options: [StatusOption]   // ordered; first option in each group is the default for that group
}

enum StatusGroupID: String, Codable, CaseIterable, Hashable, Sendable {
    case upcoming     = "upcoming"
    case inProgress   = "in_progress"
    case done         = "done"
}

struct StatusOption: Codable, Equatable, Hashable, Identifiable, Sendable {
    var value: String          // canonical key (immutable post-create) — used in stored values
    var label: String          // user-facing; renamable freely
    var color: SelectColor?    // optional override; nil means inherit group color
    var groupId: StatusGroupID // which group this option belongs to (load-bearing — drives sort + EventKit + display color)
    var id: String { value }
}
```

**Three structural groups, EventKit-aligned.** Group IDs (`upcoming` / `in_progress` / `done`) match EventKit's effective workflow model (see "EventKit semantics" below). Adding/removing groups NOT supported. Group LABELS are renamable (e.g., `"Upcoming"` → `"Queued"`). `StatusGroupID` is the load-bearing identity; label is presentation.

**`groupId` on StatusOption is source-of-truth for group membership.** Moving an option between groups rewrites its `groupId` (see "Moving options between groups").

**Multi-option-per-group fully supported.** Default seed ships ONE option per group; user adds more via schema editor or right-click "Add option…" popover. Sort + filter honor per-group option order.

**Default seed** (via `PropertyDefinition.statusDefault(name:)` static helper):

```json
{
  "name": "status",
  "type": "status",
  "status_groups": [
    {
      "id": "upcoming",
      "label": "Upcoming",
      "color": "gray",
      "options": [
        { "value": "not_started", "label": "Not started", "group_id": "upcoming" }
      ]
    },
    {
      "id": "in_progress",
      "label": "In Progress",
      "color": "blue",
      "options": [
        { "value": "in_progress", "label": "In progress", "color": "blue", "group_id": "in_progress" }
      ]
    },
    {
      "id": "done",
      "label": "Done",
      "color": "green",
      "options": [
        { "value": "done", "label": "Done", "color": "green", "group_id": "done" }
      ]
    }
  ]
}
```

##### Within-group option order (sort + drag)

Options within a group form an ordered array. Drag-reordering:
- **Drives sort order** (ascending: first-listed sorts first; same as Select)
- **Drives filter UI ordering** (v0.6.0)
- **No confirmation dialog** (within-group order is presentation, not workflow position)

##### Moving options between groups (RC-2026-05-19 — corrected semantics)

Moving an option **rewrites its `group_id`** and moves the entry between `options[]` arrays. The `value` stays immutable (entity-level `status: "<value>"` still resolves), but downstream semantics change:

- **Sort** — entities sort by NEW group's position.
- **Display color** — new group's pill color (option-level override still applies).
- **EventKit mapping** (v0.7.0) — flips. In Progress → Done means Agenda items sync as `EKReminder.isCompleted = true` / `EKEvent.status = .confirmed (past)`. Data-affecting.
- **Vault Table grouping** (v0.6.0 Group By) — entities reshuffle visually.

**Implementation:** `VaultManager.moveStatusOption(propertyName:, optionValue:, toGroup: StatusGroupID, in vault:)` (Phase 1.6). Schema-only file write (no entity rewrites; `value` unchanged) but semantically data-affecting. **Always shows a confirmation dialog** noting downstream effects (e.g., "Moving 'Blocked' from In Progress to Upcoming will affect N entities — they will now sort as Upcoming and (at v0.7.0) sync as not-completed reminders.").

##### Value storage

Page frontmatter / Item JSON / Agenda JSON:

```yaml
properties:
  status: "in_progress"       # the option's canonical value
```

Single string value (option's `value`). Resolves at render time to option's label + color (option.color > group.color).

##### PropertyValue case

New case in `PropertyValue` (analogous to `.select(String)`):

```swift
case status(String)   // option's canonical value
```

Encodes as bare string. Disambiguated from `.select` at decode via **schema lookup** — the manager knows the parent Vault's schema. Standalone decode (no schema context) defaults to `.select`; manager promotes to `.status` when binding to schema.

##### UI behavior

**Picker:** Menu-style grouped popover; three section headers (group labels) with options beneath. "+" button at bottom of each group section adds an option.

**Pill rendering:** resolved color (option.color > group.color). Same `Capsule().fill(color)` style as the refactored MultiSelectChips.

**Sort:** group position first (Awaiting < InProgress < Done ascending), then option order within group (Notion-style).

##### Schema mutations

- **Rename group label** — schema-only write (groups identified by structural `StatusGroupID`)
- **Add option** — appended to chosen group; schema-only write
- **Rename option label** — schema-only write (`value` immutable; values reference `value` not `label`)
- **Move option between groups** — schema-only write (`value` unchanged; only group reassigned)
- **Delete option** — removes + cleans up member values referencing it
- **Add/remove a group** — NOT supported (structural).

##### Where Status appears as built-in

1. **Agenda layer** (`<nexus>/Agenda/_agenda.json`) — required; cannot be deleted. Default seed adds `status` alongside `type` Select. EventKit sync (v0.7.0) maps Status group → `EKEvent.status` / `EKReminder.isCompleted`.
2. **Vault templates** — see "Vault templates" below. Default v0.3.0 template option.

##### EventKit semantics — load-bearing design rationale

The 3-group structure exists for clean EventKit mapping. Group IDs mirror EventKit's effective workflow position, not Notion's nominal "To-do / In Progress / Complete":

| Pommora StatusGroupID | `EKEvent.status` | `EKReminder.isCompleted` | Semantic meaning |
|---|---|---|---|
| `upcoming` | `.tentative` (future-dated) | `false` (not started, future due) | Not yet started |
| `in_progress` | `.confirmed` (currently happening) | `false` (today / actively due) | Actively in flight |
| `done` | `.confirmed` (past) | `true` (completed) | Complete |

Bidirectional mirror at v0.7.0 when EventKit sync ships. v0.3.0 just gets the property type live. **3 group slots locked in code** — a 4th group would create EventKit incompatibility. Workflow customization happens via adding options inside each group, not by inventing new groups.

#### Content templates (post-v1 reservation)

**v0.3.0 does NOT ship templates.** Vault-level templates were rejected RC-2026-05-19 in favor of Notion-style **content-level templates** (Page/Item templates pre-populating body + frontmatter at content-creation time, scoped to a Vault).

v0.3.0 keeps the data scaffold compatible so the future feature lands as additive. **Reservations:**

##### Storage location reserved

```
<nexus>/.nexus/templates/<ulid>.json     (per-template definition file)
<nexus>/.nexus/templates/_templates.json (registry: which template applies to which Vault)
```

Directory created on first nexus init (alongside `<nexus>/.nexus/spaces/` and `topics/`). Empty until templates ship.

@MainActor
func createPage(
    in collection: Pommora.Collection?,

##### No UI surface in v0.3.0

No "+ New Page from template…" entry, no template browser/editor. Page/Item creation flows unchanged. Template surface lands post-v1 as additive.

##### Why content templates over Vault templates

Notion's templates are per-CONTENT not per-VAULT: a Vault's schema is fundamental and rarely changes — you don't want a template auto-rewriting it. Templates pre-filling BODY + PROPERTY VALUES at content creation accelerate the repetitive case (new "Weekly Review" Page with same outline) without modifying the schema layer.

#### Phase-by-phase implementation

Each phase ships green standalone. Stub-and-progressively-replace per paradigm decision #4. Tests land alongside the code.

##### Phase 0 — Model layer extensions

Minimum data-shape changes for the locked decisions.

**0.1 — Extend `PropertyType` (PropertyType.swift)**

Add after the existing 8:
- `case lastEditedTime = "last_edited_time"` — auto-property; schema-addable but instances render read-only
- `case status` — Notion-style 3-group workflow; see Status section above

**0.2 — Reshape `RelationScope` (PropertyDefinition.swift:34-37)**

Current: `String` enum with `sameVault` + `anywhere`. Replace with associated-value enum — **three definite kinds, no `anywhere`**:

```swift
enum RelationScope: Codable, Equatable, Hashable, Sendable {
    case vault(String)            // target Vault ULID
    case collection(String)       // target Collection ULID
    case contextTier(Int)         // tier number 1/2/3

    // Codable shape: { "kind": "vault" | "collection" | "context_tier",
    //                  "vault_id": "...", "collection_id": "...", "tier": N }
}
```

Migration: most v0.2 vaults have empty `properties[]`; rest use old `sameVault`/`anywhere`. Decode-time shim: `"same_vault"` → `.vault(<this vault's ID>)`; `"anywhere"` → `.vault(<this vault's ID>)` with logged warning (zero such properties exist in current data). No separate pass.

**0.3 — Add dual-relation config to `PropertyDefinition`**

```swift
var allowsMultiple: Bool = true     // single vs multi-pick relation
var dualProperty: DualPropertyConfig?  // REQUIRED when relationScope is .vault / .collection; nil when .contextTier

struct DualPropertyConfig: Codable, Equatable, Hashable, Sendable {
    var syncedPropertyName: String   // name of the reverse property on the target
    var syncedPropertyDefinedOnVaultId: String   // target Vault ULID (or owning Vault of the Collection)
}
```

**Vault/Collection scopes REQUIRE `dualProperty != nil`**; schema editor makes reverse-name a required input. **Context-tier scopes REQUIRE `dualProperty == nil`** (no `properties[]` schema to write reverse into). Validator rejects either violation.

When a Relation with `dualProperty != nil` is saved on Vault A:
1. Target Vault B's schema gets a new Relation property (named per `syncedPropertyName`) with its own `dualProperty` pointing back at A
2. The two definitions are paired — modify one (rename/delete/type change) cascades to the other

**1.2 — Multi-file transaction support (new file `AtomicIO/SchemaTransaction.swift`)**

```swift
struct SchemaTransaction {
    let vaultURL: URL
    let originalVault: Vault
    let newVault: Vault
    let pageMutations: [(URL, PageFile) -> PageFile]
    let itemMutations: [(URL, Item) -> Item]

    func commit() async throws  // Two-phase: write all to temp files first, then atomic rename in batch
    func rollback()              // Restores originalVault to _vault.json if commit fails midway
}
```

Two-phase commit:
- Phase A: write all new content to `.tmp-<uuid>` siblings
- Phase B: atomically rename each `.tmp` → final in sequence

If Phase B fails partway: already-renamed files stay; remaining `.tmp` deleted; `_vault.json` restored via rollback. Partial inconsistency only possible in catastrophic OS failure; v0.3.3 SQLite adds launch-time consistency check.

**1.5 — Dual-relation maintenance hook (mandatory path for Vault/Collection)**

Every Vault/Collection-scoped Relation has `dualProperty != nil`. On member value change, the hook fires:
- Resolve target Vault from `dualProperty.syncedPropertyDefinedOnVaultId`
- For each added target ID: add source entity's ULID to target's reverse-property value (paired name from `dualProperty.syncedPropertyName`)
- For each removed target ID: remove source entity's ULID from target's reverse-property value
- Target rewrites use AtomicYAMLMarkdown / AtomicJSON
- Reverse property's own `dualProperty` points back — terminal hop, no further cascade

**Pair-creation** (`VaultManager.createPairedRelationProperty(sourceVault:targetScope:sourceName:reverseName:allowsMultiple:)`):
1. Validate target exists + reverse name unique in target schema
2. Append source-side PropertyDefinition with `dualProperty = DualPropertyConfig(syncedPropertyName: reverseName, syncedPropertyDefinedOnVaultId: targetVaultId)`
3. Append target-side mirror
4. Both writes through SchemaTransaction (rollback if either fails)

**Pair-deletion:** removes BOTH definitions + clears mirrored values from both vaults. Confirmation dialog ("Deleting this property will also remove '<reverseName>' from <Vault X>. Continue?").

Implemented as `propagateDualRelation(source:propertyName:oldValue:newValue:)` in ContentManager+CRUD. Called from `updatePageFrontmatter` / `updateItemProperties` after local write succeeds.

- Relation scope's target must exist at save time
- Select/Multi-select: ≥1 option; `value` unique within property

- Property add → set value → reload → preserved
- Rename → all members update; no orphan values
- Delete → all members lose value; no schema breakage
- Type change lossless (Date → Datetime) → values converted

- Dual relation create → target gets reverse; either side reflects on other
- Cross-Vault move + strip (pulled into v0.3.0 from v0.4.0 — tightly coupled to schema): warning dialog lists strips before move
