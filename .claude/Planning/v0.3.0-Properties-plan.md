### v0.3.0 Properties — Implementation Plan

> Implementation-grade plan for v0.3.0 Properties. Authored 2026-05-23 against the post-ParadigmV2 code surface (Phase 9 Task 9.2). Conceptual WHAT lives in [`v0.3.0-Properties-spec.md`](v0.3.0-Properties-spec.md) — read first; this plan is the HOW. The pre-ParadigmV2 implementation plan + uncertainty log are archived at [`Superseded/v0.3.0-Properties-implementation.md`](Superseded/v0.3.0-Properties-implementation.md) and [`Superseded/v0.3.0-Properties-uncertainty-log.md`](Superseded/v0.3.0-Properties-uncertainty-log.md). They're reference-only.

#### Vision and scope

v0.3.0 closes the Items paradigm by making properties first-class across Pages, Items, AgendaTask, and AgendaEvent. The ParadigmV2 refactor (shipped Phases 2–8) already split the operational layer into symmetric Pages-side (`PageType` / `PageCollection` / `PageContentManager`) and Items-side (`ItemType` / `ItemCollection` / `ItemContentManager`) containers, unified schema sidecars to `_schema.json`, split Agenda into `AgendaTask` + `AgendaEvent` with per-side schema sidecars at `<nexus>//Agenda//Tasks//_schema.json` and `<nexus>//Agenda//Events//_schema.json`, and stood up `SettingsManager` for renameable UI labels. This plan layers Properties on top of that foundation without reshaping the operational topology — the data layer is already symmetric end-to-end.

The plan ships in five phases (A–E). Phase A is the schema editing infrastructure (model extensions + manager schema CRUD + the cross-file transaction primitive). Phase B wires property-value editing into the per-entity detail surfaces (Page inspector, Item Window, Agenda task/event forms). Phase C lands the per-type editors and side-aware pickers. Phase D delivers the move-strip rule (cross-Type moves drop schema-orphaned values). Phase E hardens validation, integration coverage, and ships the spec doc updates.

#### Decision log

Open questions the plan resolves before Phase A begins.

##### D1 — Does `ItemTypeSettingsSheet` ship at v0.3.0?

**Choice: SHIP at v0.3.0.**

- **Options considered:**
  - **(a) Ship now.** Items-side schema editing works end-to-end via right-click "Type Settings…" the moment v0.3.0 ships.
  - **(b) Defer to the v0.3.1 Items-pane plan.** v0.3.0 focuses exclusively on Pages-side end-to-end; Item Type schemas can only be edited by hand-editing `_schema.json` until v0.3.1.
- **Rationale:** The Items-side data layer is fully symmetric (`ItemType` mirrors `PageType`, `ItemContentManager` mirrors `PageContentManager`, `ItemTypeManager` ships with full ItemType + ItemCollection CRUD); both `_schema.json` paths are live in [`NexusPaths.swift`](../../Pommora/Pommora/AtomicIO/NexusPaths.swift) (lines 173–300). Asymmetric shipping leaves a sharp edge — Item Types accepting properties via Items-side detail-pane UI but no canonical edit affordance — and forces a JSON-level workaround the entire v0.3.0 → v0.3.1 gap. The sheet itself is structurally identical to its Pages-side twin (same seven sections; same draft-and-commit boundary); building it now via a shared base view + protocol costs roughly half a session of incremental work because every primitive (NewPropertySheet, RelationPropertyWizard, the type-specific config sub-views, the option editors, the SchemaTransaction primitive) is kind-agnostic anyway.
- **What stays deferred regardless:** Items-side detail-pane Table column rendering ships with the v0.3.1 Items-pane plan. v0.3.0's Items-side surface is the Type Settings sheet + the Item Window's property panel — both work without the per-Type Table column rebuild.

##### D2 — Transaction primitive shape

**Choice: Pre-write tmp siblings + batch rename, with reverse-snapshot rollback.**

- **Options considered:**
  - **(a) Codable forward-diff + reverse-diff journal.** Apply forward-diff incrementally; replay reverse-diff on failure. Sound, but heavyweight — diff machinery, replay correctness proofs, journal format. v0.3.0 doesn't ship the SQLite indexer that would consume the journal.
  - **(b) Pre-write tmp siblings + batch rename + snapshot-based rollback.** Phase A1: write every new file body to a `.tmp-<uuid>` sibling. Phase A2: atomic-rename each tmp into place in sequence. On mid-A2 failure: rename already-renamed files back from a captured `before` snapshot, delete remaining `.tmp` siblings, restore the schema sidecar.
  - **(c) Best-effort sequential writes + log-and-warn.** No rollback. Spec explicitly forbids: "must execute as a two-phase commit … complete rollback across every affected file."
- **Rationale:** (b) matches the spec's "two-phase commit" wording and lines up with the existing rename-atomicity pattern already shipped on every Pages-side and Items-side rename path (see [`PageTypeManager.swift`](../../Pommora/Pommora/Vaults/PageTypeManager.swift) lines 134–148, [`ItemTypeManager.swift`](../../Pommora/Pommora/Items/ItemTypeManager.swift) lines 169–181, [`AgendaTaskManager.swift`](../../Pommora/Pommora/Agenda/AgendaTaskManager.swift) lines 132–144, [`AgendaEventManager.swift`](../../Pommora/Pommora/Agenda/AgendaEventManager.swift) lines 132–144). It generalizes the existing per-method "save → rollback rename on save failure" pattern across N files. The diff machinery in (a) is post-v1 — v0.3.3 SQLite adds launch-time divergence detection that subsumes it.
- **Shape:** New struct `SchemaTransaction` under `Pommora/Pommora/AtomicIO/`. Captures `originalSchema: SchemaSnapshot`, `newSchema: SchemaSnapshot`, ordered `[FileMutation]` (URL + new-content closure). `commit()` walks the mutations: writes each to its `.tmp-<uuid>` sibling, then atomic-renames in sequence. On failure mid-rename: rolls back via inverse rename of already-renamed files (the snapshot captures pre-state file URLs + payloads) and re-saves the original schema sidecar. `SchemaSnapshot` is a small enum: `.page(PageType)` / `.item(ItemType)` / `.agendaTask(AgendaTaskSchema)` / `.agendaEvent(AgendaEventSchema)` — four cases match the four `_schema.json` carriers post-ParadigmV2. Same primitive serves every Type-level schema mutation across all four sides.

##### D3 — `PropertyConfigDraft` location

**Choice: Same file as `PropertyDefinition.swift`.**

- **Options considered:** (a) same file, (b) own file under `Vaults/` or a new `Properties/` directory.
- **Rationale:** Small struct family; tightly type-paired; existing `PropertyDefinition.SelectOption` / `NumberFormat` / `SelectColor` already nest there (see [`PropertyDefinition.swift`](../../Pommora/Pommora/Vaults/PropertyDefinition.swift) lines 18–37). Keeps schema-creation types co-located.

##### D4 — `defaultSort: SortDescriptor?` location

**Choice: Top-level optional field on `PageType` / `ItemType` / `AgendaTaskSchema` / `AgendaEventSchema`.**

- **Options considered:** (a) top-level, (b) `TypeDefaults` substruct, (c) inside a v0.6.0 `views[]` first stub.
- **Rationale:** Single optional vs. substruct: v0.3.0 needs exactly one persisted field (default sort); a substruct buys nothing until v0.6.0 piles on filter / group-by defaults. When v0.6.0 does, refactor in place. Avoids the `views[]` coupling that constraints sort to live inside a saved-view config it doesn't need.

##### D5 — RelationScope legacy decode shim

**Choice: Best-effort decode-only fallback; no preserved encode path.**

- **Options considered:** (a) decode-only shim that maps legacy `"same_vault"` / `"anywhere"` strings to `.pageType(<this PageType's id>)`, log a warning; (b) deferred — assume ParadigmV2's user-data migration covers it; (c) hard failure on legacy values.
- **Rationale:** ParadigmV2 Phase 10's user-data migration (per [`ParadigmV2.md`](ParadigmV2.md)) covers structural rename of `_vault.json` → `_schema.json` but the legacy `relation_scope` string values inside the `properties[]` arrays may slip through if any v0.2-era Page Type had a Relation property. Audit shows no production data carries Relation properties yet (`PropertyEditorRow.swift:33` is a stub) — but decode-time defensiveness costs almost nothing. Encode never emits the legacy form. New encode shape is documented in the spec.

##### D6 — Items-side `panel_hidden_properties` parity

**Choice: Ship the data field + manager-signature plumbing on Items + AgendaTask + AgendaEvent in v0.3.0, mirroring the Pages-side wiring.**

- **Options considered:** (a) full parity now, (b) Pages-side only.
- **Rationale:** Spec decision #18 explicitly calls for "data scaffolding + manager method signature" across Page, Item, AgendaTask, AgendaEvent. The Items-side data layer ships the field at the same cost as Pages — `Item.swift` already carries `properties: [String: PropertyValue]`; adding `panelHiddenProperties: [String]` is a one-line Codable addition with `decodeIfPresent` default-empty. Skipping it forces a v0.3.1 schema bump.

##### D7 — Where the Type Settings router lives

**Choice: New `@MainActor @Observable SchemaEditorRouter` in `Pommora/Pommora/Sheets/`, environment-injected.**

- **Options considered:** (a) `@Environment` router, (b) per-view local sheet state, (c) a singleton.
- **Rationale:** The spec requires routed jumps from value pickers and chip context menus into Type Settings → Edit Properties at a specific property row. A small `@Observable` router with `openTarget: OpenTarget?` keeps the routing source-of-truth in one place; both `PageTypeSettingsSheet` and `ItemTypeSettingsSheet` observe it and only react to their own `typeRef` side. Singleton (c) re-introduces hidden global state we removed in Phase 7; per-view state (b) can't reach across the value-picker → sheet boundary cleanly.

#### Phase ordering

Five phases (A–E), each ships green standalone per the stub-and-progressively-replace strategy (CLAUDE.md quirk #8). Tests land alongside the code in every phase. No phase is gated behind another's UI surface — Phase A wires the data layer; Phase B reads it; Phase C reshapes the surfaces; Phase D adds cross-Type semantics; Phase E hardens and ships.

###### A.5 — `SchemaTransaction` primitive

New file: `Pommora/Pommora/AtomicIO/SchemaTransaction.swift`.

```swift
@MainActor
struct SchemaTransaction {
    let originalSchema: SchemaSnapshot
    let newSchema: SchemaSnapshot
    let schemaURL: URL
    let memberMutations: [MemberMutation]   // ordered

    enum SchemaSnapshot {
        case page(PageType)
        case item(ItemType)
        case agendaTask(AgendaTaskSchema)
        case agendaEvent(AgendaEventSchema)
    }

    enum MemberMutation {
        case page(URL, transform: @MainActor (PageFile) -> PageFile)
        case item(URL, transform: @MainActor (Item) -> Item)
        case agendaTask(URL, transform: @MainActor (AgendaTask) -> AgendaTask)
        case agendaEvent(URL, transform: @MainActor (AgendaEvent) -> AgendaEvent)
    }

    /// Two-phase commit: phase 1 writes every payload to .tmp-<uuid>; phase 2
    /// atomic-renames in sequence. On phase-2 failure mid-way: re-renames the
    /// already-renamed files back from the snapshot capture and re-writes the
    /// original schema. Throws RenameAtomicityError on cascade failure.
    func commit() async throws
}
```

The same primitive serves PageType + ItemType + AgendaTaskSchema + AgendaEventSchema mutations. The two-phase pattern matches the v0.2 rename-atomicity rollback already in use across [`PageTypeManager.swift:134`](../../Pommora/Pommora/Vaults/PageTypeManager.swift), [`ItemTypeManager.swift:169`](../../Pommora/Pommora/Items/ItemTypeManager.swift), [`AgendaTaskManager.swift:132`](../../Pommora/Pommora/Agenda/AgendaTaskManager.swift), [`AgendaEventManager.swift:132`](../../Pommora/Pommora/Agenda/AgendaEventManager.swift) — generalizes those single-file patterns across N files. Surfaces `RenameAtomicityError` (existing type at `Pommora/Pommora/AtomicIO/RenameAtomicityError.swift`) when the rollback itself fails.

#### Transaction primitive shape (consolidated reference)

The `SchemaTransaction` shape (per D2 + A.5) handles every multi-file schema mutation across all four `_schema.json` carriers. Key invariants:

- **Atomic commit boundary.** Either every cascade member's file picks up the new shape AND the schema sidecar reflects the new schema, or neither.
- **Phase 1 (pre-write).** Every payload writes to a `.tmp-<uuid>` sibling. Phase 1 failure leaves no `.tmp` orphans visible (cleanup in the `defer` block).
- **Phase 2 (rename).** Each `.tmp-<uuid>` atomic-renames in sequence into its final URL. Phase 2 failure mid-way: the SchemaTransaction's captured `originalSchema` + `originalMemberPayloads` (snapshotted from disk at commit start) re-renames the already-renamed files back, deletes remaining `.tmp` siblings, and re-writes the original schema sidecar.
- **Catastrophic-failure resilience.** If even the rollback rename fails, surface `RenameAtomicityError` (existing type, currently used in single-file rename paths). v0.3.3 SQLite adds launch-time `.tmp-<uuid>` orphan detection + divergence audit.
- **Same primitive across sides.** PageType, ItemType, AgendaTaskSchema, AgendaEventSchema all share `SchemaTransaction.SchemaSnapshot`'s four cases. No per-side variant.
- **No cross-Type batching.** Each SchemaTransaction touches exactly one Type's tree. A dual-relation create touches two transactions (source side + target side); the wizard wraps both in a parent `do { try src.commit(); try tgt.commit() } catch { try? src.rollback() }` pattern. The dual write is the only cross-Type case in v0.3.0.
