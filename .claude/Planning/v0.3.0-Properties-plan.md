Treat semantic anchors (function names, struct names, the section the citation lives in) as authoritative; re-verify line numbers with `grep` at dispatch time before any subagent edit.

> **Scope split (decided 2026-05-23 brainstorm).** v0.3.0 ships the **data layer + minimum-viable placeholder UI** to verify the data layer works end-to-end. The placeholder UI extends existing surfaces (`FrontmatterInspector`, `ItemWindow`'s `PropertyEditorRow`) to handle all 10 property types — no new design, disposable. The **real Properties Pulldown + Property Panel** is a fast-follow patch (alongside v0.3.0 or v0.3.1) — Figma-driven, host-agnostic SwiftUI component that slots into any inspector container. The **broader inspector architecture** (Claude chat as main-window inspector, PreviewWindow primitive for Page/Context Preview, Item Window redesign with pinned chips) ships in later v0.3.x patches whenever the user designs them. See spec § "Surface architecture" for the canonical reference.

#### Vision and scope

v0.3.0 closes the Items paradigm by making properties first-class across Pages, Items, AgendaTask, and AgendaEvent. The ParadigmV2 refactor (shipped Phases 2–8) split the operational layer into symmetric Pages-side (`PageType` / `PageCollection` / `PageContentManager`) and Items-side (`ItemType` / `ItemCollection` / `ItemContentManager`) containers, split Agenda into `AgendaTask` + `AgendaEvent` with per-side schemas, and stood up `SettingsManager` for renameable UI labels. The follow-on `flatlayout` refactor dropped the `Pages/` / `Items/` / `Agenda/` wrapper folders (all operational containers live at the nexus root) and split the unified `_schema.json` sidecar into six per-kind filenames: `_pagetype.json` / `_pagecollection.json` / `_itemtype.json` / `_itemcollection.json` / `_taskconfig.json` / `_eventconfig.json`. The Tasks singleton folder carries `_taskconfig.json`; the Events singleton folder carries `_eventconfig.json`. This plan layers Properties on top of that foundation without reshaping the operational topology — the data layer is already symmetric end-to-end.

The plan ships in five phases (A–E). Phase A is the schema editing infrastructure (model extensions + manager schema CRUD + the cross-file transaction primitive). Phase B wires property-value editing into the per-entity detail surfaces (Page inspector, Item Window, Agenda task/event forms). Phase C lands the per-type editors and side-aware pickers. Phase D delivers the move-strip rule (cross-Type moves drop schema-orphaned values). Phase E hardens validation, integration coverage, and ships the spec doc updates.

#### Decision log

Open questions the plan resolves before Phase A begins.

##### D1 — Does `ItemTypeSettingsSheet` (minimum-viable form) ship at v0.3.0?

**Choice: SHIP at v0.3.0, minimum-viable form.**

- **Rationale:** Items-side data layer is fully symmetric to Pages-side; both managers expose schema CRUD. Without a placeholder Items-side schema editor at v0.3.0, Item Types can only be edited by hand-editing `_itemtype.json`. The placeholder sheet (Edit Properties + Sort sections only) costs minimal incremental work because every primitive — NewPropertySheet, RelationPropertyWizard, type-specific config sub-views, option editors, SchemaTransaction — is kind-agnostic. Both Pages-side and Items-side schema editors share the same minimum-viable base.
- **What stays deferred:** Detail-pane property columns (both sides) and the full seven-section sheet redesign ship with the real Type Settings sheet UI in a later v0.3.x patch. v0.3.0 placeholder UI for Items-side property editing is the existing Item Window's `PropertyEditorRow` panel (extended to handle all 10 property types) + the placeholder Type Settings sheet (Edit Properties + Sort sections).

##### D2 — Transaction primitive shape

**Choice: Pre-write tmp siblings + batch rename, with reverse-snapshot rollback.**

- **Options considered:**
  - **(a) Codable forward-diff + reverse-diff journal.** Apply forward-diff incrementally; replay reverse-diff on failure. Sound, but heavyweight — diff machinery, replay correctness proofs, journal format. v0.3.0 doesn't ship the SQLite indexer that would consume the journal.
  - **(b) Pre-write tmp siblings + batch rename + snapshot-based rollback.** Phase A1: write every new file body to a `.tmp-<uuid>` sibling. Phase A2: atomic-rename each tmp into place in sequence. On mid-A2 failure: rename already-renamed files back from a captured `before` snapshot, delete remaining `.tmp` siblings, restore the schema sidecar.
  - **(c) Best-effort sequential writes + log-and-warn.** No rollback. Spec explicitly forbids: "must execute as a two-phase commit … complete rollback across every affected file."
- **Rationale:** (b) matches the spec's "two-phase commit" wording and lines up with the existing rename-atomicity pattern already shipped on every Pages-side and Items-side rename path (see [`PageTypeManager.swift`](../../Pommora/Pommora/Vaults/PageTypeManager.swift) lines 134–148, [`ItemTypeManager.swift`](../../Pommora/Pommora/Items/ItemTypeManager.swift) lines 169–181, [`AgendaTaskManager.swift`](../../Pommora/Pommora/Agenda/AgendaTaskManager.swift) lines 132–144, [`AgendaEventManager.swift`](../../Pommora/Pommora/Agenda/AgendaEventManager.swift) lines 132–144). It generalizes the existing per-method "save → rollback rename on save failure" pattern across N files. The diff machinery in (a) is post-v1 — v0.3.3 SQLite adds launch-time divergence detection that subsumes it.
- **Shape:** New struct `SchemaTransaction` under `Pommora/Pommora/AtomicIO/`. Carries `schemaWrites: [SchemaWrite]` + ordered `[MemberMutation]`. Single-`SchemaWrite` is the common case (every five-method mutation); two-element compound handles dual-relation creation / deletion atomically. `commit()` walks every write: phase 1 writes payloads to `.tmp-<uuid>` siblings; phase 2 atomic-renames in sequence. On mid-phase-2 failure: re-renames already-renamed files back from snapshotted pre-state and re-writes original payloads. `SchemaSnapshot` is a small enum: `.page(PageType)` / `.item(ItemType)` / `.agendaTask(AgendaTaskSchema)` / `.agendaEvent(AgendaEventSchema)` — four cases match the four per-kind sidecar carriers that hold property schemas. PageCollection / ItemCollection sidecars carry container metadata only (id, type_id, ordering), not property schemas. Full shape spec in A.5.

##### D3 — `PropertyConfigDraft` location

**Choice: Same file as `PropertyDefinition.swift`.**

- **Options considered:** (a) same file, (b) own file under `Vaults/` or a new `Properties/` directory.
- **Rationale:** Small struct family; tightly type-paired; existing `PropertyDefinition.SelectOption` / `NumberFormat` / `SelectColor` already nest there (see [`PropertyDefinition.swift`](../../Pommora/Pommora/Vaults/PropertyDefinition.swift) lines 18–37). Keeps schema-creation types co-located.

##### D4 — `defaultSort: SortDescriptor?` location

**Choice: Top-level optional field on `PageType` / `ItemType` / `AgendaTaskSchema` / `AgendaEventSchema`.**

- **Options considered:** (a) top-level, (b) `TypeDefaults` substruct, (c) inside a v0.6.0 `views[]` first stub.
- **Rationale:** Single optional vs. substruct: v0.3.0 needs exactly one persisted field (default sort); a substruct buys nothing until v0.6.0 piles on filter / group-by defaults. When v0.6.0 does, refactor in place. Avoids the `views[]` coupling that constraints sort to live inside a saved-view config it doesn't need.

##### D5 — RelationScope legacy decode shim

**Choice: Best-effort decode-only fallback; no preserved encode path.**

- **Options considered:** (a) decode-only shim that maps legacy `"same_vault"` / `"anywhere"` strings to `.pageType(<this PageType's id>)`, log a warning; (b) deferred — assume ParadigmV2's user-data migration covers it; (c) hard failure on legacy values.
- **Rationale:** The `NexusAdopter` migration (ParadigmV2's wrapper-layout adoption plus the flatlayout refactor's per-kind sidecar rename — `_vault.json` → `_pagetype.json` / `_collection.json` → `_pagecollection.json` / etc.) covers structural sidecar renames, but the legacy `relation_scope` string values inside the `properties[]` arrays may slip through if any v0.2-era Page Type had a Relation property. Audit shows no production data carries Relation properties yet (`PropertyEditorRow.swift:33` is a stub) — but decode-time defensiveness costs almost nothing. Encode never emits the legacy form. New encode shape is documented in the spec.

##### D6 — `panel_hidden_properties` field — DEFERRED from v0.3.0

**Choice: Don't ship the field at v0.3.0.**

- **Rationale:** Spec decision #18 was rewritten 2026-05-23 brainstorm. Lazy-properties UI (Properties Pulldown's populated-only + "+ Add property" model) auto-hides empty schema entries, which subsumes most of the per-entity hide use case. Explicit hide-when-populated may return as a follow-up if needed, but it's not load-bearing for the data layer or for the fast-follow Properties Pulldown / Panel patch. Ships only if the explicit-hide use case actually surfaces.
- **Impact on plan:** A.10 drops `panelHiddenProperties: [String]` from PageFrontmatter / Item / AgendaTask / AgendaEvent. A.9 drops the `panelHiddenProperties` parameter from `updatePageFrontmatter` / `updateItemProperties` manager methods.

##### D7 — `SchemaEditorRouter` — DEFERRED from v0.3.0

Cross-surface routing primitive depends on real Properties Pulldown / Panel having context-menu affordances that route into Type Settings. **Deferred to fast-follow patch.** v0.3.0 placeholder UI uses direct entry points only (Type detail-view toolbar gear button, Type-row right-click).

###### A.5 — `SchemaTransaction` primitive

New file: `Pommora/Pommora/AtomicIO/SchemaTransaction.swift`.

```swift
@MainActor
struct SchemaTransaction {
    /// One or more schema-side writes. Single-element is the common case
    /// (every five-method schema mutation). Two-element compound is the
    /// dual-relation create / delete, where two Types update atomically.
    let schemaWrites: [SchemaWrite]
    let memberMutations: [MemberMutation]   // ordered, may span Types

    struct SchemaWrite {
        let url: URL
        let original: SchemaSnapshot
        let new: SchemaSnapshot
    }

    enum SchemaSnapshot {
        case page(PageType)
        case item(ItemType)
        case agendaTask(AgendaTaskSchema)
        case agendaEvent(AgendaEventSchema)
    }

    enum MemberMutation {
        case page(URL, transform: @MainActor (PageFile) -> PageFile)
        case item(URL, transform: @MainActor (Item) -> Item)
        case agendaTask(URL, transform: @MainActor (AgendaTask) -> AgendaTask)
        case agendaEvent(URL, transform: @MainActor (AgendaEvent) -> AgendaEvent)
    }

    /// Two-phase commit: phase 1 writes every payload (schemas + members)
    /// to .tmp-<uuid> siblings; phase 2 atomic-renames in sequence. On
    /// phase-2 failure mid-way: re-renames the already-renamed files back
    /// from the snapshot capture, deletes remaining .tmp siblings, and
    /// re-writes original payloads. Rollback is internal to commit() — no
    /// separate rollback() entry point. Throws RenameAtomicityError on
    /// cascade failure.
    func commit() async throws
}
```

The same primitive serves PageType + ItemType + AgendaTaskSchema + AgendaEventSchema mutations. Compound mode (multiple `schemaWrites`) handles dual-relation creation/deletion: both sides' schema entries land atomically in one commit. The two-phase pattern generalizes the v0.2 single-file rename-atomicity rollback already in use across the per-side managers. Surfaces `RenameAtomicityError` (existing type at `Pommora/Pommora/AtomicIO/RenameAtomicityError.swift`) when the rollback itself fails.

**No public `rollback()` method** — every commit is self-contained. Callers that need "undo the schema mutation N seconds later" build a fresh inverse `SchemaTransaction` and commit it; there's no in-flight rollback state to expose.

###### A.7.5 — Load-path migrations (one shot per nexus, idempotent)

Three load-time interventions land together — all touch the `loadAll()` paths, all need to be idempotent so already-migrated nexuses no-op cheaply.

**1. AgendaTaskSchema Status injection + `type` removal.** In `AgendaTaskManager.loadAll()`, after decoding `_taskconfig.json` from disk, run a migration pass:

- If `properties` contains no `status`-typed entry → append the same default-seeded Status property from A.7. Set a dirty flag.
- If `properties` contains a `builtin: true` entry named `type` → remove it (it was the placeholder built-in superseded by Status). Set a dirty flag. User-added `type` properties (`builtin: false`) are left untouched.
- If dirty → write the migrated schema back via `AtomicJSON.write(...)`. Atomic single-file write; no SchemaTransaction needed (no member-file cascade — `type` and `status` carried no real value content in v0.2 / paradigmV2-era nexuses since no UI existed to set them).

A symmetric pass on `AgendaEventManager.loadAll()` is a no-op for Status (not auto-seeded on AgendaEvent per A.7) but covers any future event-side built-ins additively.

**2. RelationScope decode shim integration.** Per D5, custom `init(from:)` on `RelationScope` handles legacy bare strings (`"same_vault"` / `"anywhere"` → `.pageType(<this PageType's id>)` with a logged warning). The shim itself lives in A.2's struct definition; integration here means:

- The shim fires during `PageType.init(from:)` / `ItemType.init(from:)` decode automatically (Codable cascades through nested types).
- After the first `loadAll()` pass detects shimmed values (track via the same dirty flag pattern), re-encode the parent Type and write back to its per-kind sidecar. This locks the migration so future loads decode cleanly without the warning.
- Test coverage in A.12 exercises both decode paths (legacy + new) + the re-encode-and-write step.

**3. `.status` vs `.select` decode promotion.** Per A.4, `PropertyValue` decodes a bare string as `.select(String)` standalone — no schema context available at decode time. After `loadAll()` finishes decoding entity files, run `promoteStatusValues(_:against:)` per entity:

```swift
@MainActor
enum PropertyValuePromotion {
    /// Walks `entity.properties`; for every key whose parent-Type schema entry
    /// is `.status`, rewrites the value from `.select(s)` to `.status(s)`.
    /// In-memory only — no disk write. Promotion is re-run on every load
    /// because the source of truth is the schema's type field.
    static func promoteStatusValues<E: PropertyCarrier>(
        _ entity: inout E,
        against schemaProperties: [PropertyDefinition]
    )
}
```

Wire into `PageContentManager.loadAll(...)`, `ItemContentManager.loadAll(...)`, `AgendaTaskManager.loadAll()`, `AgendaEventManager.loadAll()` — right after the per-entity decode + before the in-memory cache write. `PropertyCarrier` is a tiny protocol exposing `var properties: [String: PropertyValue]` so the helper stays kind-agnostic. Promotion is read-side only — encode always emits the bare string (the schema's `type == .status` flag carries the disambiguation on disk, just as it does in memory).

**Test coverage** lands in A.12 — round-trip a `.status("done")` value through encode → decode-without-schema (resolves to `.select`) → `promoteStatusValues(...)` (resolves to `.status`) and assert equality.

###### A.8 — AgendaTask + AgendaEvent schema-mutation managers + kind-agnostic Status option helpers

**Status option helpers — generalized, not AgendaTask-only.** Status is EventKit-tailored but per-spec can be added manually to PageType, ItemType, and AgendaEvent (built-in only on AgendaTask). Status option mutations therefore live in a kind-agnostic helper layer that every schema CRUD path can call into:

New file: `Pommora/Pommora/Vaults/StatusOptionMutations.swift`.

```swift
@MainActor
enum StatusOptionMutations {
    /// Returns a new schema with the option appended to the named group.
    static func addOption(
        to property: PropertyDefinition,
        group: PropertyDefinition.StatusGroupID,
        option: PropertyDefinition.StatusOption
    ) -> PropertyDefinition

    /// Returns a new schema with the group's label renamed (structural ID preserved).
    static func renameGroupLabel(
        in property: PropertyDefinition,
        group: PropertyDefinition.StatusGroupID,
        newLabel: String
    ) -> PropertyDefinition

    /// Returns a new schema with the option's groupId rewritten. Value unchanged.
    /// Caller must follow up with a member-value resort if a saved view depends on group order.
    static func moveOption(
        in property: PropertyDefinition,
        optionValue: String,
        toGroup: PropertyDefinition.StatusGroupID
    ) -> PropertyDefinition

    /// Returns a new schema with the option removed. Caller wires the member-value
    /// void cascade through SchemaTransaction (`.status(s)` where `s == optionValue` → `.null`).
    static func deleteOption(
        from property: PropertyDefinition,
        optionValue: String
    ) -> PropertyDefinition
}
```

Per-manager wrappers (`PageTypeManager.addStatusOption(propertyName:, in pageType:, group:, option:)`, identical signatures on ItemTypeManager / AgendaTaskManager / AgendaEventManager) compose `StatusOptionMutations` with a `SchemaTransaction` to handle the member-value cascade. `addOption` / `renameGroupLabel` are schema-only (no member cascade). `moveOption` is schema-only file write but **data-semantic** (changes EventKit mapping for AgendaTask at v0.6.0; changes display color + sort everywhere). `deleteOption` voids referencing member values (`.null` for `.status`; array-shrink for `.multiSelect`).

#### Transaction primitive shape (consolidated reference)

The `SchemaTransaction` shape (per D2 + A.5) handles every multi-file schema mutation across all four per-kind sidecar carriers (`_pagetype.json` / `_itemtype.json` / `_taskconfig.json` / `_eventconfig.json`). Key invariants:

- **Atomic commit boundary.** Either every cascade member's file picks up the new shape AND the schema sidecar reflects the new schema, or neither.
- **Phase 1 (pre-write).** Every payload writes to a `.tmp-<uuid>` sibling. Phase 1 failure leaves no `.tmp` orphans visible (cleanup in the `defer` block).
- **Phase 2 (rename).** Each `.tmp-<uuid>` atomic-renames in sequence into its final URL. Phase 2 failure mid-way: the SchemaTransaction's captured `originalSchema` + `originalMemberPayloads` (snapshotted from disk at commit start) re-renames the already-renamed files back, deletes remaining `.tmp` siblings, and re-writes the original schema sidecar.
- **Catastrophic-failure resilience.** If even the rollback rename fails, surface `RenameAtomicityError` (existing type, currently used in single-file rename paths). v0.3.3 SQLite adds launch-time `.tmp-<uuid>` orphan detection + divergence audit.
- **Same primitive across sides.** PageType, ItemType, AgendaTaskSchema, AgendaEventSchema all share `SchemaTransaction.SchemaSnapshot`'s four cases. No per-side variant.
- **Compound mode for dual relations.** Single-`SchemaWrite` is the common case (every five-method mutation). Two-element compound handles dual-relation create / delete: both Types' per-kind sidecars update inside one phase-2 rename batch, so partial-state never exists on disk. The dual write is the only cross-Type case in v0.3.0 and falls out naturally from the `schemaWrites: [SchemaWrite]` shape — no orchestration wrapper, no `try? src.rollback()` dance.

#### Spec drift reconciliations flagged

During Step 1's re-read of [`v0.3.0-Properties-spec.md`](v0.3.0-Properties-spec.md) against the post-ParadigmV2 code surface, the following items needed reconciliation in this plan (none requires editing the spec — these are notes on how the spec maps onto the actual code shape):

1. **AgendaTaskSchema's `Property` struct is its own thing, not `PropertyDefinition`.** [`AgendaTaskSchema.swift`](../../Pommora/Pommora/Agenda/AgendaTaskSchema.swift) lines 15–26 carry an inline `Property` struct distinct from `PropertyDefinition` (just name/type/options/builtin/default). Same on [`AgendaEventSchema.swift`](../../Pommora/Pommora/Agenda/AgendaEventSchema.swift). Phase A.7 extends both to include `statusGroups`, `dualProperty`, `allowsMultiple`, `icon`, `relationScope` so they reach feature parity with `PropertyDefinition`. Alternative considered: drop the inline struct and use `PropertyDefinition` directly. Rejected for v0.3.0 — the existing shape has the `builtin: Bool` field that `PropertyDefinition` doesn't carry, and the two AgendaSchema structs have shipped with this shape since ParadigmV2. Phase A.7 reaches parity by extending fields, not by replacing the struct. Long-term unification is a post-v1 cleanup.

2. **`ItemValidator` is currently stubbed.** [`ItemContentManager+CRUD.swift:8–12`](../../Pommora/Pommora/Items/ItemContentManager+CRUD.swift) explicitly notes "Property validation against the Item Type's schema lands in Phase 6 once `ItemValidator` is rewired off `PageType`." Phase 6 of ParadigmV2 shipped but `ItemValidator` is still title-uniqueness only — Phase E.2 of this plan is when the property-validation rewire actually happens. The transitional `ItemCRUDError` enum at lines 276–288 stays in place during Phase A and is replaced in Phase E.

3. **`SettingsManager` ships but no UI label-site is wired through it yet.** [`SettingsManager.swift:1–68`](../../Pommora/Pommora/Settings/SettingsManager.swift) and [`SettingsLabels.swift`](../../Pommora/Pommora/Settings/SettingsLabels.swift) both ship, but the manager's call-site doc says "Consumed by every UI label-rendering site … once Phase 7 Wave 2 wires the readers — Task 7.2 ships the type definition only." Phase 7 Wave 2 didn't ship per the working state. Phase C.3 + C.7 of this plan are the first call sites that actually read `SettingsManager.labels.X` — the routing pattern lands here.

4. **`ContentManager` reference in the archived plan is now `PageContentManager`.** The archived plan refers to a unified `ContentManager`; the post-ParadigmV2 shape is the split `PageContentManager` (Pages) + `ItemContentManager` (Items). Plan uses the new names throughout.

5. **Sidecar filenames are now per-kind (post-flatlayout).** The archived plan's `_collection.json` references — and ParadigmV2's intermediate unified `_schema.json` — are both superseded by the flatlayout refactor's six per-kind sidecars: `_pagetype.json` / `_pagecollection.json` / `_itemtype.json` / `_itemcollection.json` / `_taskconfig.json` / `_eventconfig.json`. Plan uses the per-kind names throughout. The `NexusAdopter` covers legacy v0.2 + paradigmV2 wrapper input shapes and renames sidecars in place; no Properties-side concern.

6. **`Pommora.Collection` qualification retired.** Per CLAUDE.md quirk #6 (retired 2026-05-22), every `Pommora.Collection` reference in the archived plan is now bare `PageCollection` or `ItemCollection`. Plan follows the new naming.
