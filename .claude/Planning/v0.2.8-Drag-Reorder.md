### Drag-to-Reorder — Sidebar + Detail Pane

#### Context

Today every list in Pommora — the sidebar's Spaces/Topics/Vaults/Subtopics/Collections/Pages, and the detail-pane Table of a Vault or Collection — is sorted by `localizedStandardCompare` on title at every load and CRUD site. There is no drag-and-drop infrastructure (zero `.draggable`/`.dropDestination`/`.onMove`/`Transferable` hits in the codebase).

We want the user to reorder rows by drag with Finder-style feel: the dragged row lifts as a translucent chip, neighbors animate out of the way to open a gap, the array commits on release, the gesture cancels cleanly on ESC. **No system blue insertion line** — which means the surface cannot remain `List` (the blue line is owned by `List` + `.onMove`/AppKit NSTableView).

Order persists across launches via sidecar JSON — locational memory survives a stack rebuild and a cross-nexus sync, consistent with the "files are canonical" principle.

User-confirmed v1 scope:
- Type-segregated ordering inside each container (Pages reorder among Pages, Items among Items, Collections among Collections).
- Reorder-only — cross-container drags (Page → different Vault, Subtopic → different Topic) stay in the right-click menu.
- Phased ships under the stub-and-progressively-replace paradigm.

#### Architecture

##### Order persistence (hybrid)

Top-level sibling order lives in `.nexus/state.json` (the same file that already drives Pinned reorder via PinnedManager). Per-container child order lives on each container's sidecar.

| Sidecar | New optional fields |
|---|---|
| `.nexus/state.json` (NexusState) | `spaceOrder`, `topicOrder`, `vaultOrder` — each `[String]?` |
| `_vault.json` (Vault) | `collectionOrder`, `pageOrder`, `itemOrder` — each `[String]?` |
| `_collection.json` (Collection) | `pageOrder`, `itemOrder` — each `[String]?` |
| `_topic.json` (Topic) | `subtopicOrder: [String]?` |

All fields optional + encode-if-present — existing sidecars decode unchanged, fields write only after the first user drag in that container. No migration.

##### Resolver semantics

A new `OrderResolver` utility replaces every `.sorted { localizedStandardCompare }` call site:

```
resolve(ids, persistedOrder) →
  if persistedOrder == nil: return ids.sorted(by: title localizedStandardCompare)
  known    = persistedOrder.filter { ids contains it }     // honour user order, drop tombstones
  appended = (ids - known).sorted(by: title)               // new arrivals append at end
  return known + appended
```

Behavior consequences:
- Pre-drag state — resolver returns alphabetic order (current behavior preserved exactly).
- First drag in a container — captures resolver's current output, applies the user's move, writes the full array to the sidecar.
- Files added externally — appended at the end (user's curation is never disturbed silently).
- Deleted files — tombstones filtered on resolve, next write removes them from disk.

##### Cross-surface ordering (sidebar ↔ detail pane)

**Unified.** A Vault's `collectionOrder` is read by both the sidebar's `VaultRow` disclosure and the `VaultDetailView` Table. Both surfaces become read-only consumers of the same manager arrays.

##### Drag mechanics — two-path strategy

The blue insertion line is specifically a `List` + `.onMove` artifact. Never calling `.onMove` is what removes it. Two-path approach with Path A as the strong default:

**Path A — keep `List`, add `.draggable`/`.dropDestination` per row, never call `.onMove` (DEFAULT).**

- Preserves 100% of current sidebar visual chrome by construction: `.listStyle(.sidebar)`, `DisclosureGroup` chevron + automatic child indentation, `.listRowBackground(SelectionChrome(...))`, sidebar focus/key-window semantics, native section spacing — everything stays as-is.
- Each row gains `.draggable { payload } preview: { SidebarDragPreview(...) }` + `.dropDestination(for: SidebarDragPayload.self) { drops, location in ... } isTargeted: { hovering in ... }`.
- A single shared Transferable: `SidebarDragPayload { kind, id, containerID?, nexusID }`. UTType `com.pommora.sidebar-drag`, custom JSON representation; in-app only. `nexusID` gate rejects cross-window drops.
- Custom drag preview = small translucent rounded chip with the row's icon + title (not the default white card).

**Drop visualization in Path A — keep it modest, defer animation ambition:**
- On drop release inside a valid target row, the array commits via `withAnimation(.snappy)` and rows animate to their new positions through List's built-in row diffing.
- During hover, the target row gets a subtle accent border (1pt stroke, accent at 40% opacity) — no live "gap opens" reflow during hover, because `List`'s NSTableView backing does not animate row reflow on array mutation the way a `LazyVStack` would. Live mid-drag reflow is **deferred**; release-time animated commit is what ships.
- Trade-off acknowledged: the Notion-style "neighbors slide live as you drag" effect is the cost of preserving visual fidelity. The Finder feel still applies on commit — the row settles into place with a spring.

**Path C — accept the blue line, use `List.onMove` (LIGHTWEIGHT FALLBACK, user-approved 2026-05-21).**

Nathan has explicitly accepted that the standard SwiftUI/AppKit blue insertion line is acceptable as a failsafe — "it's a nitpicky visual thing on my end." Path C:

- Keeps `List` + `.listStyle(.sidebar)` + every visual primitive exactly as today (same fidelity guarantee as Path A).
- Uses native `List.onMove(perform:)` for reorder — gets the standard blue insertion line and native drag preview for free.
- Loses: custom drag chip (`SidebarDragPreview`), the explicit cross-row drop validation matrix (`onMove` is restricted to siblings in the same `ForEach` by construction — which is what v1 wants anyway).
- Wins: dramatically simpler implementation, zero risk to visual chrome, well-trodden Apple-sanctioned path. Almost certainly handles ESC-cancel + cursor + auto-scroll natively.
- The `DragValidator` matrix becomes implicit: `onMove` only fires within a single `ForEach`, so cross-container drops are impossible by construction.

**Path B — LazyVStack rebuild (LAST RESORT).**

Reserved for the case where Path A fails AND Nathan later decides the blue line in Path C is also unacceptable. Requires reconstructing the visual primitives by hand. Documented in the Visual Fidelity Contract's Path B section below.

**Fallback decision tree (in order of preference):**

1. **Path A** (custom chip, no blue line) — try first. Ships if no system indicator appears inside `List`.
2. **Path C** (native `onMove`, blue line acceptable) — fall back if Path A fails. Lightweight, low-risk.
3. **Path B** (LazyVStack rebuild) — invoke only if both A and C are unacceptable. High effort, high risk.

**Common to all three paths:**
- Drop fires `commitMove()` → manager's `reorder(...)` → `OrderPersister.set...Order(...)` → atomic write to sidecar.
- ESC or out-of-bounds release → no write, no array mutation.
- `SidebarDragPayload` Transferable, UTType, drag preview, validation matrix, manager reorder methods, persistence, OrderResolver — all unchanged across paths.
