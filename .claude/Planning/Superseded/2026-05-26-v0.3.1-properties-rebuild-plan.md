#### Post-ship polish (working tree, uncommitted, unverified by Nathan)

After the 21-commit build, a focused polish pass landed in the working tree fixing the gesture/interaction-layer defects Nathan reported ("properties don't work as intended; error message at the bottom; type picker should be a list; visibility doesn't work via interaction"). All compile-clean; none verified by Nathan.

- **Created-property route fix.** `PropertyTypePickerPane` mints the ULID in the caller via `ReservedPropertyID.mintUserPropertyID()` before `addProperty`, so the route carries a real ID instead of `""` — the old "Property not found" landing after creating a Select/Multi/Status property.
- **Snapshot → live binding.** `PropertiesListPane.resolvedProperties()`, `PropertyVisibilityPane`'s property source, and `EditPropertyPane`'s definition lookup now read live from the manager via stable IDs for all four scopes, instead of a popover-open snapshot — fixes delete-leaves-stale-row, add/rename/duplicate-not-showing in PageType/ItemType scopes.
- **Type picker → list.** `PropertyTypePicker` is a vertical `VStack`/`ForEach` (icon + name per row), no longer a `LazyVGrid` gallery.
- **Friendly errors.** New `PropertyEditorErrorMessage.string(for:)` maps `ValidationError` / manager errors to sentences; `commitError` clears on every fresh tap.
- **Property Visibility.** Reserved IDs filtered out (only user properties + `_modified_at` appear); drag-reorder via `.draggable` / `.dropDestination` on the visible section.
- **Pill/typography polish.** `StorageMenuRoot` + `EditPropertyPane` icon/title pill backgrounds; `PUI.Icon.header` → `.title3`/28pt; `PUI.Typography.sectionHeader` → `.headline`; SymbolPicker constrained to 540×460; `PropertyChipColor.selectablePalette` reordered warm-top/cool-bottom + `init(selectColor:)` added.

#### Design pivot — option editing is a double-click popover, not chevron-push

The original plan specced each Select/Multi/Status option row as `chip + chevron`, the chevron pushing `.editOption` onto the NavigationStack into `EditOptionPane`. Per Nathan's 2026-05-26 direction ("labels shouldn't require a navigation hop"), the working tree pivoted to: **chip-only rows; double-click opens an inline `OptionEditPopover`** (rename + 5×2 color grid + Delete); right-click → Delete; drag-to-reorder via `.draggable` / `.dropDestination` (Status supports cross-group drag).

Consequence: `OptionEditPopover` (new file) is the live edit surface; `EditOptionPane` and the `.editOption` route are now superseded dead code — no caller pushes the route. Removing them is a cleanup item, not a feature gap.

#### Locked UIX decisions

Carried from Nathan's prior session + this work. Cosmetic specifics (pill radii, swatch sizes) are current choices, not locked.

1. Chip-only option rows; double-click → `OptionEditPopover` for rename + recolor + delete (supersedes the chip+chevron design).
2. Drag-only reordering of options (no Sort picker); schema order is the single source of truth. Status supports cross-group drag, which moves the option's group and triggers a cascade-confirmation dialog listing affected entities.
3. 5×2 swatch grid of 10 selectable colors; tapping the selected swatch toggles back to no-color.
4. EditPropertyPane structure: combined icon+title row → Type row → per-type middle section → Duplicate + Delete footer. URL / File panes have no middle section (rename-only).
5. Date / DateTime use the 6-format DateFormatPicker for Display As; Status uses DisplayVariant `.box` / `.select` / `.chip`.
6. Chip rendering (cell display) is Status / Select / MultiSelect / Relation / File only; Dates / Links / Numbers / Checkboxes / LastEditedTime render as text or native controls.
7. `_modified_at` always visible and not draggable in the Visibility pane; reserved IDs (`_id`, `_status`, `_tier1/2/3`, `_wikilinks`, …) never appear there.
8. PropertyChipColor is a flat 12-case palette (no tiers). All 4 storage scopes surface identical popover content; the sidebar sheets stay working via the shared editors.
9. Type picker is a vertical list (icon + name, no description blurb), not a gallery.
10. Popover errors use friendly sentences, not raw enum descriptions, and clear on fresh input.
11. No drag handles anywhere — drag is initiated by direct input on the row body; the row visual stays clean. Applies to option rows and the Visibility pane.
12. Every popover-side surface reads state live from the manager via stable IDs (type ID, property ID, option value), never from a captured `ViewSettingsScope` payload. Candidate for `Guidelines/CRUD-Patterns.md` once the polish commits.
