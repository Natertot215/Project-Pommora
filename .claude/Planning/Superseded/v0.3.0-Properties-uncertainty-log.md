### v0.3.0 Properties — Uncertainty Log

> Companion to `v0.3.0-Properties-implementation.md`. Captures edge cases, Swift-language risks, SwiftUI patterns, open design questions. Authored RC-2026-05-19 from: (1) Explore agent inventory of Pommora Swift code, (2) Context7 SwiftUI docs, (3) Notion API docs.

#### Top 5 blockers (ranked)

##### 1. PropertyValue `.status` vs `.select` decode collision — **CRITICAL**

`PropertyValue.swift:33-88` decodes shape-based: bare string → `.select(String)` after URL/datetime/date probing. Status encodes as bare string too. Standalone decode can't distinguish without schema context.

**Failure mode:** `PageFile.load()` decodes `status: "in_progress"` as `.select("in_progress")`. PropertyEditorRow.`.status` case gets wrong enum → crash, silent wrong rendering, or `.select` fallback.

**Resolution:** Add `ContentManager.promoteStatusValues(in:, vault:)` post-load helper — walks entity's `properties`, rewrites `.select(s)` → `.status(s)` per schema. Called after `PageFile.load()` / `Item.load()` / `AgendaItem.load()`. Alternative `decoder.userInfo[.propertyDefinition]` rejected — breaks file-can-decode-without-schema legibility.

**Test:** Round-trip `.status("done")` → encode → decode-without-schema (`.select`) → promote-with-schema (`.status`).

##### 2. RelationScope migration — **CRITICAL**

`PropertyDefinition.swift:34-37`: `enum RelationScope: String, Codable, CaseIterable { case sameVault = "same_vault"; case anywhere }` → associated-value enum `vault(String)` / `collection(String)` / `contextTier(Int)`.

**Breaking changes:**
- `CaseIterable` drops (Swift requirement for associated-value enums)
- Disk format: `"relation_scope": "same_vault"` → `"relation_scope": {"kind": "vault", "vault_id": "01H..."}`
- `.allCases` use in picker UIs silently fails to compile

**Resolution:** Custom `init(from:)` accepts BOTH shapes. Old `"same_vault"` → `.vault(<this vault's id>)`; `"anywhere"` → `.vault(<this vault's id>)` with logged warning. New encode emits only new form. Decode-time; no separate pass. Drop `CaseIterable`; hard-code picker list (only 3 kinds).

**Test:** `RelationScopeTests.swift` — old `"same_vault"` → `.vault(parentVaultId)`; new form decodes correctly; new encode only; round-trip identity preserved.

**Mitigation cushion:** Current v0.2 vaults all have empty `properties: []`. Zero real-world exposure; test coverage matters only for forward correctness.

##### 3. Multi-file atomic-write recovery — **HIGH**

`AtomicJSON.swift:24-27` and `AtomicYAMLMarkdown.swift:46-53` use `Data.write(to:, options: [.atomic])` (single-file POSIX rename). v0.3.0 introduces multi-file ops:
- **Property rename**: N Pages + M Items = N+M+1 writes
- **Dual relation creation**: 2 `_vault.json` writes
- **Schema delete cascade**: N+M+1 writes
- **Cross-Vault move-strip**: 1 read + 1 delete + 1 create

No transaction journal; no `.tmp-*` cleanup at launch.

**Failure modes:**
- Crash mid-writes → partial state (schema updated, members not, or vice versa)
- Orphan `.tmp-<uuid>` accumulating in nexus
- Dual-relation half-writes → Vault A points to nonexistent reverse on Vault B

**Resolution (three parts):**

1. **`SchemaTransaction.swift`** (new under `AtomicIO/`): two-phase commit. Phase A: write to `.tmp-<uuid>` siblings. Phase B: atomically rename each. On Phase B failure: delete remaining `.tmp` files; already-renamed stay (best-effort rollback — true rollback needs backups of overwritten content).

2. **`<nexus>/.nexus/_transaction-log.jsonl`** (new): append-only log. Before commit, append `BEGIN` with paths + SHA256s. After all renames, append `COMMIT`. Launch scans un-committed `BEGIN` records → complete (if `.tmp` matches SHA256) or roll back (delete `.tmp`).

3. **Startup orphan cleanup** in `NexusStore.initialize()`: delete `.tmp-*` files older than X minutes.

**Tradeoff:** Full ACID not achievable in userspace POSIX without SQLite-as-source-of-truth + fsync. Best-effort recovery sufficient for personal scale; v0.3.3 SQLite adds divergence detection.

**Test:** Insert `try Task.checkCancellation()` between phases A and B; assert orphan cleanup runs on next launch; assert no torn renames.

##### 4. Swift 6 strict concurrency + Sendable for new types — **MEDIUM-HIGH**

New types (`StatusGroup`, `StatusOption`, `DualPropertyConfig`, `ContentTemplate`, `SchemaTransaction`) need `Sendable`. Stored properties are Strings, Optionals, primitives, enums — all Sendable. **Auto-synthesis should succeed.**

**Risk surface:**
- `PropertyDefinition` gains `statusGroups`, `dualProperty`, `icon`, `allowsMultiple`. Hashable + Equatable auto-synth depends on nested-type conformance. Confirmed conforming.
- New manager methods follow existing `@MainActor @escaping () -> NexusContext` pattern at `ContentManager.swift:19-46`.
- Closure capture of Vault state: views capturing Vault (e.g., "Manage options…" callback) may observe torn reads on concurrent mutation. **Mitigation:** snapshot Vault at closure-creation, not execution.

**Validate:** add fields + `xcodebuild` (trust it, not SourceKit, per quirk #3) — auto-synth either clean or concrete error.

#### SwiftUI patterns confirmed (Context7 + Apple docs)

##### Dynamic Table columns — `TableColumnForEach` (macOS 14+) — RESOLVED

The biggest UI risk has a first-party answer:

```swift
Table(rows, sortOrder: $sortOrder) {
    TableColumn("Title", value: \.title)
    TableColumn("Kind", value: \.kind)
    TableColumn("Modified", value: \.modifiedAt)
    TableColumnForEach(vault.properties) { property in
        TableColumn(property.name, value: \.value(for: property.name)) { row in
            PropertyCellView(definition: property, value: row.value(for: property.name))
        }
        .width(ideal: 120)
    }
}
.tableColumnCustomization($columnCustomization)
```

`TableColumnForEach` (`developer.apple.com/documentation/swiftui/tablecolumnforeach`) generates dynamic columns from `Identifiable`. `PropertyDefinition.id` is `String { name }` — works directly.

##### Built-in column customization — `TableColumnCustomization` — RESOLVED

Property Visibility gets implemented for free. `columnCustomization: Binding<TableColumnCustomization<Value>>` persists per-column visibility + ordering. **User gets drag-to-reorder headers AND show/hide for free.**

Recommendation: use `TableColumnCustomization` directly; persist serialized state to `_vault.json.column_customization` (new field).

##### Sortable Table — `KeyPathComparator` + `sortOrder` binding — RESOLVED

```swift
@State var sortOrder = [KeyPathComparator(\Row.modifiedAt, order: .reverse)]

Table(rows, sortOrder: $sortOrder) { ... }
    .onChange(of: sortOrder) { _, new in
        rows.sort(using: new)
    }
```

For Status's group-then-option sort, custom `SortComparator` resolves `value` → `groupId` (via schema), compares group position first, option index second. Save default to `Vault.default_sort: SortDescriptor?` (new optional field).

##### Drag-and-drop between Sections — `.draggable` + `.dropDestination` — RESOLVED

For Status's drag-between-groups (Phase 2.12):

```swift
ForEach(statusGroups) { group in
    Section(group.label) {
        ForEach(group.options) { option in
            OptionRow(option)
                .draggable(option.transferableID)
        }
    }
    .dropDestination(for: String.self) { droppedValues, _ in
        for value in droppedValues {
            moveOption(value: value, toGroup: group.id)
        }
    }
}
```

`Transferable` via raw String (option's `value`). Drop fires the dual-effect confirmation before commit.

##### Grouped Picker / Menu popover — supported via `Section` in `Picker`/`Menu` body

Status picker (Phase 2.11) is standard SwiftUI:

```swift
Menu {
    ForEach(statusGroups) { group in
        Section(group.label) {
            ForEach(group.options) { option in
                Button(option.label) { value = .status(option.value) }
            }
        }
    }
    Divider()
    Button("Manage options…") { router.openVaultSettings(at: .property(definition.name)) }
} label: { statusPill }
```

No special API needed; works on macOS 14+.

#### Open design questions — RESOLVED + REMAINING

##### Q1 — Vault Settings sheet: live writes or save/cancel? ✅ RESOLVED (RC-2026-05-19)

**Resolution: Save-required.** Edits stage into a draft; commit only on sheet-level Save. Cancel discards. SchemaTransaction wraps the flush atomically. Inline "Add option…" popovers (right-click on property label) are separate commit boundaries with their own Save — do NOT batch with VaultSettingsSheet.

##### Q2 — Concurrent Vault Settings editing (two windows of same Vault) ✅ RESOLVED

**Resolution: Disabled.** Second invocation shows disabled state ("Settings already open in <window title>") via `@MainActor` singleton `VaultSettingsRegistry` tracking by Vault ID.

##### Q3 — Last Edited Time source: file mtime or frontmatter `modified_at`? ✅ RESOLVED

**Resolution: Frontmatter `modified_at`, updates on any content/frontmatter edit, NOT view-only.** Opening never updates. Body/property/title/icon/tier1-3/panel-visibility-toggle all trigger save → `modified_at` updates. External edits update file mtime but NOT frontmatter `modified_at` until v0.3.3 watcher closes the gap.

##### Q4 — Status option deletion with referenced entities ✅ RESOLVED

**Resolution: void (cascade to `.null`) — UNIVERSAL rule.** Deleting voids referencing values. Multi-select: deleted `value` is removed from each array (shrinks; `.null` only if empty). Confirmation lists count before commit. No quarantine, no auto-reassignment, no refuse-to-delete — matches Notion. Implementation: `VaultManager.deleteSelectOption(...)` / `deleteStatusOption(...)` / `deleteMultiSelectOption(...)` cascade through SchemaTransaction.

##### Q5 — Property visibility vs sort interaction ✅ RESOLVED

**Resolution: Auto-show.** Sort referencing a hidden property auto-shows the column. Sort precedence beats visibility (no direction indicators without header). Single-write removes from `hidden_properties`.

##### Q6 — Inline option creation routing ✅ RESOLVED (refined)

**Resolution: structured inline-add via right-click on property LABEL (not value picker).** Right-click on column header / inspector row label / Item Window row label → "Add option…" (small popover: Name + Color + Group-for-Status + Save/Cancel) commits immediately on own Save. Value pickers never accept typed inline-add. Value-context right-click shows "Edit options…" only. Picker popover bottom keeps "Manage options…" link. Three commit paths:

1. **Vault Settings → Edit Properties → "+ Add option"** — batched with sheet Save (canonical)
2. **Right-click property LABEL → "Add option…"** — small popover with own Save
3. **No path from value picker** — read-only-of-options

##### Q7 — Content template wiring in v0.3.0 ✅ RESOLVED

**Resolution: UI anchor in Vault Settings (7th "Templates" section, placeholder); no manager-method parameter.** Placeholder ContentUnavailableView ("Pre-configured layouts for new Pages and Items in this Vault. Coming post-v1."). **No `template: ContentTemplate? = nil` parameter added to `ContentManager.createPage(...)`** — lands when templates ship. Preserves wiring concept (Vault Settings IS the template home) without dead code. ContentTemplate Codable sketch + `<nexus>/.nexus/templates/` reserved.

##### Q-Status — Is Status really distinct from Select? ✅ RESOLVED

**Resolution: distinct type.** (a) 3 fixed structural groups with load-bearing IDs driving EventKit mapping at v0.7.0; (b) sort behavior differs (group-position-first vs option-order-only); (c) 3-group constraint can't be expressed as "Select with defaults" without losing EventKit guarantee. Status appears as standard option in type picker — not specially recommended. Auto-seeds with 3 groups + one starter option per group.

##### Q-Disclosure — Per-entity property panel visibility (NEW, RC-2026-05-19) ✅ RESOLVED

**Resolution: wiring at v0.3.0; UI deferred.** Each Page/Item/Agenda item carries `panel_hidden_properties: [String]` (default empty). UI affordances (right-click "Hide on this entity" + footer "+ Add property") ship later; data + manager-method wiring lands at v0.3.0 so future UI is additive. Distinct from `Vault.hidden_properties` (Table column visibility, Vault-wide).

##### Q8 — `EKEvent.status = .canceled` mapping

3 groups cover most EventKit states, but `.canceled` has no Pommora equivalent. Options: (a) map cancelled → `.done`; (b) filter cancelled out of Agenda; (c) add 4th `cancelled` group (breaks "3 slots structural").

**Recommendation:** (a) for v0.7.0; document in EventKit mapping table. **Defers to v0.7.0 prep; not v0.3.0 blocker.**

#### Edge cases by feature area

##### Schema mutation during member edit

Page A's inspector shows `status: "in progress"`; another window deletes the option.

**Behavior:** Inspector refreshes on next manager observation; value silently `.null`. Display shows "—" or "Invalid value: in_progress" briefly.

**Risk:** mid-edit on a different field in Page A — schema-driven refresh may discard in-progress edits. **Mitigation:** snapshot form state separately; reconcile on focus loss.

##### Pre-SQLite (v0.3.0) relation picker performance at scale

Naive scans for `VaultMemberPicker` / `CollectionMemberPicker` / `ContextTierPicker`. ~50 Topics / ~200 Pages → sub-100ms. 1000+ entities → noticeable lag.

**Mitigation v0.3.0:** lightweight `@MainActor` LRU cache keyed by `(scope, queryString)`; clear on schema/content change. v0.3.3 SQLite retires the cache.

##### Page open during Status group label rename

Page open with "Upcoming" Status pill; user renames group to "Backlog" via Vault Settings. Pill should update on next render.

**Implementation:** Status pill resolves `option.value` → `option` → `option.group` → `group.label` at render time (uncached). Rename triggers `@Observable` propagation via VaultManager. **Verify:** PropertyEditorRow + StatusPickerEditor observe Vault changes (not just PropertyDefinition).

##### Cross-Vault move with paired relations

Page A in Vault X has paired Relation "Sources" → Vault Y. User moves A to Vault Z (no "Sources").

**Behavior:**
1. Move-strip dialog warns "Sources" will be stripped
2. On confirm: A's `properties.Sources` removed
3. B-side "Cited By" on Vault Y still has A's ULID → orphaned reference
4. Dual-relation maintenance cleans up B-side: remove A's ULID from each "Cited By" holder

**Required helper:** `cleanupOrphanedDualReferences(sourceEntityId:, sourcePropertyName:, in vault:)` called from move-strip flow.

##### Renaming property to reserved name

"status" → "id" rejected. UX: TextField red; tooltip "Reserved property name."

**Reserved:** `id`, `created_at`, `modified_at`, `tier1/2/3`, `wikilinks`. Plus uniqueness within Vault (case-insensitive).

##### Cancel during Phase 2 of two-phase commit

Cancel mid-commit (after Phase A `.tmp-*` writes, before Phase B renames): delete all `.tmp-*`; no member files changed; transaction log emits ROLLBACK. Same path as failure recovery.

##### Empty Status options list

User deletes the only option in a group. **UX:** allowed (don't force ≥1). Picker renders "Empty" placeholder for that group. Validator doesn't reject (no entity value points to nonexistent option).

#### Migration considerations

##### v0.2 → v0.3.0 migration items

1. **RelationScope string → discriminated object** (decode-time shim)
2. **AgendaSchema Status injection** — legacy `_agenda.json` auto-adds via `AgendaSchema.migrate(_:)` on load
3. **`.tmp-*` orphan cleanup** at first launch
4. **`_transaction-log.jsonl` replay** at launch (handles v0.3.0 → v0.3.x crashes)
5. **`<nexus>/.nexus/templates/`** created on first nexus init (empty; reserved)

##### v0.3.0 doesn't break v0.2.x:

- Pages/Items/Agenda items have `properties: {}` (mostly empty) — nothing to migrate
- Vaults have `properties: []` — empty schemas, no migration
- New Vault fields decode as nil/empty via `decodeIfPresent`

Considered and rejected:

1. **Vault-level templates** — replaced by content-level reservation
2. **Inline option creation** — schema editor only
3. **`anywhere` Relation scope** — every Relation has definite target
4. **Optional dual relations (toggle)** — Vault/Collection scopes MANDATORY dual
5. **Stable opaque property IDs** — name-keyed preserved for agent legibility
6. **Filter / Group By / Layout** — placeholders only; v0.6.0
7. **Multi-criterion sort** — v0.6.0
