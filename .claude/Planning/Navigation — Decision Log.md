### Frame

- **Purpose:** Settle Pommora's navigation system — how a user moves between Contexts (Areas/Topics/Projects), Collections/Sets, and Pages, and what wayfinding surfaces carry that load beyond the sidebar tree.
- **Core Value:** A user can always get from "where I am" to "where I want to be" — including cross-tree jumps (recent, pinned, searched) that the sidebar tree alone can't serve — without losing their place.
- **Success Criteria:** (1) the shared nav-state layer records every navigation into a persisted, LRU-capped recents stream, supports temp-pinning to the top and durable favorites, and resolves every entry rename/move/delete-safe from the live tree; (2) client-side title search spans the four tree kinds + Agenda; (3) the ribbon Navigation icon opens an in-app overlay surfacing search + Favorites + the history gallery (temp-pins on top), with cards showing location + icon + title; (4) tree entities route the main pane, Agenda entities route a placeholder preview window; (5) the layer is UI-agnostic — reusable by a future tab bar or toolbar without change.

### The Decomposition (Nathan's call — the surface is the invariant)

This ask bundles **two separable subsystems**, and the wayfinding surface is *independent of* the model, so it goes first:

1. **The Search/Navigation Window** (BUILD FIRST — the certain thing) — a ribbon-launched popup holding search + Recents + Pinned + a browsable tree. Paradigm-independent: tabs-or-no-tabs, the same window serves it. This is the doc's Pending "Navigation Popover," scoped up.
2. **Navigation MODEL** (SEPARATE, LATER cycle) — single-pane-replace (today) vs top-bar tabs vs split panes. Its own brainstorm; does not gate #1.

Nathan (mid-brainstorm): "lock in the one thing I DO know I want... a search or navigation window that opens when you press the ribbon button. If we do multi-tab topbar, or not, both would use a search pane that can show recents, pinned, a sidebar, all in a mini popup window."

#### E — The Plumbing (shared navigation-state layer — UI-agnostic, built regardless of the UI)

The overlay, the ribbon, the toolbar, and any future tab bar all read the SAME state layer. It's store slices + IPC + a sidecar per concern — no coupling to the overlay component. Four concerns:

- **E-1 · Recents (auto history):** [confirmed direction] Store slice `recents: RecentEntry[]`, **`RecentEntry = SelectTarget & { pinned? }`** — mirrors `SelectTarget` (so the id-less homepage is representable, R3-obs), MRU by array-order (no timestamp needed), no cached display fields (resolved live per E-8). **Recording hook:** in `select()` (`store.ts:405`) beside the existing `navStack` push — record on real navs only (`record !== false`), move-to-front + dedupe by `navKey`.
  - **[R2-F4, refined by Nathan] Record ALL selectable kinds — pages, collections, sets, contexts, AND homepage** (only `none` is skipped; it isn't a nav). `RecentEntry` **mirrors `SelectTarget`** (`store.ts:42`) so the **id-less homepage** is representable (kind-only) and dedupes by kind via `navKey` (`store.ts:50`); homepage always resolves (singleton, always present). Whether each kind is *displayed* as a card is a separate open call → **D-12**.
  - **[R2-F1] Resolution is RENDER-prune, never STORAGE-prune.** A `reconcileSelection` → `none` entry is **hidden at render only**; it is NEVER deleted from the stored array on a failed resolve (a cross-nexus or mid-load transient would wipe real data). Storage changes ONLY via record, **LRU roll-off of oldest un-pinned**, and explicit unpin. A genuinely-deleted entity just stops rendering and rolls off naturally.
  - **[R2-F5 + R2-F2] Debounce lives in MAIN, not the renderer.** The renderer sends each nav fire-and-forget over IPC; **main** debounces the disk write, keys it by the current nexus root, and **flushes on `before-quit`**. A renderer `beforeunload` can't await the async IPC write (sandbox + contextIsolation), so a renderer-side flush races and loses the tail — main-side is the only sound place, and it also can't leak a pending write across a nexus switch.
  - **[R3-F1] The `before-quit` flush must actually await.** The current `before-quit` (`index.ts:1543`) is **synchronous fire-and-forget** (fine for the regeneratable index, NOT for a durable write) — Electron won't await an async `writeJson`. So the flush must `event.preventDefault()` → `await` the pending write → `app.quit()` behind a re-entry guard. Otherwise the last debounce window (~1 nav) is silently lost on ⌘Q.
  - **[R1-F8] Back/Forward does NOT bump MRU** — `stepHistory` re-selects with `{record:false}`, so recents order reflects forward selections only (recents = "places selected anew"). Explicit decision, not an accident.
- **E-2 · Pinning (temp-pins → top):** [confirmed direction] A `pinned: boolean` flag ON a `RecentEntry` — NOT a separate list (matches D-3's collapse). Action `togglePin(ref)`; persists until unpinned (D-7); sorts to top at render; exempt from LRU roll-off. **[R1-F6] Pin mutations write immediately (un-debounced)** — only the hot nav-stream debounces (in main, E-1) — so a pin can't be clobbered by a later-landing debounced write.
- **E-3 · Favorites (durable, sidebar):** [confirmed direction] Store slice `favorites: FavoriteEntry[]` (`{ kind, id }`); **storage mutated ONLY by explicit add/remove/reorder, written immediately.** **Persistence:** a synced sidecar (E-5). **[R2-F1] Resolution is RENDER-prune only** — a `none`-resolving favorite is hidden at render, NEVER auto-deleted from disk (a cross-nexus switch resolves every favorite to `none` against the wrong tree — auto-prune would wipe the highest-value tier). **[R1-F4] Naming:** disambiguate from the existing app-level `appConfig.recents` (nexus paths) and the dormant `tree.saved` `SavedNode{key:'recents'}` — pick names that don't collide (e.g. `navRecents.json`/`navFavorites.json`). **[R3-F2] v1 favorites-add is scoped to tree kinds** (page/collection/set/context/homepage) — NOT agenda: an agenda favorite would render through the E-9(b)-deferred resolver Map and can't display. Agenda-favoriting ships WITH that resolver in Agenda's feature (follows from the E-9 lean split; keeps the (a)/(b) boundary clean).
- **E-4 · Search:** [confirmed direction] **v1 = client-side fuzzy filter** over a memoized flat `{ id, title, kind, path }` list built from `store.tree` via the existing flatteners (`allCollections`/`allSets`/`allPages` + `tree.contexts`) **plus the cached `agenda:list` snapshot** (Tasks/Events — the once-fetched cache from E-9(a), not a per-summon walk). Title/kind match, zero new backend. Match/rank semantics (weighting, empty-query) are a build detail (empty query = the gallery is the default view, search filters it). **Deferred (Prospect):** full-text body search + context-tag-filtered search → needs a SQLite query layer + FTS/unified table + IPC, none of which exists.

- **E-9 · Agenda-aware plumbing (LEAN split — R1-F3 resolved; F-3 boundary corrected):** [confirmed] Two halves:
  - **(a) BUILD NOW — hold + find agenda.** The ref `kind` union widens to `page | collection | set | context | task | event` (a widenable string union, ~free). **Search lists agenda items** — its one standalone v1 value: you can *find* a Task/Event. **[R2-F3] This requires a CACHED `agenda:list` snapshot in build-now scope** — `agenda:list` (`collectAgendaEntries`) is a full disk-walk with no cache/by-id resolver, so it's fetched **once** and reused across every NavPane/NavMenu open, invalidated on an agenda-file change (never re-walked per summon or per keystroke — that'd be "expensive on every X" on a constantly-summoned surface).
  - **(b) DEFER to Agenda's own feature — resolve + display + ROUTE a task/event card.** Recents can't produce agenda entries in v1 (F-3), and an agenda click has no real destination (F-2). Deferred = **only** the stored-ref → card resolve/display + the placeholder-window routing (NOT the fetch, which is in (a)). **Reuse blueprint (documented, not built):** from the (a) cached snapshot build an **id→entry Map** → resolve agenda cards from it → route the click to the placeholder preview window (F-2). Slots into the already-widened union with no rewrite when Agenda ships.

- **E-5 · Sync boundary:** [confirmed — reaffirmed over R1-F1] **All of it syncs** — `recents.json` (incl. temp-pins) and `favorites.json` are both canonical/synced; NEITHER goes in `DEVICE_LOCAL_NEXUS_FILES`. **Rationale (dissolves R1-F1):** last-writer-wins IS Pommora's deliberate sync model; this is a **single-user** app where two devices are never edited live at once, so the concurrent-write loss R1-F1 describes is **outside the threat model** — an accepted tradeoff, not an oversight. No merge-on-read needed. The `activeViews` device-local precedent addressed churn for a per-container pointer; here Nathan wants the working set + history to travel, and accepts the model. *(Nathan)* **Do not re-flag** without new information.
- **E-8 · Display resolution:** [confirmed] Stored entry = `{ kind, id }` only; the card's **location + icon + title all resolve live from the tree** (icon off the node, location via `crumbs.ts`, title via flatteners) — nothing display-related is persisted, so cards stay auto-correct on rename/move. *(Nathan, D-9)*. Per-kind location-slot rules (for the kinds D-12 chooses to display): **pages/sets** → their collection›set chain (`crumbs.ts`); **collections** → root; **contexts** → their **tier** (Area/Topic/Project) fills the location slot — `crumbs.ts:66` returns only the context's own name, so the tier stands in (mirrors agenda's group-as-location, D-10); **homepage** → special-cased (id-less singleton); **agenda** → group, via the pre-fetched `agenda:list` Map (E-9).
- **E-10 · Second consumer (Insert / Link-Page):** [confirmed] **Yes — Insert / Link-Page IS a consumer of layer E**, via a **`page`-kind-filtered** recents view behind its Page branch. **Deferred, not urgent:** the current disclosure-based picker works for now; the recents-backed swap comes later. Noted in `SurfacePM.md` Pending (the stale `state.json`-recents claim there was corrected — the layer is the real source). *(Nathan)*
- **E-6 · navStack stays separate:** [confirmed] The in-memory `navStack`/`navIndex` Back/Forward history is a DISTINCT structure (directional session history) from the persisted MRU recents. Both fed from `select()`; don't conflate. *(code)*
- **E-11 · Nexus-switch reset:** [confirmed direction, R2-F2] On a nexus switch, `load()` must **replace recents/favorites in-memory wholesale** (as it already does for `activeViews`/`subfield`, `store.ts:248-268`) BEFORE the first render or record — else nexus A's entries linger and the first nav in B appends to A's list. The main-side debounce (E-1) keys pending writes by nexus root, so a nexus-A write can't land in nexus B's file. This is the write-cadence gap "mirror activeViews" hid: activeViews is safe because it writes only on explicit action; recents writes on every nav.

#### Lessons
- **Don't trust preserved-but-unread on-disk keys as "working plumbing."** The `state.json` recents/pinned/cursor block looked live (populated, file freshly written) but is dead Swift-era data the JSON writer spreads forward — `setStateOrder` only ever writes order keys. A populated field ≠ a maintained field; trace the writer. → routes to `Guidelines/`.
- **[R1-F4] Inventory the whole name-space before claiming greenfield.** The first grounding caught the `state.json` keys but missed TWO other "recents" — a dormant `tree.saved` `SavedNode{key:'recents'}` (readNexus.ts:415, renameable via `saved-config.json`) and a live app-level `appConfig.recents` (nexus paths, cap 10). A new `recents.json` name-collides with the latter. Grep the concept name across `src/main` before naming a new file.
- **[R1-F1] A synced file under last-writer-wins isn't "just synced" — it's lossy.** Whole-file read-merge-write is only in-process safe; across devices iCloud keeps one copy. High-value durable data (favorites) needs merge-on-read, and hot streams (recents) belong device-local — `activeViews.ts` already made that exact call for the same reason. → routes to `Guidelines/`.
