## Block Surfaces — Decision Log

> **This log is a prerequisite to any Contexts / Spaces architectural rethink.** It specifies only the block-like surfaces and how they work — the host-agnostic tile system. The contexts resolution (keep the current design / user-created groups / a separate Spaces entity), the sidebar surfaces, and Homepage are deliberately parked for their own pass, and nothing here may depend on how they land.
>
> **Status: REVIEW-CERTIFIED · Section H reopened.** Three adversarial rounds ran (the Review-Discipline cap); every finding was independently verified against the code and folded. Chrome adopted MarkdownPM's handle method (G-8), so no design gate remains. **Section H (embed mechanics, end to end) is reopened at Nathan's direction** — its `[open]` entries are the live questions; everything else stands certified.

### Frame

- **Purpose:** Specify the **block/tile system** — Pommora's composable dashboard surface — as a host-agnostic core that works identically under every resolution of the contexts question, and record the contexts direction only as far as it's actually settled.
- **Core Value:** The block system's architecture (document shape, host abstraction, grid engine, tile types, storage, chrome, config surfaces) is fully understood and buildable *before* the contexts decision — nothing in it forecloses any contexts future.
- **Success Criteria:** (1) The block system is spec'd end-to-end and review-certified; (2) every contexts option maps cleanly onto it via the host abstraction; (3) the deliberately-parked decisions are enumerated with nothing silently depending on them; (4) no `[assumed]` tag survives ratification.

### Decisions

#### A — The Diagnosis + Direction

- **A-1:** [confirmed] The rethink decomposes into two independent axes: (1) the **tag vocabulary**; (2) the **dashboard surface**. Nathan's original Ideas A and B were different pairings of positions on these axes.

- **A-2:** [confirmed — Nathan] The pain is the **fusion**: the ratified design makes the three tag tiers the *only* holders of the block-dashboard capability (plus Homepage), forcing one entity to be both organizational tag and composed surface. The three-level vocabulary itself is fine — Areas/Topics/Projects stay; other dimensions (People, Courses…) belong to properties.

- **A-3:** [confirmed — Nathan] What's actually locked: **block functionality reserved exclusively to the three tag tiers is the no-go.** The leading candidate resolution is Idea-B-shaped — Contexts simplify to structural tag surfaces while Spaces ship as their own block-dashboard entity — but the final contexts resolution is deliberately parked (→ D-1); the block system may not depend on any particular outcome.

- **A-4:** [confirmed — Nathan] Contexts do **not** get page-style properties. Context→context linking is directionally sensible (scope TBD — Prospects).

#### B — Surface Model

- **B-1:** [confirmed — Nathan, by example] The block surface is a **flowing-length tessellated mosaic**: the page grows as short or long as its content, blocks are drag-to-position, drag-to-resize tiles — and **interior empty space cannot exist**. Tiles span freely (edges need not globally align, regions run different lengths), but every region is covered by exactly one tile; ragged ends are legal, holes never. Columns are emergent from tile placement, never structured column blocks.

- **B-2:** [open — parked with the contexts decision] Whether Context detail views are themselves block surfaces (Nathan: they "shouldn't be excluded" from it) and/or carry a structured Linked-From default.

- **B-3:** [confirmed — Nathan] **Homepage is deferred out of this arc entirely** (graph-view host idea logged as a Prospect). Nothing here depends on it.

- **B-4:** [confirmed — Nathan] **The tessellation invariant replaces compaction.** There is no gravity pass because there is nothing to compact: the layout is a **split tree** (nested row/column splits with ratio arrays, tiles as leaves), where deletion makes the sibling absorb the space, insertion splits the target region, and **resize moves a shared divider — redistributing between both sides with a minimum-size clamp** (splitter semantics; "resizing one next to another always resizes the other"). The exact vertical semantics (stack-internal edges, ragged ends, the flowing bottom) are deliberately **build-tuned live** (the live-HMR cadence), not spec'd in prose.

- **B-5:** [confirmed — Nathan] **Markdown-block tiles edit in place** — the tile itself is the editor surface. **Page-embed tiles are scrollable AND editable in place**, gated by a **hover-lock in the tile's top-right corner**; a separate open action navigates to the real Page. The hover-lock is **kind-specific**: a locked **page** block prevents editing the page's content *and clicking into it*; a locked **view** block prevents **configuring** the view (layout, groups, sorts, filters…) while **interacting** with its content — dragging pages, assigning values — stays live.

- **B-6:** [confirmed — Nathan, revised] Tile drags are handle-scoped (`dragConfig.handle`); `cancel` guards interactive innards. The handle is a **hover handle set into the block's border itself** (left edge, per Nathan's sketch) working like the existing MarkdownPM drag handles: **drag it to move the block, click it to open the block-actions menu** (Turn Into a different block type, among others). Embedded views' internal drag surfaces (PommoraDND rows/bands/chips) must never fight the grid.

- **B-7:** [confirmed — Nathan] **The engine is `SurfacePM`** — a genuinely in-house layout engine, the MarkdownPM precedent applied. The tessellation geometry (B-4) retired react-grid-layout's brain (collision cascades, compaction, coordinate math all exist to manage floating tiles with empty space — a problem this geometry doesn't have): SurfacePM is a **split-tree layout model + PommoraDND-pattern pointer-capture sensors + flex/percent rendering** — no collision solver, insertion is a tree splice, resize is one ratio write. The full RGL v2.2.3 teardown (→ `7-10 - SurfacePM — RGL Teardown.md`) informs it: patterns inherited (placeholder ghost, snapshot→preview→commit/abort drag state, scale-aware pointer math, recompute-from-snapshot discipline — their accumulation bug class), small pieces possibly verbatim (resize-edge composition shape, the container-width hook shape), the clone kept as MIT reference. SurfacePM sits behind the thin engine seam like every other library (no-dependency-lock-in rule).

- **B-8:** [confirmed — Nathan] **Embed row-title navigation honors the configured Open In:** a title click opens the page as a **page preview or full main-pane page depending on what's configured** — Linked views follow the source Collection's Open In; **Custom views carry their own "Open In" knob in their configuration**. Full-page opens are ordinary single-pane navigation (Back returns to the host). **Honest dependency (round 3):** the `open_in` config is stored and editable but **no surface consumes it yet** — `select()` has no preview branch (`store.ts`); the preview path (and Custom View's Open In knob) is inert until the page-preview surface ships, for embeds and ordinary Collections alike. Full-page is the working behavior until then.

#### C — Invariant Work (direction-independent)

- **C-1:** [confirmed — code-grounded, re-verified twice] The two genuinely new subsystems are: (1) the **block/tile renderer** (`blocks[]` reserved on disk, zero modeling/rendering exists), and (2) the **embedded-view host**: since embeds are fully interactive with write-backs *intended* (D-12), the mechanism is the existing container-view machinery **pointed at the embedded source from inside a tile**, plus **three seams** (adversarial round 2 falsified "persistence is the one refactor"):
  - **View resolution is per-instance, never global** — today which view renders derives solely from the container-keyed `activeViews[source.id]` slot (`useActiveView.ts:14`) and `pickView` searches only `source.views` (`TableView.tsx:92-96`), so a payload-owned config is unreachable and two mounts of one collection fight over one slot. The embed host takes its view from the **tile payload** (the copied config, D-12) and never touches `activeViews` — independent views per mount by construction; the main pane keeps its slot.
  - **View persistence is payload-only** — embed config edits write the block payload, never the source sidecar (D-12: copied, not synced); `saveViewAdopting(source)` is never called from an embed.
  - **Embed chrome** — tile bounds, banner excluded, toggleable in-line title (G-4).

  TableView's remaining `source` coupling (values, set paths, mutations) is *correct behavior* for embeds.

- **C-2:** [confirmed — code-grounded] Cost asymmetry: a new sibling entity kind is cheap (generic `folderEntity` CRUD, kind-agnostic banner/icon/order plumbing); user-creatable tier *vocabulary* is invasive (~20 hardcoded-3 files). Informs the parked contexts pass, not this one.

#### D — The Block System

- **D-1:** [confirmed — Nathan] This spec is **pre-contexts-decision**: the block system must work identically under do-nothing / Idea A / Idea B. The block system is the core; contexts wiring is the variable.

- **D-2:** [confirmed — engineering shape, no product choice in it] The system binds to a **BlockHost abstraction**: any folder entity whose sidecar carries the block document. All contexts options produce valid hosts; tier contexts' reserved `blocks[]` means even "do nothing" provides one. (User-facing consequences — any-entity-can-host, contexts-decision independence — separately confirmed in D-1/A-3.) **One uniform block system across Spaces / Homepage / Contexts for now** [confirmed — Nathan]; the abstraction must leave room for **per-host-kind rules down the road** (differing tile allowances, defaults, or behaviors once the contexts fork resolves) — don't build the divergence, don't foreclose it.

- **D-3:** [confirmed — engineering shape, no product choice in it] The **block document** is host-agnostic: a **layout tree** (nested `row`/`column` split nodes carrying ratio arrays, leaves referencing tile ids) plus a `blocks[]` of tagged-union tile payloads `{ id (ULID), type, …type payload }` — zod-validated in shared types, unknown entries preserved (E-1, separately confirmed).

- **D-4:** [confirmed — Nathan] **Markdown-block tiles are file-backed:** a real `.md` in the host's own folder — no properties, non-discoverable by search. The host folder generalizes the folder-with-sidecar idiom: sidecar = layout, loose `.md` = its prose.

- **D-5:** [confirmed — Nathan] **View-embed tiles come in two kinds, and the difference is the seed, nothing else** (vocabulary locked): a **Linked View** copies an existing saved view of a Collection/depth-1 Set into the block at pick time; a **Custom View** starts from a blank default. Both are **block-owned `SavedView`-shaped config against a single source container** (Notion's one-database-per-view model), resolved against that source's schema, diverging freely from the moment the tile exists (D-12: copied, never synced). A view embed additionally carries its own Open In (B-8, gated). **Authoring-pane UIX pending design (G-8); the plumbing must exist.**
  - **D-5a [confirmed — Nathan]: every view has a home container, because creation needs one.** A view surface's + New Page must know where the page is born — which Collection's folder and schema. A sourceless view could render rows but never mint one, and a view that can't create isn't a view surface. The tile payload therefore always carries the source ref, both kinds; **+ Custom** lives inside the picker's container drill (→ G-9): pick the source first, customize after.
  - **D-5b [confirmed — Nathan]: cross-nexus Custom Views are a Prospect, not v1** — parked until a creation story exists (→ Prospects). The `SavedView`-shaped payload and per-instance resolution leave that door open with zero v1 work.

- **D-6:** [confirmed — Nathan] Block hosts live under **`.nexus/`** — shielded from other apps. Root-lift is **not** additive (→ Prospects; verified: the adoption pass would stamp a root-level host's `.md` blocks into Pages and the host into a Collection).

- **D-7:** [confirmed — Nathan] Prose tiles come two ways: **page-embed** (a scrollable embedded view of a collection-managed Page) and **markdown block** (host-local `.md`, no properties, non-searchable).

- **D-8:** [confirmed — Nathan] Local markdown blocks **join the SQLite link graph** — their outbound `[[connections]]` index — but are **deliberately excluded from any graph-view surface** and stay **non-discoverable by search**. They're link *sources*, not destinations (→ D-11: nothing links to them). **Verified cost (round 2):** two `.nexus`-skipping subsystems must each gain a host-folder pass to deliver this — the connection indexer builds only from walked pages (`index/build.ts:316-326`), and `renameCascade` skips `.nexus` wholesale (`crud/cascade.ts:20`), so without both extensions block links would never index AND would silently go stale when a target page renames. Both extensions are Core work, not options.

- **D-9:** [assumed] v1 tile types: **markdown block · page embed · view embed.** Widgets are Prospects.

- **D-10:** [open — parked with the contexts decision] Sidebar/navigation surfaces for hosts.

- **D-11:** [confirmed — Nathan] **Markdown-block files are ULID-named** (`<tile-ulid>.md`) — no title, no frontmatter needed. Nothing ever links *to* a markdown block (they only link outward, → D-8), so filename-is-title buys nothing; the ULID name makes the tile↔file reference direct and rename-proof by construction.

- **D-12:** [confirmed — Nathan] **Embedded views are fully interactive on data, and fully detached on config.** Controlling a Collection's *content* from a block surface is the point: data writes always flow through in both kinds — value-cell edits AND drag-a-page-between-property-groups (a property write to the page file); structural set-reparent drags (real file moves between Sets) also live inside embeds [confirmed — Nathan: "unified structural continuity, it's consistent"]. **Config is the opposite: embedding a view copies its `SavedView` config into the block payload, and from then on the embed and the source view evolve independently** — an embed config edit never touches the Collection's saved view, and a source-view edit never reshapes the embed. An embed is a small viewport that needs tile-local tailoring (columns, sorts, widths); synced config would corrupt the canonical view on every fit-the-tile tweak, and reshape dashboards on every source edit. Config persistence has exactly one home: the block payload.

#### E — Sweep Outcomes (robustness + interaction)

- **E-1:** [assumed] **Unknown/malformed tile entries are preserved and rendered inert** — never stripped, never crashing the host (the filter-registry lesson applied deliberately).

- **E-2:** [assumed] **Dead references render inert placeholders** (deleted Page / Collection / Set — a deleted source *view* can't orphan an embed, the config is copied per D-12); the tile persists until the user removes it. No auto-cleanup cascade.

- **E-3:** [confirmed — Nathan] **Write + read cadence:** layout writes once on drag/resize-stop, debounced; mid-drag layout is transient renderer state. **Block content is a targeted per-host load** (the `loadValues(source.path)` pattern), never folded into the whole-tree walk; block mutations refresh through the store's direct load path. Named honestly: any write under `.nexus/` still triggers the watcher's full re-walk today (same cost page edits already pay) — the walk must simply never *parse* block content; watcher-side handling is a plan-stage knob. **Value coherence — most recent wins:** the single-pane model makes "main pane + embed of the same collection" impossible by construction (the pane shows the host OR the collection, never both) — the only multi-mount case is one host embedding a collection twice, and there **the most recent edit/load wins**; no shared value store is mandated (live sync between duplicate tiles is a plan-stage option, not a requirement).

- **E-4:** [confirmed — Nathan] **Every prose tile IS a CM6 portal.** Markdown tiles and page embeds mount one MarkdownPM view each, **read-only at rest** — full decorations always, no second render pipeline to keep in sync — with editability flipped in place through a live-reconfigured compartment: entering edit is a facet change on the same view, never a remount (jitter-free by construction). One tile holds the caret at a time; edit exits on click-out/Esc. The perf trade (N views over one) is accepted deliberately; visibility-capped mounting is the escalation if big surfaces ever drag.

- **E-5:** [confirmed — Nathan] **Removing a markdown-block tile trashes its backing `.md`** — a **new file-level trash op** (verified: `deleteFolderEntity` is folder-only; Pages delete via the page op), `.trash` destination settled at plan time. **Removing a page-embed or view-embed tile deletes only the tile entry — the real Page/view is never touched.** Deleting a host folder trashes the folder with its blocks inside.

- **E-6:** [confirmed — Nathan] **Right-click is the primary creation surface: block surfaces get their own fully-equipped context menus** — insert-block entries at minimum, per the app's native-menu convention. Exact menu contents are a design-pass detail.

- **E-7:** [confirmed — resolved by B-7] **Esc-abort for tile drags ships natively in SurfacePM** — owning the sensor layer means the app's Esc-abort gesture law applies by construction (PommoraDND's discipline), closing the gap RGL's component layer never documented.

- **E-8:** [confirmed — verified against the real Nexus] **Swift left no block content:** every `blocks[]` in Nathan's live Nexus (homepage.json + all Area sidecars) is an empty array; Topics/Projects hold no entities. The codec starts from a clean slate — no legacy shape to honor; E-1's preservation still guards any foreign future content.

- **E-9:** [confirmed] **Layout undo is out of v1** — a mis-drag reverses by dragging back (the tree restores on the inverse splice), destructive acts route to `.trash`; layout history is a Prospect. The surface never captures ⌘Z.

#### G — Surface Chrome + Config

- **G-1:** [confirmed — Nathan] Tiles render as **standard rectangular bordered blocks using the separator border token** (`--separator-border`); **during a resize drag the border highlights in accent**. Docs name the tokens; exact values live in code.

- **G-2:** [confirmed — Nathan] **Block components live in their own components directory** — a dedicated `Blocks/` family, not scattered into Detail/.

- **G-3:** [confirmed — Nathan] **The SettingsPane for any block-content surface is WIRED but not built** — the only real control at first is a **footing "lock" icon that prevents resize and reorder** (grid-wide static mode: drags + resizes disabled). All locks — this one and G-5's container lock — **persist in their entity's sidecar (synced)**, never per-machine [confirmed — Nathan].

- **G-4:** [confirmed — Nathan] **Per-tile hover chrome:** the border drag handle (B-6, drag = move · click = block actions); a **vertical-ellipsis (⋮) actions menu in the top-right on hover** carrying at minimum the banner toggle (page embeds) and show/hide in-line title; page and view embeds carry the kind-specific hover-lock (B-5). **Banners: view embeds never show banners** — a banner's controls are liquid-glass buttons, which don't belong inside a view — while **page embeds keep their banners** (no glass buttons needed), toggleable via the ⋮. In-line titles stay on both kinds.

- **G-6:** [confirmed — Nathan] **A view embed's in-line title defaults to the source Collection/Set's name; renaming it renames only the block's display title** (a per-block payload override), never the actual Collection or Set.

- **G-7:** [confirmed — Nathan] **New blocks default to markdown blocks**, and linking (Link Page / Link View via the handle menu) is the one conversion — markdown → embed, trashing the backing `.md` recoverably (E-5's discipline). Conversion is one-way and once: a configured embed's type is fixed for the tile's life; source management follows G-14 (page: re-pick via the handle; view: per-view, never the handle). Linking never touches the embedded Page/view (E-5).

- **G-8:** [confirmed — Nathan] **The handle + hover chrome adopts MarkdownPM's established drag-handle method** — the same visual treatment and the same interaction grammar (drag the handle = move; click / right-click the handle = the interactive block menus). No Figma gate: the precedent is the spec, and the feature arc's non-design-dependent work proceeds on it.

- **G-9:** [confirmed — Nathan] **The right-click Insert menu** offers three entries: **Page · View · Block.** *Block* creates a new markdown block directly (the G-7 default). *Page* always opens a search over the Collections' pages to pick the embed target. *View* opens the **View Source picker**: a menu listing the Collections as drill-in rows. Drilling into a Collection shows its Sets as chevron rows above that Collection's own views, with a **+ Custom** footer beneath them (the Custom View's entry point — source picked by the drill, per D-5a) — **no create-new of saved views inside the picker** (Linked means pre-configured; + Custom mints block-owned config, never a saved view on the source). A Set's chevron opens a compact picker-menu of that Set's views rather than another full drill level; Sub-Sets carry no views (existing spec). Exact visuals ride the G-8 Figma gate.

- **G-10:** [confirmed — Nathan] **Embedded views render slightly zoomed out** relative to a full container surface — the existing zoom mechanism, at a **fixed amount** tuned at build time. The zoom never varies with tile size (→ H-10).

- **G-11:** [confirmed — Nathan] **Page embedding is one shared framework:** the machinery that renders a Page inside a foreign surface serves both SurfacePM's page-embed tiles AND MarkdownPM's future `![[Embed]]` in-page embeds (the Canvas `![[canvas:ULID]]` directive family). One seam, two consumers — built with the block arc, consumed by the editor later.

- **G-12:** [confirmed — Nathan] **The Homepage is the persistence dev host** — block documents write to `homepage.json` behind the BlockHost seam while the contexts decision stays parked. Explicitly removable: the seam makes the host swappable, and Nathan clears it when the real hosts land.

- **G-13:** [confirmed — Nathan] **Alignment form-lock:** resize boundaries magnetize to other blocks' edges when within a few px of perfect alignment (`snapPx` knob, default 6) — shipped in the engine.

- **G-14:** [confirmed — Nathan] **The handle menu + per-tile Style.** Click / right-click on the drag handle opens the block menu, root rows state-dependent (→ G-16): an unconfigured markdown block offers **Link View ›** · **Link Page ›** — linking IS the conversion, trashing the block's `.md` recoverably per E-5; a configured **page** embed offers **Source ›** instead (the search pane re-picks the target page); a **view** embed's Source row renders **disabled** — views are per-block and each carries its own source (D-5a), so source management is per-view, never per-tile. The type is fixed for the tile's life (no embed→embed or embed→markdown conversions). Both states carry **Style ▸** (Bordered · Borderless) · **Duplicate** · **Delete** (confirmed before resolving). **Borderless** is a per-tile entry field: the chassis hides until you reach for it — border and notch return on border/handle hover and while dragging or resizing — and a locked host pins it hidden (wires with G-3).

- **G-15:** [confirmed — Nathan] **Background create + wedge fill + band-seam negotiation.** Right-clicking the surface background creates a block (the G-7 markdown default; the full G-9 Insert menu layers on later): inside a **ragged wedge** the new block fits flush to the row bottom under the tile above the click; on open background it appends as a new band. A **full-width block's north edge negotiates the band seam** when the band above is also a single tile — one grows what the other gives, blocks below stay put (the same one-height-to-give decline rule as nested stacks); south keeps stretch-and-flow. Tiles also **track pane toggles 1:1** — while the surface width animates (sidebar/inspector), tile transitions gate off so blocks fill the edges at the pane's own rate.

- **G-16:** [confirmed — Nathan] **The block menu is an in-app `PickerMenu`, never a native menu** — native menus can't hold search fields, pinned rows, or styled drill panes. The handle menu (and the G-9 Insert menu when it lands) composes existing primitives only: a `PickerMenu` shell notch-anchored to the drag handle — dropping down or opening out the side via its existing collision flip, native-feel placement — hosting the menu family's DRY pane mechanics: `PaneSlider` between panes, the ‹ `MenuPaneTopRow` back header, the pinned-edge scroll frame (pinned top/bottom rows, one scrollable middle). Built **after** the plumbing; the native `blockHandleMenu`/`blockPagePicker` are interim scaffolding until it lands.
  - **Link Page pane** [confirmed — Nathan]: search-first and scrolling. Each result row: (icon)(title) in label at `control` size, over a tight `Collection › Set` path breadcrumb in label-secondary `footnote.emphasized`. The list shows recently-opened pages in descending order until the search bar is used. Page-level recents don't exist yet — and are built as **shared architecture** [confirmed — Nathan], not a Link-Page-local list: one recently-opened-pages record (reserved home in `state.json`'s session-state Recents → [[Architecture]]) that this pane, Navigation, and any future recents consumer all read. Whoever builds first (Navigation or this pane) lays the shared record; the other adopts it.
  - **Link View pane**: the G-9 source drill (Collections → Sets chevron → that container's views, + Custom pinned footer) inside the same pane mechanics.
  - **Duplicate** [confirmed — Nathan]: copies the tile — a markdown block mints a copy of its `.md`, embeds copy the payload — landing directly below via the attach logic.
  - **Root rows are state-dependent** [confirmed — Nathan]: an unconfigured markdown block shows **Link View ›** · **Link Page ›** (the conversion moment); once configured, both collapse to a single **Source ›** — live on page embeds (the search pane re-picks the page), **disabled on view embeds** (views are per-block, each with its own source — source changes live with the view's config surfaces, never the tile handle). The type never changes. Style › · ─ · Duplicate · Delete ride both states; Delete keeps the native confirm for now.

- **G-5:** [confirmed — Nathan] **The view lock generalizes to a first-class per-container feature** — all views of any Collection/Set get it, not just embeds: a **lock icon in the ViewPane's `MenuBottomRow` footer** (`Toolbar/ViewPane.tsx` — the "+ create · … more" strip) that, when engaged, **dims the SettingsPane and ViewPanes across that entire Collection or Set** (view configuration disabled container-wide; interaction untouched). Same configure-vs-interact line as the embed hover-lock. Persists in the container's sidecar (synced) [confirmed — Nathan]. **The lock's scope includes view CRUD** — create/rename/delete/reorder live in the panes it dims, so they lock with it (per Nathan's "dims the SettingPane and ViewPanes"). Standalone-buildable — doesn't wait for the block system.

#### H — Embed Mechanics, End To End (reopened)

How a page or view actually lives inside a tile — the walkthrough the D/G decisions imply, with the fine-grained calls pinned. Certified decisions are restated with their tags; the `[open]` entries are what this reopening exists to settle.

- **H-1 — Creation** [confirmed — restates G-9/G-7]: embeds are **references, not files** — Insert → *Page* searches every Collection's pages and stores the pick as `page_id` (ULID, rename-proof); Insert → *View* runs the source drill (Collections → Sets chevron → that container's views, **+ Custom** footer) and stores `source_id` + the config — copied from the picked view (Linked) or the blank default (Custom); the source ref is never optional (D-5a), and view entries are list-shaped for the tabbed future. Only markdown blocks mint files; an embed's tile payload is the entire artifact.

- **H-2 — Page embed anatomy** [confirmed — B-5/E-4/G-4; scroll resolved]: the tile is an **editable window onto the real page** — the CM6 portal (E-4), header chrome parked pending the ⋮ pass; edits flow through the same debounced body-write path as the full page editor, so an embed edit IS a page edit. The same page embedded twice resolves by most-recent-wins (E-3). **Scrolling is at-rest edge-release** [confirmed — Nathan]: the embed scrolls its own overflow on hover (no click / no caret needed), releasing to the host page at its top/bottom bound (default scroll-chaining); a page that fits passes the wheel straight through, so a short embed never hijacks the surface scroll. Editing still traps the wheel (`overscroll-behavior: contain`) so a live edit can't leak scroll to the page. This unifies page embeds with **view embeds** onto one model — B-5's original at-rest-scroll wording — and dissolves the H-3 compound (a locked embed still scrolls to read; it never needed edit-mode to reach the tail). Both edges carry the shared **scroll-edge fade** (top + bottom).

- **H-3 — Page-embed lock + navigation** [confirmed — B-5/B-8/Nathan]: the hover-lock freezes *content* — no editing, no click-into-the-body — while the **open action stays available** (a lock guards the page's text, not your ability to visit it). Opening honors the source Collection's Open In; full-page is the working behavior until the preview surface ships.

- **H-4 — View embed anatomy** [confirmed — C-1/D-12/G-4/G-6/G-10]: the tile renders the **view body only** — heading band + rows, slightly zoomed out, never a banner, in-line title defaulting to the source's name (renaming it retitles only the block). The view resolves **per-instance from the tile payload** — never the global active-view slot (`activeViews[source.id]`, read today by the table, the properties pane, the view pane, and the settings pane: `TableView.tsx:151`, `PropertiesPane.tsx:166`, `ViewPane.tsx:68`, `SettingsPane.tsx:74` — so every config surface an embed opens must receive the embed's resolution, not the slot's).

- **H-5 — View-embed header + configuration surface** [confirmed — Nathan, by Notion screenshot + anatomy follow-up]: the embed's header is up to two rows — an optional **title row** (the ####-scale in-line title, click-to-rename like a doc title, G-6; the view's icon leads it, the config affordance trails, a hairline beneath — the header's ONLY divider) above the **switcher row**: the view pills plus a + New View affordance — or, in Dropdown style, a single pill opening the ViewPane's row list as a PickerMenu, switch-only with no edit chevrons (per-view editing lives in the settings pane). The entry reuses the container presentation vocabulary (`view_button` icon/labeled — embeds default labeled — and `view_style` toolbar/dropdown, → [[Views]]). No toolbar row. **Right-clicking the title row offers Hide Icon / Hide Title; right-clicking the switcher area opens Hide/Show Titles · New View · Style ▸ Toolbar/Dropdown — plus Show Title while the row is hidden (collapsed, the switcher line is the whole header, config affordance trailing, no dividers anywhere).** Embedded tables shed the column-header band chrome — no heading fill, no divider beneath it (embed-scoped; container tables keep the band). The config affordance rides the tile's hover chrome (G-4's top-right family) and opens the existing **settings dropdown scoped to the embed** — the view-shaped leaves (Layout · Visibility · Group · Sort) reading the embed's per-instance resolution (H-4), never the global slot. [confirmed — Nathan] The header's icon + title are **the view's own** — payload edits, never `submitRename`/`setIcon` on the source folder (G-6's law; you can't rename a Collection or its icon from a view). The **Properties leaf renders and wires to the source Collection — deliberately**: a property assigned from an embed must register in the collection schema or pages carry properties the schema doesn't know; it's the one intentionally global leaf. The Configuration leaf waits on B-8's preview surface. The pane's **footer**: the source path breadcrumb on the left — each container with its icon, `(icon) Collection › (icon) Set`, custom icon or the kind default (never empty) — the lock on the right; both label-secondary `footnote.emphasized`. **The footer lock IS the tile's B-5 per-tile config lock** [confirmed — Nathan] — freezing this view embed's configuration only (not the host, not the source Collection's other views). **Per-view source re-pointing is parked** [confirmed — Nathan, deferred]: changing which Collection/view an embed points at after minting has no v1 affordance (→ Prospects) — an embed's source is fixed at pick time, re-picked by deleting + re-inserting for now.
  - **Switcher pills** [confirmed — Nathan, in-chat design]: each pill is icon + label-control title on the quinary fill, 4pt radius, 1pt segment hairline; the active pill's border wears the accent tint. Pills display **icon + title (default) or icon-only** — the Hide Titles toggle, the per-view `view_button` concept. A pill's own right-click carries the ViewPane row family (Rename / Edit Icon / Delete, Delete gated on the last view); **+ New View mints a blank view on the active view's source and becomes active**. Pills **drag to reorder** (reordering the payload's `views[]`) `[pending — rides the interaction pass]`. **Dropdown mode copies the ViewPane's logic** (the list + its row CRUD, operating on the tile's copied views, never the source's). The config affordance is the toolbar **SettingsPane button — same icon, same dropdown — except its (icon)(title) header references the *view*, not the source Collection/Set.**

- **H-6 — Write-through routing** [confirmed — D-12/C-1]: data edits (cell values, drag-between-groups, set-reparent drags) write to the source from either kind, always. Config edits write the block payload, always — the source's saved views are untouchable from inside an embed (copied, not synced). The view lock (B-5) freezes the config half only; interaction stays live.

- **H-7 — View-embed runtime** [confirmed — D-5a/D-12]: both kinds resolve against their source container through the exact same rows/schema/values pipeline, config always from the payload — the seed (copied view vs blank default) is the only difference, and it's over by the time the tile exists. New pages minted from a view embed are born in the source container — creation always has a home. Carries its own Open In (inert until the preview surface, like every Open In).

- **H-8 — Death + conversion** [confirmed — E-2/E-5/G-7]: removing an embed tile deletes the tile entry only — the page/view is never touched. A dead reference (deleted page, deleted Collection/Set) renders an inert placeholder that holds its space until the user removes it; no auto-cleanup. A deleted source view can't kill an embed — the config is copied (D-12). Linking a markdown block away trashes its `.md` recoverably and never touches the embedded source; a configured embed's type is fixed thereafter — source management follows G-14.

- **H-9 — Embed scroll + sizing feel** [confirmed — Nathan]: a scrolling tile scrolls its own overflow and releases to the host page at its bound (edge-release, exhausts before the page moves); view embeds want a wider practical minimum than markdown tiles (live-HMR knobs, not spec). The "many scrollable embeds on one board" feel **works as-is for now** [confirmed — Nathan]; the hover-intent capture escalation (a page-scroll-in-progress skips over embeds, only a deliberate at-rest scroll captures) stays **documented but unbuilt**, revisited only if the plain model bites in real use.

- **H-11 — Embedded popups escape the tile** [confirmed — Nathan]: any popup or dropdown born inside an embed (the `[[` autocomplete, future menus) must never clip against the tile's borders — they render through a **body-level portal**. The engineering rule underneath: a SurfacePM tile is a `transform`ed ancestor, which re-anchors `position: fixed` descendants to itself — so fixed-position UI inside a tile's subtree is broken by construction; portal out, always. The portal also carries every MarkdownPM affordance: tiles define their own `--fold-gutter` strip so grips + fold chevrons render exactly as they do on a page.

- **H-10 — Resize is a viewport, never a scale** [confirmed — Nathan]: expanding or resizing an embed tile — page or view — changes only **how much of the content is visible** (the window's width/height onto it), never how big the content renders. The embed zoom (G-10) is one fixed amount; content never rescales with tile size. A narrower view embed shows fewer columns' worth of width, a shorter one fewer rows — like resizing an OS window, not zooming a canvas.

#### Considered & Rejected

- **Idea A — user-creatable context groups replacing the three tiers** — rejected by Nathan: the three-level vocabulary isn't the pain; property dimensions cover extra groupings; expanding tag entities to carry dashboard needs is the wrong fix.
- **Contexts carrying page-style properties** — scrapped by Nathan.
- **Block-editor libraries (BlockNote / Plate / Lexical / Editor.js / Yoopta)** — second editor runtime beside CM6, lossy markdown, GPL'd columns, DOM-owning cores, or bus-factor-1.
- **Read-only view embeds** — reviewer-proposed, rejected against PRD:205 and Nathan's intent: controlling a Collection's content from a block surface is the point. Embeds are fully interactive (D-12).
- **A new pure/read-only render path as the embed mechanism** — briefly held after adversarial round 1 flagged TableView's `source` write-backs as dangerous from inside a tile; dissolved when D-12 established the write-backs are *intended*. The mechanism is TableView-against-the-source with a view-persistence seam (C-1), not a parallel renderer.
- **Always-nexus-wide Custom Views** (no source ref in the payload; a Location filter as the only scoping tool) — held through certification, rejected by Nathan on the creation test: with no home container, a view's + New Page has nowhere to be born, so the design ships a view surface that structurally can't create. One-container-per-view (Notion's model) replaces it; cross-nexus returns as a Prospect gated on a creation story.
- **Two-way write-through Linked Views** ("it IS the collection's view, edited from anywhere") — held through certification, rejected by Nathan: an embed is a small viewport needing tile-local tailoring, so every fit-the-tile tweak would silently rewrite the Collection's canonical view, and every source edit would silently reshape dashboards built around the tile. Copied-not-synced replaces it (D-12); push-to-source/re-pull affordances stay open as Prospects.

#### Lessons

- **A capability granted exclusively to one entity kind quietly fuses two jobs onto it.** Reserving the block surface to the tag tiers made "organizer" and "dashboard" one entity and capped the surface's reach. Spec capabilities host-agnostic; let entities opt in. → routes to `Guidelines/`.
- **Run every content surface through the creation test before certifying it.** A surface spec'd from the rendering side alone can pass three adversarial rounds while structurally unable to answer "where does + New put the new thing" — display, filtering, and grouping all work without a home container; creation is the one operation that can't. → routes to `Guidelines/`.
