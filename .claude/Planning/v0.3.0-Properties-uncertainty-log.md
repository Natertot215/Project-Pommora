### v0.3.0 Properties — Uncertainty Log

> Companion to `v0.3.0-Properties-implementation.md`. Captures edge cases, Swift-language risks, SwiftUI pattern confirmations, and open design questions that need answers before implementation begins. Authored RC-2026-05-19 from: (1) Explore agent inventory of existing Pommora Swift code, (2) Context7 query against Apple SwiftUI docs, (3) Notion API docs for property semantics.

#### Top 5 blockers (ranked)

##### 1. PropertyValue `.status` vs `.select` decode collision — **CRITICAL**

`PropertyValue.swift:33-88` decodes shape-based: a bare string defaults to `.select(String)` after URL / datetime / date probing. Status values encode as bare strings too (matching Select's storage shape). Standalone decode cannot distinguish them without schema context.

**Failure mode:** Page loaded via `PageFile.load()` → frontmatter decodes `status: "in_progress"` as `.select("in_progress")` instead of `.status("in_progress")`. When the inspector tries to render via `PropertyEditorRow.case .status`, the value is the wrong enum case → either crash, silent wrong rendering, or fallback to `.select` UI.

**Resolution (recommended):** Add `ContentManager.promoteStatusValues(in:, vault:)` post-load helper that walks the entity's `properties` dict, checks each key's schema entry, and rewrites `.select(s)` → `.status(s)` where the schema declares `.status` type. Called immediately after `PageFile.load()` / `Item.load()` / `AgendaItem.load()` inside ContentManager's load paths. Alternative: thread schema via `decoder.userInfo[.propertyDefinition]` — more invasive, breaks the file-can-decode-without-schema legibility constraint.

**Test:** Round-trip a `.status("done")` → encode → decode-without-schema (verify `.select`) → promote-with-schema (verify `.status`).

##### 2. RelationScope migration — **CRITICAL**

`PropertyDefinition.swift:34-37` currently declares `enum RelationScope: String, Codable, CaseIterable { case sameVault = "same_vault"; case anywhere }`. v0.3.0 reshapes to associated-value enum with `vault(String)` / `collection(String)` / `contextTier(Int)`.

**Breaking changes:**
- `CaseIterable` conformance drops (Swift requirement for associated-value enums).
- Stored format on disk changes from `"relation_scope": "same_vault"` (bare string) to `"relation_scope": {"kind": "vault", "vault_id": "01H..."}` (discriminated object).
- Any code iterating `RelationScope.allCases` for picker UIs silently fails to compile.

**Resolution:** Custom `init(from:)` accepts BOTH shapes. Old bare-string `"same_vault"` decodes as `.vault(<this vault's id>)`; `"anywhere"` decodes as `.vault(<this vault's id>)` with a logged warning. New encode always emits the discriminated-object form. No separate migration pass — happens at decode time. Drop `CaseIterable`; replace any `.allCases` use with a hard-coded picker list (only 3 kinds; rare in code).

**Test:** `RelationScopeTests.swift` — decode old `"same_vault"` → `.vault(parentVaultId)`; decode new discriminated-object → correct case; new encode emits only new format; round-trip identity preserved.

**Mitigation cushion:** Current v0.2 vaults all have empty `properties: []` arrays (no relation properties exist in shipped vaults). So real-world data has zero exposure to this migration. Test coverage matters only for forward correctness.

##### 3. Multi-file atomic-write recovery — **HIGH**

`AtomicJSON.swift:24-27` and `AtomicYAMLMarkdown.swift:46-53` both use `Data.write(to:, options: [.atomic])` (single-file POSIX rename atomicity). v0.3.0 introduces multi-file operations:
- **Property rename** in a Vault with N Pages + M Items = N+M+1 file writes
- **Dual relation creation** = 2 `_vault.json` writes
- **Schema delete cascade** = N+M+1 writes
- **Cross-Vault move-strip** = 1 read + 1 delete (source) + 1 create (target)

No transaction journal exists. No `.tmp-*` cleanup pass at launch.

**Failure modes:**
- Power loss / crash between file writes → partial state where Vault schema is updated but member files aren't (or vice versa).
- Orphan `.tmp-<uuid>` files accumulate over time, gradually filling the nexus.
- Dual-relation half-writes: Vault A's relation property added, Vault B's reverse property not yet written → Vault A points to nonexistent reverse property; UI breaks.

**Resolution (two parts):**

1. **`SchemaTransaction.swift`** (new file under `AtomicIO/`): two-phase commit pattern. Phase A: write all new content to `.tmp-<uuid>` siblings. Phase B: atomically rename each `.tmp` → final name in sequence. On any failure in Phase B: existing `.tmp` files of the failed transaction are deleted; already-renamed files stay (the rollback is best-effort — true rollback would require backups of overwritten content).

2. **`<nexus>/.nexus/_transaction-log.jsonl`** (new): append-only log of intended multi-file operations. Before SchemaTransaction.commit(), append a `BEGIN` record listing target file paths + new SHA256s. After all renames succeed, append `COMMIT`. On launch, scan for un-committed `BEGIN` records → either complete the renames (if `.tmp` files exist and match SHA256) or roll back (delete `.tmp` files).

3. **Startup orphan cleanup** in `NexusStore.initialize()`: scan nexus root for any `.tmp-*` file older than X minutes → delete. Quick safeguard against orphan accumulation.

**Tradeoff:** Full ACID isn't achievable in a userspace POSIX-files model without serious infrastructure (SQLite-as-source-of-truth, fsync everywhere). Best-effort recovery is sufficient for personal-scale Pommora; serious data integrity arrives with v0.3.3 SQLite when the index can detect divergence from files.

**Test:** Simulate crash by inserting `try Task.checkCancellation()` between phase A and B in SchemaTransaction; assert orphan cleanup runs on next launch; assert no torn renames.

##### 4. Swift 6 strict concurrency + Sendable for new types — **MEDIUM-HIGH**

All new types (`StatusGroup`, `StatusOption`, `DualPropertyConfig`, `ContentTemplate`, `SchemaTransaction`) need `Sendable` conformance. Auto-synthesis works if all stored properties are Sendable. New types contain only Strings, Optionals, primitives, enums — all Sendable. **Auto-synthesis should succeed.**

**Risk surface:**
- `PropertyDefinition` gains `var statusGroups: [StatusGroup]?` + `var dualProperty: DualPropertyConfig?` + `var icon: String?` + `var allowsMultiple: Bool`. Hashable + Equatable auto-synthesis depends on all nested types conforming. Confirmed conforming.
- New manager methods (`createPairedRelationProperty`, `propagateDualRelation`, `updatePageFrontmatter`) follow the existing `@MainActor @escaping () -> NexusContext` closure pattern at `ContentManager.swift:19-46`.
- Closure capture of Vault state: if a view captures the current Vault in a closure (e.g., for "Manage options…" callback), and Vault.properties is mutated concurrently, the closure may observe a torn read. Mitigation: snapshot Vault at closure-creation, not closure-execution.

**Validate:** Adding the new fields to PropertyDefinition.swift and running `xcodebuild` (`trust xcodebuild not SourceKit` per CLAUDE.md quirk #3) — auto-synthesis either succeeds clean or produces concrete error to address.

#### SwiftUI patterns confirmed (Context7 + Apple docs)

##### Dynamic Table columns — `TableColumnForEach` (macOS 14+) — RESOLVED

The biggest UI risk for v0.3.0 (rendering `vault.properties` as dynamic Table columns) has a first-party answer. Pattern:

```swift
Table(rows, sortOrder: $sortOrder) {
    TableColumn("Title", value: \.title)
    TableColumn("Kind", value: \.kind)
    TableColumn("Modified", value: \.modifiedAt)
    TableColumnForEach(vault.properties) { property in
        TableColumn(property.name, value: \.value(for: property.name)) { row in
            PropertyCellView(definition: property, value: row.value(for: property.name))
        }
        .width(ideal: 120)
    }
}
.tableColumnCustomization($columnCustomization)
```

`TableColumnForEach` (Apple docs URL: `developer.apple.com/documentation/swiftui/tablecolumnforeach`) generates dynamic columns from an `Identifiable` collection. `PropertyDefinition.id` is already `String { name }` — works directly.

##### Built-in column customization — `TableColumnCustomization` — RESOLVED

The **Property Visibility** feature gets implemented for free. SwiftUI Table has `columnCustomization: Binding<TableColumnCustomization<Value>>` that persists per-column visibility + ordering. Persists to `_vault.json.hidden_properties` via observation. **The user gets drag-to-reorder column headers AND show/hide for free.**

Recommendation: Use `TableColumnCustomization` directly; don't invent our own `hidden_properties` mechanism. Persist its serialized state to `_vault.json.column_customization` (new field).

##### Sortable Table — `KeyPathComparator` + `sortOrder` binding — RESOLVED

```swift
@State var sortOrder = [KeyPathComparator(\Row.modifiedAt, order: .reverse)]

Table(rows, sortOrder: $sortOrder) { ... }
    .onChange(of: sortOrder) { _, new in
        rows.sort(using: new)
    }
```

For Status's group-position-then-option-order sort, implement a custom `SortComparator` that resolves an option's `value` to its `groupId` (via Vault schema) then compares group position first, option index second. Save the default sort to `Vault.default_sort: SortDescriptor?` (new optional field).

##### Drag-and-drop between Sections — `.draggable` + `.dropDestination` — RESOLVED

For the Status option editor's drag-between-groups (Phase 2.12):

```swift
ForEach(statusGroups) { group in
    Section(group.label) {
        ForEach(group.options) { option in
            OptionRow(option)
                .draggable(option.transferableID)
        }
    }
    .dropDestination(for: String.self) { droppedValues, _ in
        for value in droppedValues {
            moveOption(value: value, toGroup: group.id)
        }
    }
}
```

`Transferable` conformance via raw String (option's `value`). Drop fires the dual-effect confirmation dialog before commit.

##### Grouped Picker / Menu popover — supported via `Section` in `Picker`/`Menu` body

Status picker (Phase 2.11) uses standard SwiftUI:

```swift
Menu {
    ForEach(statusGroups) { group in
        Section(group.label) {
            ForEach(group.options) { option in
                Button(option.label) { value = .status(option.value) }
            }
        }
    }
    Divider()
    Button("Manage options…") { router.openVaultSettings(at: .property(definition.name)) }
} label: { statusPill }
```

No special API needed; works on macOS 14+.

#### Open design questions — RESOLVED + REMAINING

##### Q1 — Vault Settings sheet: live writes or save/cancel? ✅ RESOLVED (RC-2026-05-19)

**Resolution: Save-required.** All edits in the Vault Settings sheet stage into a draft model; commits happen only on the sheet-level Save button. Cancel discards the entire draft. SchemaTransaction wraps the flush so all property edits land atomically. Inline "Add option…" popovers (reached via right-click on a property label) are separate commit boundaries with their own Save buttons — they do NOT batch with VaultSettingsSheet.

##### Q2 — Concurrent Vault Settings editing (two windows of the same Vault) ✅ RESOLVED

**Resolution: Disabled.** Second invocation of Vault Settings on a Vault that already has one open shows a disabled state ("Settings already open in <window title>"). Implemented via `@MainActor` singleton `VaultSettingsRegistry` tracking open sheets by Vault ID.

##### Q3 — Last Edited Time source: file mtime or frontmatter `modified_at`? ✅ RESOLVED

**Resolution: Frontmatter `modified_at`, updates on any content / frontmatter edit, NOT on view-only actions.** Opening a Page or Item never updates `modified_at`. Body edits, property value changes, title renames, icon changes, tier1/2/3 edits, panel-visibility toggles all trigger save → `modified_at` updates. External edits (file modified outside Pommora) update file mtime but NOT frontmatter `modified_at` until v0.3.3 watcher closes the gap.

##### Q4 — Status option deletion with referenced entities ✅ RESOLVED

**Resolution: void (cascade to `.null`) — UNIVERSAL rule for all property types.** Deleting an option voids the affected entities' property values (sets to `.null`). Multi-select case: deleted option's `value` is removed from each entity's array (the array shrinks; only fully voids if the array becomes empty). Confirmation dialog lists affected entity count before commit. Same delete cascade pattern applies to Select / Status / Multi-select. No quarantine, no auto-reassignment, no refuse-to-delete — matches Notion behavior. Implementation: `VaultManager.deleteSelectOption(...)` / `deleteStatusOption(...)` / `deleteMultiSelectOption(...)` all cascade through SchemaTransaction.

##### Q5 — Property visibility vs sort interaction ✅ RESOLVED

**Resolution: Auto-show.** When sort references a property in `hidden_properties`, the column auto-shows. Sort precedence beats visibility (user can't see direction indicators without the header). Single-write removes the property name from `hidden_properties`.

##### Q6 — Inline option creation routing ✅ RESOLVED (refined)

**Resolution: structured inline-add via right-click on property LABEL (not value picker).** Decision #14 refined: right-click on column header (in Vault Table), inspector row label (Pages inspector), or Item Window row label exposes "Add option…" — opens a small structured popover (Name + Color + Group-for-Status + Save / Cancel) that commits the new option immediately on its own Save. Value pickers (Select dropdown, Multi-select chip rows, Status grouped popover) still never accept typed inline-add. Value-context right-click shows "Edit options…" only (jumps to Vault Settings). Picker popover bottom keeps the "Manage options…" link. Three commit paths:

1. **Vault Settings → Edit Properties → "+ Add option"** — batched with sheet-level Save (canonical, batch surface)
2. **Right-click on property LABEL → "Add option…"** — small popover with own Save (shortcut, single-option commit)
3. **No path from value picker** — picker is read-only-of-options

##### Q7 — Content template wiring in v0.3.0 ✅ RESOLVED

**Resolution: UI anchor in Vault Settings (7th "Templates" section, placeholder); no manager-method parameter.** v0.3.0 adds a "Templates" section to VaultSettingsSheet as the 7th tab — placeholder ContentUnavailableView with "Pre-configured layouts for new Pages and Items in this Vault. Coming post-v1." **No `template: ContentTemplate? = nil` parameter is added to `ContentManager.createPage(...)`** — that signature change lands when content templates actually ship. The placeholder section preserves the wiring concept (Vault Settings IS the template-management home) without dead code in the manager layer. The previously-defined ContentTemplate Codable sketch + `<nexus>/.nexus/templates/` storage location remain reserved for the future implementation.

##### Q-Status — Is Status really distinct from Select? ✅ RESOLVED

**Resolution: distinct type.** Status is structurally distinct from Select because (a) it has 3 fixed structural groups with load-bearing IDs that drive EventKit mapping at v0.7.0, (b) sort behavior differs (group-position-first vs option-order-only), and (c) the 3-group constraint can't be expressed as a "Select with defaults" without losing the EventKit guarantee. Status appears as a standard option in the property type picker — NOT specially recommended or featured. When selected, auto-seeds with 3 default groups + one starter option per group.

##### Q-Disclosure — Per-entity property panel visibility (NEW, RC-2026-05-19) ✅ RESOLVED

**Resolution: wiring required at v0.3.0; UI deferred.** Each Page / Item / Agenda item carries `panel_hidden_properties: [String]` (default empty). UI affordances (right-click → "Hide on this entity" + footer "+ Add property" button) ship later but the data + manager-method wiring lands at v0.3.0 so the future UI is additive. Distinct from `Vault.hidden_properties` (Vault Table column visibility, Vault-wide).

##### Q8 — `EKEvent.status = .canceled` mapping

The Status type's 3 groups (upcoming / in_progress / done) cover most EventKit states. But `.canceled` events have no Pommora group equivalent. Options:

- (a) Map cancelled events to `.done` (treat cancellation as a form of completion)
- (b) Filter cancelled events out of Pommora's Agenda entirely (not synced)
- (c) Add a 4th structural group `cancelled` (breaks "3 slots structural" decision)

**Recommendation:** (a) for v0.7.0 sync. Document in EventKit mapping table. **Defers to v0.7.0 prep; not a v0.3.0 blocker.**

#### Edge cases by feature area

##### Schema mutation during member edit

User opens Page A's inspector. Property panel shows `status: "in progress"`. In another window, user (or another process) deletes the "in progress" option from the schema.

**Behavior:** Page A's inspector should refresh on next manager observation. The value silently becomes `.null` (because the option no longer exists). Display shows "—" or "Invalid value: in_progress" briefly until user clears it.

**Risk:** If user is mid-edit on a different field in Page A, the schema-update-driven refresh may discard in-progress edits. **Mitigation:** Snapshot the form state separately from the model; reconcile on focus loss.

##### Pre-SQLite (v0.3.0) relation picker performance at scale

Naive scans for `VaultMemberPicker` / `CollectionMemberPicker` / `ContextTierPicker` query managers directly. At personal scale (~50 Topics, ~200 Pages per Vault), latency is sub-100ms. At larger scale (1000+ entities), the picker opens with noticeable lag.

**Mitigation for v0.3.0:** Add a lightweight in-memory cache (`@MainActor` LRU dictionary keyed by `(scope, queryString)`) — clear on schema/content change. v0.3.3 SQLite swap-in retires the cache.

##### Page open during Status group label rename

User has Page open with Status pill rendering "Upcoming" (default group label). Renames group label to "Backlog" via Vault Settings. Page's pill should update to "Backlog" on next render.

**Implementation:** Status pill resolves `option.value` → `option` → `option.group` → `group.label` at render time (not cached). Renaming triggers `@Observable` propagation via VaultManager. **Verify:** PropertyEditorRow + StatusPickerEditor properly observe Vault changes (not just PropertyDefinition).

##### Cross-Vault move with paired relations

Page A in Vault X has a paired Relation property "Sources" pointing to Vault Y. User moves Page A to Vault Z (which doesn't have the "Sources" property).

**Behavior:**
1. Move-strip dialog warns "Sources" will be stripped
2. On confirm: Page A's `properties.Sources` value (list of B-side ULIDs) is removed
3. The B-side reverse property "Cited By" on Vault Y still has A's ULID listed → orphaned reference
4. Dual-relation maintenance must clean up B-side: for each B-side entity holding A's ULID in "Cited By", remove that ULID

**Required helper:** `cleanupOrphanedDualReferences(sourceEntityId:, sourcePropertyName:, in vault:)` called from move-strip flow.

##### Renaming property to reserved name

User renames property "status" → "id" (reserved). Validator rejects. UX: TextField goes red; tooltip shows "Reserved property name."

**Reserved names per spec:** `id`, `created_at`, `modified_at`, `tier1`, `tier2`, `tier3`, `wikilinks`. Plus uniqueness within Vault (case-insensitive).

##### Cancel during Phase 2 of two-phase commit

If user cancels (esc/cancel button) midway through a multi-file `SchemaTransaction.commit()` — say, after phase A `.tmp-*` writes succeed but before phase B renames begin:

**Behavior:** All `.tmp-*` files deleted; no member files changed; transaction log emits ROLLBACK. Same path as failure recovery.

##### Empty Status options list

User adds a Status property. Each group gets exactly one default option. User then deletes the "In progress" group's only option. Now group has empty options.

**UX:** Allowed (don't force at-least-one). Picker UI renders an "Empty" placeholder for the group section. Schema validator doesn't reject (no Status entity-level value would point to a non-existent option).

#### Migration considerations

##### v0.2 → v0.3.0 migration items

1. **RelationScope string → discriminated object** (decode-time shim; no separate pass)
2. **AgendaSchema Status injection** — legacy `_agenda.json` without Status gets it auto-added via `AgendaSchema.migrate(_:)` on load
3. **`.tmp-*` orphan cleanup** at first launch after v0.3.0 install
4. **`_transaction-log.jsonl` replay** at launch (handles v0.3.0 → v0.3.x crashes)
5. **`<nexus>/.nexus/templates/` directory** created on first nexus init (empty; reserved)

##### v0.3.0 doesn't break v0.2.x:

- Existing Pages / Items / Agenda items have `properties: {}` (mostly empty) — nothing to migrate
- Existing Vaults have `properties: []` — empty schemas, no migration needed
- New Vault fields (`default_sort`, `hidden_properties`, etc.) decode as nil/empty defaults via existing `decodeIfPresent` pattern

#### Items deferred OUT of v0.3.0 (explicit non-goals)

These were considered and rejected; listed so they don't get scope-crept back in:

1. **Vault-level templates** — replaced by content-level template reservation
2. **Inline option creation** — schema editor is the only authoritative surface
3. **`anywhere` Relation scope** — every Relation has a definite target
