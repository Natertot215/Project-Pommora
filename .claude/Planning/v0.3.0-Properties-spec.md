### v0.3.0 Properties — Spec

> Conceptual spec for v0.3.0 Properties. Locked decisions and behavior; **no implementation detail** (file:line citations, class shapes, phase ordering, test names, effort estimates). Feature-layer context at [[Properties]]. Pre-ParadigmV2 implementation plan archived at [`Planning/Superseded/v0.3.0-Properties-implementation.md`](Superseded/v0.3.0-Properties-implementation.md).

> **Scope split (decided 2026-05-23 brainstorm):** v0.3.0 ships the **data layer + minimum-viable placeholder UI** to verify the data layer works. The **real Properties Pulldown + Property Panel** UI is a fast-follow (alongside v0.3.0 or v0.3.1, Figma-driven). The **broader inspector architecture** (Claude chat as main-window inspector, PreviewWindow primitive for Page/Context Preview, Item Window redesign with pinned chips at Item Collection level) ships in later v0.3.x patches whenever the user designs them. See "Surface architecture" below for the three-tier UI roadmap.

#### Surface architecture (locked direction, tiered timing)

Properties live in three different surfaces depending on context. UI design + visual specifics are deferred to Figma; the architecture is locked.

| Surface | Property home | Timing |
|---|---|---|
| **Page in main window** | NavDropdown-style pulldown at top of content (populated-only + "+ Add property" picker over schema; lazy properties — empty schema entries don't render) | Real UI: v0.3.0 fast-follow / v0.3.1 |
| **Page Preview** (standalone window) | Property panel inside the window's inspector (toggle button; **default closed**) | UI ships when PreviewWindow primitive ships (v0.3.x) |
| **Item Window** (popover) | Property panel inside the popover's inspector (toggle button; **default closed**) + **pinned-property chips** above title, saved at the Item Collection level (shared across all Items in that Collection; cohabits with future Item Templates) | UI ships when Item Window redesign ships (v0.3.x) |
| **Context Preview** (standalone window) | n/a — Contexts have no properties. Inspector slot reserved for a TBD purpose. | n/a |
| **Main window inspector** | Claude chat (CLI subprocess bridge — frontend to local CLI, not API integration). Property panel NEVER lives in the main-window inspector under the new direction. | Ships independently, "whenever" |

**Three timing tiers:**

1. **v0.3.0** — data layer (model extensions + SchemaTransaction + manager schema CRUD + load-path migrations + validators + move-strip primitive) + minimum-viable placeholder UI (existing FrontmatterInspector and ItemWindow PropertyEditorRow patterns extended to handle all 10 property types; minimum-viable Type Settings sheet with just Edit Properties + Sort sections). Verifies data layer works end-to-end. Placeholder UI is disposable; no design polish.
2. **v0.3.0 fast-follow / v0.3.1** — Properties Pulldown (Pages main view) + Property Panel SwiftUI component (host-agnostic; slots into any inspector container). Figma-designed, quick implementation. Replaces placeholder UI.
3. **v0.3.x+ (whenever)** — Claude chat as main-window inspector; PreviewWindow primitive (Page Preview, Context Preview); Item Window redesign with pinned chips + inspector toggle; Item Collection's `pinned_properties` data field ships alongside Item Window redesign.

`_itemcollection.json` `pinned_properties: [String]` field and per-entity `panel_hidden_properties: [String]` field are **deferred out of v0.3.0** — both depend on UI surfaces that ship in tier 3. Lazy-properties UI may obsolete `panel_hidden_properties` entirely.

#### Property type catalog

Ten types. Same catalog across Pages, Items, AgendaTask, and AgendaEvent — only the storage substrate differs (Page frontmatter is YAML; Items / AgendaTask / AgendaEvent are JSON `properties` blocks). Each type has a fixed config shape stored inside the property's entry in the parent Type's per-kind sidecar (`_pagetype.json` / `_itemtype.json` / `_taskconfig.json` / `_eventconfig.json`) `properties` array.

| Type | Value shape on disk | Config shape | UI behavior |
|---|---|---|---|
| **Number** | `42` or `3.14` | `{ "number_format": "integer" \| "decimal" \| "percent" \| "currency" }` | Numeric input; formatted output. |
| **Checkbox** | `true` / `false` | `{}` | Toggle. |
| **Date** | `"2026-06-15"` | `{}` | Date picker; UTC-anchored on disk. |
| **Date & Time** | `"2026-06-15T14:30:00Z"` (ISO-8601 with timezone) | `{}` | Date + time picker. |
| **Select** | `"<option value>"` | `{ "select_options": [{ "value", "label", "color" }, ...] }` | Single-pick colored pill. Option `value` immutable post-create; `label` renameable. |
| **Multi-select** | `["<value>", ...]` | `{ "select_options": [...] }` (same shape as Select) | Multi-pick colored chips, **each chip in its option's color**. |
| **URL** | `"https://..."` | `{}` | Clickable link with favicon. |
| **Relation** | `{"$rel": "<ULID>"}` (single) or `[{"$rel": "<ULID>"}, ...]` (multi) | `{ "relation_scope": {...}, "allows_multiple": Bool, "dual_property": {...}? }` | Scope-aware picker; resolved target's current title rendered as styled colored inline text. |
| **Status** | `"<option value>"` | `{ "status_groups": [...] }` (3 fixed groups; user-editable options inside) | Grouped picker popover; single-pick; pill color resolves option override > group default. |
| **Last Edited Time** | *(not stored — derived from `modified_at`)* | `{}` | Read-only, sortable. v0.3.0 default sort, descending. |

##### Title is not a property

The only pure text field is the **filename** (Pages and Items both). Where a Notion-style free-text property would appear, Pommora uses Select or Multi-select with creatable options.

##### PropertyValue.relation encoding

Relations are stored as a **tagged JSON object** `{"$rel": "<ULID>"}` (locked 2026-05-16). The tag lets external agents and the future graph-view indexer identify cross-entity edges from any file without consulting schema. Display resolves the ULID to the target's current title (rename-safe).

##### Wikilinks are not a property type

Body-text wikilinks (`[[Title]]`) ship at v0.3.2 with their own derived `wikilinks: [...]` frontmatter mirror. The schema editor does not offer Wikilink as a creatable type.

#### Status property type

Single-pick workflow property with **three EventKit-aligned structural groups**, each containing user-editable options.

##### Fixed groups (EventKit-aligned)

| Group ID | Default label | Default color | EventKit meaning |
|---|---|---|---|
| `upcoming` | "Upcoming" | gray | `EKReminder.isCompleted = false` + future-dated; `EKEvent` future start |
| `in_progress` | "In Progress" | blue | Actively due / currently happening |
| `done` | "Done" | green | `EKReminder.isCompleted = true`; events in the past |

**Group IDs are load-bearing; group LABELS are user-renameable.** The three structural slots are fixed — adding or removing groups is not supported (preserves EventKit compatibility for v0.6.0+ sync). Workflow customization happens by adding options within groups.

##### Options within groups

Each group holds an ordered list of options. Each option carries `value` (canonical, immutable post-create), `label` (renameable), optional `color` override (nil inherits the group's color), and `group_id` (load-bearing — drives sort, EventKit mapping, display color).

##### Where Status is built-in

| Schema | Status built-in? |
|---|---|
| AgendaTask (Tasks singleton folder at root, sidecar `_taskconfig.json`) | **Yes — required, non-deletable.** Default seed includes the three groups with one starter option each. |
| AgendaEvent (Events singleton folder at root, sidecar `_eventconfig.json`) | **No.** Completion isn't an event concept; events derive effective state from `start_at` / `end_at` relative to now. |
| Page Types and Item Types | **No.** Not auto-seeded. Users add manually via Type Settings. |

##### Schema mutations on a Status property

- **Rename a group label** — schema-only (groups identified by structural ID).
- **Add an option** — schema-only.
- **Rename an option label** — schema-only (`value` immutable).
- **Move an option between groups** — schema-only file write but **data-semantic** (`group_id` rewrites; affects sort, display color, EventKit mapping, future Group By). Triggers a confirmation dialog listing affected entities.
- **Delete an option** — voids referencing values (`.null`). Confirmation dialog lists count.
- **Add or remove a group** — not supported.

##### Sort behavior

Group position first (`upcoming < in_progress < done` ascending), then option order within group.

#### Relation scope and dual relations

Each Relation property targets exactly one container at creation. Five scope kinds; no `anywhere` fallback.

| Scope kind | Picker source | Pairing |
|---|---|---|
| `page_type` | All Pages in the specified Page Type | **Required dual** — paired reverse property on target Page Type's `_pagetype.json` |
| `item_type` | All Items in the specified Item Type | **Required dual** — paired reverse property on target Item Type's `_itemtype.json` |
| `page_collection` | All Pages in the specified Page Collection | **Required dual** — reverse stored on the parent Page Type's `_pagetype.json` (Page Collections don't carry their own `properties[]`) |
| `item_collection` | All Items in the specified Item Collection | **Required dual** — reverse stored on the parent Item Type's `_itemtype.json` |
| `context_tier` | All Contexts at the specified tier (1 = Spaces / 2 = Topics / 3 = Projects) | **One-way** — Contexts have no `properties[]` schema; reverse view is query-derived (same as `tier1` / `tier2` / `tier3` backlinks) |

##### Dual-relation lifecycle

Creating a Type- or Collection-scoped Relation always produces two paired property definitions written atomically — one on each side. There is no opt-out (locked RC-2026-05-19, supersedes the earlier optional-toggle framing).

- **Creation** — user provides BOTH names (source + target) in the schema editor; both writes atomic.
- **Value setting** — setting a relation on entity A mirrors a back-reference on target entity B; removing removes both ends.
- **Renaming either side** — schema-only write that updates the other side's reverse-name reference. Paired identity survives.
- **Deleting either side** — confirmation dialog ("Deleting this property will also remove '<reverseName>' from <Type X>"). On confirm, both definitions deleted and mirrored values cleared on both sides.
- **Moving an entity across Types with a paired relation** — strip rule applies; the orphaned reverse value is cleaned up on the target side.

##### Cross-side relation guard (locked)

Pages-side schemas cannot target Item Types or Item Collections. Items-side schemas cannot target Page Types or Page Collections. The schema editor's picker is side-locked and the validator rejects cross-side scope at save. Cross-side Item ↔ Page relations are a post-v1 Prospect (coupled to cross-side promotion — see [[Prospects]]).

#### Options management (Select / Multi-select / Status)

Option creation, renaming, recoloring, deletion, and reordering happen **only via the schema editor** — never inline in the value picker. Notion's pattern.

##### Three commit paths

1. **Type Settings → Edit Properties → "+ Add option"** — canonical; batched with the sheet's Save.
2. **Right-click a property LABEL** (pulldown row, panel row, Item Window row) → **"Add option…"** — small structured popover (Name + Color + Group-for-Status + Save / Cancel) with its own commit boundary, separate from the sheet's Save. (Ships with the real Properties Pulldown / Panel UI; placeholder UI v0.3.0 doesn't include this affordance.)
3. **Right-click a property VALUE** (pill, chip, status indicator) → **"Edit options…"** — routes to Type Settings → Edit Properties at that property's row. Value pickers themselves do not accept typed new options; they show a "**Manage options…**" link at the bottom that routes to the same destination.

##### Option `value` immutable; `label` renameable

Each option carries a canonical `value` set at creation (never changes) and a user-facing `label` (renameable freely). Stored frontmatter / JSON references `value`; renames are schema-only. Same stable-target-with-renameable-display pattern as Pommora's wikilinks and relations.

##### Schema option order drives sort

For Select and Multi-select, the order options appear in the schema editor defines sort behavior — ascending returns first-listed first. Replaces alphabetical sort (wrong for workflow stages) and is clearer than Notion's separate "manual sort" mode. Status combines this with group position (group first, option order second).

##### Universal void-on-delete

Deleting an option voids referencing entity values (`.null`). Multi-select differs only in that the deleted value is removed from each array (the array shrinks; `.null` only if it becomes empty). No quarantine, no auto-reassignment, no refuse-to-delete. Confirmation dialog lists affected entity count before commit.

#### Property icons

Every property carries an optional SF Symbol. Shown next to the property name in the schema editor, in property panel rows, and as the column header glyph in detail-pane Table views. Settable via per-property `IconPickerField` (reuses SymbolPicker integration). Additive — existing schemas decode `nil`; no migration.

---

#### Move-strip rule

Moving a Page across Page Types (or an Item across Item Types) strips properties not in the destination schema — Notion-style; no quarantine. Confirmation warning lists what will be stripped; user can cancel, add the property to the destination first, or accept. Within the same Type (between Page Collections, or between Item Collections), no strip — schema is shared.

Cross-side promotion (Page ↔ Item) is not supported at v0.3.0 — see [[Prospects]]. **Ships v0.3.0** (pulled forward from the original v0.4.0 plan; coupled to schema mutations).

#### Sort by property

v0.3.0 ships single-criterion sort in the Pages-side detail-pane Table view. Click a column header to sort; click again to reverse. Type-aware comparators:

- **Number** — numeric.
- **Checkbox** — false-first vs true-first.
- **Date / Date & Time** — chronological.
- **Last Edited Time** — chronological; **descending is the v0.3.0 default sort** for all Type Tables.
- **Select / Multi-select** — schema option order.
- **Status** — group position first (Upcoming < In Progress < Done), then option order within group.
- **URL** — alphabetical on the absolute string.
- **Relation** — alphabetical on the resolved current title of the target.

##### Per-Type default sort

Persisted on the parent Type's per-kind sidecar (`_pagetype.json` / `_itemtype.json` / `_taskconfig.json` / `_eventconfig.json`) — one default per Page Type / Item Type / AgendaTask schema / AgendaEvent schema. Full per-view sort and multi-criterion sort with saved-view configs land at v0.6.0.

##### Hidden-property-used-for-sort = auto-show — deferred

This rule only applies when `hidden_properties` (per-Type column visibility) exists. Deferred to whenever the detail-pane view shape gets reimagined and brings column visibility back online.

#### Schema mutations

Five mutations supported at v0.3.0 on every Type's per-kind sidecar (`_pagetype.json` / `_itemtype.json` / `_taskconfig.json` / `_eventconfig.json`):

| Mutation | Effect on existing values |
|---|---|
| **Add a property** | Appears empty on every member; no member writes until a value is set. |
| **Rename a property** | Transactional rewrite: schema entry plus every member file in the Type's tree (root + every Collection). Values preserved under the new key. |
| **Change a property's type** | **Lossless only** — Date → Date & Time, Select → Multi-select. Lossy conversions require explicit confirmation and drop conflicting values. |
| **Delete a property** | Schema row removed; values removed from every member. No quarantine. |
| **Reorder properties** | Schema-only (members are dictionary-keyed); affects property panel order. |

##### Editing Select / Multi-select / Status options

Adding / reordering / renaming option labels is schema-only. Deleting an option voids member references per the universal void-on-delete rule above.

##### Atomicity requirement

Schema mutations that touch multiple files (rename / type-change / delete / paired-relation create / paired-relation delete) **must execute as a two-phase commit**: write all new content to `.tmp-<uuid>` siblings, then atomically rename in batch; on any rename failure, complete rollback across every affected file. A best-effort transaction log under `.nexus/` handles crash-mid-commit recovery; v0.3.3 SQLite adds launch-time divergence detection. The specific transaction primitive (its class shape, parameter types, internal phases) is an implementation concern; the requirement is that the commit boundary is atomic across all affected files and that orphan `.tmp-` siblings are cleaned up on next launch.

##### Validation

Enforced at every schema write and every member-file write:

**Schema-level**

1. Property name unique within the Type (case-insensitive), non-empty, no reserved characters (`/`, `.`, leading underscore).
2. Reserved property names (universal): `id`, `created_at`, `modified_at`, `tier1`, `tier2`, `tier3`, `wikilinks`. AgendaTask additionally reserves `status` (built-in, non-deletable).
3. Dual relation requires `page_type` / `item_type` / `page_collection` / `item_collection` scope; `context_tier` scope rejects dual.
4. Relation scope's target ULID must resolve to a live entity at save time, and must be same-side (cross-side guard).
5. Select / Multi-select require at least one option; option `value` unique within the property.
6. Built-in `status` on the AgendaTask schema is non-deletable; not auto-seeded on the AgendaEvent schema, Page Types, or Item Types. (Users may add a Status property manually to any of those — only the auto-seeded AgendaTask `status` is reserved.)

**Value-level**

1. Every value's shape matches its schema entry's type.
2. Relation `$rel` ULIDs must resolve to a live entity (warned, not enforced — broken-link semantics).
3. Select / Multi-select / Status values must reference live option `value`s (cleaned up on schema mutation).

#### Per-Type schema editor (Type Settings sheet)

Central edit surface per Type, reached via:

- The Type detail view's toolbar gear button.
- The Type row's right-click → "<Label> Settings…" (UI label per side: "Vault Settings…" on Page Types by default; "Type Settings…" on Item Types by default — both renameable via the Settings scaffold).
- "**+ Property**" column header in the detail-pane Table view (jumps to Edit Properties + the Add Property flow).
- Column header right-click in the detail-pane Table → "Edit property…" (jumps to the relevant row).

##### Minimum-viable sections at v0.3.0 (placeholder UI)

| Section | v0.3.0 status |
|---|---|
| Edit Properties | Functional (add / rename / type-change / delete / reorder / icon / per-type config). |
| Sort | Functional (per-Type default, single-criterion). |

That's it for v0.3.0 placeholder UI. Property Visibility, Filter, Group By, Layout, Templates sections are **not rendered at v0.3.0** — they ship with the real Type Settings sheet redesign in a later v0.3.x patch. Per-Type column visibility for the detail-pane view comes back online when the detail-pane view shape gets reimagined (v0.6.0 alongside the five view types).

##### Commit boundary

Each Type Settings sheet is a **batched edit** — edits stage into a draft; commit only on sheet Save. Cancel discards. The sheet's flush wraps the staged mutations in a single atomic transaction. The right-click "Add option…" popover is a **separate** commit boundary with its own Save (Notion-parity carved-out path).

##### Concurrent open forbidden

A given Type's Settings sheet can be open in at most one window at a time. A second invocation shows a disabled state ("Settings already open in <window title>").

##### Parallel Pages and Items surfaces

Two parallel sheets — one on the Pages side (over Page Types) and one on the Items side (over Item Types). UI label strings read from the Settings scaffold; default labels match the ParadigmV2-locked defaults. Both sheets ship at v0.3.0 (resolved in plan D1) so Item Type property editing works end-to-end via right-click → Type Settings the moment v0.3.0 ships. The Items-side detail-pane Table column UI is the only piece gated behind the v0.3.1 Items-pane plan; the sheet is not.

#### Per-entity property panel visibility — deferred

`panel_hidden_properties` was originally scoped as v0.3.0 data scaffolding. **Deferred** under the new direction: the lazy-properties model used by the Properties Pulldown (populated-only + "+ Add property" picker) auto-hides empty schema entries, which subsumes most of the use case. Explicit per-entity hide-when-populated may return as a follow-up if needed; the data field doesn't ship at v0.3.0.

Per-Type column visibility (`<Type>.hidden_properties: [String]`) also doesn't ship at v0.3.0 — comes back online when the detail-pane view shape gets reimagined.

#### Auto-managed properties

On every Page, Item, AgendaTask, and AgendaEvent — not user-creatable:

- `id` — ULID assigned at creation, never changes.
- `created_at`, `modified_at` — ISO-8601 timestamps maintained by Pommora.

`id` and `created_at` appear at the bottom of whichever property surface shows them (pulldown / inspector panel / Item Window), collapsed by default. `modified_at` is exposed alongside user-defined properties as the **Last Edited Time** property for sortability — same underlying value, two surfacings.

Items, AgendaTask, and AgendaEvent additionally carry one built-in field that is not a property:

- `description` — plain text, hard cap 250 characters. Not Markdown, not property-editable; rendered alongside the title in views.

(Filename plays the title role; there is no `name` field on any of these kinds.)

##### `modified_at` trigger semantics

Updates on **any content or frontmatter edit** — body, property value, title rename, icon, tier1/2/3, panel-visibility toggle. View-only actions never update. External edits update file mtime but do NOT update frontmatter `modified_at` until v0.3.3 watcher closes the gap.

#### Settings scaffold integration

UI label strings throughout the Properties surface (sheet titles, picker headings, add-button labels, section headings) read from the Settings scaffold introduced in ParadigmV2 Phase 7 — not hardcoded. When the user renames "Set" to "Library" in Settings, the Items-side picker title updates everywhere. v0.3.0 ships the storage + label wiring; the Settings editing UI lands v0.6.0. Default labels match the ParadigmV2-locked defaults ([[CLAUDE]] § Overview, [[Domain-Model]]).

#### Type templates rejected; content templates reserved

Page Type and Item Type creation seeds **no default properties** at v0.3.0 (locked RC-2026-05-19; supersedes earlier toggle proposals). Users add properties manually. The only exception is the AgendaTask schema's built-in Status (EventKit-required).

Notion-style **content-level templates** (Page / Item templates pre-filling body + property values at creation) are reserved for post-v1. v0.3.0 keeps the data scaffold compatible: `<nexus>/.nexus/templates/` is reserved (empty) and the Type Settings sheet's 7th section is a placeholder anchor. No manager-method `template:` parameter is added until templates ship — the placeholder preserves the wiring concept without dead code. Full reservation lives in [[Prospects]].

#### Locked decisions (conceptual)

Numbered for cross-reference. Each survives any code reshape; the implementation detail is deferred to the fresh post-ParadigmV2 plan.

1. **Last Edited Time = auto-tracked, first-class, sortable.** Read-only, derived from `modified_at`. Visible alongside user-defined properties in whichever surface shows them (pulldown / inspector panel / Item Window). Default sort descending v0.3.0.
2. **Number included.** Full Notion parity for the catalog.
3. **Relation scope = Notion-style single target.** Each Relation targets one container at creation: one Page Type, Item Type, Page Collection, Item Collection, or Context tier. Multiple containers → multiple Relation properties. No `anywhere` fallback.
4. **Dual relations mandatory for Page-Type / Item-Type / Page-Collection / Item-Collection scopes.** No opt-out; user provides both names at creation; mirrored values stay synced. Cross-side (Item ↔ Page) NOT supported at v0.3.0. Context-tier stays one-way (Contexts have no `properties[]` schema). Collection-scoped reverses are stored on the parent Type's per-kind sidecar (`_pagetype.json` / `_itemtype.json`).
5. **Property names are the key.** Renaming triggers a transactional cross-member rewrite. Stable opaque IDs were considered and rejected — they violate the agent-legibility constraint on frontmatter.
6. **Wikilinks are NOT a property type.** Body-text feature with a derived `wikilinks: [...]` frontmatter mirror. Implementation v0.3.2.
7. **Status is a new, distinct property type with EventKit-aligned groups.** Three fixed structural slots (`upcoming` / `in_progress` / `done`); per-group user-editable options; group labels renameable; group slots fixed. Standard option in the type picker — not featured.
8. **Multi-select pills render per-option color.** Same 9-color palette as Select.
9. **No Type templates at creation.** Page Type / Item Type creation seeds no default properties. Content-level templates reserved post-v1.
10. **Schema CRUD coverage explicit.** The v0.3.0 schema editor supports add / rename / type-change / delete / reorder for every property type, including Status (operating on options and group labels, not the structural slots).
11. **Property icons.** Optional SF Symbol per property. Schema-additive; no migration.
12. **Type Settings = central edit surface; v0.3.0 ships minimum-viable form.** v0.3.0 placeholder UI: Edit Properties + Sort sections only. The full seven-section design (adding Property Visibility / Filter / Group By / Layout / Templates) ships with the real Type Settings sheet redesign in a later v0.3.x patch.
13. **Collection picker UX = two-step Type → Collection in the schema editor; Collection-grouped value picker at value time.** Avoids flat nexus-wide lists.
14. **No inline option creation in value pickers; structured inline-add from property LABELS.** Three commit paths (Type Settings, label-right-click popover, value-right-click → routes to Type Settings). Value pickers are read-only over the option set.
15. **Type Settings sheet = save-required + concurrent-disabled.** Edits stage into a draft, committed on sheet Save inside an atomic transaction. Cancel discards. The inline label-right-click "Add option…" popover is a separate commit boundary with its own Save.
16. **`modified_at` trigger semantics.** Any content or frontmatter edit updates it. View-only actions never update. External edits update file mtime but not frontmatter `modified_at` until v0.3.3 watcher closes the gap.
17. **Hidden-property-used-for-sort = auto-show.** Sort precedence beats visibility.
18. **Per-entity property panel visibility deferred from v0.3.0.** `panel_hidden_properties` doesn't ship — lazy-properties UI in the Properties Pulldown auto-hides empties, subsuming the use case. Revisit if explicit hide-when-populated becomes needed.
19. **Universal void-on-delete for property options.** Multi-select shrinks the array; all others null the value. No quarantine. Confirmation dialog lists affected count.
20. **Type Settings "Templates" 7th section = placeholder + unwired.** UI anchor for future content templates; no manager-method parameter added until templates actually ship.

##### Decisions added 2026-05-23 brainstorm (Properties Pulldown direction)

21. **"+ Add property" picker scope: existing schema properties only (v0.3.0).** The picker over the parent Type's schema lists properties NOT yet populated on this entity. Selecting one populates it (with empty / default value, ready to edit). Adding a brand-new schema property requires opening Type Settings → Edit Properties. Future updates may surface inline schema-create from the picker.
22. **Property panel order persists per-Type, not per-entity.** Drag-reorder in any surface (pulldown / inspector panel) writes to the parent Type's per-kind sidecar declaration order — affects every entity of that Type. Per-entity overrides not in v0.3.0.
23. **Empty property surface renders an explicit "No properties" state.** When a page / item / task / event has zero populated properties, the pulldown / inspector panel renders an empty-state message ("No properties") + "+ Add property" affordance. Surface stays visible — never collapses to invisible.
24. **Pinning interaction: right-click property row → "Pin Property".** From any property surface (inspector panel / future Item Window). Right-click chip → "Unpin Property" to remove. Per spec § Surface architecture, pinned chips save at the Item Collection level.
25. **Status property is universal across all schemas — silently informational off-Agenda.** Users may add Status to PageType / ItemType / AgendaEvent manually. EventKit sync only applies to AgendaTask's built-in Status. UI does NOT signal EventKit non-relevance on non-Agenda Types — Status is just a data shape there, agent-readable ("AI can still read `status` on an item or page and understand it"). Future updates may extend EventKit bridging to Items / Pages if useful.
26. **Validation feedback is live (red-border on invalid input).** Matches the live-save model. Invalid values get rendered with a red border as the user types; failed saves silently revert; recovery happens on next valid keystroke. No on-blur or on-save-only validation gating.
