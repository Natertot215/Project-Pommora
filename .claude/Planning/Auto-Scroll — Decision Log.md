## App-Wide Auto-Scroll on Drag — Decision Log

*Status: RATIFIED. Two adversarial review rounds run — round 1 reshaped the design (fixed-scroller reversal), round 2 verified it (0 High) + 3 clean folds. No open decisions. Ready for planning.*

### Frame

- **Purpose:** one app-wide auto-scroll-on-drag engine that every drag source feeds, replacing today's duplicated per-surface copies — token-driven so an `autoscroll.css` is the single tunable source.
- **Core Value:** dragging anything near a scroll container's edge scrolls that container, on every surface, from one mechanism.
- **Success Criteria:** a drag held near an edge scrolls smoothly (frame-synced, ramped, limit-aware) so an off-screen target becomes reachable; the same module serves sidebar · MarkdownPM (block drag) · table (rows/columns) · SurfacePM · settings panes · the PommoraDND engine; params live in tokens, not scattered JS literals; no second copy remains.

### Sources

- [[PommoraDND]] — the in-house drag engine + its "Auto-scroll (`autoscroll.ts`) — rAF loop, nearest scrollable ancestor, ease-in ramp, limit-aware; nearest ancestor only, not the page/window." The doc says the sort *engine* isn't app-consumed yet (stale re: autoscroll.ts specifically — see below).
- [[Interaction]] — motion law; §"Drag Motion" names a "quadratic edge-proximity auto-scroll ramp (`interactions/autoscroll.ts`, rAF)" for the engine AND a separate "edge auto-scroll" for the editor block-drag (`editor/blockDrag.ts`). Tokens: `motion.ts → theme-vars.css.ts`; auto-scroll constants are NOT tokenized (JS literals).
- `design-system/interactions/autoscroll.ts` — the existing engine module (42 lines): pure `findScroller(el)` (nearest scrollable ancestor, walks UP from `el`, both axes, real-overflow check) + `autoScroll(scroller, x, y)` (`EDGE=48`px, `MAX=14`px/frame, quadratic `MAX*p²` ramp, zeroes a maxed-out edge). No rAF loop of its own — the engine's loop calls it each frame.
- `design-system/interactions/autoscroll.ts` consumers TODAY (review L1 — precise): `SurfacePM/SurfaceView.tsx` + `Components/Detail/paneDnd.tsx` call `autoScroll` INLINE-per-pointermove on LIVE surfaces (the math is proven, the loop is not); `engine.tsx` imports it into the rAF-loop `Zone`, but `Zone` has zero call sites (unmounted). So: math live on 2 surfaces, loop live on 0.
- `MarkdownPM/editor/blockDrag.ts` — a SECOND, independent auto-scroll: same `EDGE=48`, its own linear `edgeStep` (max 14), y-axis only, its own rAF loop (`g.raf`), scrolls CM6's `view.scrollDOM` (`.cm-scroller`), and re-measures drop candidates on the `scroll` event (CM6 only renders ~viewport, so far targets don't exist until scrolled in). The duplication to collapse.
- `styles.css:67-78` — `.shell` is `overflow: clip`: the app shell is deliberately NOT a scroll container. No page/window scroller exists → "scroll the screen" = scroll the inner container under the pointer.
- Scroll containers (the auto-scroll targets): `.sidebar` (y), `.detail` / `.detail-scroll` (y, the main content scroller — also what a SurfacePM block drag scrolls, since the surface host never scrolls itself), `.cm-scroller` (CM6-internal, y — the editor), `.table-view` (x only; table ROW vertical scroll comes from the enclosing `.detail-scroll`), menu `scrollFrameBody` (y). `.detail-page`, `.mdpm-shell`, inspector root are `overflow:hidden` (non-scrollers).
- Token system: vanilla-extract `.css.ts` authored in `design-system/tokens/`; `theme-vars.css.ts` bridges them to `:root` CSS vars via `globalStyle(':root', {vars})` (motion tokens at lines 96-103). Plain `.css` consumes `var(--…)`. Per-feature scoped-var precedent: `Detail/Views/Table/table-tokens.css` (scoped to `.table-view`, its own tunable numbers, imported in `main.tsx`). Global plain-CSS `:root` vars live in `styles.css` + `interactions.css`; the one top-level design-system plain CSS is `scroll-edge-fade.css`. A new `autoscroll.css` registers as a side-effect import in `main.tsx` (order matters — after the token bridge).

- **Drag-system census (13 mechanisms, ALL Pointer Events + `setPointerCapture`, ZERO native HTML5 DnD).** Auto-scrolls today via the shared `autoscroll.ts`: PommoraDND single-zone (`engine.tsx`, a real rAF loop — the reference), SurfacePM block move (`SurfaceView.tsx`), settings pane reorder (`paneDnd.tsx`). Own DUPLICATE: MarkdownPM block drag (`editor/blockDrag.ts`, same 48/14 constants, scrolls CM6 `view.scrollDOM`, re-measures on its own `scroll` event). NONE (retrofit targets): Sidebar (`sidebarDnd.tsx`), Table row (`tableDnd.tsx`), Table band (`bandDnd.tsx`), Table column reorder (`TableView.tsx startColumnDrag`, horizontal), MarkdownPM list drag (`editor/listDrag.ts`), MarkdownPM GFM-table drag (`Tables/TableView.tsx`), grouping pane (`groupingDnd.tsx`), the cross-list board (`group.tsx`). Two families: engine/sensor drags with a `drag` ref (`.active`/`.lastX/Y`), and window-listener drags that already hold the pointer in every `pointermove`. CM6 drags (block/list) operate on `view.scrollDOM` (not a React ref) — the module needs an explicit-scroller override, not always `findScroller`.

- **External research (native + prior art).** Native edge-scroll exists ONLY for HTML5 DnD (`dragover`), which Pommora doesn't use — pointer-capture drags get NOTHING free from Chromium/Electron. Hand-build. The canonical pattern every lib (dnd-kit, rbd/@hello-pangea, Atlassian pragmatic, dom-autoscroller) converges on: a rAF loop **independent of `pointermove`** (move records coords ONLY; the loop scrolls every frame — so holding still at the edge keeps scrolling), find the container via `document.elementsFromPoint(x,y)` (plural → nested stack) walked innermost→outer, falling through to the next ancestor when the inner is maxed (natural inner-then-outer chaining). `setPointerCapture` does NOT break `elementsFromPoint` (hit-test by coords, never `event.target`). Two UX wins to steal: **time-dampening** (speed ramps from ~0 over the first ~300ms of a drag — rbd/pragmatic; without it, entering the zone slams to max and feels violent) and **direction-intent** (don't scroll a direction until the user has dragged that way — dnd-kit; kills the "grabbed an item near the bottom → page rockets" bug). Frame-rate independence: measure speed in px/**second** × `dt` (rAF timestamp), not px/frame — px/frame double-speeds on 120Hz ProMotion. Guard sub-pixel: accumulate the fractional remainder so slow ramps don't round to 0.

### Decisions

#### A — Build vs Extract
- **A-1:** [confirmed] Extract + generalize `autoscroll.ts` into a standalone, drag-agnostic module — don't build fresh. Honest framing (per review M3): the ramp + limit-awareness MATH is proven (`autoScroll` runs inline-per-move on 2 LIVE surfaces — `SurfaceView.tsx:432`, `paneDnd.tsx:181`); the rAF-**loop-owns-the-scroll** lifecycle is NEW — its only home, `engine.tsx`'s `Zone`, has zero call sites (unmounted/dead code). So the loop deserves real test scrutiny; "app-proven" covers the math, not the lifecycle.

#### E — The loop owns the scroll (not per-move inline)
- **E-1:** [confirmed] The module OWNS a singleton rAF loop that scrolls every frame off the last-recorded point; a drag's `pointermove` only feeds the point. The `paneDnd` inline-per-move pattern is INCOMPLETE — it stops scrolling when the pointer holds still at the edge (no move → no call). Fixing that (and killing the block-drag duplicate) is the DRY payoff, so `paneDnd` + the engine + SurfacePM migrate onto the module too.

#### B — Which scroller (REVISED post-review — fixed scroller, resolved once)
- **B-1:** [confirmed] Resolve the scroller ONCE at drag start and keep it for the whole drag — NOT per-frame under the pointer. Rationale: **no CORE drag crosses scroll containers** (each lives in exactly one — sidebar in `.sidebar`, table rows/bands in `.detail-scroll`, block in `.cm-scroller`, SurfacePM/panes in `.detail-scroll`/the pane; verified round 2). The ONE exception is the cross-list board (`group.tsx`, a Prospect) which resolves the zone under the pointer per-move — it alone would need the rejected per-frame path, so promoting it is NOT a "feed the point" one-liner (see D-3). The rAF loop computes edge-proximity from that fixed scroller's rect vs. the raw pointer (the existing `autoScroll(scroller,x,y)` signature) — a pointer past the viewport edge just reads as max ramp, no lookup. This dissolves the review's H1 (`elementsFromPoint` returns `[]` for out-of-viewport coords → loop would stall exactly when you overshoot and hold), M1, M2, and L2 at once. `document.elementsFromPoint` + chain-to-outer was imported from general libraries (dnd-kit handles arbitrary nested scrollers); Pommora doesn't need it.
- **B-2:** [confirmed] `findScroller` becomes AXIS-AWARE — `findScroller(el, axis)` returns the nearest ancestor scrollable IN THE NEEDED AXIS. Fixes the real bug (review L2 + verified): a vertical table-row drag's nearest either-axis scrollable is `.table-view` (`overflow-x:auto`, x-only) whenever columns overflow, so the current `findScroller` would resolve the wrong container and never scroll `.detail-scroll` (y). Each drag passes its axis(es); SurfacePM passes both.
- **B-3:** [confirmed] Two ways to supply the scroller to `start(getPoint, {scroller?, axis?})`: (a) an EXPLICIT scroller — required for CM drags (`view.scrollDOM`, since `findScroller` won't derive it) AND for every consumer that folds scroll-delta into its OWN pointer math (`engine.tsx track()`, `SurfaceView.tsx:433-436`), so the module scrolls the SAME container the consumer compensates against (review M1); or (b) `findScroller(dragEl, axis)` when the drag doesn't self-compensate (sidebar, table row/band — they re-measure on scroll instead). Consumers keep their own scroll-event re-measure (scrolling `.cm-scroller` fires CM's `scroll` → block-drag re-measures, unchanged).

#### C — Token surface
- **C-1:** [confirmed] Params live in a new `design-system/interactions/autoscroll.css` (`:root` vars, registered in `main.tsx` after the token bridge), read via `getComputedStyle(DRAG-ELEMENT).getPropertyValue()` ONCE at drag start + cached — never per frame (`getComputedStyle` forces layout; the "never on every X" rule).
- **C-2:** [confirmed] Per-surface override seam: read the tokens from the **drag element**, NOT the scroller (review round-2 finding 1). CSS custom properties inherit downward only, and for table/band/SurfacePM the scroller is the shared `.detail-scroll` — an ancestor the surface doesn't own — so an override set on the surface's own element (a descendant) would be invisible if read off the scroller, and `.detail-scroll` is shared so can't be scoped either. The drag element inherits the `:root` global AND any `--autoscroll-*` override set on itself or an ancestor (e.g. `.sidebar { --autoscroll-speed: … }`) — exactly where a per-surface override lives. Ship ONE global default; tune per-surface only when one proves it needs it (Prospect).

#### F — Feel / correctness (research-backed)
- **F-1:** [confirmed] Speed authored in px/**second**, applied `× dt` from the rAF timestamp (not px/frame) — px/frame double-speeds on 120Hz ProMotion. Accumulate the sub-pixel remainder so slow ramps don't round to 0.
- **F-2:** [confirmed — Nathan chose Polished] Include BOTH prior-art UX wins: **time-dampening** (speed ramps from ~0 over `--autoscroll-dampen-ms` from drag start) + **direction-intent** (a direction doesn't scroll until the user has dragged that way). Intent is baked on (a correctness behavior, not a tuning value) — it doesn't get a token.
- **F-3:** [confirmed] Token set (numeric, in `autoscroll.css` `:root`, per-surface-overridable): `--autoscroll-edge` (edge-zone band, px) · `--autoscroll-speed` (px/**sec** at the true edge) · `--autoscroll-ramp` (proximity ramp exponent; 2 = the current quadratic) · `--autoscroll-dampen-ms` (time-dampening window). Flat-px edge (not %-of-container) is the simpler sufficient default given Pommora's scrollers are all reasonably sized; a %-with-px-cap edge is a Prospect if a surface ever needs it.

#### D — Per-surface wiring
- **D-1:** [confirmed — Nathan chose Recommended core; listDrag demoted post-review] Core surface set. Every drag calls `start(getPoint, opts)`/`stop()`; the module owns the loop. **Core:** the module + Sidebar (`sidebarDnd.tsx`) + Table rows (`tableDnd.tsx`) + Table bands (`bandDnd.tsx`) + MarkdownPM block (`editor/blockDrag.ts` — migrate off + DELETE its dup loop; it ALREADY has the scroll→re-measure path auto-scroll needs) + migrate the 3 existing consumers (`engine.tsx`, `SurfaceView.tsx`, `paneDnd.tsx`) onto the module. Window-listener drags feed the point from their existing `pointermove`; the engine feeds from its `drag` ref; CM/compensating drags pass an explicit scroller (B-3).
- **D-2:** [confirmed] MarkdownPM **list drag** (`editor/listDrag.ts`) → demoted from Core to Prospect (review H2, verified `listDrag.ts:155`): it measures drop candidates ONCE at activation in viewport coords with NO scroll listener. Auto-scrolling it would leave the insertion line pointing at stale gaps and make scrolled-in items un-targetable. It needs `blockDrag`'s scroll→re-measure path first — its own scoped task, not a "feed the point" one-liner. Symmetric-with-blockDrag was the wrong read.
- **D-3:** [confirmed] **Prospects:** MarkdownPM list drag (D-2, needs the block-drag re-measure path) · Table column-reorder horizontal · MarkdownPM GFM-table drag (`Tables/TableView.tsx`) · grouping pane (`groupingDnd.tsx`) · the cross-list board (`group.tsx` — **architecturally distinct**: its per-move `zoneAt` crosses scrollers, so it alone needs per-frame scroller resolution reintroduced, NOT the fixed-scroller path; do not scope it as a simple retrofit). Each Prospect names its real cost, not "feed the point later."

- **G-1:** [confirmed] [[PommoraDND]] §"Constraints, Auto-Scroll, Accessibility" says auto-scroll is `autoscroll.ts`, "nearest scrollable ancestor … not the page/window" — becomes false (the module is now app-wide, under-pointer via `elementsFromPoint`, token-driven, dampened). - **G-2:** [confirmed] [[Interaction]] §"Drag Motion" cites the engine's "quadratic edge-proximity auto-scroll ramp (`autoscroll.ts`)" AND the editor block-drag's separate "edge auto-scroll (`blockDrag.ts`)" — the second is deleted; restate as one shared engine. §"Duration Inventory" lists `autoscroll.ts:5-6` `EDGE=48`/`MAX=14` literals as "engine timing, keep local" — those become tokens; update that entry.

### Core (must-have)

- One standalone module: singleton shared rAF loop + `start(getPoint, {scroller?, axis?})/stop()` + FIXED scroller resolved once at start (explicit, or axis-aware `findScroller(dragEl, axis)`) + token-read params (cached at start) + px/sec × dt + sub-pixel accumulation + time-dampening + direction-intent. Edge-proximity from the fixed scroller's rect (raw pointer, no viewport clamp needed — past-the-edge = max ramp).
- **The module OWNS a defensive backstop for the LOOP ONLY** (review H3 + round-2 finding 2): `start()` installs a `blur` / `visibilitychange` / `pointercancel` fallback that calls `stop()` — which stops the auto-scroll LOOP, nothing else. This is a backstop against a leaked rAF loop (a cmd-tab / OS-modal focus-steal mid-drag would otherwise scroll a container forever, since per-surface abort sets are uneven — only `blockDrag` has a blur handler; sidebar/table/band lack blur + `lostpointercapture`). **It does NOT abort the surface's drag** — each surface stays fully responsible for aborting its OWN gesture (keep `blockDrag`'s `blur → finish(false)` etc.; do not delete them). Each surface also calls `stop()` on its normal up/cancel; the module's fallback is purely the loop's safety net.
- Singleton: `start()` while a loop runs replaces it (pointer capture means one drag at a time; be defensive).
- Wired into the recommended core surfaces (D-1); the block-drag duplicate deleted and the 3 existing consumers migrated — NO second copy remains.
- Params in `autoscroll.css` tokens.

#### Prospects (allowed later)

- The D-1 Prospect surfaces (column horizontal, GFM-table, grouping pane, board).
- Per-surface override VALUES (seam shipped, values tuned only when a surface demands it).
- Non-DOM scroll target via a registry `scroll(dx,dy)` callback (a future canvas that pans its own viewport) — not needed now (SurfacePM scrolls a real DOM ancestor).

#### Out of Scope

- Native HTML5 DnD anything — Pommora is 100% pointer-capture drags; native `dragover` edge-scroll never applies.

#### Considered & Rejected

- **Per-move inline `autoScroll` (the current `paneDnd` pattern), no shared loop** — stops scrolling when the pointer holds still at the edge (`pointermove` doesn't fire when stationary). The rAF-owns-the-scroll model (E-1) is why every library runs a loop.
- **A global "active-drag" React context + one always-on app-level rAF loop** — over-machinery: drags are self-contained today; an explicit `start/stop` module call is simpler and just as DRY.
- **Per-frame "scroller under the pointer" via `document.elementsFromPoint` + chain-to-outer (the initial B-1/B-2, from dnd-kit/general-library prior art)** — rejected after the adversarial review: it stalls the moment the pointer leaves the viewport (`elementsFromPoint` → `[]`), needs viewport-clamp + per-scroller token-cache-keying + broke scroll-compensation, all to solve nested-scroller crossing that **no Pommora drag actually does**. Replaced by the fixed-scroller-at-start model (B-1 revised), which is simpler AND the already-proven pattern.

#### Lessons

- **Prior-art patterns carry assumptions that may not be yours.** The research's "resolve the scroller under the pointer each frame" is correct for a general DnD library (arbitrary nested scrollers) but over-built for Pommora, where every CORE drag lives in one container — a fixed scroller resolved once is simpler and dodges a class of bugs (viewport-exit stall, token-cache incoherence). Borrow the algorithm's math, not its generality. (The lone crosser, the board, stays a Prospect precisely because it'd need that generality back.)
- **A latent gap becomes an active leak when you add a persistent loop.** Uneven per-surface abort handling (missing `blur`/`lostpointercapture`) is harmless with inline-per-move scrolling but strands a singleton rAF loop on a focus-steal. A shared always-running mechanism must own its own termination, not inherit each caller's.
