### Plan: v0.2.7 — Strip Pallepadehat fork; rewire to swift-markdown (parser) + swift-markdown-engine (live-preview foundation)

#### Context

Pommora's v0.2.7 Page editor shipped through Phase G on the Pallepadehat fork (`Natertot215/PageEditorMD@addaa23` — CodeMirror 6 inside WKWebView). Phase G fortified the visual baseline (Apple typography, markdown auto-pair, transparent bg) but the smoke test on `main@1989fac` confirmed the WKWebView path can't deliver the Notion/Obsidian-style native feel Pommora's principles demand. Two pivots followed: first to Milkdown+Crepe (also WKWebView; documentation prepared at `// Planning//v0.2.7-milkdown-swap.md`), then — after Nathan saw the swift-markdown-engine demo render natively on his Mac — to a TextKit-2 native architecture.

The locked architecture: **Apple's swift-markdown is the default for parsing AND for rendering — Apple primitives (NSAttributedString, NSTextView, NSTextLayoutManager) do all the visual styling. swift-markdown-engine is used ONLY for the live-preview substrate Apple doesn't ship: NSTextView subclass UX polish (caret, drag-select, bottom overscroll, paste), custom NSTextLayoutFragment for inline image rendering, paragraph-scoped per-keystroke restyle loop. Engine = chassis. Apple = engine.** Nathan-locked rationale: trust Apple's text preview + UI integration over a third-party design.

**The two load-bearing engine contributions** (the reasons we use it at all, not just NSTextView raw):

1. **Dynamic syntax** — markers (`**`, `__`, `[[`, `` ` ``, `#`, `$`, table pipes, blockquote `>`, list bullets) **shrink to a near-invisible inline marker font when the caret leaves the node** and **expand to full size when the caret enters**. This is the live-preview pattern Bear / Notion / iA Writer use; users see formatted output on inactive paragraphs but full source on the line they're editing. Apple's bare NSTextView doesn't ship this — it's editor-grade UX that must be implemented as a per-keystroke restyle that maps caret position to AST node containment. The engine's `shrinkInactiveMarkers` pattern (in the deleted `MarkdownStyler`) is the proof-of-concept; **`PommoraMarkdownStyler.MarkersShrinker` reimplements it on top of Apple's AST**, applied via NSAttributedString attributes — Apple-rendered, engine-patterned.

2. **Markdown-aware typing helpers** — three behaviors typing-time, all flowing through the engine's `Input/` surface (the `NSTextViewDelegate.textView(_:shouldChangeTextIn:replacementString:)` slot):
   - **List continuation** (engine ships, [MarkdownListHandler.swift]): hit Enter at end of `- item` → next line is `- ` (or `1.` → `2.` for ordered, including indent + checkbox preservation). KEEP unchanged after porting tokenizer dependency.
   - **Block-token auto-wrap** (engine ships, [MarkdownInputHandler.swift]): typing adjacent to `$$...$$` or `![[...]]` auto-inserts newlines so the block stays on its own line. KEEP after porting from `MarkdownTokenizer.parseTokens(in:)` to Apple AST queries (`Document.descendants(of: BlockDirective.self)`-style walks).
   - **Character-pair auto-pairing** (ADD in Phase 4.5, doesn't ship in engine): typing `**` → `**\|**` with caret between; `[[` → `[[\|]]`; `` `` `` → `` ``|`` ``; `__` → `__\|__`; selection-wrap when text is selected. This is the Phase G.4 CodeMirror work, freshly ported to NSTextViewDelegate. The engine's `Input/` surface is the slot we extend; Apple's bare NSTextView gives us nothing here.

Everything else the engine provides (drag-select boost, bottom overscroll, paste handling, caret workarounds, custom NSTextLayoutFragment for inline image rendering) is editor-polish gravy — nice to keep but not the load-bearing reason. **If we strip the engine to its essence: dynamic syntax + the Markdown-aware typing-helper PATTERN are why it exists in Pommora. The actual character-pair logic gets added on top.**

Apple's `Document(parsing:)` yields a full GFM AST covering Heading, Strong, Emphasis, Strikethrough, BlockQuote, Table (+ Head/Body/Row/Cell), ThematicBreak, CodeBlock, InlineCode, Link, OrderedList, UnorderedList, ListItem, Image, LineBreak, SoftBreak, HTMLBlock, BlockDirective — every block type Pommora needs for v0.2.7 + every directive Pommora will need in v0.2.9. Apple's `NSAttributedString` + `NSTextView` + `NSTextLayoutManager` do the actual rendering (font, color, paragraph styling, link rendering, selection, find — all via Apple's native APIs).

**Scoping discovery (Phase 1 verification):** the engine's `MarkdownToken` type is NOT internal-to-styling — it's the load-bearing data abstraction the entire coordinator architecture reaches through. 11 non-styling files reference `MarkdownToken`/`MarkdownTokenizer`/`MarkdownDetection`: every coordinator extension (`+Restyling`, `+TextDelegate`, `+CodeBlocks`, `+Autocorrect`, `+InlineSelection`), the main `NativeTextViewCoordinator`, `Input/MarkdownInputHandler`, `Input/MarkdownListHandler`, `TextView/ContextMenu`, `TextView/NativeTextView/+SpellingPolicy`. They use these types for: cached selection state (wikiLink / imageEmbed), code-block-aware autocorrect suppression, inside-code/latex spell-policy, context-menu bold/italic toggle, paragraph-scoped restyle re-trigger when a token's active state flips. **Wholesale-deleting the types would force rewriting all 11 files** — losing the engine's coordinator polish in the process.

**Correct architecture: preserve types, replace implementations.** Keep `MarkdownToken`, `MarkdownTokenKind`, `MarkdownTokenizer.parseTokens(in:)`, `MarkdownDetection.isInside…`/`computeActiveTokenIndices` as PUBLIC API. **Rewrite their internals** to back onto Apple AST: `MarkdownTokenizer.parseTokens(in:)` walks `Document(parsing: text)` and emits `[MarkdownToken]` shims (one shim per AST node Pommora cares about — `Strong → .bold`, `Emphasis → .italic`, `CodeBlock → .codeBlock`, `Link → .link`, etc., plus a supplemental scan for wikilinks/image-embeds). `MarkdownDetection` helpers query the same `[MarkdownToken]` array. `MarkdownStyler.styleAttributes` is the ONE exception — fully replace its body with `PommoraMarkdownStyler` that builds attributes from Apple AST + iterates Pommora-side token shims for inactive-marker shrinking. Net effect: every coordinator extension keeps working unchanged; the parser's regex internals get wholesale-replaced with Apple AST; the styler's regex internals get wholesale-replaced with AST-driven attribute construction.

This rewires every part of the v0.2.7 editor surface without touching any v0.2.7 domain code. All 198 tests, the inspector, the sidebar wiring, the page-switch flush, the atomic-write contract, and the frontmatter preservation rule all survive unchanged because they target the `PageEditorViewModel` ↔ `ContentManager` ↔ `PageFile` ↔ `AtomicYAMLMarkdown` chain — the editor library is just the leaf.

#### Load-bearing assurances (carried verbatim from prior plan; library-agnostic)

The "files are canonical" principle. The editor MUST write the actual `.md` file out of the box. The wiring chain stays:

1. **Load:** sidebar click → `ContentManager` reads `<vault>/<collection?>/<title>.md` from disk → `PageFile.load(from:)` calls `AtomicYAMLMarkdown.load` which splits `(frontmatter: PageFrontmatter, body: String)` → `PageEditorViewModel(page:body:saver:)` holds both; passes `body` to the editor via `Binding<String>`.
2. **Type:** every keystroke in `NativeTextView` → text-storage edit → Coordinator restyle → `text` binding updates Swift-side `Binding<String>` → `PageEditorViewModel.body` `didSet` → `scheduleSave()` enqueues a 300ms debounced save Task.
3. **Save (debounced):** Task awakes → `ContentManagerPageSaver.save` → `ContentManager.updatePage(_:body:in:vault:)` (or `inVaultRoot:`) → reconstructs `PageFile(frontmatter:body:title:)` → `AtomicYAMLMarkdown.write(frontmatter:body:to:)` (atomic temp-file + rename, project-locked v0.2.5 standard) → in-memory cache updates.
4. **Flush on context loss:** page-switch (PageEditorHost `.task(id:)` awaits `old.close()`), window-close (`PageEditorView.onDisappear`), `NSApplication.willResignActiveNotification`, `willTerminateNotification`, `⌘S`. All paths existing and untouched.
5. **Failure handling:** existing `pendingError` alert pattern; draft body preserved; retry re-schedules.

**Frontmatter preservation:** Editor binds only to `body` (pure Markdown — YAML stripped by `AtomicYAMLMarkdown.load` before reaching the editor). Frontmatter is held in `viewModel.page.frontmatter` and re-serialized on save from the typed struct, never from a string-prefix. **The user cannot destroy frontmatter via the editor; YAML is never visible.**

#### Phase 4.5 — Extend MarkdownInputHandler with markdown-aware auto-pair (~45 min, load-bearing UX)

**Why this gets its own phase:** auto-pair is one of the two load-bearing engine contributions (per the architecture intro). The engine ships partial auto-pairing (`$`, `$$`, `![[`) but not the broader markdown-aware patterns. Phase G.4 ported these to the CodeMirror fork via a custom `EditorView.inputHandler.of(...)` — we re-port to NSTextViewDelegate.

**Extend `Pommora/Pommora/PageEditor/Engine/Input/MarkdownInputHandler.swift`:**

Add four auto-pair patterns matching Phase G.4's behavior, implemented via the existing input-handler dispatch (the file already routes typing-time interception per character class — slot in alongside `$`/`$$`/`![[`):

| Trigger | Result | Notes |
|---|---|---|
| Type `*` after a preceding `*` (so `**`) on a blank/empty selection | `**\|**` (caret between) | Detection: look at preceding character in storage |
| Type `_` after a preceding `_` (so `__`) on empty selection | `__\|__` | Same pattern |
| Type `[` after a preceding `[` (so `[[`) on empty selection | `[[\|]]` | Wikilink prep |
| Type `` ` `` after a preceding `` ` `` (so `` `` ``) on empty selection | `` ``|`` `` | Inline code prep |
| Type any of `*` `_` `` ` `` ON a non-empty selection | Wrap selection: `*word*` / `**word**` if doubled, `_word_`, `` `word` `` | Selection-wrap (Obsidian/Bear muscle memory) |

**Auto-exit on whitespace boundary (Nathan-locked):** after auto-pair inserts `**\|**` and the user types `bold`, the caret sits at `**bold\|**`. If the next keystroke is **whitespace** (space, tab, newline), the handler treats it as "user is done with this span" and **auto-advances the caret PAST the closing `**` before inserting the space**. Net result: `**bold** ` with caret after the space, outside the bold region — typing more text adds non-bold prose. Same logic for `__ `, `[[` (Tab/Enter typically; space inside a wikilink is valid so detect carefully), `` `` `` for inline code. This matches Notion's "type-naturally" feel: the user never has to press Tab/Right-arrow to escape a freshly-paired region.

**State tracking required:** the handler maintains a per-text-view `freshPairContext: (closeRange: NSRange, closeMarker: String)?` — set when an auto-pair fires; cleared when the user moves the caret (arrow keys, click, selection change), types the closing character literally (to make `***`), or completes the auto-exit. The state is cheap and lives on the `NativeTextViewCoordinator` alongside existing per-coordinator caches.

**Edge cases (from Phase G.4 research + Phase 4.5 additions, locked):**
- If caret already sits before a matching close char (e.g., user types `*` and the next char is `*`), suppress double-insert — advance caret past existing close
- If typing inside a fenced code block (detect via Apple AST `Markup.parent` chain: `InlineCode` / `CodeBlock`), skip auto-pair entirely (preserves literal asterisks in code)
- If typing `***` (already `**`, adding 3rd `*`), the middle `*` shouldn't trigger double-insert — emphasis-marker resolution belongs to the parser, not the input handler
- If user types whitespace at the `freshPair` close boundary, auto-advance past close (per "Auto-exit on whitespace" above)
- If user types ANY non-whitespace inside the fresh pair, keep filling the span normally (no auto-exit)
- If user types the close character literally inside the fresh pair (e.g., `**word*` then `*` again → `**word**`), recognize this as manual close + clear `freshPairContext` (don't double-insert another `*`)

**Implementation surface:** add helpers to `MarkdownInputHandler` (the existing dispatch struct/enum) for each pattern. The dispatch is called from `NativeTextViewCoordinator+TextDelegate.swift` `textView(_:shouldChangeTextIn:replacementString:)`; verify the existing dispatch wiring routes `**` etc. through. If the existing dispatch is tightly coupled to `$`/`![[` specifics, refactor minimally to accept a "pair detector" registry.

#### Why this design wins

**Architecture cleanliness:** ONE parser (Apple), ONE styler (Pommora-side AST walker), ONE substrate (vendored engine). No regex hybrid, no fragmented detection. Future features land in one place: extend `PommoraMarkdownStyler.visit()` for new Apple AST types, or `PommoraInlineScanner` for new Obsidian-style scans.

**Native everything:** TextKit 2 gives Apple typography, Writing Tools (15.1+), system spell-check, autocorrect, IME, accessibility, dynamic system colors, native context menu, drag-select boost, bottom overscroll — all for free. Phase G's transparent-bg hacks, JS bridges, and CSS approximations all go away.

**Future-proof:** swift-markdown's AST already covers tables, blockquotes, ThematicBreak, strikethrough — features Pommora needed but the engine's regex tokenizer couldn't deliver. Also covers `BlockDirective` (v0.2.9 `:::callout` / `@Columns`) and `Strong`/`Emphasis`/`Heading`/etc. — every block type Pommora will ever need is already parsed; we just style.

**Risk-minimized:** All 198 v0.2.7 tests survive unchanged (editor library is a leaf, not a dependency of the domain layer). The atomic-write contract, frontmatter preservation, page-switch flush, lifecycle observers — all untouched. Failure budget concentrates entirely in `PommoraMarkdownStyler` + `PommoraInlineScanner` + `SourceRangeToNSRange`, all of which are unit-testable in isolation.

**Source-preserving:** the engine's "markers shrink, never disappear" invariant carries over directly. Page bytes stay byte-identical to disk — "files are canonical" survives intact.
