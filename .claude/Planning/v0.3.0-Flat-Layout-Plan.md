### v0.3.0 Flat-Layout Refactor — Implementation Plan

> A v0.3.0 refinement, not a new paradigm. Keeps all post-ParadigmV2 Swift struct names (`PageType`, `PageCollection`, `ItemType`, `ItemCollection`, `AgendaTask`, `AgendaEvent`). Drops the wrapper folders (`<nexus>/Pages/`, `<nexus>/Items/`, `<nexus>/Agenda/`) and splits the unified `_schema.json` sidecar into six per-kind filenames. Ships before v0.3.0 Properties because Properties' schema-editing operates on sidecar files — doing this against the flat layout is cleaner than against the wrapper layout.

**Status:** drafted 2026-05-23, tightened in a brainstorming pass same day. Locked discipline: each phase ships green standalone (stub-and-progressively-replace per quirk #8). Subagent-driven dispatch (per `superpowers:subagent-driven-development`).

**Tag target:** `flatlayout` (annotated, between v0.2.8 / `paradigmV2` and v0.3.0).

---

#### Locked decisions (this session — brainstorming pass)

1. **Wrapper folders disappear.** No `<nexus>/Pages/`, no `<nexus>/Items/`, no `<nexus>/Agenda/`. Types live at root.
2. **Six per-kind sidecar filenames** — no unified `_schema.json` anymore:
   - `_pagetype.json` — PageType folder
   - `_pagecollection.json` — PageCollection sub-folder (inside a PageType)
   - `_itemtype.json` — ItemType folder
   - `_itemcollection.json` — ItemCollection sub-folder (inside an ItemType)
   - `_taskconfig.json` — Tasks singleton (AgendaTask schema)
   - `_eventconfig.json` — Events singleton (AgendaEvent schema)
3. **Asymmetric `config` suffix on Agenda is intentional.** Agenda's per-entity files use `.task.json` / `.event.json` extensions; bare `_task.json` would visually clash. Pages-side / Items-side sidecars never collide with entity extensions (`.md` for Pages, `.json` for Items), so no suffix needed there.
4. **Swift struct names unchanged.** `PageType` / `PageCollection` / `ItemType` / `ItemCollection` / `AgendaTask` / `AgendaEvent` stay as shipped at `paradigmV2`. No `Swift.Collection` collision returns; no bare `Type` paper cut.
5. **Agenda stays singleton (sidecar-driven discovery).** Exactly ONE Tasks folder + ONE Events folder per Nexus. **Discovery walks root for any folder carrying `_taskconfig.json` (= the Tasks singleton) or `_eventconfig.json` (= the Events singleton), regardless of folder name.** Same pattern as PageType/ItemType. Folder rename via Finder Just Works. If two folders carry the same Agenda sidecar (pathological), first-found wins with a warning logged. Multiple Task/Event types remain a post-v1 Prospect.
6. **Sidebar grouping** reads from sidecar filename, not from folder location. Items section of the sidebar = every root folder carrying `_itemtype.json`; Pages section = every root folder carrying `_pagetype.json`. No wrapper presence needed.
7. **Adopter handles FOUR input shapes** (best-effort, log warnings on ambiguity):
   - **Fresh** — folder with no recognized sidecar; content-sniff (`.md` dominant → PageType candidate; `.json` dominant → ItemType candidate; empty → default PageType).
   - **Legacy v0.2** — folder at root carrying `_vault.json` (pre-ParadigmV2 PageType sidecar) and/or sub-folder `_collection.json`. Rename in place to `_pagetype.json` / `_pagecollection.json`. Nathan's real Nexus is in this shape today.
   - **ParadigmV2 wrapper layout** — `<nexus>/Pages/<X>/_schema.json` / `<nexus>/Items/<X>/_schema.json` / `<nexus>/Agenda/Tasks/_schema.json` / `<nexus>/Agenda/Events/_schema.json`. Move each child up to root + rename `_schema.json` to the per-kind sidecar; delete wrapper when empty.
   - **Already flat (target)** — folder with any of the six per-kind sidecars at the right depth. No-op.
   - Mixed input shapes coexist (e.g. some folders legacy, others wrapped, others flat) — adopter handles each folder independently; failures on one don't block the rest. Final summary report surfaced.
8. **Pathological case policy: best-effort + log warnings.** Folder with two sidecars → first-found in the discovery order wins, warning logged. Collision on unwrap (destination root path already exists) → append a timestamp-discriminator suffix to the moved folder name (mirrors `Filesystem.moveToTrash` discriminator pattern). Adoption never aborts on a single bad folder.
9. **Tasks/Events folders eagerly created on launch** (current behavior preserved). `AgendaTaskManager.loadAll` and `AgendaEventManager.loadAll` continue calling `NexusPaths.ensureDirectoryExists` + seeding `_taskconfig.json` / `_eventconfig.json` if absent. A fresh Nexus shows both folders at root even when empty — predictable for the user, uniform for discovery.
10. **Agenda collapse is EventKit-aligned, not just structural.** `EKEvent` and `EKReminder` are peer types in EventKit (separate `EKEntityType` buckets, separate `requestFullAccessTo*` APIs, separate Apple apps — Calendar.app vs Reminders.app). Pommora's UI already collapses Agenda (no Agenda sidebar heading per locked Sidebar.md; Calendar pin consolidates). The disk layout now matches the same semantic peer-relationship.
11. **Adopter atomicity: best-effort + idempotent.** No two-phase transaction across folders. Each folder migration is self-atomic (one filesystem op or a tight pair). If interrupted mid-apply, re-launch picks up where it left off — already-migrated folders are recognized as "already flat" and skipped. No partial-state corruption because each folder transitions through legal states only.
12. **Documentation ships FIRST** (before code) — this plan's Phase 1. Subagents executing code phases read CLAUDE.md, Handoff.md, Features/*, PRD, etc. as canonical spec. If those docs still describe the wrapper layout while subagents are asked to build the flat layout, the docs and the task contradict — bad for subagent reliability. Pre-updating docs means subagents read the target state cleanly and just make code match. Handoff.md is the one doc that EXPLICITLY signals "flatlayout in-flight; code catches up over phases 2–6"; everything else describes the flat layout as canonical with no caveats.
13. **Phase 1 → Phase 2 gates on Nathan's explicit approval (remote-review pattern).** Phase 1 completes with mirror-to-Nexus; main session then PAUSES and waits for Nathan's "docs look good, proceed" signal via remote (Obsidian-readable on iPhone) before dispatching Phase 2. Catches subagent mis-interpretations of the spec at the cheapest stage. Other phases run autonomously per Auto Mode default; Nathan can interrupt any time.

#### Resulting on-disk layout

```
<nexus>/
  .nexus/                              (unchanged — settings.json, state.json, tier-config.json, …)
  .trash/                              (unchanged)

  Recipes/                             ← PageType (UI: "Vault")
    _pagetype.json
    Breakfast/                         ← PageCollection (UI: "Collection")
      _pagecollection.json
      Pancakes.md
    Standalone.md                      ← Page in PageType root

  Contacts/                            ← ItemType (UI: "Type")
    _itemtype.json
    Friends/                           ← ItemCollection (UI: "Set")
      _itemcollection.json
      Alice.json
    Bob.json                           ← Item in ItemType root

  Tasks/                               ← AgendaTask singleton
    _taskconfig.json
    Pay-bills.task.json

  Events/                              ← AgendaEvent singleton
    _eventconfig.json
    Standup.event.json
```

What this delivers:
- `ls <nexus>` shows your real content immediately — no traversal needed
- Sidecar filename alone classifies every folder
- LLM directory-walks land on what they want at depth 1
- Folders without a recognized sidecar are unrecognized; adopter offers adoption
