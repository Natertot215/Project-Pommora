### Frame

- **Purpose:** Decide whether and how Pommora gains **warm, state-preserving tabs** in the toolbar — a working set of open entities you flip between like browser tabs, each keeping its own view state (scroll, editor state, its own back/forward), superseding today's single-pane-replace model.
- **Core Value:** You can hold several entities open at once and switch between them instantly *without losing your place in any of them* — the thing single-pane-replace and the temp-pin "fake tabs" can't deliver.
- **Success Criteria:** (1) opening entities builds an ordered, warm tab set in the toolbar; (2) switching a tab restores its scroll + editor state (perceived warmth), with only one view mounted at a time; (3) pinned vs unpinned tabs govern replace-vs-spawn on navigate; (4) the tab set is per-window and reuses the existing nav layer (recents/favorites) rather than duplicating it; (5) it holds to the perf rules — no N-views-mounted, no expensive work on every switch.

#### B — The Warm-State Mechanism (load-bearing)
- **B-1:** [recommend, unopposed] **Option B — one view mounted, per-tab serialized state cache, rehydrate on switch.** Rejected: Option A (N live views) — forces `selection`/`pageDetail` off their singular shape across ~15 consumers AND keeps N un-virtualized tables alive (perf hard-rule violation). Rehydrate-on-switch also re-reconciles each tab against the live tree (fresher than a stale live view for a file-canonical app). Nathan's warmth spec (B-2) is agnostic to A-vs-B; B is the perf/architecture call.
- **B-2:** [confirmed — Nathan; corrected by review-F1] **Warmth = scroll + undo + heading-fold.** But the warm **cache** serializes only **`historyField` (undo) + scrollTop** — **folds are NOT in the cache**: heading folds already persist per-page to `folds.json` and re-fold on mount (`PageView.tsx` folds prop → `applySavedFolds`, `MarkdownPM/index.tsx`), so fold-warmth is **free** and survives both switch and relaunch. Scroll + undo are session-only (in-memory).
  - **B-2a:** [confirmed — Nathan; corrected by review-F1] **Split by durability.** The **tab SET** persists + syncs (D-8). **Scroll + undo warmth is session-only** — a relaunched tab is cold (top of scroll, empty undo). **Folds are durable** (per-page `folds.json`), so a relaunched tab re-folds where you left it — NOT "unfolded" (the earlier claim was false against shipped code). No warm-EditorState to disk (sidesteps the sync-desync risk); folds ride their existing durable store.
- **B-3:** [confirmed — grounded by review] Implementation, corrected against code:
  - **Seed the fresh mount from a cached `EditorState` serializing `historyField` ONLY** (`@codemirror/commands`, verified exported + round-trips) + restore scrollTop post-mount. **Do NOT serialize `foldField`** — it's module-private, unexported, has no `toJSON`, and would throw (review-F1); folds come from `folds.json`.
  - **Freeze `(tabId, navKey)` at the CM6 mount** (in the `[]`-dep effect / a set-once ref); the unmount capture writes under those FROZEN values, never live `activeTabId` — else the switch races and a tab captures under the *next* tab's id (review-F2, `select` sets selection synchronously before the `openPage` await, `store.ts:591-596`).
  - **Warm switch short-circuits the refetch (kills the flash).** For a warm, unchanged tab, `activateTab` reuses the cached `pageDetail` and skips `openPage` + the `loading` placeholder (review-F8, Nathan: "flashing is weird, prevent it"). A change-bus-invalidated tab (I-4) refetches — the only legitimate flash, and only when the file genuinely changed. **This makes `select()` warm-aware** (a small, deliberate dent in "select untouched" — accepted for the feel).

#### C — Tabs vs the Existing Nav Layer (reconciliation) — REVISED by Nathan
- **C-1:** [confirmed — Nathan; grounded R1-F1] **The existing durable pins store IS the pinned-tabs set.** Pins are already a **separate, synced, per-pin-file store** — `PinEntry = NavTarget & { order, deleted? }` (`shared/types.ts:343`), one file per pin under `.nexus/pins/`, exposed as the `pins: PinEntry[]` slice (`store.ts:160`) with `pinTarget` / `unpinTarget` / `reorderPin` each persisting a single file immediately. **Pinning a tab = `pinTarget(target)`; unpinning = `unpinTarget(key)`; reordering pinned tabs = `reorderPin` (fractional `order`)** — the identical calls the NavGallery pins already use. So the pin set is **one working set surfaced in two places** (the tab bar + the NavPane gallery), rendered off the same `pins` slice. *(The legacy `RecentEntry.pinned` flag is one-time-migration-only — `navRecents.ts:2-4`, `resolveRecents` no longer floats it; superseded the earlier "graduate the flag" wording.)*
- **C-2:** [confirmed — Nathan] **Pinned tabs dock LEFT** and are the persisted working set. Existing pins appear there immediately when the feature ships — **zero migration**, the `.nexus/pins/` store already holds them. Pinned-tab reorder writes the same fractional `order` as the gallery — DRY onto `reorderPin`.
- **C-3:** [confirmed] Tiers: **Recents** (full history stream, feeds NavPane gallery + the new-tab page) · **Pinned tabs** (= the pin set, persisted, left-docked) · **Unpinned tabs** (ephemeral session scratch, right of the pins) · **Favorites** (durable bookmarks, unchanged).
- **C-5:** [assumed] **Recording semantics under tabs.** A recent records on a genuine *navigation* (opening an entity into a tab, or navigating within one), NOT on **activating an already-open tab** — switching tabs is re-surfacing, not a new nav, so it must call `select` with `record:false` (mirrors how Back/Forward already suppresses recording). Otherwise flipping between two open tabs pollutes recents. — confirm the intent; the mechanism (`record:false` on tab-activate) is settled either way.
- **C-4:** [confirmed — Nathan] **Pin surfacing in two places is intentional** — one working set, two views (left-docked pinned tab + pins-on-top in the NavPane gallery). Not redundant; deliberate.
- **C-6:** [confirmed — review, both agents] **`isPinned` is DERIVED, never stored.** `isPinned = pins.some(p => navKey(p) === navKey(tab.target))`, computed on read. The synced `tabs.json` stores **only unpinned tabs + `activeTabId`**; pinned tabs are rendered **off the `pins` slice** (identity + fractional `order`). On load, drop any `tabs.json` tab whose `navKey` is in the pins set. **Why:** storing `isPinned` in a second synced file re-introduces the whole-array-LWW desync the per-file `.nexus/pins/` store was built to dodge (a pin tombstoned on device A vs a `tabs.json` `isPinned=true` from device B → duplicate/mismatch, violating I-1). I-1 (≤1 tab per entity) guarantees the derivation has no conflict.

#### D — Tab Behavior (Nathan's stated model — to pressure-test)
- **D-1:** [confirmed — Nathan] **Unpinned tab = ephemeral/scratch:** navigating to another entity **replaces** its content in place (unless "Open in New Tab"). Avoids junk-tab accumulation — one browsing tab gets reused.
- **D-2:** [confirmed — Nathan] **Pinned tab = protected:** navigating while a pinned tab is active **opens a new tab** instead of replacing. Pinning = the explicit "keep this open" gesture.
- **D-3:** [confirmed — Nathan] **"Open in New Tab"** = a right-click item across Sidebar, NavPane, and views — forces a new tab regardless of pin state. *(Cost: 4 menu touch points, 2 across IPC — see Sources.)*
- **D-3b:** [confirmed — Nathan] **One primitive, one predicate.** `openTab(target, { newTab })`: **(1) dedup first** — if `target` is already in a tab, focus it and stop (I-1); **(2) else** `newTab = explicitOpenInNewTab || activeTab.isPinned` decides replace-active vs spawn. "Open in New Tab," "active tab is pinned so don't clobber it," and "Back/Forward off a pinned tab" (I-6) all funnel through the same spawn path — pin state, the menu item, and history-steps-off-a-pin are inputs to the one boolean. No duplicate logic anywhere.
- **D-4:** [confirmed — Nathan] Pinned tabs show a **pin icon on the left** (and dock left, C-2); a **hover `X` on the right** closes (DRY onto the chip × — J-1); create/remove use **one DRY'd animation** (J-6).
- **D-4b:** [confirmed — Nathan] **Tab drag-reorder uses the gallery's reflow animation** — DRY to the same `SortableZone`/`useDragItem` reflow the NavGallery pins use (`design-system/interactions/drag`). Not a new drag mechanism. **Zone-aware** (I-13): reorder within a zone, or drag across the pinned│unpinned divider to pin/unpin at the drop position.
- **D-5:** [confirmed — Nathan] **New-tab `+`** = a hover-icon right of the rightmost tab; when tabs span the full toolbar width it parks at the trailing end and **overlays the tab beneath it** (J-2/J-4).
- **D-6:** [confirmed — Nathan] **Single tab open → toolbar can stay blank** (no strip until ≥2, or until a pin).
- **D-7:** [confirmed — Nathan; A/B adjudicated] **Per-tab Back/Forward** — chosen over a single shared history (A vs B put to Nathan explicitly). Each tab owns its own history → lift `navStack`/`navIndex` into the tab; toolbar arrows act on the **active tab's** history. Matches browser muscle memory; avoids Back teleporting across tabs. (Single-shared was simpler state but cross-tab Back is disorienting — see Considered & Rejected.)
- **D-8:** [confirmed — Nathan] **The FULL tab set persists across restart — closing Pommora NEVER resets tabs.** Both pinned AND unpinned tabs, their order, the active tab, and each tab's current target + Back/Forward *targets* survive a quit/relaunch. On relaunch every tab reopens **in order, at its last-viewed entry, COLD** (warm view-state is session-only, B-2a). Warm state does NOT persist; the *structure* (what's open, where, in what order) does.
  - **D-8a:** [confirmed — Nathan] **The tab-set sidecar SYNCS cross-device.** A persisted **tab-state sidecar** holds the ordered tab list (targets + pin references), `activeTabId`, and per-tab {current target, history targets, index} — **NO warm state**. It is **canonical/synced** (NOT in `DEVICE_LOCAL_NEXUS_FILES`), following the nav layer's E-5 model: churny writes **debounced in main** (like recents), **last-writer-wins**, single-user so concurrent live edits are outside the threat model. So your open tabs **travel across devices**; pinned identities ride the already-synced `.nexus/pins/`. *(Supersedes the earlier device-local lean — Nathan: tabs must sync.)*

#### D′ — Tab Lifecycle (close · pin-gate · unpin placement)
- **D-9:** [confirmed — Nathan] **Closing the active tab focuses the most-recently-used tab** — an MRU focus stack, NOT the spatial neighbor. (Each activation pushes onto the stack; closing the active pops to the previous.) *(Visual tab ORDER stays stable on plain switching — the MRU stack governs close-focus only, not reshuffling; consistent with the "fixed order, don't jump on switch" call.)*
- **D-10:** [confirmed — Nathan] **Pin state gates closability.** The hover-`X` shows ONLY on unpinned tabs; a pinned tab is not directly closable — you toggle the pin off first, which reveals the `X`. (Browser pinned-tab behavior.)
- **D-11:** [confirmed — Nathan] **Unpin placement — promote to front.** The unpinned zone's **most-recent end is the LEFT** (adjacent to the pinned tabs). Unpinning a tab inserts it at **position 1 (leftmost / most-recent)** — UNLESS the **active tab is currently position 1**, in which case it inserts at **position 2 (directly behind the active-first tab)** so the active tab keeps its spot. "From anything besides the already-first → first; from the first → behind it." Example: active = A and A is position 1; unpin B → **B lands at position 2, right behind A, regardless of what was there.**
- **D-12:** [confirmed — Nathan] **Two placement rules, intentionally asymmetric.** **New tab-opens append to the RIGHT and hold fixed order** (no reshuffle on plain switching) — a fresh open is new work parked at the end. **Unpins promote to the front-left** (D-11) — unpinning says "I'm working this now." So the unpinned zone is *not* a single recency sort: opens grow it rightward, unpins insert at the left. Plain tab-switching never reorders; only open / unpin / close move things.

#### I — Interaction & Edge Cases (Nathan-confirmed)
- **I-1:** [confirmed — Nathan] **No duplicate tabs.** Opening an entity already in a tab **focuses that tab** (never a second copy, never replacing your scratch). This dissolves the "same entity is the active view *and* in tabs" concern — each entity maps to **≤ 1 tab**. The context menu is **stateful**: an already-open entity shows **"Open"** (focus its tab), a not-open entity shows **"Open in New Tab."**
- **I-2:** [confirmed — Nathan; grounded R1-F2] **Live entity mutation while open — no delay.** Rename / move re-resolve the tab's label + path **live and immediately** (live tree resolve). **Delete splits by pin state:** an **unpinned** tab whose entity is deleted **actually closes** (active → MRU-focus, D-9); a **pinned** tab whose entity is deleted **render-HIDES but keeps its pin file** — the shipped **render-prune, never storage-prune** rule (`navResolve.ts:104` `resolvePins`; the sole pin-storage mutation is an explicit `unpinTarget`, `store.ts`). So a synced/undeleted-elsewhere entity's pin survives; the same render-hide covers the relaunch case. Auto-close **never** drops a pin from storage.
  - **I-2a:** [confirmed — review-F3] **Every tab's target reconciles, not just the active selection.** `applyTree` today reconciles only the singular `selection` (`store.ts:346`) — so an inactive tab holds a raw stale `path`, and activating it after a rename/move calls `openPage(stalePath)` → error pane. Fix: on each `applyTree`, map `reconcileSelection` over **every tab's target + navStack** (applying the I-2 close-vs-render-hide split). This is real tab-awareness in `applyTree` — the "consumers untouched" framing (B-1) holds for the *render* consumers, NOT for tree-reconciliation.
- **I-3:** [confirmed — Nathan] **Autosave flush on replace/close — but no input delay.** Replacing or closing a tab **fires the pending autosave immediately** (cancel the debounce, write the captured doc snapshot **fire-and-forget — no await, no main-thread block**) before discarding the warm state. Zero input delay; the write lands in the background; the `before-quit` flush (nav-layer pattern) catches an unlanded tail. Data guard with no lag.
- **I-4:** [confirmed — Nathan] **Warm yields to disk truth.** A changed file **drops the warm cache and reloads fresh** (never a stale doc over a divergent undo stack). **Invalidation rides the EXISTING file-change bus** (`nav:changed` / tree-refresh watcher), not a separate mtime poll — one change-detection path, not two.
- **I-5:** [confirmed — Nathan] **Closing the last tab → the new-tab page** (never a blank window).
- **I-6:** [confirmed — Nathan] **Pinned tabs never change content in place — this extends to Back/Forward.** On an **unpinned** tab, Back/Forward navigates **in place** between its history entries. On a **pinned** tab, the first Back/Forward **spawns one new unpinned tab that inherits the history stack** and navigates *there*; the pinned tab stays put. (Generalizes D-3b: *any* navigation off a pinned tab — new-open or history-step — spawns.)
- **I-7:** [confirmed — Nathan] **The back-stack is WARM; the ~20 cap is on BACK-FORTH DEPTH per tab, NOT on open-tab count.** Open tabs are **uncapped** — you can open as many as you want. Within a tab, Back/Forward restores each history entry's warm state (scroll + undo + fold), and only the **last ~20 back-forth entries** stay warm — a Back beyond that re-opens cold. Warm state is serialized JSON for inactive tabs (only the **active** tab is mounted), so an inactive tab holds its current + up to ~20 back-forth blobs — far lighter than Option A's N mounted views. If aggregate warm memory ever bites, the fallback is a **global LRU that trims warm DEPTH, never closes tabs** (open tabs stay uncapped). *(Escalates B-2's warm scope from current-content-only to the capped back-stack.)* Note the split with persistence: a tab's history **targets** persist across restart (D-8), but their **warm** state is session-only + ~20-capped — on relaunch Back still navigates, just cold.
- **I-8:** [confirmed — Nathan] **Pinned tab = compact icon.** The entity icon with the **pin as an accent** (no title inline); the **full name reveals on hover.** The pinned zone stays **fixed-left** (compact icons cost little); only the **unpinned zone overflow-scrolls.**
- **I-9:** [confirmed — Nathan] **NavPane preview-mode COEXISTS with tabs** — a preview is an ephemeral **peek** that does **NOT** open a tab and does **NOT** touch Back/Forward. (Resolves G-4; supersedes the earlier "retire preview" lean — preview stays a distinct tab-neutral gesture.)
- **I-10:** [confirmed — Nathan; revised by D-8] **Nexus switch** — the current nexus's **full tab set persists to its synced sidecar**, then the new nexus's tab set loads (its pinned tabs from that nexus's synced `.nexus/pins/`). Returning to a nexus **restores its tabs** — not discarded. In-memory reset on switch still mirrors the nav-layer's per-nexus reset (E-11), but backed by the persisted set.
- **I-11:** [confirmed — Nathan] **Keyboard = tab-cycling ONLY** — no ⌘T/⌘W/⌘1–9, only next/prev-tab cycling. **Cycling wraps around and includes all tabs (pinned + unpinned).** Binding: **`Ctrl`+`Tab` / `Ctrl`+`Shift`+`Tab`** (Nathan-signed-off — the one approved keybinding). Everything else stays mouse-driven.

### Core (must-have)
- **Tab store slice** — `tabs` (each: id, target, `isPinned`, own `navStack`/`navIndex`, in-memory warm cache) + `activeTabId`; the singular `selection`/`pageDetail`/`liveBody` stay as-is and always mean the active tab (B-1's small blast radius). Pinned tab identities come from the synced `.nexus/pins/` store (C-1/C-2).
- **Persisted tab-set store** (D-8/D-8a) — a **synced** sidecar of the ordered tab list + active + per-tab history *targets* (no warm state); the whole set reopens on relaunch AND travels cross-device (debounced main-side, last-writer-wins). Closing never resets; restore is cold (warm view-state is session-only, B-2a).
- **Warm rehydration path** (B-1/B-2/B-3/I-7) — one view mounted; serialize scroll + undo + fold **per tab-history-entry** (warm back-stack, I-7) on switch-away, seed the fresh mount from it on switch-to; disk-change invalidates (I-4); session-scoped, resets on quit; per-tab history depth-capped to bound the cache.
- **The `openTab(target, {newTab})` primitive** with the single `newTab = explicitOpenInNewTab || activeTab.isPinned` predicate (D-3b), driving `select()`; per-tab Back/Forward on top (D-7).
- **The real tab bar UI** (grown from `TabBarPreview`): pinned left-docked as **compact icon + pin accent, name-on-hover** (I-8); hover-`X` close; trailing `+` overlay; DRY create/remove animation; **zone-aware** drag (reorder + cross-zone pin/unpin) via the gallery's `SortableZone` reflow (D-4b/I-13); a **tab context menu** (Pin/Unpin · Close · Close to the Right, I-12); `Ctrl`+`Tab` cycling (I-11); blank when a single tab (D-1…D-6, D-9…D-12).
- **The four context-menu "Open in New Tab" touch points** (D-3) — Sidebar (native push-back), TableView (native cell action), NavList (PickerMenu item), NavGallery (new menu).
- **The new-tab page** (E-1) — full-window NavGallery + `--main-bg` + a search input.
- **The reveal-on-hover setting** (F-1) — a `personalization` boolean, default false.

#### Prospects (allowed later, not now)
- Cross-restart warm state (persist scroll/undo/fold to disk) — explicitly rejected for v1 (B-2a); session-only warmth is the call, only the tab *set* persists.
- **Drag-reorder recents** (rewrite recents order without opening/pinning) — **NAV-LAYER scope, not multi-tab.** Clean + cheap: reuse the pin `SortableZone` in the same gallery + a `reorderRecent` that rewrites the recents array — **the nudge IS the recents order.** No MRU conflict: `recordRecent`'s move-to-front only re-fronts the ONE visited entry (correct — it's recent again); every other nudged entry keeps its position. Recents ⟂ history (E-6), so it never touches Back/Forward. Only nuance: an unpinned nudge still rolls off at the cap (pins alone are roll-off-exempt — the real pin-vs-nudge line). Deferred purely because it's nav-layer, not multi-tab. *(Nathan QOL — genuinely small; could ride as its own quick task.)*
- Per-window tab sets once multi-window lands (D-8a keys the sidecar per window ref).
- **Cross-divider drag-to-pin/unpin** (I-13) — deferred: it's bespoke horizontal cross-zone engine work (the cross-zone engine is vertical + portal-overlay). v1 pins via affordance + menu; within-zone reorder drag ships.
- Drag a tab out into its own window (G-5).
- ⌘1–9 / ⌘W / ⌘T shortcuts — proposed separately per the no-keybinding-without-signoff rule.

#### Out of Scope
- Split panes (the third B-1 branch) — not this cycle.
- Table row virtualization — its own standing perf debt; warm-B avoids needing it (one view mounted).

#### Considered & Rejected
- **Option A (N live mounted views)** — perf hard-rule violation (N un-virtualized tables) + ~15-consumer blast radius. (B-1)
- **Retiring the nav-layer `pinned` flag** (my initial recommendation) — Nathan overrode: pins *graduate* into pinned tabs instead, one shared working set surfaced in two places. (C-1)
- **Persisting warm VIEW-STATE to disk** (scroll/undo/fold) — rejected; warm state is session-memory only, resets on relaunch (B-2a). The tab *set* (structure) DOES persist (D-8) — distinct concern.
- **Single shared Back/Forward history** across all tabs — rejected (A/B put to Nathan). Simpler state, but Back teleporting between tabs fights browser muscle memory. Per-tab history won (D-7).
- **A merged `NavSurface` for the new-tab page + NavPane** — rejected (E-3); reuse the gallery style, keep the shells separate to avoid coupling two surfaces that may diverge.
- **Collapsing Favorites into pinned tabs** — rejected (E-4); favorites are a future jump-to surface, not a held-open working set.
- **The chip melt/blur `×` reveal on tabs** — rejected (review-F2); the melt needs a solid `--chip-fill` a frosted-glass tab lacks. Plain `×` hover-fade instead (J-1).
- **Serializing `foldField` for warm state** — rejected (review-F1); it's unexported/non-serializable and would throw, AND folds already persist durably via `folds.json`. Warm cache serializes `historyField` only (B-3).
- **Storing `isPinned` in `tabs.json`** — rejected (review, both agents); derive it from the pins set to avoid a cross-device LWW desync class (C-6).

#### Lessons
- The nav layer was built UI-agnostic *specifically* to feed this tab bar — the reconciliation is a fulfillment, not a rewrite. Ground new features against prior decision logs, not just code.
