### Markdown Editor — Full Cleanup Pass

#### Context

Nathan asked for a read-through of Pommora's markdown editor (vendored `swift-markdown-engine` at `External/MarkdownEngine/` + Pommora-side wiring under `Pommora/Pommora/Pages/`) cross-referenced against `// Features//PageEditor.md` and `// Guidelines//Markdown.md`, surfacing routing-unclear / bloated / over-complicated areas.

Three Explore agents read ~40 Swift files. The editor is **fundamentally sound** — the dynamic-syntax pattern is locked and correct, the save pipeline is intact, `isProgrammaticEdit` is consistently applied, and editor coupling to Pommora is genuinely confined to `PageEditorView.swift` (~13 lines).

This plan executes **all 9 cleanup items** the audit surfaced. **Hard constraint:** zero impact on user-visible interaction. Every change is either a code move, a dead-code deletion, a comment addition, or a behavior-preserving refactor verified against the existing 197+ test suite.

---

#### Architecture as-built (1-line per layer)

- **Parser** — Apple `swift-markdown` 0.8.0 produces the AST; `MarkdownTokenizer` runs a parallel regex pass for inline marks (hot path).
- **Styler** — Two stylers coexist: `MarkdownStyler` + 6 extensions (regex-driven inline marks) and `AppleASTSupplementalStyler` (AST walker for BlockQuote / Strikethrough / Table / ThematicBreak).
- **Renderer** — `MarkdownTextLayoutFragment` overrides `draw(at:in:)` for code-block bg + LaTeX + HR line + task checkboxes via TextKit 2.
- **Coordinator** — `NativeTextViewCoordinator` (NSTextViewDelegate) split across 10 files.
- **Services** — `HRVisibility` is the canonical caret-awareness service; `WikiLinkService` / `MarkdownEditorServices` are clean DI containers.
- **Input** — `MarkdownInputHandler` (char-pair auto-pair, block auto-wrap) + `MarkdownListHandler` (list continuation, three-stage detection).
- **NSTextView subclass** — `NativeTextView` + 7 single-purpose extensions (well-executed; not touched in this pass).
- **Pommora-side** — `PageEditorView` hosts `NativeTextViewWrapper`; `PageEditorViewModel` debounces saves; `PageEditorHost` flushes on page-switch.

#### Cleanup items — full scope

##### Phase 1 — Pure rename / comment / dead-key deletion (zero risk)

**1.1. Rename `cachedSpellingDisabled` → `previousSpellingDisabled`**
- `NativeTextViewCoordinator.swift:80` (declaration) + `+Autocorrect.swift:47, 50` (read + write).
- The flag isn't a cache — it's "remember the last value to skip redundant setter calls." Cache implies invalidation logic that doesn't exist. Pure rename clarifies intent.

**1.2. Delete `.pommoraThematicBreak` dead declaration**
- `MarkdownTextLayoutFragment.swift:26` (declaration only — no read/write sites anywhere in the engine).
- Keep the surrounding cautionary comment block (lines 21-25 + 27) as historical documentation of the failed first HR attempt. The lesson lives in the comment, not the dead identifier.

**1.3. Add `bareMarkerRegex` docstring**
- `MarkdownListHandler.swift:37`.
- One block comment explaining the regex matches lines with ONLY a bare list marker (`-` / `*` / `+` / `\d+\.`), used to detect the "incomplete marker" case where Enter completes the marker + creates a new item.

##### Phase 2 — Code consolidation (low risk; pure moves)

**2.1. Consolidate 6 coordinator extensions into `+Services.swift`**
- Merge: `+Find.swift`, `+Notifications.swift`, `+Autocorrect.swift`, `+CodeBlocks.swift`, `+WritingTools.swift`, `+InlineSelection.swift` → single `NativeTextViewCoordinator+Services.swift`.
- LEAVE separate: `NativeTextViewCoordinator.swift` (state), `+TextDelegate.swift` (hot delegate path), `+Restyling.swift` (restyle pipeline), `+HRVisibility.swift` (canonical caret-awareness service). These four ARE the locked dynamic-syntax pipeline boundary.
- Net: 10 files → 5. No logic change; pure file consolidation.

**2.2. Inline `MarkdownStyler+Code` + `MarkdownStyler+TaskCheckboxes` back into `MarkdownStyler.swift`**
- 68 + 93 lines respectively. The split was organizational, but neither has cross-file coupling that justifies the file boundary. Inlining keeps these small concerns next to the rest of the regex styler.

##### Phase 3 — Shared helper + dead-code removal (the audit's strongest finding)

**3.1. Extract shared HR detection into `MarkdownDetection`**
- Currently duplicated: `MarkdownTextLayoutFragment.swift:69-87` (`hasThematicBreak`) ↔ `NativeTextViewCoordinator+HRVisibility.swift:87-107` (`isThematicBreakParagraph`). Markdown.md L2 documents this as "fragile but works."
- Add `static func isThematicBreakLine(_ line: String, hasCodeBlockBackground: Bool) -> Bool` to `External/MarkdownEngine/Sources/MarkdownEngine/Parser/MarkdownDetection.swift`. Implements the three-stage detection (Stage 0 code-block guard → Stage 1 string prefilter → Stage 2 `Document(parsing:)` AST check).
- Replace inline detection at both call sites with the shared helper.

**3.2. Delete dead `MarkdownStyler.styleHorizontalRules`**
- `MarkdownStyler.swift:328-343` (method) + `:171` (call site in `styleAttributes`).
- Method emits `.foregroundColor: .clear`, thick strikethrough, mutable paragraphStyle over the HR range. The HRVisibility service is the sole writer of HR visual state per Markdown.md §3.2 + L3 — these emissions conflict (anti-pattern 6.2). Visible flicker is masked today only because the service writes after the styler in the restyle pipeline.

##### Phase 4 — Behavior-preserving hot-path refactors (require careful verification)

**4.1. Replace `previousBacktickCount` heuristic with token-count delta**
- `NativeTextViewCoordinator.swift:65` (state) + `+TextDelegate.swift:108-110` (read + compute).
- Current: counts `` ``` `` substrings to detect code-block structure changes; trips on edge cases (spaces inside triple-backticks, escaped backticks).
- Replacement: compare `parsed.codeTokens.count` against previous value (or deeper equality for position changes if needed). The tokenizer already produces `codeTokens`; no new parsing work.
- **Behavior parity check:** for well-formed Markdown the two heuristics produce identical answers; the difference is only at malformed/transient states where the new approach is more robust. Risk: zero behavior change for valid content.

**4.2. Fix UTF-8 / UTF-16 column-offset bug in `LineOffsetIndex`**
- `AppleASTSupplementalStyler.swift:206-251` (the documented latent bug).
- cmark-gfm reports `SourceRange.column` as UTF-8 byte offsets; current code treats them as UTF-16 code-unit offsets. ASCII coincides; multi-byte content misaligns.
- Fix: in `utf16Offset(line:column:)`, walk the line's bytes up to `column` and accumulate the UTF-16 offset codepoint-by-codepoint instead of treating the column as a direct UTF-16 index.
- **Verification:** add one targeted test in `PommoraTests` with multi-byte content (one accented char inside a blockquote AND one emoji in a strikethrough range) — confirms range alignment is correct. ASCII-only existing tests continue to pass unchanged.
- No current shipping content triggers this bug, so user-facing impact is zero today; the fix is preemptive correctness for when Tables / non-Latin content lands.

#### Sequencing — 4 phases, each shipping green standalone

Phases run in this order. After each phase: build + lint + test (commands below). If any phase regresses, the previous phase is already green — clean rollback target. Phases 1-3 commit per phase; Phase 4 commits per item (4.1 and 4.2 separately).

1. **Phase 1** — items 1.1 + 1.2 + 1.3 (rename + dead-key + docstring). Single commit.
2. **Phase 2** — items 2.1 + 2.2 (file consolidations). Two commits, one per item.
3. **Phase 3** — items 3.1 + 3.2 in this order (extract helper first, then delete the conflicting styler method that the new helper would expose). One commit (the two items are coupled — 3.2's deletion is what 3.1's centralization enables).
4. **Phase 4** — items 4.1 then 4.2. Two commits. 4.2 includes the new test case.

This matches the locked stub-and-progressively-replace strategy (paradigm decision #4).
