### Paradigm Decisions

Pommora's value depends on its on-disk format, schemas, and cross-entity contracts surviving a stack rebuild and a future cloud sync (load-bearing constraints #1 and #2). Code locking those shapes is **paradigm-solidifying** â€” once data exists in the wild, migrating is expensive.

#### Operating rule

**Stop and surface paradigm choices for Nathan's confirmation before the code lands** â€” use `AskUserQuestion` with concrete trade-offs and your recommendation.

Applies even when a written plan proposes one path â€” if you spot ambiguity, a real downside, or an alternative worth weighing, surface it first. Spec drift is acceptable; silent commitment is not.

#### What counts as paradigm-solidifying

- **On-disk schema shapes** â€” fields, types, naming conventions, snake_case vs camelCase per-key choices, nesting structures inside the per-kind sidecars (`_pagetype.json` / `_pagecollection.json` / `_taskconfig.json` / `_eventconfig.json`) / `.space.json` / `.project.json` / `.task.json` / `.event.json` / etc.
- **Wire encodings for ambiguous types** â€” tagged-object vs bare-string discrimination (e.g. `.relation` vs `.select` strings), date format choices (ISO-8601 vs Unix epoch vs human-readable), null-vs-missing semantics.
- **Identifier conventions** â€” ULID format, filename-as-title rule, ID-vs-title display split, relation key shape (e.g. `{"$rel": "..."}`).
- **Default values that become locked once data exists** â€” seeded per-kind sidecar shapes (`_pagetype.json` / `_pagecollection.json` / `_taskconfig.json` / `_eventconfig.json`), `defaultSeed()` outputs, default property catalog per Type.
- **File layout choices** â€” folder vs file boundaries for entities, filename extension conventions (`.project.json` vs `_project.json`), sidecar metadata file naming.
- **Cross-entity contracts** â€” tier1/tier2/tier3 array semantics, parent-pointer conventions, move-strip rules.
- **Error semantics at file load** â€” silent recovery (e.g. missing field â†’ default) vs hard throw, malformed-file handling, validation timing.
- **Behavioral defaults that change user-visible outcomes downstream** â€” delete promote-vs-cascade default, filename collision handling (reject vs auto-suffix), move-across-Type property-strip behavior.

#### What does NOT count

- **Internal implementation choices** that don't affect on-disk shape â€” use of `@Observable` vs `ObservableObject`, value types vs reference types, manager-per-entity vs unified store.
- **UI structure** â€” view extraction, sheet vs popover, sidebar layout â€” these can be refactored freely without data migration.
- **Test strategy** â€” Swift Testing vs XCTest, test file organization, fixture patterns.
- **Build configuration** â€” Swift version, strict concurrency settings (already locked).
- **Naming of types in code** â€” internal CodingKeys names, struct rename refactors that preserve on-disk shape.

If in doubt, surface it. Better to overconfirm than retrofit.

#### Confirmation protocol

1. **Stop** before writing the locking line.
2. **State the choice in user-facing terms** â€” on-disk shape, user-visible behavior, migration cost. Not jargon.
3. **Present 2â€“3 options** with concrete on-disk samples. Lead with your recommendation.
4. **Wait for confirmation.** Update the spec/plan before dispatching implementation.
5. **Record the locked decision** in `History.md` (the canonical log for confirmed paradigm decisions and the surrounding session context), then add a one-line entry to the registry below.

#### Registry

Numbered, chronological. Citations elsewhere ("paradigm decision  # ") resolve here. Each entry is the decision plus a one-line rationale; full session context lives in `History.md`.

1. **`PropertyValue.relation` encodes as an array of tagged objects `[{"$rel": "<ULID>"}]`** (always multi-value; a single link is a one-element array), not bare strings â€” relation edges stay legible to external agents and the graph indexer without consulting a Type's schema (constraint #3).
2. **Collections persist a sidecar** (`_pagecollection.json`, carrying id + parent-Type id + ordering) â€” the parent relation is explicit on-disk rather than inferred from folder nesting.
3. **SF Symbol picker = Pommora's own native `IconPicker`** â€” superseded (2026-05-30) the `xnth97/SymbolPicker` SPM dep, which hardcoded a 540pt macOS frame and kept its catalog `internal` (neither resizable nor re-skinnable). A compact Liquid-Glass dropdown over the full SF Symbols catalog (`IconCatalog`, bundled as source) with search + Saved/favorites; hosted via the single `.iconPickerPopover` modifier / `IconPickerSheet` / `IconPickerField`. The call-site-isolation goal now lives in that one modifier; the SPM dep is unused (removable). Saved icons persist in app-level UserDefaults (`IconFavorites`) â€” a UI convenience, not on-disk user content.
4. **Stub-and-progressively-replace** â€” each task ships as a green commit; when an earlier task's file references a type built in a later task, inline a throwaway stub at that commit and replace it in place when the real type lands. Rejected alternative: batch-commit all tasks at the end of the branch â€” one break anywhere contaminates the whole batch and isn't found until the final task.
5. **Sidebar creation is via right-click context menus scoped to the cursor**, not "+ New" buttons â€” right-clicking a row binds "New X" to that exact entity, keeping the entry point unambiguous.
6. **Sidebar selection chrome lives at the row-file level via `.listRowBackground(SelectionChrome(...))`**, not in-content `.background` â€” the row-level background covers the disclosure-chevron gutter, and an in-content `.background` regressed a launch crash in `OutlineListCoordinator.recursivelyDiffRows`.
7. **Pages editor stack** â€” superseded. The locked direction was originally Tiptap (ProseMirror) in a WKWebView; the editor shipped instead as **TextKit 2 + Apple `swift-markdown` + the Pommora-owned MarkdownPM** package (`External/MarkdownPM/`, originally vendored from `swift-markdown-engine`, Apache 2.0) (native NSTextView, no web view). Spec â†’ `// Features//PageEditor.md`; engine rules â†’ `Markdown.md`.

8. **Relations are always multi-value** â€” the single/multi `allows_multiple` toggle is gone; every relation property holds an array (see #1), so one uniform value shape exists on disk and in the editor.
9. **The relation schema target uses the `relation_target` key; `context_tier` is the only surviving case** â€” user-creatable targets (`page_type` / `item_type` / `agenda_tasks` / `agenda_events`) were removed in the Contextv2 refactor (2026-06-04); `context_tier` (tiers 1/2/3) is the sole target kind. `page_collection` / `item_collection` are read-tolerated for legacy sidecars and cleared during the migration walk; the decode is tolerant (`try?`) so unknown legacy `relation_target` shapes degrade to `nil` without failing sidecar load.
10. **Tiers are the sole relation-type connection** â€” `tier1` / `tier2` / `tier3` (root frontmatter arrays) are the only relation-property edges; they emit into the SQLite `context_links` table and share one reverse-lookup path (`IndexQuery.incomingContextLinks`). User-creatable relation properties are removed; there is no separate tier-links table.
11. **Deleting a Context cascades source-side** â€” every operational entity's `tier1/2/3` reference to the deleted Context is removed from its own file (not promoted, not orphaned). Orchestrated at the delete call site, which holds the content-manager references.
12. **~Agenda Tasks and Events as relation targets â€” superseded.~** This decision was retired by #16 (Contextv2); Agenda is no longer a valid user-selectable relation target.
13. **Same-container title collisions are rejected** â€” creating, renaming, or moving an operational entity to a title a sibling already holds in the same container is refused (case-insensitive), not auto-suffixed and never overwriting. Because `filename = title`, a collision resolves to the same path and the atomic write would silently destroy the other entity's file; rejection is the safe behavior absent an independent title field (a Prospect). One shared `NameCollisionValidator` enforces it across Pages and Agenda Tasks/Events; the container types (Spaces / Topics / Projects, Page Types + Collections) enforce the same rule through their own validators. Self-recasing an entity's own title (e.g. `notes` â†’ `Notes`) is allowed â€” the rename guard compares on-disk file identity, not the case-folded name.
14. **~Items are Markdown (Shape A) â€” superseded by #17 (the Items entity no longer exists).~** Original decision: Item content files are plain `.md` (YAML frontmatter + body), not `.json`, sharing Pages' single `AtomicYAMLMarkdown` pipeline. The capped description IS the body (single source of truth; no frontmatter-description field, no mirror). Kind authority is the parent Type folder's sidecar (`_itemtype.json` / `_pagetype.json`); a reserved, UI-hidden, **non-authoritative** `Class` stamp (`item`|`page`) marks the form. Three-state resolution: absent â†’ write from folder; agrees â†’ proceed; disagrees (or homeless) â†’ relocate to a hidden `.unsorted` inbox (future-UI-surfaced). **Foreign frontmatter keys are preserved by value â€” never culled â€” on every Item AND Page write path** (a reversal of the deleted Session-Context cull; Yams reflows style + drops comments, content safe). A mandatory auto-run launch migration normalizes legacy `.json` Items â†’ `.md` (idempotent, resumable; not a declinable consent-gate). **Agenda stays JSON** (`.task.json` / `.event.json`); sidecars, Projects, Spaces, Settings stay JSON â€” only Item content became `.md`. Char cap = 500 markdown-source chars (`ItemValidator.maxDescriptionLength`), per-Type `description_cap` override; validated on save. Pairs with #13 (`filename = title` + collision rejection).

15. **~`template_config` schema (ItemsV2) â€” superseded by #17 (no `template_config` ships; `open_in` landed instead as a plain `PageType` field with `compact|window` values).~** Original decision: both per-kind sidecars carry a reserved, all-optional, null-round-tripping `template_config`. Items: `{ layout, promoted_properties: [{id, display}], cover_property_id, description_cap, default_description }`. `layout` is the `LayoutArchetype` enum, on-disk strings `compact|standard|banner_two_column|gallery|wide|reserved` (tolerant `unknown` round-trips; only `standard` ships, the rest are muted-until-promoted). Per-property `display` is the tolerant `PropertyDisplay` enum (`inline|thumbnail|banner|chips|list|unknown`); the archetype sets a default a non-nil `display` overrides. `cover_property_id` names an existing `.file` property whose `accept` admits an image â€” no new PropertyType/PropertyValue case. Pages: reserved parity `{ layout (null â€” Pages have no archetype), default_body, open_in }`; `open_in` (`OpenInMode`: `preview|full_page`) is INERT until Page preview ships. **Collection-level `template_config` overrides its parent Type; absent â‡’ Type default.** Additive: existing files with no `template_config` render the standard Item Window.


16. **Contextv2 â€” user relations dropped; tiers are the sole relation-type connection (2026-06-04).** User-creatable `PropertyType.relation` properties are retired at decode (`.droppingUserRelations()` filter, tier-safe via reserved `_tier1/2/3` IDs). `RelationTarget` collapses to `.contextTier`-only; `RelationTargetKind.string()` reduced to match. The SQLite table `relations` renamed to `context_links` (`idx_relations_*` â†’ `idx_context_links_*`); all surviving Relation-named symbols renamed `Context*` (`RelationChip`â†’`ContextChip`, `RelationValueEditor`â†’`ContextValueEditor`, `RelationPicker`â†’`ContextPicker`, `RelationDisplayResolver`â†’`ContextDisplayResolver`, `BuiltInRelationProperties`â†’`BuiltInContextLinkProperties`, `incomingRelations`â†’`incomingContextLinks`, `reconcileRelations`â†’`reconcileContextLinks`, `entitiesByTarget`â†’`entitiesByContextTarget`). `Project.linked_relations`â†’`project_links` (dual-key tolerant decode: accepts both keys, always writes `project_links`). Orphaned `$rel` member values are schema-invisible (the property panel iterates the schema, not the member dict), unindexed, and round-trip-stable; cleared opportunistically during the migration member-walk for Types that undergo the ID-mint migration. The legacy Collectionâ†’Type migration (`applyRelationTransforms`) deleted as dead-after-filter. Substrate kept: `$rel` token, `PropertyValue.relation` codec, `RelationTarget.contextTier`, `TierRelationCarrying`, `RelationTargetKind`, `PropertyType.relation` (tier-only). **Roadmap:** a separate connection-model layer (per-shape tables, weight-at-query, contexts-as-cores "spiderweb") is planned but not yet built or slotted.

17. **PagesV2 collapse â€” Page is the sole operational entity beside Agenda (2026-06-09/10).** The Items side (entity, `ItemType` / `ItemCollection` containers, Item Window, `_itemtype.json` / `_itemcollection.json` sidecars, item index tables, "Type"/"Set" labels) is deleted, not migrated â€” supersedes #14 and #15. Consequences locked with it: the `Class` frontmatter stamp is retired (an on-disk `Class` key is preserved foreign frontmatter, never written); `[[Title]]` is the **sole** connection syntax (`{{ }}` retired to plain text; the chip visual survives only as the dormant Component Library design file `Properties/Chips/ChipLink.swift`); per-vault presentation replaces the separate entity â€” `PageType.open_in` (`compact` | `window`, absent = `window`) routes page-taps to the in-window `PagePreview` card or the main detail pane; the standalone `PreviewWindow` primitive is eliminated in favor of the in-window `PreviewStack` overlay; index schema v10 â†’ v11 (delete-and-rebuild, page-only `connections`); legacy `_itemtype.json` folders adopt as sidecar-less Page Types with the stale sidecar left inert.
