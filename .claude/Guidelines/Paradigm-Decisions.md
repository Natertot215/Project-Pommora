### Paradigm Decisions

Pommora's value depends on its on-disk format, schemas, and cross-entity contracts surviving a stack rebuild and a future cloud sync (load-bearing constraints #1 and #2). Code locking those shapes is **paradigm-solidifying** — once data exists in the wild, migrating is expensive.

#### Operating rule

**Stop and surface paradigm choices for Nathan's confirmation before the code lands** — use `AskUserQuestion` with concrete trade-offs and your recommendation.

Applies even when a written plan proposes one path — if you spot ambiguity, a real downside, or an alternative worth weighing, surface it first. Spec drift is acceptable; silent commitment is not.

#### What counts as paradigm-solidifying

- **On-disk schema shapes** — fields, types, naming conventions, snake_case vs camelCase per-key choices, nesting structures inside the per-kind sidecars (`_pagetype.json` / `_pagecollection.json` / `_pageset.json` / `_taskconfig.json` / `_eventconfig.json` / `_area.json` / `_topic.json` / `_project.json`) / `.task.json` / `.event.json` / etc.
- **Wire encodings for ambiguous types** — tagged-object vs bare-string discrimination (e.g. `.relation` vs `.select` strings), date format choices (ISO-8601 vs Unix epoch vs human-readable), null-vs-missing semantics.
- **Identifier conventions** — ULID format, filename-as-title rule, ID-vs-title display split, relation key shape (e.g. `{"$rel": "..."}`).
- **Default values that become locked once data exists** — seeded per-kind sidecar shapes (`_pagetype.json` / `_pagecollection.json` / `_taskconfig.json` / `_eventconfig.json`), `defaultSeed()` outputs, default property catalog per Type.
- **File layout choices** — folder vs file boundaries for entities, filename extension conventions (`.project.json` vs `_project.json`), sidecar metadata file naming.
- **Cross-entity contracts** — tier1/tier2/tier3 array semantics, parent-pointer conventions, move-strip rules.
- **Error semantics at file load** — silent recovery (e.g. missing field → default) vs hard throw, malformed-file handling, validation timing.
- **Behavioral defaults that change user-visible outcomes downstream** — delete promote-vs-cascade default, filename collision handling (reject vs auto-suffix), move-across-Type property-strip behavior.

#### What does NOT count

- **Internal implementation choices** that don't affect on-disk shape — use of `@Observable` vs `ObservableObject`, value types vs reference types, manager-per-entity vs unified store.
- **UI structure** — view extraction, sheet vs popover, sidebar layout — these can be refactored freely without data migration.
- **Test strategy** — Swift Testing vs XCTest, test file organization, fixture patterns.
- **Build configuration** — Swift version, strict concurrency settings (already locked).
- **Naming of types in code** — internal CodingKeys names, struct rename refactors that preserve on-disk shape.

If in doubt, surface it. Better to overconfirm than retrofit.

#### Confirmation protocol

1. **Stop** before writing the locking line.
2. **State the choice in user-facing terms** — on-disk shape, user-visible behavior, migration cost. Not jargon.
3. **Present 2–3 options** with concrete on-disk samples. Lead with your recommendation.
4. **Wait for confirmation.** Update the spec/plan before dispatching implementation.
5. **Record the locked decision** in `History.md` (the canonical log for confirmed paradigm decisions and the surrounding session context), then add a one-line entry to the registry below.

#### Registry

Numbered, chronological. Citations elsewhere ("paradigm decision  # ") resolve here. Each entry is the decision plus a one-line rationale; full session context lives in `History.md`.

1. **`PropertyValue.relation` encodes as an array of tagged objects `[{"$rel": "<ULID>"}]`** (always multi-value; a single link is a one-element array), not bare strings — relation edges stay legible to external agents and the graph indexer without consulting a Type's schema (constraint #3).
2. **Collections persist a sidecar** (`_pagecollection.json`, carrying id + parent-Type id + ordering) — the parent relation is explicit on-disk rather than inferred from folder nesting.
3. **SF Symbol picker = Pommora's own native `IconPicker`** — superseded (2026-05-30) the `xnth97/SymbolPicker` SPM dep, which hardcoded a 540pt macOS frame and kept its catalog `internal` (neither resizable nor re-skinnable). A compact Liquid-Glass dropdown over the full SF Symbols catalog (`IconCatalog`, bundled as source) with search + Saved/favorites; hosted via the single `.iconPickerPopover` modifier / `IconPickerSheet` / `IconPickerField`. The call-site-isolation goal now lives in that one modifier; the SPM dep is unused (removable). Saved icons persist in app-level UserDefaults (`IconFavorites`) — a UI convenience, not on-disk user content.
4. **Stub-and-progressively-replace** — each task ships as a green commit; when an earlier task's file references a type built in a later task, inline a throwaway stub at that commit and replace it in place when the real type lands. Rejected alternative: batch-commit all tasks at the end of the branch — one break anywhere contaminates the whole batch and isn't found until the final task.
5. **Sidebar creation is via right-click context menus scoped to the cursor**, not "+ New" buttons — right-clicking a row binds "New X" to that exact entity, keeping the entry point unambiguous.
6. **Sidebar selection chrome lives at the row-file level via `.listRowBackground(SelectionChrome(...))`**, not in-content `.background` — the row-level background covers the disclosure-chevron gutter, and an in-content `.background` regressed a launch crash in `OutlineListCoordinator.recursivelyDiffRows`.
7. **Pages editor stack** — superseded. The locked direction was originally Tiptap (ProseMirror) in a WKWebView; the editor shipped instead as **TextKit 2 + Apple `swift-markdown` + the Pommora-owned MarkdownPM** package (`External/MarkdownPM/`, originally vendored from `swift-markdown-engine`, Apache 2.0) (native NSTextView, no web view). Spec → `// Features//PageEditor.md`; engine rules → `MarkdownPM.md`.

8. **Relations are always multi-value** — the single/multi `allows_multiple` toggle is gone; every relation property holds an array (see #1), so one uniform value shape exists on disk and in the editor.
9. **The relation schema target uses the `relation_target` key; `context_tier` is the only surviving case** — user-creatable targets (`page_type` / `item_type` / `agenda_tasks` / `agenda_events`) were removed in the Contextv2 refactor (2026-06-04); `context_tier` (tiers 1/2/3) is the sole target kind. `page_collection` / `item_collection` are read-tolerated for legacy sidecars and cleared during the migration walk; the decode is tolerant (`try?`) so unknown legacy `relation_target` shapes degrade to `nil` without failing sidecar load.
10. **Tiers are the sole relation-type connection** — `tier1` / `tier2` / `tier3` (root frontmatter arrays) are the only relation-property edges; they emit into the SQLite `context_links` table and share one reverse-lookup path (`IndexQuery.incomingContextLinks`). User-creatable relation properties are removed; there is no separate tier-links table.
11. **Deleting a Context cascades source-side** — every operational entity's `tier1/2/3` reference to the deleted Context is removed from its own file (not promoted, not orphaned). Orchestrated at the delete call site, which holds the content-manager references.
12. **~Agenda Tasks and Events as relation targets — superseded.~** This decision was retired by #16 (Contextv2); Agenda is no longer a valid user-selectable relation target.
13. **Same-container title collisions are rejected** — creating, renaming, or moving an operational entity to a title a sibling already holds in the same container is refused (case-insensitive), not auto-suffixed and never overwriting. Because `filename = title`, a collision resolves to the same path and the atomic write would silently destroy the other entity's file; rejection is the safe behavior absent an independent title field (a Prospect). One shared `NameCollisionValidator` enforces it across Pages and Agenda Tasks/Events; the container types (Areas / Topics / Projects, Page Types + Collections) enforce the same rule through their own validators. Self-recasing an entity's own title (e.g. `notes` → `Notes`) is allowed — the rename guard compares on-disk file identity, not the case-folded name.
14. **~Items are Markdown (Shape A) — superseded by #17 (the Items entity no longer exists).~** Original decision: Item content files are plain `.md` (YAML frontmatter + body), not `.json`, sharing Pages' single `AtomicYAMLMarkdown` pipeline. The capped description IS the body (single source of truth; no frontmatter-description field, no mirror). Kind authority is the parent Type folder's sidecar (`_itemtype.json` / `_pagetype.json`); a reserved, UI-hidden, **non-authoritative** `Class` stamp (`item`|`page`) marks the form. Three-state resolution: absent → write from folder; agrees → proceed; disagrees (or homeless) → relocate to a hidden `.unsorted` inbox (future-UI-surfaced). **Foreign frontmatter keys are preserved by value — never culled — on every Item AND Page write path** (a reversal of the deleted Session-Context cull; Yams reflows style + drops comments, content safe). A mandatory auto-run launch migration normalizes legacy `.json` Items → `.md` (idempotent, resumable; not a declinable consent-gate). **Agenda stays JSON** (`.task.json` / `.event.json`); sidecars, Projects, Spaces, Settings stay JSON — only Item content became `.md`. Char cap = 500 markdown-source chars (`ItemValidator.maxDescriptionLength`), per-Type `description_cap` override; validated on save. Pairs with #13 (`filename = title` + collision rejection).

15. **~`template_config` schema (ItemsV2) — superseded by #17 (no `template_config` ships; `open_in` landed instead as a plain `PageType` field with `compact|window` values).~** Original decision: both per-kind sidecars carry a reserved, all-optional, null-round-tripping `template_config`. Items: `{ layout, promoted_properties: [{id, display}], cover_property_id, description_cap, default_description }`. `layout` is the `LayoutArchetype` enum, on-disk strings `compact|standard|banner_two_column|gallery|wide|reserved` (tolerant `unknown` round-trips; only `standard` ships, the rest are muted-until-promoted). Per-property `display` is the tolerant `PropertyDisplay` enum (`inline|thumbnail|banner|chips|list|unknown`); the archetype sets a default a non-nil `display` overrides. `cover_property_id` names an existing `.file` property whose `accept` admits an image — no new PropertyType/PropertyValue case. Pages: reserved parity `{ layout (null — Pages have no archetype), default_body, open_in }`; `open_in` (`OpenInMode`: `preview|full_page`) is INERT until Page preview ships. **Collection-level `template_config` overrides its parent Type; absent ⇒ Type default.** Additive: existing files with no `template_config` render the standard Item Window.


16. **Contextv2 — user relations dropped; tiers are the sole relation-type connection (2026-06-04).** User-creatable `PropertyType.relation` properties are retired at decode (`.droppingUserRelations()` filter, tier-safe via reserved `_tier1/2/3` IDs). `RelationTarget` collapses to `.contextTier`-only; `RelationTargetKind.string()` reduced to match. The SQLite table `relations` renamed to `context_links` (`idx_relations_*` → `idx_context_links_*`); all surviving Relation-named symbols renamed `Context*` (`RelationChip`→`ContextChip`, `RelationValueEditor`→`ContextValueEditor`, `RelationPicker`→`ContextPicker`, `RelationDisplayResolver`→`ContextDisplayResolver`, `BuiltInRelationProperties`→`BuiltInContextLinkProperties`, `incomingRelations`→`incomingContextLinks`, `reconcileRelations`→`reconcileContextLinks`, `entitiesByTarget`→`entitiesByContextTarget`). `Project.linked_relations`→`project_links` (dual-key tolerant decode: accepts both keys, always writes `project_links`). Orphaned `$rel` member values are schema-invisible (the property panel iterates the schema, not the member dict), unindexed, and round-trip-stable; cleared opportunistically during the migration member-walk for Types that undergo the ID-mint migration. The legacy Collection→Type migration (`applyRelationTransforms`) deleted as dead-after-filter. Substrate kept: `$rel` token, `PropertyValue.relation` codec, `RelationTarget.contextTier`, `TierRelationCarrying`, `RelationTargetKind`, `PropertyType.relation` (tier-only). **Roadmap:** a separate connection-model layer (per-shape tables, weight-at-query, contexts-as-cores "spiderweb") is planned but not yet built or slotted.

17. **PagesV2 collapse — Page is the sole operational entity beside Agenda (2026-06-09/10).** The Items side (entity, `ItemType` / `ItemCollection` containers, Item Window, `_itemtype.json` / `_itemcollection.json` sidecars, item index tables, "Type"/"Set" labels) is deleted, not migrated — supersedes #14 and #15. Consequences locked with it: the `Class` frontmatter stamp is retired (an on-disk `Class` key is preserved foreign frontmatter, never written); `[[Title]]` is the **sole** connection syntax (`{{ }}` retired to plain text; the chip visual survives only as the dormant Component Library design file `Properties/Chips/ChipLink.swift`); per-vault presentation replaces the separate entity — `PageType.open_in` (`compact` | `window`, absent = `window`) routes page-taps to the in-window `PagePreview` card or the main detail pane; the standalone `PreviewWindow` primitive is eliminated in favor of the in-window `PreviewStack` overlay; index schema v10 → v11 (delete-and-rebuild, page-only `connections`); legacy `_itemtype.json` folders adopt as sidecar-less Page Types with the stale sidecar left inert. No data migration — the index is regeneratable and no `.md` content changes shape. Full record → `History.md`. *Superseded in part at v0.4.0 (`History.md` § "v0.4.0"): the in-window card/`PreviewStack` primitive was rebuilt as a real child-attached `PagePreview` window, and The Nexus's legacy `_itemtype.json` sidecars were repaired away (Status defs merged into `_pagetype.json`, inert files deleted) — the sidecar-less adoption semantic itself stands.*

18. **Contexts Decoupling — free-standing Areas / Topics / Projects (2026-06-10).** The three context tiers stop relating to each other: **Projects decouple from Topics** (no containment, no `parents`, no `project_links`, no Project→Topic promotion), **Topics drop `parents`**, and **tier-1 Space is renamed to Area** (type, files, kind strings, on-disk tokens `areas/` · `_area.json` · `area_order`). Each tier is a folder with a config sidecar (`_area.json` / `_topic.json` / `_project.json`), owned by three sibling managers (`AreaManager` / `TopicManager` / `ProjectManager`) — supersedes the ParadigmV2 containment model (Projects-inside-Topics, tier-parent rules, parent-derived tagging). Index schema v11 → v12 (drop `contexts.parent_topic_id`) → v13 (Area kind strings); delete-and-rebuild on open, no data migration (greenfield). Substrate kept: `tier1/2/3` cross-layer relations (Pages/Agenda → Context) untouched, the `context_links` table, the `context_tier` target, and the `blocks` field (empty, reserved). **Deferred to a future design pass:** context→context relations (tiers linking upward via settings), transitive page roll-up, the composed-blocks surface. Full record → `History.md`.

19. **Sets — a distinct third-tier entity inside Collections (v0.4.1, 2026-06-11).** The Pages-side hierarchy is **Vault → Collection → Set (optional) → Pages**, strictly three levels (depth decides kind; depth-3+ folders are sidecar-less, their pages roll up into the nearest Set). A Set is a **distinct `PageSet` entity, not a recursive Collection** — the type carries no views / settings / open-in fields, so the compiler enforces that Sets never grow Collection behavior; everything inherits from the parent Collection. The nested operational tiers vs free-standing Context tiers asymmetry is intentional — operational containment is physical (file location), Contexts are categorical. Overrides the 2026-05-27 Folders revert: Sets are non-navigable (expandable, never selectable, no detail view) + settings-inheriting, and the view-grouping interplay is specified in `Planning/06-11-Views-Spec.md` — the two reasons Folders was reverted. Sidecar: `_pageset.json` (`id` / `collection_id` / `icon` / `page_order`); parent Collection holds `set_order`. Full record → `History.md`; spec → `Features/Sets.md`.

20. **Views — custom renderers over a pure pipeline; SavedView v2; covers/banners (v0.5.0, 2026-06-12).** A **custom SwiftUI Table replaces native `Table`** for all detail views (the SDK still gives no row reorder and no column-width readback, plus active Tahoe memory-leak bugs) — `DetailRow` / `PropertyColumnBuilder` / native `Table` bodies retired. Both Table and Gallery render from **one pure, in-memory view pipeline** (filter → group → sort) over the `@Observable` manager caches, so property edits recompute instantly with no extra disk reads. **SavedView v2** stores a single `property_order` + a `hidden_properties` set (replacing the old visible/hidden two-array model): `_title` is movable but never hideable, and the page `cover` field is never a column. **Group drops carry dual semantics** — a drop into a structural group is a real file move, a drop into a property group rewrites that frontmatter value. **Covers/banners store in `.nexus/assets/<entityID>/`** — cover is a per-page root frontmatter field (never in any properties UI; per-view Gallery display only), banner is a container-sidecar field. Enabled by the **macOS 26.4 deployment floor**, which unlocks the native drag-session APIs (continuous hover `location`, native multi-select) used un-gated. Full spec → `Features/Views.md`.
