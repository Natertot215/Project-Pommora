## History — Pommora React

What shipped + locked decisions, newest first. Brief by design. Cross-cutting locked decisions live in `CLAUDE.md`; current architecture in `Features/`; roadmap in `Framework.md`; deep specs + ledgers in `Planning/`.

### Subfield (footer) + window chrome (2026-06-25)

**The Subfield (Swift's footer) shipped + its locked decisions.** A breadcrumb (depth-aware navigation + a dimmed forward **ghost crumb** for the last-visited page) and per-view items (pages → `lines·words·chars`; Collections/Sets → an add-menu). Type is a new **Subline** scale (8px/10px). **Persistence is per-nexus in `.nexus/settings.json` under a React-owned `subfield` foreign key** (`{order, expanded}`) — Swift ignores unknown keys, so it round-trips; surfaced over a `subfield:get/set` IPC modelled on `folds`. **Collapse is one app-level flag** (every view toggles together). Homepage + Contexts carry no Subfield until they have content. Spec → `Features/Subfield.md`.

**Glass split into two materials (locked).** `glass-controls` is now Apple **"Liquid Glass"** via `@samasante/liquid-glass` (real `feDisplacementMap` edge-refraction over the live app), chosen over four alternatives + the prior frost in a homepage tuning lab (since removed); `CONTROL_OPTICS` is baked static in `materials/glass-controls.tsx`. Window + Surface stay the frost recipe. → `Features/Design.md` § Glass.

**Inspector + locked-header content-views.** A full-height **Inspector** pane (`Detail/InspectorPanel`, `GlassWindow`) is the sidebar's right-side twin — it pushes/reflows content when open (`--content-inset-right` mirrors the sidebar's left inset), is edge-resizable, and toggles app-level. **Content-views (Collections/Sets/Contexts) lock their banner + title** while the table scrolls (`DetailScaffold` `lockedHeader`); the title now **always renders** (banner-less views, e.g. Areas, no longer go titleless), the divider is full-width always, and the title size is one DRY value across banner/no-banner + Collection/Set. The Add-Banner button was unified into one shared component. **Scrollbars are hidden app-wide** (Electron's default reads heavy; native overlay unreliable).

### Tables — full GFM table editing (2026-06-22 → 06-23)

GFM pipe-tables became a real interactive table: a block-replace CodeMirror 6 widget over the canonical GFM source (which never leaves `EditorState.doc`), every cell a live nested CodeMirror editor (caret / IME / spellcheck / connections + `[[` autocomplete inside cells; in-cell Cmd-Z scopes to the page). Structure changes only through hover **grips** — drag to reorder a row/column, right-click for an OS-native menu (align / insert / clear / delete / delete-table), and drag a column boundary to **resize** (whole dashes redistribute between adjacent columns, 1-dash floor, pixel-exact preview). Keyboard-uncorruptable via one shape-preserving `transactionFilter` plus a second guard that refuses the blank-line deletion which would fuse two tables. The headless core (`model`/`codec`/`regions`/`operations`) imports neither CodeMirror nor React and is unit-tested standalone (51 Tables tests). Closed out with a simplify + code-review + comment-audit pass.

**Provenance + locked.** A React port of **ckant/codemirror-markdown-tables** (MIT) — the widget-over-GFM + nested-cell architecture is theirs; **dash-count-as-width columns and column width-resize are ours** (ckant has content-sized columns and a *dimension* resize, not width). On-disk, **column width = the delimiter dash count** (`<colgroup>` ratios + `table-layout:fixed`; best-effort cosmetic, never data) — width rides the portable source because CSS renders custom widths natively (the Swift build stores widths in frontmatter instead). `@tanstack/react-table` was evaluated and rejected (its resize grows the table, opposite our conserve-total model). Spec → `Features/MarkdownPM.md` § Tables.

### Page banner + shared title header (2026-06-21)

**Locked — a page's banner is the Swift-compatible `cover` frontmatter key.** Setting a page banner writes `cover: .nexus/assets/<page-id>/banner-<token>.<ext>` into the `.md` frontmatter (foreign keys + body preserved via `mergeFrontmatter`) — the same asset layout containers use under `banner`. `setBanner` gained a `page` owner kind. `cover` is the field Swift's `PageFrontmatter` already uses for the gallery card, so the page banner + the (future) card are one image — Notion-style, conceptually portable. The page editor renders a full-width 230px cover band with the title overlaid bottom-left, its bottom edge the full-width divider; no banner → a hover Add-Banner strip + inset divider. The header scroll-tracks over its measured height (body padding follows via a `ResizeObserver`).

**Shared `DetailTitleHeader` (Swift parity).** The title is a displayed `[icon] [name]`, not an always-input — right-click → native **Rename / Edit Icon** (Rename flips to inline edit). The banner keeps its own right-click → **Change / Remove Banner**: two separate, non-overlapping native menus. Built reusable; the container/context/homepage banners still render the bare `.banner-title` text, so swapping them onto `DetailTitleHeader` (with their icon + rename wiring) is the immediate next step. **Edit Icon** routes to a `Components/IconPicker` stub — the real symbol grid is designed in Figma and wants a shared **dropdown animation primitive** in the interactions layer for its open/close.

This session also landed: heading folding (chevron + `.nexus/folds.json`, sidebar Reveal animation), the native editor context menu (formatting block above Speech/Share, Connection + Heading 5, ⌘B/I/E/K/⌘⇧X/⌘⇧K shortcuts), unfocused-clean render + caret-unplaced-on-fold + the title-bounce fix, raw `[[ ]]` for phantom connections, and Nathan's heading em scale (h1 1.8 → h5 1.075). `refactoring-phase-b` (the Swift module regroup) was merged into `main`.

### Page editor — MarkdownPM (2026-06-20)

The Swift `MarkdownPM` package ported faithfully onto CodeMirror 6: a dynamic-syntax Markdown editor where markers reveal as raw source on the caret line and render styled when the caret leaves. CM6 substrate + a micromark/mdast parser, both behind seams; the behavior layer (`parser`/`detect`/`tokens`/`input`/`decorations`) is framework-free + unit-tested. Shipped: inline marks (bold/italic/strike/code/link/connection), caret-aware headings, bullet/ordered/task lists, HR, blockquote cards, fenced code blocks; an inline title + divider in a scroll-tracking top zone with rename-on-Enter; input transforms (list continuation, smart-backspace, auto-pair, dash/arrow, checkbox-canonicalize, Tab-indent, Shift+Enter exit); a built zoom slider (not yet placed). List markers are **editable source, not atomic widgets** — the marker rework that fixed swallowed-space + caret-jump bugs.

**Locked — per-page editor UI state lives in dedicated local files, not frontmatter.** Heading-fold state goes in its own `.nexus/folds.json`, keyed by entity id, sync-excluded (per-machine, Obsidian's `workspace.json` model); each per-page UI-state concern gets its own dedicated JSON so a page-count-scaling map can't bloat an unrelated store. Frontmatter would leak Pommora UI noise into the portable `.md`; the index can't hold it (regeneratable). **Deliberate divergence from Swift** (which uses `folded_headings` in frontmatter) — Swift should align. Spec → `Planning/MarkdownPM.md`.

### Container views + banner + Swift-aligned renderer reorg (2026-06-19)

Selecting a Vault or Collection renders its pages in a table; Context and the nexus-header **Homepage** render their own views. Vault + Collection share one `ContainerView` (`source.kind` the divergence seam); every banner-bearing view sits in a shared `DetailScaffold` (banner + body + divider). Homepage + Collections became selectable sidebar entities (the nexus header *is* the homepage). The renderer tree mirrors Swift — `Detail/` (router · `DetailScaffold`≈ViewSurface · `Scope`≈DetailScope · `ContainerView`/`HomepageView`/`ContextView`/`PageView`) · `Detail/Table/` · `Detail/Banner/` · `Sidebar/` · `Components/`; the `styles.css` monolith split into co-located per-area stylesheets.

**Banner** — a shared image cover *behind the glass* on Vault/Collection/Context/Homepage: native picker → `.nexus/assets/<id>/banner-<token>.<ext>` → served over a registered `nexus-asset://` protocol; native Change/Remove menu; one `setBanner` mutate op (the homepage's banner lives in the `.nexus/homepage.json` singleton). Spec + detail → `Features/Architecture.md`.

### Sidebar drag-and-drop — the "sidebar" behavior (2026-06-19)

After an 8-variant feel spike, Nathan chose **insertion-line-solid + grab-ghost** as the canonical "sidebar" drag behavior: an accent line marks the exact drop, the picked-up row stays muted in place, a ghost rides the cursor — no row displacement. Adopted into the real app, resolving the prior deferred tree-cross-level + real-app-adoption note: the sidebar tree reorders pages within a folder *and* reparents cross-set / cross-collection, position-aware. Bespoke (`components/sidebarDnd.tsx` — live-rect hit-test, portal ghost to escape the glass `backdrop-filter` containing block; pure unit-tested model in `sidebarDndModel.ts`); it **replaced** the container-only `useTreeMove` (since removed), leaving the PommoraDND sort engines + Interaction Lab untouched.

Generalized so **every sidebar entity reorders within its parent heading** — collections/sets (`collection_order`/`set_order` via `reorderChildren`), vaults + the three context tiers (`.nexus/state.json` via `reorderTop`), pages (`page_order` via `movePage`); only the code-fixed Saved pins stay inert. `movePage` gained an optional `order` (the destination's full page-id list after the drop); Main moves the file then persists `page_order` to the destination sidecar, a stale source id self-drops on the next read. Cleanup folded `SetRow`/`CollectionRow`/`VaultRow` onto a shared `ContainerRow` and the drag gesture into a discriminated union. Spec → `Features/DragAndDrop.md`.

### Design system — showcase shell + color primitives (2026-06-18)

The standalone DesignSystem page, Interaction Lab, and CSS glass editor became one hash-routed showcase (`design-system.html`) with a glass sidebar mirroring the app and leaves driven by a single registry; the glass editor became the Glass leaf's frost tuner. Colors gained **system-grey / system-white / system-black** primitives — labels derive from system-white, fills/states/separators from system-grey, each at an opacity via `color-mix` rather than a baked hex. Tints are tokenized the same way — an opacity scale in `tint.ts` applied to any base via `tintAt(base, step)`; chips re-map onto it (fill = primary, border = secondary, text = label washed with quaternary). Spectrum solids and opaque surfaces stay literal values.

### In-house drag engine — PommoraDND (2026-06-18)

Decided to **own** drag-and-drop rather than rent dnd-kit — the same "own the layer, rent the engine then rebuild it" path as MarkdownPM, backed by a line-by-line dissection of dnd-kit's dev source and an adversarially-reviewed plan. The engine lives behind the `interactions/drag.tsx` seam and was built + verified in the **Interaction Lab** (`interactions.html`), not the app. One Pointer-Events sensor with `setPointerCapture` (no mouse/touch split, no document listeners); rects measured once at drag start; closest-centre collision with hysteresis; decide-then-animate drop committing on the lifted item's `transitionend`. Drives list / grid / table / tree-within-level + a cross-list board (`group.tsx`, portal overlay, slot-pitch shifts, single commit — no mid-drag mutation). Plus constraints (axis/bounds/swap/async `canReorder`), auto-scroll (rAF, ease-in ramp), and keyboard + a11y (Space/arrow/Esc, ARIA live region, focus restore).

**Cutover complete** — Nathan Lab-inspected + approved; `@dnd-kit/*` uninstalled and import-free. All code identifiers are brand-free (`engine.tsx`/`group.tsx`/`Zone`/…); "PommoraDND" lives only in docs. **Not a 1:1 port** — drops dnd-kit generality we don't need (framework-agnostic core, 3 sensors, 4 collision strategies, modifier pipeline, SSR guards, continuous re-measuring), adds pointer capture + hysteresis + a frame-accurate commit. **Deferred:** tree cross-level reparenting, board keyboard access. Spec → `Features/DragAndDrop.md`.

### Glass — CSS frost + window chrome (2026-06-17)

Built a comparison lab (a custom edge-bevel SVG lens + 8 npm liquid-glass libraries) and an Apple-recipe research pass (the refractive look is an edge-shaped displacement map, not turbulence; clear center, refractive rim, near-zero tint). **CSS frost won** for the shipped material — it adds its own light, so it never collapses to a dark slab over a dark window the way clear refraction does. A clear backdrop blur with a slight dim, no fill, no saturate, plus a glassy edge (specular top rim + hairline inner ring + soft lower-rim light); one shared material spread by `GlassSurface`/`GlassControls`. The Apple-style SVG **edge-lens** (true rim refraction) was explored and **shelved** — kept as `materials/edge-lens.tsx` and surfaced in the glass tuner; `liquid-dom` (WebGPU) shelved earlier. **Window chrome:** native frame kept (standard macOS corner radius + shadow), title bar hidden, traffic lights positioned into the sidebar, opaque window background so the sidebar glass samples the app, not the desktop.

### Desktop + filesystem write path (2026-06-16)

Made the React app a real Mac app over a real on-disk nexus: app config + session (no hardcoded path; restore-or-empty, never a launch modal); native folder picker + recents + `app.addRecentDocument`; drag-a-folder-to-open; native macOS menu bar. The write path is one `mutate` IPC (create page/container/context + root-create · rename · delete · movePage; **relative paths only**, resolved under root via realpath — `pathSafety.resolveUnderRoot`, the single guard). Cascade policy is owned at the mutate layer — page rename → `renameCascade` with revert-on-failure; context delete → `unlinkTier` before the folder. Delete branches on `trashMode` (in-nexus `.trash` vs `shell.trashItem`). Native right-click menus (New / Delete-confirm / Reveal) on every Sidebar row; **New Page ⌘N** end-to-end. Live SQLite index via **full-refresh** after each mutation.

**Locked:** renderer sends RELATIVE paths only · cascade policy lives at the mutate layer, not call sites · index = full-refresh, not incremental (reuses `buildIndex`, zero row-logic duplication; incremental is a later perf optimization) · `db.ts` lazy-requires better-sqlite3 so a native-ABI mismatch degrades to a cold index, never crashes · `node_modules` stays Node-ABI for the test gate; electron-builder rebuilds for Electron at package time. **Deferred:** inline rename UI (op + menu exist) + top-level create affordances.

### Data layer — headless, caught up to Swift (2026-06-15)

The complete write/mutation side, built tests-only (no UI wired), grounded in a 20-agent dual-research pass with load-bearing claims verified against real Swift:

- **Contracts + atomic I/O** — `Result` envelope; the value codec in the locked Swift precedence (table-driven round-trip); the single ID owner (ulidx monotonic + `adoptedId`); atomic write (sorted/stable JSON, mutateJson, trash).
- **Page file engine** — the envelope + foreign-preserving write via the yaml Document API; foreign frontmatter keys **and YAML comments** survive (comments being the win over Swift, whose hand-rolled merge drops them).
- **Sidecars** — zod v4 schemas (schema = codec = type; `z.looseObject` defensively retains foreign sidecar keys; shared builders collapse Swift's triplicated context managers); path-based stateless kind authority.
- **CRUD** — ONE create/rename/delete/updateSidecar for all six folder entities + page CRUD + reorder. filename = title; fresh ULID on create; delete → in-nexus `.trash`; partial updates govern only their keys; all return `Result`, never throw.
- **Properties** — value write + schema CRUD on the type sidecar (add/rename/reorder/delete/changeType) with atomic member strips (two-phase commit) + tier synthesis.
- **Connections + tier relations** — pure-Map `[[link]]` engine (scan/rewrite/resolve/edges) + rename cascade + unlinkTier + `setPageTier`. Resolution has **no** SQLite dependency — the index is a pure accelerator.
- **SQLite index** — `better-sqlite3` behind `db.ts`; an 11-table schema structurally identical to Swift's + version handshake + per-entity upserts + cold build.
- **Agenda** — Tasks + Events CRUD + index pop + config-schema CRUD, generalizing the schema engine over a `SchemaTarget` (one five-op core serves pages + agenda).

A 7-agent adversarial review + hard DRY pass fixed the criticals (rollback hole, a dangerous `.bak-` sweep, index tier `target_kind`, lenient colors, name guards, self-link skip) and collapsed duplicated helpers to single owners. **Swift→React data-layer LOC: 8,552 → 2,383 (~72%, shared-functionality-only).** **Deviation theme:** every shipped enhancement is Swift framework-complexity deleted, with a capability (comment-preserving writes, index-independent connection resolution) falling out for free. Architecture → `Features/Architecture.md`.

### Navigation + view pipeline (2026-06-15)

The tested logic spine, with visual renderers left as honest placeholders. `page:open` IPC — path-traversal-guarded (rejects non-string/empty/absolute/`..`-climbing), never throws across the boundary. `readPage` — on-demand single-page read (lenient frontmatter split, stable `adopted-<sha256>` id). A side-effect-free **view pipeline** (`filter (AND) → group → sort` over `ViewRow[]`; empties sort last; `ViewRow.frontmatter` optional so frontmatter-keyed columns light up later with no pipeline change). Selection → detail routing wired; the Table/Gallery and page-render branches stubbed.
